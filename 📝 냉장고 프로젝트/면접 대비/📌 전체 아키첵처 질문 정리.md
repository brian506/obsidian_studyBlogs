#면접
## Q. Naver Clova API를 비동기로 처리했는데, 구체적으로 어떻게 구현하셨나요?

### 꼬리: @Async를 쓸 때 같은 클래스 내에서 호출하면 왜 동작하지 않나요? 
- @Async 메서드를 가진 빈을 프록시로 감싸서 등록하는데, 같은 클래스 내 호출은 this.method() 형태라 프록시를 거치지 않고 실제 객체를 호출하게 됩니다.
### 꼬리: 비동기 작업의 예외는 어떻게 처리했나요? @Async에서 예외가 발생하면 어떻게 되나요?
- @Retryable 어노테이션으로 재시도 처리를 3번 수행합니다.
### 꼬리: 비동기로 처리하면 사용자에게 레시피 결과를 어떻게 전달했나요? 
- CompletableFuture<>의 thenApply() 로 비동기적으로 AI 에서 응답이 오면 결과를 반환해줍니다.

## Q. LLM API 응답 시간이 가변적일 텐데, 타임아웃은 어떻게 처리했나요?

### 꼬리: LLM API가 다운됐을 때 Fallback 처리는 어떻게 했나요?
- Spring AI 가 call() 하는 재시도 횟수를 3번으로 정했고, LLM API 의 타임아웃은 5초로 지정했습니다.
- 또한 나중에 다시 시도하라는 응답 객체를 보냄으로써 서비스가 멈추는 것을 방지했습니다.
### 꼬리: LLM API 호출 비용이 발생하는데, 동일한 식재료 조합에 대한 캐싱은 고려하셨나요?
- 추후에 API 호출 빈도수가 많게 되면, 로컬 캐시로 동일한 식재료 조합에 대해서 호출 없이 반환할 수 있도록 리팩토링 예정입니다.

## Q. 4계층 아키텍처를 도입하여 계층 간 의존성을 낮췄다고 하셨는데, 기존 3계층과 어떻게 다른가요?

### 꼬리: 4계층에서 추가된 계층이 어떤 역할을 하나요? 
- service 와 dataaccess 사이의 implementaion 계층이 추가됐습니다. 해당 계층은 비즈니스 로직이 아닌 도메인 로직의 구현을 담당합니다.
### 꼬리: 각 계층 간 데이터를 전달할 때 DTO 변환은 어느 계층에서 했나요?
- 표현 계층에 최대한 의존하지 않도록 표현-응용 계층 영역에서 DTO 변환을 했습니다. 또한, DB 에 의존하지 않도록 도메인과 데이터 접근 계층에서도 DTO변환 했습니다.
### 꼬리: 계층 간 의존 방향을 어떻게 강제했나요? 상위 계층이 하위 계층을 의존하는 것을 코드 레벨에서 어떻게 보장했나요?
- 상위 계층이 하위 계층을 의존하는 방향으로 강제했고, 의존성 주입 시에 분류되어 있는 디렉토리 기준으로 보장했습니다.

## Q. 도메인 계층에서 JPA Entity를 그대로 사용했나요, 별도 도메인 객체를 만들었나요?

### 꼬리: JPA Entity를 도메인 객체로 쓰는 것의 문제점은 뭔가요?
- 별도 도메인 객체를 만들었습니다. 만약 현재 사용하고 있는 관계형 데이터베이스에서 다른 비관계형 데이터베이스로 사용하게 되면 JPA에 의존하게 되기 때문에 변경 사항이 있을 때 힘듭니다.

### 꼬리: Entity와 도메인 객체를 분리했다면 변환 비용이 생기는데, 그 트레이드오프를 어떻게 판단했나요?
- 저도 초반에는 매우 불편함을 느꼈고, 코드 구현 시에도 시간이 많이 걸렸습니다. 하지만 프로젝트가 진행될수록 비즈니스 로직 변경에 따른 DB 스키마 변경에 영향을 줄 때 낮은 결합도로 인해 DB의 안정적인 운영이 가능했습니다. 또한 테스트 코드 작성 시에, DB 없이 로직만 테스트할 수 있어 수월했습니다.

## Q. Prometheus와 Grafana를 어떻게 활용했나요? 어떤 지표를 모니터링했나요?

### 꼬리: Prometheus의 pull 방식과 push 방식의 차이를 설명해주세요.
- 현재 방식에서는 prometheus 로의 pull 방식으로 매트릭을 scrape하고 있습니다.
- pull 방식은 /actuaor 경로로 scrape하는 방식이고, push 방식은 중간 컴포넌트에 능동적으로 데이터를 밀어넣는 방식입니다.
### 꼬리: Spring Boot Actuator의 어떤 메트릭을 주로 봤나요? 커스텀 메트릭을 만든 적 있나요?
- http 요청,응답과 cpu 활성 상태, JVM 메모리 상태 등의 매트릭을 봅니다. 
### 꼬리: 알람 설정은 어떻게 했나요? 특정 임계치 초과 시 어떻게 감지했나요?
- http 500번대 에러 응답 비율이 5프로 이상 차지하거나, CPU 사용량이 80%를 지속적으로 넘길 때를 감지하여 알림을 보내도록 했습니다.

## Q. 유통기한 알림을 매일 정오에 보내는 스케줄러를 어떻게 구현했나요?

### 꼬리: @Scheduled와 Quartz, Spring Batch의 차이를 설명해주세요. 왜 @Scheduled를 선택했나요?
- spring batch 는 대용량 데이터를 청크 단위로 나누어 처리하고 트랜잭션을 관리하는 데이터 처리에 특화된 프레임워크이고, 스케줄러는 아닙니다. 반면 Scheduled 와 Quartz는 작업을 특정 시간에 실행시키는 스케룰러입니다. 서비스 초기 상황에서 데이터가 많이 없을 때 batch 를 사용하는 것은 오버엔지니어링이라고 생각했고, Quartz 는 추후에 다중 서버로 변환 시 고려할 것 같습니다.
### 꼬리: 서버가 2대 이상으로 Scale-out 됐을 때 스케줄러가 중복 실행되는 문제는 어떻게 처리할 건가요?
- 해당 스케줄러를 Quartz 로 변환하여 해당 문제를 자체적으로 막거나, 분산락을 도입하여 중복 처리되지 않도록 할 것입니다. 현재 구조에서는 레디스를 사용하지 않기 때문에, 레디스 분산락이 아닌 **shedLock** 을 통해서 동시 처리 문제를 방지할 것 입니다.
### 꼬리: 서버가 정오에 다운됐다가 12시 5분에 재시작되면 그날 알림이 발송되나요?
- 현재 scheduled 를 통해서 구현되어 있기 때문에 서버가 12시를 지나서 재시작되면 알림이 발송되지 않습니다. 서비스 규모가 커지게 된다면 해당 누락 문제를 해결하기 위해 재발송 로직이 가능한 스케줄러인 Quartz를 도입하여 해결할 것입니다.

## Q. 알림을 받기 싫은 사용자는 어떻게 처리했나요?

### 꼬리: FCM 토큰이 만료됐거나 앱을 삭제한 사용자의 토큰은 어떻게 관리했나요?
- FCM 토큰을 발급할 때와 메시지를 전송할 때 해당 토큰의 유효성을 검증합니다. 따라서 만료됐거나 앱을 삭제한 사용자의 토큰은 Firebase 응답을 통해 식별하여 deleteAll()로 일괄삭제합니다.
### 꼬리: 알림 발송 이력은 DB에 저장했나요? 재발송 처리는요?
- 알림 발송 이력은 따로 DB에 저장하지 않습니다. 비동기로 처리되는 메시지 전송 건들에 대해서 재발송 3회 처리가 이루어집니다.

## 인프라

### 꼬리: Public Subnet과 Private Subnet을 나눈 이유가 뭔가요?
- 외부 위험 요청이나 매크로 챗봇의 요청을 막기 위해 시스템 운영 상 중요한 모니터링 환경과 DB 환경을 외부에서 접근하지 못하도록 private subnet으로 나누었습니다. 추후에 애플리케이션 서버도 private subnet 에 두고, nginx 만 public subnet 으로 두어 안정적인 운영이 가능한 환경을 구축할 수 있습니다.
### 꼬리: 운영 서버와 개발 서버를 분리한 이유와, 실제로 어떤 차이점이 있었나요?
- 프론트엔드와 작업할 때 swagger 로 작업을 했습니다. 개발 서버에서 해당 API에 대한 JSON 정보에 대한 공유 편리성, 그리고 운영 서버와 비슷한 개발 서버를 둠으로써 서버의 병목 지점과 API 요청 에러에 대한 부분을 미리 파악할 수 있는 편의성이 있었습니다. 비용은 더 들겠지만, 무중단 배포를 하기 위해서는 필수적이라고 생각합니다.

### 꼬리: NAT Gateway를 사용한 이유가 뭔가요? Private Subnet에서 외부 통신이 필요한 이유는요?
- 현재 모니터링 시스템과 DB가 private subnet 에 있는데, 외부 통신이 필요한 이유로는 장애 상황이 발생했을 시에 해당 알림을 외부 인터넷을 통해 아웃바운드 통신이 필요합니다. 
- 통신이 필요하다고 해서 DB나 모니터링 서버를 public subnet 에 배치하거나 공인 IP를 주게되면 외부로부터 보안 위협이 발생합니다. NAT Gateway 를 둠으로써 알림 발송을 위해 나가는 트래픽(아웃바운드)은 IP를 변환해 허용해주지만, 반대로 외부에서 들어오는(인바운드) 통신은 차단할 수 있습니다.
