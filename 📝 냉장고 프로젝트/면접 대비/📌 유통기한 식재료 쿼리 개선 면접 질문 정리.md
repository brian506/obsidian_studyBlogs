# 복합 인덱스 & 쿼리 최적화

## Q. [memberKey, status] 인덱스에서 ref scan이 나왔는데, range scan보다 왜 느렸나요?

### 꼬리: ref와 range의 차이를 실행 계획 관점에서 설명해주세요.
-  ref 는 인덱스 조합으로 같은 row 의 비고유한 데이터가 스캔될때 발생하고, 주로 동등조건일때 발생합니다.
- range 는 인덱스 조합이 where 절에서 범위 조건일 때 리프 노드에서 수평 탐색할 때 주로 발생합니다.
### 꼬리: status 컬럼의 카디널리티가 낮다고 하셨는데, 카디널리티가 인덱스 선택에 미치는 영향을 설명해주세요.
- 카디널리티가 낮은 컬럼은  값의 고유도가 낮습니다. 카디널리티가 낮은 컬럼에 대해서 탐색을 할 떄 그만큼 필터링 되는 데이터량이 적기 때문에 스캔해야 하는 데이터 수가 늘어나게 됩니다. 따라서 카디널리티가 높은 컬럼 즉, 고유도가 높고 식별성이 높은 컬럼에 대해서 인덱스를 선택해야 합니다.

### 📌 꼬리: 카디널리티가 낮은 컬럼을 인덱스 선행 컬럼으로 두면 어떤 문제가 생기나요?
- 선행 컬럼이 카디널리티가 낮게 되면, 첫 번째 수직 탐색 이후 같은 값을 가진 row가 많아서 수평 탐색 범위가 넓어집니다. 후행 칼럼의 카디널리티가 아무리 높아도, 선행 컬럼에서 이미 많은 데이터를 스캔하게 되므로 인덱스 효율이 떨어집니다.

## Q. 최종적으로 [memberKey, expiry_date]를 선택했는데, status 필터링은 어떻게 처리했나요?

### 꼬리: status 조건이 WHERE 절에 있는데 인덱스에 없으면 어떻게 처리되나요? (Index Filter vs Table Filter)
- status 가 인덱스에 없으면 해당 memberKey 와 expiry_date 조건에 부합하는 데이터를 먼저 인덱스 테이블에서 필터링하고 해당 데이터들의 참조 주소값으로 원본 테이블에 엑세스해서 status 가 'ACTIVE'인 데이터를 스캔하게 됩니다.
### 꼬리: [memberKey, status, expiry_date]가 25ms, [memberKey, expiry_date]가 27ms로 거의 비슷한데 왜 2컬럼 인덱스를 선택했나요?
- 현재 카디널리티가 작은 status 는 필터링 되는 데이터의 비율이 적습니다. 현재로서는 인덱스 생성 비용과 쓰기 비용을 감안해서 status를 뺀 상태의 인덱스로 선택했습니다. 추후에 데이터 분석 후에 status = 'ACTIVE'인 데이터량이 20프로 이하일 경우에 인덱스에 추가할 것 같습니다.

### 📌 꼬리: 커버링 인덱스를 고려해보셨나요? SELECT * 대신 필요한 컬럼만 조회하면 어떻게 달라졌을까요?
- 커버링 인덱스는 select, where, order by 에 사용되는 모든 컬럼이 인덱스 안에 포함되어야 합니다. 현재 쿼리는 식재료 전체 컬럼을 select 하고 있어서, 인덱스에 없는 컬럼들을 원본 테이블에서 Random I/O 로 가져올 수 밖에 없습니다. 따라서 커버링 인덱스 적용이 불가능합니다.

## Q. 옵티마이저가 항상 여러분이 만든 인덱스를 선택하나요? 옵티마이저가 인덱스를 무시하는 경우는 언제인가요?

### 📌 꼬리: 인덱스가 있는데도 Full Table Scan이 선택되는 상황을 설명해주세요.
- 인덱스를 통한 Random I/O 비용이 Full Scan의 Sequential I/O 비용보다 크다고 옵티마이저가 판단할 때 Full Table Scan을 선택합니다. 대표적으로 조회 대상 비율이 높을 때, 테이블 자체가 매우 작을 때, 통계 정보가 오래되어 부정확할 때 발생합니다.

## Q. 조건절에 함수를 쓰면 인덱스를 못 탄다고 하셨는데, 그 이유를 원리적으로 설명해주세요.

### 꼬리: DATEDIFF(expiry_date, CURDATE()) = n 이 인덱스를 못 타는 이유와, expiry_date = DATE_ADD(CURDATE(), INTERVAL n DAY)가 인덱스를 탈 수 있는 이유의 차이는요?
- 연산함수 안에 컬럼을 넣게 되면 가공된 컬럼을 토대로 탐색하기 때문에 인덱싱 되어 있는 테이블에서 순차적으로 스캔하지 못하게 됩니다. 후자는 DATE_ADD() 함수가 먼저 선행적으로 실행되고 나서 실행된 값으로 정렬되어 있는 expiry_date 컬럼에 맞는 행을 찾기 때문에 인덱스를 탈 수 있게 됩니다.

### 꼬리: Generated Column(가상 컬럼)으로 함수 결과를 인덱싱하는 방법도 있는데, 이 방법은 왜 선택하지 않으셨나요?
- 현재 쿼리에서는 칼럼을 가공하지 않고 동등 조건으로 변환하여 인덱스를 탈 수 있었기 때문에 가상 컬럼이 필요하지 않습니다. 함수 가공을 피할 수 없는 상황일 때 고려할 것 같습니다.

## 380ms -> 25ms 개선을 어떻게 측정하고 검증했나?

### 운영 환경에 인덱스를 추가할 때 주의할 점은?

인덱스를 생성할 때 DB는 모든 데이터를 읽고 정렬해야 합니다. 이때 데이터 정합성을 지키기 위해 DB는 해당 테이블에 락을 겁니다. 이렇게 되면 모든 요청이 실패를 응답하기 때문에, 테이블에 락을 걸지 않도록 Online DDL로 락 없이 인덱스를 생성해야 합니다.

### 개선 전후 비교를 위해 어떤 지표를 사용했나?

DB 쿼리 시간 조회를 위해서는 옵티마이저 실행 계획을 분석하였고, 해당 쿼리를 실행하여 비교했습니다.
API 응답 시간을 비교하기 위해서는 postman을 통해 해당 로직의 API를 호출하여 응답시간을 비교했습니다.

## DB 쿼리 외에 API 응답 시간에 영향을 주는 요소는?

API 응답 시간은 크게 세 구간으로 나뉩니다. 첫번째는 네트워크 진입 구간, 둘째는 애플리케이션 내부 로직, 셋째로는 외부 시스템 연동입니다.
지금 상황에서는 외부 시스템 연동은 하지 않기 때문에 고려하지 않아도 됩니다.
처음 API 요청이 발생하면, 원래는 해당 도메인을 ip 주소로 변환하기 위해 DNS 서버에 요청하는 과정이 있지만, 도메인 주소를 따로 사용하지 않기 때문에 HTTPS 보안 연결을 맺기 위해 핸드셰이크 과정을 통한 지연이 발생합니다.
또한, DB 쿼리 시간이 줄어들면서 커넥션 풀 점유 시간도 줄어들고, 대기하고 있던 다른 요청들의 처리 속도도 연쇄적으로 개선됐을 것이라고 생각합니다.
