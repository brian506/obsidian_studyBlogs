## 인덱스가 있어도 Full Table Scan이 발생하는 경우는?

데이터 스캔 범위가 20-25프로 이상 스캔하게 되면, 옵티마이저는 인덱스를 사용하여 조회하는 것보다 Full Table Scan으로 스캔하는 것이 더 효율적이라고 판단을 하기 때문입니다.

## 옵티마이저 실행 계획에서 type 컬럼들의 차이점은?

먼저 type이 ALL일 경우에는 Full Table Scan을 합니다. 인덱스가 있어도 데이터의 분포도, 스캔 해야 하는 데이터의 개수에 따라 옵티마이저는 ALL로 스캔을 합니다.
그리고 type이 range 일 경우에는 Index Range Scan을 합니다. 조건절에 있는 범위 조건을 인덱스 테이블 내에서 스캔하게 될때 range로 스캔합니다.
마지막으로 type이 ref 일 경우에는 Non-Unique 인덱스에서 동등조건으로 탐색될 때 사용되며, 같은 값을 가진 row 가 여러 개일 수 있는 경우에 사용됩니다.

## [memberKey, status, expiry_date] 가 아닌 [memberKey, expiry_date] 복합인덱스를 선택한 이유는?

status 는 'ACTIVE'과 'DELETED' 값만을 가지는 카디널리티가 낮은 컬럼입니다. 
인덱스를 사용할 컬럼은 중복도가 낮고, 고유값을 가지는 컬럼을 선택하는 것이 유리합니다.
현재 상황에서는 status 를 인덱스에 포함시키는 상황과 포함시키지 않는 상황을 비교했을 때 쿼리 속도 차이도 없었고, status 를 인덱스에 포함시키면 쓰기 성능을 저하하고 인덱스 크기 비용을 불필요하게 늘릴 수 있다고 판단이 들어서 [memberKey, expiry_date]로 선택했습니다.
추후에 status 컬럼의 'ACTIVE'인 데이터가 20프로 이하일 때 적용하는 것에 대해서 고려할 것 같습니다.

## 카디널리티가 낮은 컬럼에 단독 인덱스를 걸면 어떻게 되나?

카디널리티가 낮은 컬럼에 단독 인덱스를 걸게 되면 옵티마이저가 결국엔 Table Full Scan을 선택할 것입니다.

## BETWEEN 과 range scan의 관계를 설명하라

where 절에 있는 BETWEEN 은 해당 컬럼의 조건에 대해서 범위 조건으로 데이터를 탐색하게 됩니다.
range scan 도 마찬가지로 인덱스 테이블에서 범위 조건에 해당하는 컬럼에 대해서 수평탐색을 합니다.

## 연산함수를 Where 절에 쓰면 인덱스에 영향이 있나?

인덱스에 포함되어 있는 컬럼에 연산함수를 사용하여 가공하게 되면, 인덱스 스캔을 하지 못합니다.
인덱스를 사용하게 되면 해당 컬럼에 대해서 정렬된 상태로 인덱스 테이블에 저장되어 순차적으로 탐색을 할 수 있지만, 해당 컬럼이 가공되어 버리면 모든 데이터에 대해 가공된 상태의 데이터와 비교하게 되므로 모든 데이터를 탐색해야 합니다.

## 커버링 인덱스 적용이 가능한가?

해당 쿼리에서는 커버링 인덱스를 적용할 수 없습니다. 커버링 인덱스는 Random I/O 없이 인덱스 테이블 내에서 select 절에 있는 모든 컬럼이 조회되어야 하는데, 복합 인덱스 구조에는 2개의 컬럼만 존재하는 상황이고, where 조건절에서는 3개의 컬럼만이 사용되고 있습니다. 따라서 모든 식재료 컬럼을 조회하는 쿼리에서는 커버링 인덱스를 적용할 수 없습니다.

## 380ms -> 25ms 개선을 어떻게 측정하고 검증했나?

### 운영 환경에 인덱스를 추가할 때 주의할 점은?

인덱스를 생성할 때 DB는 모든 데이터를 읽고 정렬해야 합니다. 이때 데이터 정합성을 지키기 위해 DB는 해당 테이블에 락을 겁니다. 이렇게 되면 모든 요청이 실패를 응답하기 때문에, 테이블에 락을 걸지 않도록 Online DDL로 락 없이 인덱스를 생성해야 합니다.

### 개선 전후 비교를 위해 어떤 지표를 사용했나?

DB 쿼리 시간 조회를 위해서는 옵티마이저 실행 계획을 분석하였고, 해당 쿼리를 실행하여 비교했습니다.
API 응답 시간을 비교하기 위해서는 postman을 통해 해당 로직의 API를 호출하여 응답시간을 비교했습니다.

## DB 쿼리 외에 API 응답 시간에 영향을 주는 요소는?

API 응답 시간은 크게 세 구간으로 나뉩니다. 첫번째는 네트워크 진입 구간, 둘째는 애플리케이션 내부 로직, 셋째로는 외부 시스템 연동입니다.
지금 상황에서는 외부 시스템 연동은 하지 않기 때문에 고려하지 않아도 됩니다.
처음 API 요청이 발생하면, 원래는 해당 도메인을 ip 주소로 변환하기 위해 DNS 서버에 요청하는 과정이 있지만, 도메인 주소를 따로 사용하지 않기 때문에 HTTPS 보안 연결을 맺기 위해 핸드셰이크 과정을 통한 지연이 발생합니다.
또한, DB 쿼리 시간이 줄어들면서 커넥션 풀 점유 시간도 줄어들고, 대기하고 있던 다른 요청들의 처리 속도도 연쇄적으로 개선됐을 것이라고 생각합니다.
