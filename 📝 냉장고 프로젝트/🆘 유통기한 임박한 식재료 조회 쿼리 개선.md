메인페이지로 전환 시 유통기한이 임박한 식재료를 사용자가 알게 하기 위해 매번 조회된다.
호출 빈도가 높은 API 요청 지연시간이 500ms로 메인페이지에서의 병목 지점이 될 수 있다.
응답시간을 줄이기 위해 해당 쿼리에 인덱스를 적용해서 개선하고자 한다.
더미 데이터는 100만건 넣었다.
유통기한이 임박한 식재료는 유통기한 = 오늘 날짜 + 7 의 조건을 만족해야 한다.

![](../images/스크린샷%202026-03-02%2017.33.49.png)

**쿼리 순서 상황 설명**
1. 식재료에 대한 사용자 식별을 하기 위해 먼저 memberKey의 동등 조건으로 사용자를 찾는다.
2. 해당 식재료가 soft delete 되지 않은 `ACTIVE`인 식재료를 동등 조건으로 조회한다.
3. `curDate(), date_add()`함수를 먼저 실행한다.
4. 유통기한 = 오늘 날짜 + 7인 데이터의 범위 조건으로 식재료를 조회한다.

## 인덱스 선택

`status`는 'ACTIVE' 와 'DELETE' 두 개의 컬럼 값이 존재한다.
### case1) 인덱스 [memberKey, status]
#### 인덱스 탐색 과정
1. 인덱스 테이블에서  `memberKey = 'user_90'` 와 `status = 'ACTIVE'`인 데이터들의 위치(주소값)를 찾기 위해 해당 조건을 만족하는 첫번째 데이터로 수직탐색을 하고, 조건이 끝날 때까지 수평탐색을 한다.
2. `expiry_date` 컬럼은 인덱스에 없으므로, 인덱스에서 찾은 위치값을 토대로 모든 데이터에 대해서 각각 원본 테이블에 접근(Random I/O)하여 해당 조건에 맞는 모든 식재료를 조회하여 유통기한 조건을 필터링한다.
3. 찾은 데이터를 `expiry_date`를 기준으로 메모리 상에 올려 정렬한다.

**Scan Type** : ref
**지연 시간** : 140ms

#### 해당 인덱스를 선택하지 않은 이유
- **불필요한 원본 테이블 접근** : `memberKey = 'user_90', status = 'ACTIVE'` 인 데이터가 많게 되면, 이 조건절을 기반으로 찾은 데이터 수만큼 Random I/O가 발생하게 되므로 성능 저하가 우려된다.
- **정렬 발생** : 인덱스 안에 `expiry_date` 가 없으므로 찾아낸 데이터들을 메모리상에서 정렬하게 됨으로 추가 오버헤드가 발생한다.

### case 2) 인덱스 [memberKey, status, expiry_date]

#### 인덱스 탐색 과정
1. 인덱스 테이블에서 모든 조건을 만족하는 첫번째 데이터에 수직 탐색을 하고, `expiry_date`에 대해서 범위 조건에 맞는 데이터들을 수평탐색한다.
2. 별도의 정렬 연산은 수행하지 않는다.

**Scan Type** : range
**지연 시간** : 25ms

#### 해당 인덱스를 선택하지 않은 이유
- **낮은 카디널리티** : 낮은 카디널리티인 `status` 컬럼에 대해서 인덱스 구조에 포함시키게 되면 쓰기 성능만 저하된다. 또한, 해당 쿼리가 커버링 인덱스가 되려면 select 절에 모든 컬럼이 포함되어 있어야 하는데, 그러지 않기 때문에 어차피 Random I/O는 발생하게 된다. 
- **비슷한 지연시간** : status 를 인덱스에 포함시키지 않았을 때와 크게 성능 차이도 없다. 만약 해당 status 컬럼에서 만족하는 'ACTIVE'인 데이터가 20프로 이내라면 인덱스에 포함하는 것이 유리하다고 생각한다.
### case 3) 인덱스 [memberKey, expiry_date]

#### 인덱스 탐색 과정
1. 인덱스 테이블에서 `memberKey = 'user_90'`와 `expiry_date` 범위 조건에 맞는 첫번째 데이터에 대해서 수직탐색을 하고, 인덱스 테이블 내에서 범위 조건에 만족하는 데이터들을 수평탐색한다.
2. 이미 인덱스를 적용했기 때문에 `expiry_date` 는 정렬되어 있으므로 별도의 정렬 연산은 수행하지 않는다.

**Scan Type** : range
**지연 시간** : 27ms

#### 해당 인덱스를 선택한 이유
- 인덱스가 적용된 컬럼에 대해서 이미 정렬이 이루어져 있으므로 별도의 정렬 연산을 수행하지 않아도 된다.
