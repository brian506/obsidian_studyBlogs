알림 전송은 스케줄러를 이용해 매일 낮 12시에 firebase 를 통해 전송된다.
모든 DB에 있는 식재료들을 대상으로 유통기한과 사용자가 설정한 알림시간을 기준으로 데이터들을 조회한다.

현재 FcmToken을 이용해 메시지 전송을 할 때 다음과 같은 순서로 동작한다.

1. expiryDate = today + expiryAlaramDays 인 식재료들을 조회한다.
2. 식재료의 memberKey들을 기반으로 fcmToken들을 찾아서 Map에 저장한다.
3. 해당 fcmToken으로 `.forEach()` 로 각 fcmToken 들에 대해서 FirebaseMessaging.send()를 수행한다. - 가상 스레드 위에서 수행한다.


## 식재료 조회 쿼리 개선

### 1. 기존 식재료 조회 쿼리

```java
 @Override
    public List<FoodEntity> findFoodsToNotify(LocalDate targetDate) {
        return selectFrom(foodEntity)
                .where(isActive(),
                        dateTemplate(
                                Integer.class,
                                "DATEDIFF({0}, {1})",
                                foodEntity.expiryDate,
                                targetDate
                        ).eq(foodEntity.expiryAlarmDays)
                )
                .fetch();
    }
```
![](../images/스크린샷%202026-02-21%2020.26.59.png)
- `DATEDIFF` 함수로 인해 모든 행마다 값을 연산하므로 미리 색인하지 못하고 풀스캔을 하게 된다. 쿼리 내에 있는 연산함수로 인해 인덱스를 사용할 수 없는 쿼리이다.
- 2초가 넘는 응답시간으로 개선이 필요하다.

**기존 쿼리로는 식재료 데이터가 많으면 풀 테이블 스캔을 하게 된다. 인덱스를 활용할 수 있는 쿼리로 지연시간을 줄여야한다.**

### 2. 개선된 식재료 조회 쿼리

알림시간 설정은 당일,1,2,3,7,14일 전에 설정 가능하다.
```java
@Override  
public SliceObject<FoodEntity> findFoodsToNotify(Cursorable<Long> cursorable, LocalDate targetDate) {  
    List<FoodEntity> content = selectFrom(foodEntity)  
            .where(  
                    isActive(),  
                    eqExpiryAlarmDays(targetDate),  
                    ltCursor(cursorable.cursor())  
            )  
            .limit(cursorable.limit() + 1)  
            .orderBy(foodEntity.id.desc())  
            .fetch();  
    return new SliceObject<>(content, cursorable, hasNext(cursorable, content));  
}

private static BooleanBuilder eqExpiryAlarmDays(LocalDate targetDate) {  
    List<Integer> alarmDays = List.of(0,1,2,3,7,14);  
    BooleanBuilder alarmBuilder = new BooleanBuilder();  
  
    for(Integer expiryAlarmDays : alarmDays) {  
        alarmBuilder.or(  
                foodEntity.expiryAlarmDays.eq(expiryAlarmDays)  
                        .and(foodEntity.expiryDate.eq(targetDate.plusDays(expiryAlarmDays)))  
        );  
    }  
    return alarmBuilder;  
}
```

- BooleanBuilder 를 이용하여 알림 시간으로 가능한 경우들에 대해 or 조건으로 맞는 식재료를 바로 동등 조건을 이용해 인덱스를 탈 수 있게끔 쿼리를 변경했다.
	- `foodEntity.expiryAlarmDays.eq(expiryAlarmDays)` : 사용자가 선택한 알림 시간 설정에 대해서 맞는 지 동등조건으로 먼저 판별
	- `foodEntity.expiryDate.eq(targetDate.plusDays(expiryAlarmDays))` : 유통기한 = 오늘 + 알림시간 인 데이터를 동등 조건으로 판별
- 한 번에 많은 데이터를 불러오면 메모리 초과가 발생한다. 따라서 페이징 객체를 이용하여 커서 기반으로 데이터를 500개씩 가져와서 처리하도록 한다.

아래는 위의 쿼리를 sql 문으로 나타낸 것
```sql
SELECT * FROM food  
WHERE status = 'ACTIVE'  
  AND (  
    (expiry_alarm_days = 0  AND expiry_date = CURDATE())  
        OR (expiry_alarm_days = 1  AND expiry_date = DATE_ADD(CURDATE(), INTERVAL 1 DAY))  
        OR (expiry_alarm_days = 2  AND expiry_date = DATE_ADD(CURDATE(), INTERVAL 2 DAY))  
        OR (expiry_alarm_days = 3  AND expiry_date = DATE_ADD(CURDATE(), INTERVAL 3 DAY))  
        OR (expiry_alarm_days = 7  AND expiry_date = DATE_ADD(CURDATE(), INTERVAL 7 DAY))  
        OR (expiry_alarm_days = 14 AND expiry_date = DATE_ADD(CURDATE(), INTERVAL 14 DAY))  
    )  
  AND food_id < null  
ORDER BY food_id desc ;  
limit 500;
```

### 인덱스 선정 기준

| **인덱스 조합**                               | **Scan Type** | **지연 시간(ms)** | **설명**                                                                             |
| ---------------------------------------- | ------------- | ------------- | ---------------------------------------------------------------------------------- |
| [status, expiry_alarm_days]              | All           | 991ms         | 결국 각 expiryAlarm 에 대해서 모든 데이터의expiry_date 조건이 맞는지 전부 확인하므로 풀 스캔 때려버림               |
| [status, expiry_date]                    | All           | 965ms         | 마찬가지로 정렬된 expiry_date에 대해서 모든 데이터의 expiryAlarm이 원하는 조건과 일치하는지 전부 확인하게 되므로 풀스캔 때려버림 |
| [status, expiry_alarm_days, expiry_date] | range         | 14ms          | 원하는 조건절이 모두 인덱스 안에 있으므로 원본 테이블을 엑세스하지 않고 인덱스 내에서 Range Scan                        |
🆘 foodId 를 선두컬럼에 넣게 되면, foodId 값을 기준으로 내림차순으로 정렬되어있을 것이다. 여기서 인덱스 첫 수직탐색으로 foodId 역순으로 탐색해서 나머지 where 조건절에 부합한지 끝부터 처음까지 다 스캔하게 된다.

## Cursor Pagination

커서 기반으로 받아온 데이터들을 FcmMessage에서 효율적으로 보내기 위해서는 단건씩 처리하는 것보다 한번에 묶음 단위로 보내는 것이 처리량을 극대화할 수 있을 것이라고 생각했다.
FirebaseMessaging에서 배치 처리를 통해 보낼 수 있는 메시지량은 500개가 최대이다. 
따라서 넉넉하게 식재료 조회 쿼리에서 받아오는 데이터를 450개로 정했다.
