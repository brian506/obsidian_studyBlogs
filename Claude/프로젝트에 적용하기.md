## 📄 CLAUDE.md (프로젝트 규칙)

````markdown
# My Awesome Project - 개발 가이드

## 프로젝트 개요

- **목표**: [프로젝트 설명]
- **스택**: TypeScript, React, Node.js
- **팀 규모**: 4명

## 코딩 컨벤션

### 명명 규칙

- 컴포넌트: PascalCase (`UserProfile.tsx`)
- 함수: camelCase (`getUserData()`)
- 상수: UPPER_SNAKE_CASE (`API_TIMEOUT`)
- 파일: kebab-case (`user-profile.tsx`)

### 타입스크립트

- 모든 함수에 타입 지정
- `any` 사용 금지
- strict mode 필수

### React 컴포넌트

```typescript
// 좋은 예
export const UserCard: React.FC<UserCardProps> = ({ user }) => {
  return <div>{user.name}</div>;
};

// 피해야 할 것
export const userCard = (user) => {
  return <div>{user.name}</div>;
};
````

## 테스트

- 모든 유틸리티는 단위 테스트 필수
- API 엔드포인트는 통합 테스트 필수
- 커버리지 목표: 80% 이상

## Git 컨벤션

```
feat: 새로운 기능
fix: 버그 수정
docs: 문서
refactor: 코드 구조 개선
test: 테스트 추가
perf: 성능 개선
chore: 빌드, 의존성 등
```

## 배포 프로세스

1. feature 브랜치에서 개발
2. PR 생성 (자동 테스트 통과 필수)
3. 코드 리뷰 (최소 1명)
4. main에 머지
5. CI/CD 자동 배포

## 연락처

- **팀 리더**: @leader
- **백엔드**: @backend
- **프론트엔드**: @frontend

````

### 1. api-conventions

**경로**: `.claude/skills/api-conventions/SKILL.md`

```yaml
---
name: api-conventions
description: Our REST API design guidelines. Use when designing endpoints or reviewing API code.
when_to_use: "API endpoint", "REST design", "response format"
user-invocable: false
paths: ["src/api/**/*.ts", "src/routes/**/*.ts"]
---

# API 설계 가이드

## 엔드포인트 명명

```

GET /api/v1/users # 목록 GET /api/v1/users/:id # 단일 항목 POST /api/v1/users # 생성 PUT /api/v1/users/:id # 전체 업데이트 PATCH /api/v1/users/:id # 부분 업데이트 DELETE /api/v1/users/:id # 삭제

````

## 응답 형식

성공 (200, 201):
```json
{
  "success": true,
  "data": { ... },
  "meta": { "timestamp": "..." }
}
````

에러 (4xx, 5xx):

```json
{
  "success": false,
  "error": {
    "code": "ERROR_CODE",
    "message": "설명"
  }
}
```

상세 가이드: [response-schemas.md](https://claude.ai/chat/response-schemas.md) 예시: [examples.md](https://claude.ai/chat/examples.md)

# 테스트 가이드

## 단위 테스트 (Jest)

```typescript
describe('getUserData', () => {
  it('should fetch user data', async () => {
    const user = await getUserData('123');
    expect(user.id).toBe('123');
  });

  it('should handle errors', async () => {
    await expect(getUserData('invalid')).rejects.toThrow();
  });
});
````

## 통합 테스트 (API)

```typescript
describe('POST /api/users', () => {
  it('should create user', async () => {
    const res = await request(app)
      .post('/api/users')
      .send({ name: 'John', email: 'john@example.com' });
    
    expect(res.status).toBe(201);
    expect(res.body.data.id).toBeDefined();
  });
});
```

  "hooks": [
    {

      "commands": [
        {
          "type": "Bash",

        },
        {
          "type": "Bash",

## 🎯 실제 사용 시나리오

### 시나리오 1: 새 API 엔드포인트 만들기

```
1. TypeScript 파일 작성
   └─> api-conventions Skill 자동 로드
   └─> Claude가 컨벤션 적용

2. 구현 완료
   └─> /code-style로 검토
   └─> code-reviewer subagent 호출

3. 배포 준비
   └─> /deploy-checklist production 실행
```

### 시나리오 2: 버그 수정

```
1. 코드 작성
   └─> code-reviewer로 자동 검토
   └─> security-auditor로 보안 검사

2. 커밋
   └─> 자동으로 lint + type-check (hook)

3. PR 생성
   └─> documentation-writer로 문서 작성
```

### 시나리오 3: 배포

```
1. main 브랜치 업데이트
   └─> CI/CD 자동 실행

2. 수동 배포 필요 시
   └─> /deploy-checklist production

3. 이슈 발생 시
   └─> security-auditor로 원인 분석
   └─> code-reviewer로 수정 검토
```
