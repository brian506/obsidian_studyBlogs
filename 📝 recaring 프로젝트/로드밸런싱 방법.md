30초에 한번씩 위치 좌표값을 수신받고 MVP 목표로 처음엔 500명을 대상으로 생각하고 있다.
초당 20TPS의 여건을 마련해야한다

ECS 컨테이너 위에서 로드밸런싱의 역할로 ALB와 Nginx 둘 중 하나를 고려했다.

### ALB 대신 Nginx 선택한 이유?

**ALB가 유리한 부분**
먼저 ALB를 선택했을 때는 인증서 자동 갱신과 AWS에서 자동으로 관리해준다. 
또한 별도의 Green-Blue 배포 스크립트 없이 Rolling Update으로 무중단 배포가 가능하고, AZ 다중화, 자동 헬스 체크, WAF 연동이 내장되어 있어 사용자 규모가 커지면 ALB를 선택하는것이 적합해 보인다. 

**Nginx가 유리한 부분**
- ALB는 매달 최소 16달러로 비용이 꽤 나간다. NGINX 는 공짜임 ㅋ
- SSL 인증서 Certbot과 연동해서 90일마다 자동갱신할 수 있긴한데, 추가 구성이 필요
-  ALB는 idle timeout(최대 4000초)과 connection draining 정책이 고정되어 있어 SSE 장기 연결 튜닝이 제한적. nginx는 proxy_read_timeout, keepalive_timeout 등을 완전히 제어 가능

## 주의사항

### 1. Blue-Green 전환 — nginx가 제대로 바꾸는가?
- 현재 GPS 전송 기능은 한 번 연결되면 10분, 20분 계속 연결을 유지하는 SSE 방식임. 근데 배포를 진행할 때 Nginx가 방향을 신버전의 컨테이너로 전환하면, 기존 구버전을 쓰고 있던 사용자들은 계속해서 구버전의 컨테이너와 SSE 연결을 맺으면서 GPS 데이터를 받고 있을 것임. 근데 이제 배포 끝나면 구버전은 종료시키니까 SSE 연결이 강제로 끊기면서 더 이상 데이터를 받을 수 없게됨. 
	-> 프론트에서 서버와 SSE 연결이 끊기게 되면 즉시 다시 연결하는 로직을 별도로 구현해야됨
	

### 2. ECS 전환 후 host.docker.internal:8080 에러
#### Nginx를 도커 컨테이너로 띄울때
Nginx는 자기만의 독립된 방을 갖게됨
- **localhost** 호출 시 : 내 PC의 8080 포트를 찾는 게 아니라 자신의 컨테이너 안에서 8080포트를 찾게 되어 Spring:8080을 못찾음
- **host.docker.internal** 호출 시 : Nginx 컨테이너 안에서 내 PC에 띄워둔 다른 서비스 컨테이너들에 접속하기 위해 host.docker.internal:8080로 다른 호스트 컴퓨터에 접속할 수 이게 됨

#### Nginx를 하나의 ECS task 로 띄울때
- 하나의 ECS task에 두었으므로 같은 네트워크 망을 공유하게 된다.
	- localhost:8080 or 127.0.0.1:8080 으로 통신 가능

### 3. 인증서 자동 갱신
#### 전략 1: EventBridge + ECS Scheduled Task (권장: 관심사 분리)

가장 "AWS다운" 방식입니다. Certbot을 평소에는 꺼두었다가 갱신 시점에만 잠깐 띄워 비용과 리소스를 아낍니다.

- **구조:**
    
    1. **EventBridge:** 매달 1일(또는 15일마다) ECS Task를 실행하는 규칙 생성.
        
    2. **Certbot Task:** EFS를 마운트하여 실행 (`certbot renew`).
        
    3. **Reload 문제 해결:** Certbot 작업이 끝난 후 Nginx에게 "설정 다시 읽어!"라고 알려줘야 합니다.
        
- **구현 팁:** Certbot 컨테이너의 `command` 마지막에 AWS CLI를 포함하여 Nginx 서비스의 배포를 강제로 업데이트(`aws ecs update-service --force-new-deployment`)하거나, 특정 Lambda를 호출해 Nginx 컨테이너 내부에 `nginx -s reload`를 전송합니다.
    

#### 전략 2: Nginx Task 내 '사이드카' 컨테이너 (실시간 감시)

Nginx와 Certbot(또는 가벼운 스크립트)을 **하나의 Task 정의** 안에 나란히 두는 방식입니다.

- **구조:**
    
    - **Container A (Nginx):** 웹 서비스 서빙.
        
    - **Container B (Renew-Watcher):** 아주 가벼운 이미지로, 인증서 파일의 수정을 감시(inotify)하거나 주기적으로 갱신 명령을 내립니다.
        
- **장점:** 같은 Task 내에 있으므로 `localhost` 통신이 가능하고, 설정을 공유하기 매우 쉽습니다
- **단점** : certbot 컨테이너가 계속 켜져있어야함. 얘는 인증서 갱신만 하는데 쓸데없이 리소스 갉아먹음
    

#### 전략 3: Entrypoint 수정 

질문자님이 언급하신 "관심사 분리가 깨지는" 방식이지만, **단일 EC2 MVP**에서는 관리 포인트가 가장 적어 매우 효율적입니다.

- **방법:** Nginx 이미지의 `entrypoint.sh`를 커스텀하여 내부에 백그라운드로 도는 루프 스크립트를 넣습니다

### 3. nginx 다운 시 ECS 자동 복구

  맞음. ECS Service(desiredCount: 1)로 nginx를 관리하면 Task 비정상 종료 시 ECS가 자동으로 새 Task 기동.

  단, 조건이 있음:
  - EFS 마운트 성공해야 nginx가 뜸 (인증서 파일 접근 필요). EFS 마운트 실패하면 Task가 계속 재시작 루프에 빠질 수 있음 → CloudWatch Alarm 설정 권장
  - 재시작 중 30~60초 다운타임 발생 (architecture.md에도 명시됨, MVP 수준 감수)

Nginx 는 24시간 가동되어야 하기 때문에, task가 아닌 service로 ECS 구성되어야함.

## Certbot Task 만들기

### 1. EFS 생성
인증서 자체를 EFS에 넣어둘 것이다.

EFS의 엑세스 포인트에 2개 생성
- letsencrypt 
- webroot
소유자 UID/GID : 0
권한 : 755

### 2. Certbot Docker 이미지

#### Dockerfile
```yml
FROM certbot/certbot:latest  
  
RUN apk add --no-cache aws-cli  
  
COPY entrypoint.sh /entrypoint.sh  
RUN chmod +x /entrypoint.sh  
  
ENTRYPOINT ["/entrypoint.sh"]
```

#### entrypoint.sh

- Nginx는 처음 켜질 때 SSL 인증서를 메모리에 한번만 읽어온다.
- 현재 Certbot 컨테이너와 Nginx 컨테이너가 나누어져있기 때문에, EFS에 읽어온 인증서를 Nginx가 반영하려면 컨테이너를 재시작해야 하는데, 이 스크립트로 ECS에게 Nginx 컨테이너를 재시작하도록 명령하는 것이다.

```bash
#!/bin/sh  
  set -e  
  
  DOMAIN="${DOMAIN:-re-caring.duckdns.org}"  
  EMAIL="${EMAIL:-choiyngmin506@gmail.com}"  
  AWS_REGION="${AWS_REGION:-ap-northeast-2}"  
  
  echo "[Certbot : 시작]: domain=${DOMAIN}"  
  
  # 최초 발급 vs 갱신 분기  
  if [ -f "/etc/letsencrypt/live/${DOMAIN}/fullchain.pem" ]; then  
    echo "[Certbot : 갱신 시도]"  
    certbot renew --non-interactive  
  else  
    echo "[Certbot : 최초 발급 시도]"  
    certbot certonly \  
      --webroot \  
      --webroot-path /var/www/certbot \  
      -d "${DOMAIN}" \  
      --email "${EMAIL}" \  
      --agree-tos \  
      --non-interactive  
  fi  
  
  echo "[Certbot : 인증서 처리 완료]"  
  
  # Nginx force-new-deployment (갱신된 인증서 로드)  
  echo "[ECS : Nginx 재배포 시작]: cluster=${ECS_CLUSTER} service=${ECS_SERVICE}"  
  aws ecs update-service \  
    --cluster "${ECS_CLUSTER}" \  
    --service "${ECS_SERVICE}" \  
    --force-new-deployment \  
    --region "${AWS_REGION}" \  
    --no-cli-pager  
  
  echo "[ECS : Nginx 재배포 요청 완료]"
```
**최초 발급 vs 갱신 분기**
- 인증서가 있을 경우에만 갱신 시도를 하고, 없으면 발급
- 무조건 발급 명령만 하면, Let's Encrypt 발급 제한 횟수에 걸려 차단당함
- 무조건 갱신 명령만 하면, 처음에 아무것도 없을 때 실패

**Nginx force-new-deployment**
- certbot 컨테이너 안에서 certbot Dockerfile에 적어둔 AWS CLI을 설정해둬서 Nginx 컨테이너 재시동한다.
- Certbot ECS Task에 부여되는 역할에서 ecs:UpdateService 권한을 허용하는 IAM 정책을 붙여야한다.
#### Certbot 컨테이너에게 Nginx 컨테이너 update 권한 부여
- 인라인 정책 추가

```
  {
    "Version": "2012-10-17",
    "Statement": [
      {
        "Effect": "Allow",
        "Action": [
          "ecs:UpdateService",
          "ecs:DescribeServices"
        ],
        "Resource": "arn:aws:ecs:ap-northeast-2:<ACCOUNT_ID>:service/<CLUSTER_NAME>/<NGINX_SERVICE_NAME>"
      },
      {
        "Effect": "Allow",
        "Action": [
          "logs:CreateLogGroup",
          "logs:CreateLogStream",
          "logs:PutLogEvents"
        ],
        "Resource": "*"
      }
    ]
  }
```


**동작 순서**
1. Nginx 컨테이너는 부팅될 때 설정파일에 적힌 경로에서 인증서를 찾는다. 이때 해당 경로는 EFS와 연결되어 있다.
2. Certbot 태스크가 실행되면 /etc/letsencrypt 폴더 안의 파일들을 검사한다.만료일이 얼마 안남았으면, 새로운 인증서를 받아와서 덮어쓰기 한다.
3. EventBridge로 설정한 시간마다 Certbot 태스크를 실행해서 인증서를 갱신한다.
4. EFS 안의 파일이 바뀌어도, Nginx는 예전 인증서를 읽고 있기 때문에, Nginx를 강제로 다시 새로 띄우고, EFS에 업데이트된 새 인증서를 읽게 된다.
