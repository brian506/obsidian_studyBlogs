## ECS 개념

**실행 모드**
- Service : 24시간 상시 실행
- Task : 단발성 실행

### 1. EC2에 ECS 에이전트 설치 및 클러스터 등록

```bash
# ubuntu ecs 에이전트 설치
sudo apt-get update && sudo apt-get install -y amazon-ecs-init
 # 2. 클러스터 이름 설정
 sudo bash -c 'echo ECS_CLUSTER=recaring-cluster >> /etc/ecs/ecs.config'

 # 3. ECS 에이전트 시작
 sudo systemctl enable --now ecs

 # 4. 등록 확인 (잠시 후)
 sudo systemctl status ecs
```

## ENI(Elastic Network Interface)
>VPC 안에서 동작하는 가상 랜카드

ENI 하나가 가지는 것들:

- **Primary Private IP** (필수, 변경 불가)
- **Secondary Private IP들** (선택, 여러 개 가능)
- **Public IP / EIP** (선택, ENI당 하나)
- **MAC 주소** (ENI 만들 때 고정)
- **Security Group** (ENI당 최대 5개)
- **소속 서브넷** (만들 때 정해지고 변경 불가)

EC2를 만들면 자동으로 호스트인 primary ENI(eth0)가 생긴다. 

**태스크 안의 컨테이너는 EC2 host의 다른 ENI를 못 본다.**
- 호스트 OS가 ENI를 컨테이너의 네트워크 namespace로 격리시키기 때문
- 그래서 별도 EC2처럼 **보안 그룹을 따로 받고, IP도 따로 가진다**


## 네트워크 모드에 따른 장단점

![](../images/스크린샷%202026-04-25%2017.18.15.png)
### bridge 모드
특징 : 로컬에서 쓰던 일반적인 도커 네트워크. EC2 인스턴스의 IP 1개를 모든 컨테이너가 공유하며, 도커가 내부적으로 가상 브리지를 만들어 통신
- 컨테이너는 별도의 IP를 갖지 않고, EC2 호스트의 프라이빗 IP와 포트 매핑을 통해 통신
- EC2 프라이빗 IP + 매핑된 포트 구조로 사용
장점 : ENI 개수 제한 없음. 동적 포트 매핑을 지원하여, 포트 충돌 없이 여러 ㅓㄴ테이너를 무한정 띄울 수 있음.
단점 : 컨테이너 개별로 AWS 보안 그룹을 걸 수 없고, EC2 호스트 수준에서만 방어 가능
- Max : 100, Min : 0 으로 해야됨. 그래야 다시 배포하는 과정에서 포트충돌이 일어나지 않음
	- 가용영역 리밸런싱 꺼줘야됨. 이건 멀티 AZ일때 태스크를 균등하게 분산시키는 기능임
### awsvpc 모드
특징 : 컨테이너 하나당 독립적인 ENI와 VPC 프라이빗 IP를 하나씩 꽂아줌
장점 : 컨테이너가 하나의 독립된 EC2 인스턴스처럼 동작하므로 보안과 네트워크 추적 완벽
단점 : EC2 인스턴스 타입마다 꽂을 수 있는 ENI 최대 개수 제한
- 컨테이너가 죽었다 살아나면 IP가 매번 무조건 바뀜

### host 모드
특징 : 도커의 가상 네트워크 계층을 무시하고, 호스트의 네트워크를 그대로 씀
장점 : 네트워크 병목 없어서 개빠름
단점 : 포트 충돌에 취약.(하나의 포트를 쓰는 컨테이너를 두개 못띄움)


## 1. ECS의 4단계 계층 구조: "관리와 실행의 위계"

ECS는 애플리케이션을 논리적인 단위로 감싸 관리합니다. 이를 **마트료시카(러시아 인형)** 구조로 이해하면 쉽습니다.

- **Cluster (클러스터):** 최상위 논리적 경계. EC2 인스턴스나 Fargate 리소스가 모여 있는 '사무실 건물'입니다.
    
- **Service (서비스):** 핵심 매니저. 특정 태스크 정의(설계도)를 바탕으로 '항상 N개의 태스크가 살아있도록 유지'하는 감독관입니다. 장애 발생 시 자동 복구(Auto-healing)와 배포(Rolling Update)를 주관합니다.
    
- **Task (태스크):** 배포의 최소 단위. 서비스 매니저에 의해 복제되는 '작업 그룹'입니다. 하나의 태스크 안에는 하나 또는 여러 개의 컨테이너가 포함될 수 있습니다.
    
- **Container (컨테이너):** 실제 로직이 구동되는 **'실무자'**. 도커 이미지로 규격화된 Spring Boot, Nginx, Redis 등이 여기에 해당합니다.
    

> **핵심 원칙:** 하나의 서비스는 오직 **하나의 태스크 정의(설계도)**만 관리합니다. "Nginx 서비스 하나가 Nginx 태스크와 App 태스크를 동시에 관리할 수는 없습니다." 역할이 다르면 서비스도 분리되어야 합니다.



## 서킷 브레이커

### 문제 상황
새 컨테이너가 에러가 발생하여 죽는 상황이 발생했을 때, 설정된 유지 개수를 맞추기 위해 실패한 컨테이너를 계속 띄우고 죽고 하는 상황이 반복되게 된다.

### 서킷 브레이커와 자동 롤백
새 컨테이너의 실패 상태가 누적되어 임계치에 도달하면, ECS는 정상적인 배포를 할 수 없다고 판단하여 서킷 브레이커를 동작하게 되어 무의미한 재시도를 즉시 중단한다.
또한, 자동 롤백으로 가장 마지막으로 정상 작동했던 이전 버전으로 스위치를 자동으로 되돌려 놓는다.



## 트러블슈팅

### ENI는 EIP를 가질 수 있으니까 NAT Gateway 없이 EIP 할당해서 외부와 연결하면 안되나?

#### 안되는 이유들

**1. 태스크의 ENI는 일회용**

awsvpc 태스크는 죽고 살아날 때마다 **새 ENI가** 만들어진다. 이전의 ENI는 사라지기 때문에, ENI에 EIP를 할당해도 EIP는 계속 이전의 ENI에 할당되고 새 ENI에는 EIP가 없는 상태가 지속된다.
