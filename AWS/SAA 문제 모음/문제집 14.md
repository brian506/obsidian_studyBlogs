## 문제 146

한 회사가 Application Load Balancer 뒤에 있는 Amazon EC2 온디맨드 인스턴스 그룹의 프로덕션 환경에서 상태를 저장하지 않는(stateless) 웹 애플리케이션을 실행하고 있습니다. 이 애플리케이션은 영업일 기준 매일 8시간 동안 사용량이 많습니다. 야간에는 애플리케이션 사용량이 보통 수준이고 안정적입니다. 주말에는 애플리케이션 사용량이 적습니다. 회사는 애플리케이션의 가용성에 영향을 미치지 않으면서 EC2 비용을 최소화하려고 합니다. 어떤 솔루션이 이러한 요구 사항을 충족합니까?

- **A.** 전체 워크로드에 스팟 인스턴스(Spot Instances)를 사용합니다.
    
- **B.** 기본(baseline) 사용 수준에는 예약 인스턴스(Reserved Instances)를 사용합니다. 애플리케이션에 필요한 추가 용량에는 스팟 인스턴스를 사용합니다.
    
- **C.** 기본 사용 수준에는 온디맨드 인스턴스를 사용합니다. 애플리케이션에 필요한 추가 용량에는 스팟 인스턴스를 사용합니다.
    
- **D.** 기본 사용 수준에는 전용 인스턴스(Dedicated Instances)를 사용합니다. 애플리케이션에 필요한 추가 용량에는 온디맨드 인스턴스를 사용합니다.

>[!.questions]- 정답
>정답 : B
>정답 조건 : 언제나 필요한 최소한의 자원 + 추가로 필요한 변동 자원
>꾸준히 사용되는 워크로드
>- 예약 인스턴스 or Saving Plans
>stateless + 일정 시간 사용량 급증

## 문제 150

한 회사가 애플리케이션을 온프레미스 서버에서 Amazon EC2 인스턴스로 마이그레이션하고 있습니다. 마이그레이션 설계 요구 사항의 일부로 솔루션 아키텍트는 인프라 지표 경보(metric alarms)를 구현해야 합니다. 회사는 짧은 시간 동안 CPU 사용률이 50% 이상으로 증가하더라도 조치를 취할 필요가 없습니다. 하지만 CPU 사용률이 50%를 초과하는 동시에 디스크의 읽기 IOPS가 높을 경우에는 가능한 한 빨리 조치를 취해야 합니다. 또한 솔루션 아키텍트는 오경보(false alarms)를 줄여야 합니다. 솔루션 아키텍트는 이러한 요구 사항을 충족하기 위해 무엇을 해야 합니까?

- **A.** 가능한 경우 Amazon CloudWatch 복합 경보(composite alarms)를 생성합니다.
    
- **B.** 지표를 시각화하고 문제에 신속하게 대응할 수 있도록 Amazon CloudWatch 대시보드를 생성합니다.
    
- **C.** 애플리케이션을 모니터링하고 경보를 발생시키도록 Amazon CloudWatch Synthetics 카나리아(canaries)를 생성합니다.
    
- **D.** 가능한 경우 여러 지표 임계값을 사용하여 단일 Amazon CloudWatch 지표 경보를 생성합니다.

>[!.questions]- 정답
>정답 : A
>정답 조건 : 서로 다른 지표가 특정 조건을 동시에 만족할 때만 알림을 보내는 로직이 무엇인가?
>Amazon CloudWatch 복합 경보
>기능 : 여러 개의 개별 경보를 논리 연산자로 결합하여 하나의 상위 경보를 만드는 기능
>경보 방식 
>경보 1 : CPU 사용률 > 50 일 때 발생
>경보 2 : 디스크 읽기 IOPS 높으면 발생
>복합 경보 : 경보1 AND 경보2 
>-> 이렇게 하면 복합 경보일 때의 조건만 알림을 발생시킴
>

>[!.questions]- 문항별 풀이
>C : 카나리아는 사용자 입장에서 웹사이트의 엔드포인트나 URL이 정상 작동하는지 외부에서 테스트하는 스크립트. 
>D : CloudWatch 지표 경보는 하나의 지표만 모니터링 가능. 한 경보 안에 서로 다른 두 지표를 섞어서 울리는 건 불가능
