## 문제 1 - AWS Config

AWS에서 웹 애플리케이션을 호스팅하는 한 회사가 모든 Amazon EC2 인스턴스, Amazon RDS DB 인스턴스, Amazon Redshift 클러스터에 태그가 올바르게 설정되어 있는지 확인하고자 합니다. 회사는 이 점검을 구성하고 운영하는 데 드는 노력을 최소화하려 합니다.

**솔루션 아키텍트는 이를 달성하기 위해 무엇을 해야 합니까?**

- A. AWS Config 규칙을 사용하여 태그가 올바르게 설정되지 않은 리소스를 정의하고 감지합니다.
- B. Cost Explorer를 사용하여 태그가 올바르게 설정되지 않은 리소스를 표시합니다. 해당 리소스에 수동으로 태그를 지정합니다.
- C. 모든 리소스의 태그 할당이 올바른지 확인하는 API 호출 코드를 작성합니다. EC2 인스턴스에서 주기적으로 코드를 실행합니다.
- D. 모든 리소스의 태그 할당이 올바른지 확인하는 API 호출 코드를 작성합니다. Amazon CloudWatch를 통해 AWS Lambda 함수를 예약하여 주기적으로 코드를 실행합니다.

>[!.questions]- 정답
>정답 : A
>AWS Config
> 규정 준수 자동화 : AWS 리소스 설정이 정의한 규칙에 맞는지 지속적으로 모니터링하는 서비스

## 문제 3 - Kinesis

한 회사가 AWS에서 온라인 마켓플레이스 웹 애플리케이션을 운영하고 있습니다. 이 애플리케이션은 피크 시간대에 수십만 명의 사용자에게 서비스를 제공합니다. 회사는 수백만 건의 금융 거래 세부 정보를 여러 내부 애플리케이션과 공유하기 위해 확장 가능하고 거의 실시간에 가까운 솔루션이 필요합니다. 또한 거래 데이터를 저장하기 전에 민감한 정보를 제거하여 낮은 지연 시간으로 조회 가능한 문서 데이터베이스에 저장해야 합니다.

**솔루션 아키텍트는 이 요구 사항을 충족하기 위해 무엇을 권장해야 합니까?**

- A. 거래 데이터를 Amazon DynamoDB에 저장합니다. 쓰기 시마다 민감한 데이터를 제거하는 DynamoDB 규칙을 설정합니다. DynamoDB Streams를 사용하여 다른 애플리케이션과 거래 데이터를 공유합니다.
- B. Amazon Kinesis Data Firehose로 거래 데이터를 스트리밍하여 Amazon DynamoDB와 Amazon S3에 저장합니다. Kinesis Data Firehose와 AWS Lambda를 통합하여 민감한 데이터를 제거합니다. 다른 애플리케이션은 Amazon S3에 저장된 데이터를 사용합니다.
- C. Amazon Kinesis Data Streams로 거래 데이터를 스트리밍합니다. AWS Lambda를 통합하여 각 거래에서 민감한 데이터를 제거한 후 Amazon DynamoDB에 저장합니다. 다른 애플리케이션은 Kinesis 데이터 스트림에서 거래 데이터를 사용합니다.
- D. 배치 처리된 거래 데이터를 Amazon S3에 파일로 저장합니다. AWS Lambda를 사용하여 각 파일을 처리하고 민감한 데이터를 제거한 후 S3에 업데이트합니다. Lambda 함수는 데이터를 Amazon DynamoDB에 저장합니다. 다른 애플리케이션은 Amazon S3에 저장된 거래 파일을 사용합니다.

>[!.questions]- 정답
>정답 : C
> 정답 조건
> - Kinesis Data Stream는 여러 개의 앱이 동시에 같은 데이터를 읽어가는 '멀티 컨슈머' 기능 제공 (1:N 분산)하여 실시간 공유가 가능
> - AWS Lambda를 Kinesis에 통합하여 실시간으로 메시지를 가공할 수 있음
> - DynamoDB는 문서형 DB로, 굉장히 낮은 지연시간 보장

>[!.questions]- 문항별 풀이
>A : DynamoDB는 스스로 데이터를 가공하는 기능이 없음.Lambda와 결합해야함
>B : Kinesis Data Firehose 는 S3나 Redshift 같은 서비스에 로드하는데 특화된 서비스. 주로 분석용 저장소로 보낼 때 씀

## 문제 5 - AWS Shield

한 회사가 AWS 클라우드에서 퍼블릭 웹 애플리케이션을 출시할 준비를 하고 있습니다. 아키텍처는 Elastic Load Balancer(ELB) 뒤의 VPC 내 Amazon EC2 인스턴스로 구성됩니다. DNS에는 서드파티 서비스가 사용됩니다. 회사의 솔루션 아키텍트는 대규모 DDoS 공격을 탐지하고 방어하는 솔루션을 권장해야 합니다.

**이 요구 사항을 충족하는 솔루션은 무엇입니까?**

- A. 계정에서 Amazon GuardDuty를 활성화합니다.
- B. EC2 인스턴스에서 Amazon Inspector를 활성화합니다.
- C. AWS Shield를 활성화하고 Amazon Route 53에 할당합니다.
- D. AWS Shield Advanced를 활성화하고 ELB에 할당합니다.

>[!.questions]- 정답
>정답 : D
>정답 조건 : 서드파티 DNS 사용 + 대규모 DDoS 방어
> 대규모 DDos : 대규모 공격을 받을 땐 Shield 'Advanced'를 써야됨
> 서드파티 DNS : DNS는 외부업체를 쓰기 때문에 공격이 처음으로 들어오는 ELB에 Shield Advanced 활성화

>[!.questions]- 문항별 풀이
>A : GuardDuty는 계정 내에 이상한 행동을 찾아내는 위협 탐지 서비스
>B : Inspector는 보안과 취약점이 없는지 검사하는 서비스
>C : DNS는 외부 업체를 쓰기 때문에 Route53은 제외됨

참고 - [AWS Security & Encryption](../SAA/AWS%20Security%20&%20Encryption.md)

## 문제 6 - KMS

한 회사가 AWS 클라우드에서 애플리케이션을 구축하고 있습니다. 이 애플리케이션은 두 개의 AWS 리전에 있는 Amazon S3 버킷에 데이터를 저장합니다. 회사는 S3 버킷에 저장되는 모든 데이터를 AWS Key Management Service(AWS KMS) 고객 관리형 키로 암호화해야 합니다. 두 S3 버킷의 데이터는 동일한 KMS 키로 암호화 및 복호화되어야 합니다. 데이터와 키는 두 리전 각각에 저장되어야 합니다.

**운영 오버헤드가 가장 적은 솔루션은 무엇입니까?**

- A. 각 리전에 S3 버킷을 생성합니다. Amazon S3 관리형 암호화 키(SSE-S3)를 사용한 서버 측 암호화를 사용하도록 S3 버킷을 구성합니다. S3 버킷 간 복제를 구성합니다.
- B. 고객 관리형 다중 리전 KMS 키를 생성합니다. 각 리전에 S3 버킷을 생성합니다. S3 버킷 간 복제를 구성합니다. 클라이언트 측 암호화와 함께 KMS 키를 사용하도록 애플리케이션을 구성합니다.
- C. 각 리전에 고객 관리형 KMS 키와 S3 버킷을 생성합니다. Amazon S3 관리형 암호화 키(SSE-S3)를 사용한 서버 측 암호화를 사용하도록 S3 버킷을 구성합니다. S3 버킷 간 복제를 구성합니다.
- D. 각 리전에 고객 관리형 KMS 키와 S3 버킷을 생성합니다. AWS KMS 키(SSE-KMS)를 사용한 서버 측 암호화를 사용하도록 S3 버킷을 구성합니다. S3 버킷 간 복제를 구성합니다.

>[!.questions]- 정답
>정답 : B
> 다중 리전 KMS 키
> - 동일한 키 사용 : 일반적으로 KMS는 리전별로 고유함. 고객 관리형 다중 리전 KMS 키로 여러 리전에서 동일한 키를 가짐.
> - 운영 효율성 : 다중 리전 키를 사용하면 리전 간 데이터를 옮길 때마다 다시 암호화할 필요가 없음. 

>[!.questions]- 문항별 풀이
>A & C : 관리형 암호화 키(SSE-S3)가 아닌 고객 관리형 키(KMS)를 사용해야함
>D : 각 리전에 KMS를 따로 생성하게 되면 다른 키값을 가지게 되어 동일한 암호화 및 복호화가 되지 않는다.

참고 - [AWS Security & Encryption](../SAA/AWS%20Security%20&%20Encryption.md)

## 문제 7 - SSH

한 회사가 최근 AWS 계정의 Amazon EC2 인스턴스에서 다양한 새 워크로드를 시작했습니다. 회사는 인스턴스에 원격으로 안전하게 접근하고 관리하는 전략을 수립해야 합니다. 또한 기본 AWS 서비스와 함께 작동하고 AWS Well-Architected Framework를 따르는 반복 가능한 프로세스를 구현해야 합니다.

**운영 오버헤드가 가장 적은 솔루션은 무엇입니까?**

- A. EC2 직렬 콘솔을 사용하여 각 인스턴스의 터미널 인터페이스에 직접 접근하여 관리합니다.
- B. 기존 인스턴스와 새 인스턴스 각각에 적절한 IAM 역할을 연결합니다. AWS Systems Manager Session Manager를 사용하여 원격 SSH 세션을 설정합니다.
- C. 관리용 SSH 키 페어를 생성합니다. 각 EC2 인스턴스에 공개 키를 로드합니다. 퍼블릭 서브넷에 배스천 호스트를 배포하여 각 인스턴스 관리를 위한 터널을 제공합니다.
- D. AWS Site-to-Site VPN 연결을 설정합니다. 관리자가 VPN 터널을 통해 SSH 키를 사용하여 온프레미스 머신에서 인스턴스에 직접 연결하도록 지시합니다.

>[!.questions]- 정답
>정답 : B
>AWS Systems Manager Session Manager 특징
>키 관리 불필요 : 전통적인 SSH방식은 pem 키 파일을 분실 및 유출 위험과 관리자가 바뀔때마다 키를 갱신해야 하는 오버헤드 비용 있음. Session Manager 는 키 대신 IAM 역할을 통해 권한을 제어
>보안성 : 인스턴스에 퍼블릭 IP 없어도 접근 가능. SSH (22번)포트 안열어도됨
>감사 기능 

>[!.questions]- 문항별 풀이
> A : 직렬 콘솔은 서버 부팅이 안되거나 네트워크 설정이 꼬여서 먹통일 때 쓰는 최후의 수단
> C : 별도의 서버를 배포하는 것은 운영 오버헤드 증가
> D : VPN 새로 파는 것도 운영 오버헤드 증가


## 문제 8 

한 회사가 Amazon S3에 정적 웹사이트를 호스팅하고 있으며 DNS에는 Amazon Route 53을 사용하고 있습니다. 전 세계에서 웹사이트에 대한 수요가 증가하고 있습니다. 회사는 웹사이트에 접근하는 사용자들의 지연 시간을 줄여야 합니다.

**가장 비용 효율적으로 이 요구 사항을 충족하는 솔루션은 무엇입니까?**

- A. 모든 AWS 리전에 웹사이트가 포함된 S3 버킷을 복제합니다. Route 53 지리적 위치 라우팅 항목을 추가합니다.
- B. AWS Global Accelerator에서 액셀러레이터를 프로비저닝합니다. 제공된 IP 주소를 S3 버킷에 연결합니다. Route 53 항목을 편집하여 액셀러레이터의 IP 주소를 가리키도록 합니다.
- C. S3 버킷 앞에 Amazon CloudFront 배포를 추가합니다. Route 53 항목을 편집하여 CloudFront 배포를 가리키도록 합니다.
- D. 버킷에서 S3 Transfer Acceleration을 활성화합니다. Route 53 항목을 편집하여 새 엔드포인트를 가리키도록 합니다.

>[!.questions]- 문항별 풀이
> B : Global Accelerator는 캐싱 기능이 없음. 주로 UDP,TCP 트래픽을 AWS 전용망으로 빠르게 보내는 용도
> D : S3 Transfer Acceleration은 업로드 속도 향상을 위한 서비스. 멀리 떨어진 곳에서 S3로 대용량 파일을 올릴 때 엣지를 거쳐서 올리는 기술

## 문제 10 - Kinesis Data Firehose

한 회사에 수천 개의 엣지 디바이스가 있으며, 이 디바이스들은 매일 총 1TB의 상태 알림을 생성합니다. 각 알림의 크기는 약 2KB입니다. 솔루션 아키텍트는 향후 분석을 위해 알림을 수집하고 저장하는 솔루션을 구현해야 합니다.

회사는 고가용성 솔루션을 원합니다. 단, 비용을 최소화하고 추가 인프라를 관리하지 않으려 합니다. 또한 즉각적인 분석을 위해 14일치 데이터를 사용 가능한 상태로 유지하고, 14일이 지난 데이터는 아카이브하려 합니다.

**이 요구 사항을 충족하는 가장 운영 효율적인 솔루션은 무엇입니까?**
