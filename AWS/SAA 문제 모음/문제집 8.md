## 문제 1

한 회사가 Amazon DynamoDB를 사용하여 고객 정보를 저장하는 쇼핑 애플리케이션을 실행합니다. 데이터 손상 시 솔루션 아키텍트는 15분의 복구 지점 목표(RPO)와 1시간의 복구 시간 목표(RTO)를 충족하는 솔루션을 설계해야 합니다. 솔루션 아키텍트는 이러한 요구 사항을 충족하기 위해 무엇을 권장해야 합니까?

- **A.** DynamoDB 글로벌 테이블(global tables)을 구성합니다. RPO 복구를 위해 애플리케이션이 다른 AWS 리전을 가리키도록 합니다.
    
- **B.** DynamoDB 특정 시점 복구(point-in-time recovery)를 구성합니다. RPO 복구를 위해 원하는 시점으로 복원합니다.
    
- **C.** 매일 DynamoDB 데이터를 Amazon S3 Glacier로 내보냅니다(export). RPO 복구를 위해 S3 Glacier에서 DynamoDB로 데이터를 가져옵니다(import).
    
- **D.** 15분마다 DynamoDB 테이블에 대한 Amazon Elastic Block Store(Amazon EBS) 스냅샷을 예약합니다. RPO 복구를 위해 EBS 스냅샷을 사용하여 DynamoDB 테이블을 복원합니다.

>[!.questions]- 정답
>정답 : B
>RPO : 장애 발생 시 최대 얼마만큼의 데이터 소신을 감수할 수 있는가
>RTO : 장애 발생 후 얼마 만에 서비스를 정상화해야 하는가
>DynamoDB PITR : 초 단위로 백업하고, AWS 관리형 서비스

>[!.questions]- 문항별 풀이
>A : 글로벌 테이블은 데이터 오염이 발생했을 때 이것도 똑같이 모든 리전에 복제되므로 복제 솔루션으로는 부적합
>D : DynamoDB는 AWS가 관리하는 서비스이므로 스냅샷 생성 불가

## 문제 2 - 동일 리전 내 데이터 전송 비용 최적화

한 회사가 동일한 AWS 리전에 위치한 Amazon S3 버킷에서 사진을 자주 업로드하고 다운로드해야 하는 사진 처리 애플리케이션을 실행하고 있습니다. 솔루션 아키텍트는 데이터 전송 수수료 비용이 증가한 것을 발견했으며 이 비용을 줄이기 위한 솔루션을 구현해야 합니다. 솔루션 아키텍트는 이 요구 사항을 어떻게 충족할 수 있습니까?

- **A.** 퍼블릭 서브넷에 Amazon API Gateway를 배포하고 이를 통해 S3 호출을 라우팅하도록 라우팅 테이블을 조정합니다.
    
- **B.** 퍼블릭 서브넷에 NAT 게이트웨이를 배포하고 S3 버킷에 대한 액세스를 허용하는 엔드포인트 정책을 연결합니다.
    
- **C.** 퍼블릭 서브넷에 애플리케이션을 배포하고 인터넷 게이트웨이를 통해 S3 버킷에 액세스할 수 있도록 허용합니다.
    
- **D.** VPC에 S3 VPC 게이트웨이 엔드포인트를 배포하고 S3 버킷에 대한 액세스를 허용하는 엔드포인트 정책을 연결합니다.

>[!.questions]- 정답
>정답 : D 
>S3 VPC 게이트웨이 엔드포인트
>- 게이트웨이 엔드포인트는 무료
>- AWS 전용 네트워크에서 이동

## 문제 3 - VPC 통신

한 회사가 최근 VPC의 프라이빗 서브넷에 있는 Amazon EC2에서 Linux 기반 애플리케이션 인스턴스를 시작하고 퍼블릭 서브넷의 Amazon EC2 인스턴스에서 Linux 기반 배스천 호스트(bastion host)를 시작했습니다. 솔루션 아키텍트는 회사의 인터넷 연결을 통해 온프레미스 네트워크에서 배스천 호스트와 애플리케이션 서버에 연결해야 합니다. 솔루션 아키텍트는 모든 EC2 인스턴스의 보안 그룹이 해당 액세스를 허용하는지 확인해야 합니다. 솔루션 아키텍트가 이러한 요구 사항을 충족하기 위해 취해야 하는 단계 조합은 무엇입니까? (2개 선택)

- **A.** 배스천 호스트의 현재 보안 그룹을 애플리케이션 인스턴스에서 들어오는 인바운드 액세스만 허용하는 그룹으로 교체합니다.
    
- **B.** 배스천 호스트의 현재 보안 그룹을 회사의 내부 IP 범위에서 들어오는 인바운드 액세스만 허용하는 그룹으로 교체합니다.
    
- **C.** 배스천 호스트의 현재 보안 그룹을 회사의 외부 IP 범위에서 들어오는 인바운드 액세스만 허용하는 그룹으로 교체합니다.
    
- **D.** 애플리케이션 인스턴스의 현재 보안 그룹을 배스천 호스트의 프라이빗 IP 주소에서만 들어오는 인바운드 SSH 액세스를 허용하는 그룹으로 교체합니다.
    
- **E.** 애플리케이션 인스턴스의 현재 보안 그룹을 배스천 호스트의 퍼블릭 IP 주소에서만 들어오는 인바운드 SSH 액세스를 허용하는 그룹으로 교체합니다.

>[!.questions]- 정답
>정답 : C, D
>정답 조건 : 온프레미스 -> 배스천 -> APP
>배스천 호스트는 프라이빗 서브넷에 있는 APP에 안전하게 접속하기 위한 징검다리
>
>온프레미스 -> 배스천 서버
>- 사무실 네트워크는 공용 네트워크를 통과하므로, 배스천 보안 그룹은 외부 IP로부터 인바운드 SSH 요청만 허용하도록 하는 것이 안전
>
>배스천 서버 -> APP 서버
>여기서 배스천 서버와 APP 서버는 같은 VPC 안에 있으므로 프라이빗 네트워크를 통해 통신한다. 따라서 배스천 호스트의 프라이빗 IP주소로부터 오는 SSH만 허용

>[!.questions]- 문항별 풀이
>A & B : 베스천 호스트는 외부에서 안으로 들어오기 위한 입구. 내부 IP나 APP인스턴스의 접속만 허용하면 외부에서 접근 불가
>E : VPC 내부 통신은 퍼블릭 IP가 아닌 프라이빗 IP 사용


## 문제 4 - DB 포트 매핑

솔루션 아키텍트가 2계층(two-tier) 웹 애플리케이션을 설계하고 있습니다. 이 애플리케이션은 퍼블릭 서브넷의 Amazon EC2에서 호스팅되는 퍼블릭 대상(public-facing) 웹 계층으로 구성됩니다. 데이터베이스 계층은 프라이빗 서브넷의 Amazon EC2에서 실행되는 Microsoft SQL Server로 구성됩니다. 회사에서 보안은 최우선 순위입니다. 이 상황에서 보안 그룹을 어떻게 구성해야 합니까? (2개 선택)

- **A.** 웹 계층의 보안 그룹이 0.0.0.0/0에서 포트 443으로의 인바운드 트래픽을 허용하도록 구성합니다.
    
- **B.** 웹 계층의 보안 그룹이 0.0.0.0/0에서 포트 443으로의 아웃바운드 트래픽을 허용하도록 구성합니다.
    
- **C.** 데이터베이스 계층의 보안 그룹이 웹 계층의 보안 그룹에서 포트 1433으로의 인바운드 트래픽을 허용하도록 구성합니다.
    
- **D.** 데이터베이스 계층의 보안 그룹이 웹 계층의 보안 그룹으로 포트 443 및 1433의 아웃바운드 트래픽을 허용하도록 구성합니다.
    
- **E.** 데이터베이스 계층의 보안 그룹이 웹 계층의 보안 그룹에서 포트 443 및 1433으로의 인바운드 트래픽을 허용하도록 구성합니다.

>[!.questions]- 정답
>정답 : A,C

## 문제 5

한 회사가 애플리케이션 성능을 향상시키기 위해 온프레미스에서 AWS 클라우드로 다중 계층 애플리케이션을 이동하고자 합니다. 애플리케이션은 RESTful 서비스를 통해 서로 통신하는 애플리케이션 계층으로 구성됩니다. 한 계층이 과부하되면 트랜잭션이 삭제됩니다. 솔루션 아키텍트는 이러한 문제를 해결하고 애플리케이션을 현대화하는 솔루션을 설계해야 합니다. 이러한 요구 사항을 충족하며 운영상 가장 효율적인 솔루션은 무엇입니까?

- **A.** Amazon API Gateway를 사용하고 애플리케이션 계층으로서 트랜잭션을 AWS Lambda 함수로 보냅니다. 애플리케이션 서비스 간의 통신 계층으로 Amazon Simple Queue Service(Amazon SQS)를 사용합니다.
    
- **B.** Amazon CloudWatch 지표를 사용하여 애플리케이션 성능 내역을 분석하고 성능 장애 중 서버의 최고 사용률을 확인합니다. 피크 요구 사항을 충족하도록 애플리케이션 서버의 Amazon EC2 인스턴스 크기를 늘립니다.
    
- **C.** Amazon Simple Notification Service(Amazon SNS)를 사용하여 Auto Scaling 그룹의 Amazon EC2에서 실행되는 애플리케이션 서버 간의 메시징을 처리합니다. Amazon CloudWatch를 사용하여 SNS 대기열 길이를 모니터링하고 필요에 따라 스케일 업 및 스케일 다운합니다.
    
- **D.** Amazon Simple Queue Service(Amazon SQS)를 사용하여 Auto Scaling 그룹의 Amazon EC2에서 실행되는 애플리케이션 서버 간의 메시징을 처리합니다. Amazon CloudWatch를 사용하여 SQS 대기열 길이를 모니터링하고 통신 실패가 감지되면 스케일 업합니다.

>[!.questions]- 정답
>정답 : A
>다중 계층 애플리케이션은 Web-App-DB 같은 구조. (MSA 아님)
>정답 조건 
>- 트랜잭션 유실 방지 (SQS)
>	- '한 계층이 과부하되면 트랜잭션 삭제' -> 동기식으로 통신하고 있기 때문
>	- 처리 과부하 오지 않도록 큐에 메시지를 쌓아 두는 방식이 적합
>- 운영 효율이 가장 좋은 것은 '서버리스' 조합은 API Gateway + Lambda

## 문제 6

>-증분 복사, 데이터 무결성 검사, 네트워크 대역폭 관리 기능 -> 안정적인 전송
>

>[!.questions]- 문항별 풀이

## 문제 8

한 회사가 사용자 트랜잭션 데이터를 Amazon DynamoDB 테이블에 보관해야 합니다. 회사는 이 데이터를 7년 동안 보존해야 합니다. 이러한 요구 사항을 충족하는 운영상 가장 효율적인 솔루션은 무엇입니까?

- **A.** DynamoDB 특정 시점 복구(point-in-time recovery)를 사용하여 테이블을 지속적으로 백업합니다.
    
- **B.** AWS Backup을 사용하여 테이블에 대한 백업 일정 및 보존 정책을 생성합니다.
    
- **C.** DynamoDB 콘솔을 사용하여 테이블의 온디맨드 백업을 생성합니다. Amazon S3 버킷에 백업을 저장합니다. S3 버킷에 대한 S3 수명 주기 구성을 설정합니다.
    
- **D.** AWS Lambda 함수를 호출하는 Amazon EventBridge(Amazon CloudWatch Events) 규칙을 생성합니다. 테이블을 백업하고 Amazon S3 버킷에 백업을 저장하도록 Lambda 함수를 구성합니다. S3 버킷에 대한 S3 수명 주기 구성을 설정합니다.

>[!.questions]- 정답
>정답 : B
>AWS Backup : DyanmoDB, RDS, EFS, S3등의 서비스 백엡을 한곳에서 관리해주는 서비스
>- 보존 규칙을 백엡 플랜에서 설정하여 자동 백업 가능
>- 운영 오버헤드 적음

>[!.questions]- 문항별 풀이
> A : 특정 시점 복구(PITR)는 실수로 데이터를 지웠을 때 복구하기 위한 용도로 최대 35일까지만 데이터를 보관
> C & D : 가능은 하지만, 코드 구현이나 관리해야 되는 서비스가 아닌, 이미 만들어져 있는 서비스를 사용하여 운영 효율성을 갖춰야함

## 문제 9

한 회사가 데이터 스토리지로 Amazon DynamoDB 테이블을 사용할 계획입니다. 회사는 비용 최적화에 관심이 있습니다. 이 테이블은 대부분의 아침에는 사용되지 않습니다. 저녁에는 읽기 및 쓰기 트래픽을 예측할 수 없는 경우가 많습니다. 트래픽 급증이 발생하면 매우 빠르게 발생합니다. 솔루션 아키텍트는 무엇을 권장해야 합니까?

- **A.** 온디맨드 용량 모드(on-demand capacity mode)로 DynamoDB 테이블을 생성합니다.
    
- **B.** 글로벌 보조 인덱스(global secondary index)를 사용하여 DynamoDB 테이블을 생성합니다.
    
- **C.** 프로비저닝된 용량(provisioned capacity) 및 Auto Scaling을 사용하여 DynamoDB 테이블을 생성합니다.
    
- **D.** 프로비저닝된 용량 모드(provisioned capacity mode)로 DynamoDB 테이블을 생성하고 글로벌 테이블로 구성합니다.

>[!.questions]- 정답
>정답 : A
>온디맨드 vs 프로비저닝
>
>온디맨드 용량 모드
>비용 최적화 : 오전에는 사용되지 않으므로 비용이 0원
>예측 불가능한 트래픽 : 사전에 용량을 예약할 필요 없이 들어오는 요청을 즉시 처리
>급격한 트래픽 급증 수용 가능
>

## 문제 10

한 회사가 최근 애플리케이션 마이그레이션 이니셔티브에 대한 지원을 받기 위해 AWS Managed Service Provider(MSP) 파트너와 계약을 맺었습니다. 솔루션 아키텍트는 기존 AWS 계정의 Amazon Machine Image(AMI)를 MSP 파트너의 AWS 계정과 공유해야 합니다. AMI는 Amazon Elastic Block Store(Amazon EBS) 기반이며 AWS Key Management Service(AWS KMS) 고객 관리형 키를 사용하여 EBS 볼륨 스냅샷을 암호화합니다. 솔루션 아키텍트가 MSP 파트너의 AWS 계정과 AMI를 가장 안전하게 공유하는 방법은 무엇입니까?

- **A.** 암호화된 AMI 및 스냅샷을 퍼블릭으로 사용할 수 있게 만듭니다. MSP 파트너의 AWS 계정이 키를 사용할 수 있도록 키 정책을 수정합니다.
    
- **B.** AMI의 launchPermission 속성을 수정합니다. MSP 파트너의 AWS 계정과만 AMI를 공유합니다. MSP 파트너의 AWS 계정이 키를 사용할 수 있도록 키 정책을 수정합니다.
    
- **C.** AMI의 launchPermission 속성을 수정합니다. MSP 파트너의 AWS 계정과만 AMI를 공유합니다. 암호화를 위해 MSP 파트너가 소유한 새 KMS 키를 신뢰하도록 키 정책을 수정합니다.
    
- **D.** 소스 계정에서 MSP 파트너의 AWS 계정에 있는 Amazon S3 버킷으로 AMI를 내보냅니다. MSP 파트너가 소유한 새 KMS 키로 S3 버킷을 암호화합니다. MSP 파트너의 AWS 계정에서 AMI를 복사하고 시작합니다.

마이그레이션 이니셔티브 : 우리 회사 서버들을 AWS로 옮기자
MSP(Managed Service Provider) : 인프라 운영 전문 업체 (ex) 삼성SDS, LGCNS 등
