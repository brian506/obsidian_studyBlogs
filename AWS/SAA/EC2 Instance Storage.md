### 1. EBS (Elastic Block Store)

- **네트워크 드라이브** — 물리 드라이브가 아님. 약간의 지연 발생 가능
- **특정 AZ에 바인딩** — us-east-1a의 EBS는 us-east-1b에 연결 불가
- 다른 AZ로 이동하려면 **반드시 스냅샷 → 복원** 과정 필요
- 하나의 인스턴스에 EBS 여러 개 연결 가능. 반대(하나의 EBS → 여러 인스턴스)는 기본적으로 불가 (Multi-Attach 예외)
- **Delete on Termination**: 루트 볼륨은 기본적으로 인스턴스 종료 시 삭제됨. 추가 볼륨은 유지됨
 
**Lazy Loading** : EBS 스냅샷으로 새로운 불륨을 만들면, 데이터가 즉시 복사되는게 아니라 S3에 머물러 있음. 사용자가 처음 데이터에 접근할 때 S3에서 EBS로 데이터를 가져오는 상황
- FSR 사용해서 지연 최소화

---

### 2. EBS Snapshots

- 스냅샷 생성 시 볼륨 분리 권장 (필수는 아님)
- **AZ / 리전 간 복사 가능** → 이걸 이용해서 다른 AZ로 데이터 이동

|기능|핵심 내용|
|---|---|
|**Snapshot Archive**|최대 75% 저렴. 복원에 24~72시간 소요|
|**Recycle Bin**|삭제된 스냅샷 보존 (1일~1년 설정 가능)|
|**Fast Snapshot Restore (FSR)**|첫 사용 지연 없애는 기능, 비용 비쌈 ($$$)|

---

### 3. AMI (Amazon Machine Image)

- 특정 **리전**에 귀속 (리전 간 복사는 가능)
- AMI 빌드 시 EBS 스냅샷이 함께 생성됨
- 종류: Public AMI (AWS 제공) / 직접 만든 AMI / Marketplace AMI

> **AMI 생성 프로세스**: EC2 시작 → 설정 → 인스턴스 중지 → AMI 빌드 → 새 인스턴스 시작

---

### 4. EC2 Instance Store ⭐

EBS와 비교 문제로 자주 출제

- **물리적으로 연결된 디스크** — 네트워크 아님 → 최고 I/O 성능
- **임시 스토리지** — 인스턴스 중지/종료 시 데이터 삭제됨
- 버퍼, 캐시, 임시 데이터 용도에 적합
- 백업/복제는 **사용자 책임**
- 하드웨어 장애 시 데이터 손실 위험

---

### 5. EBS Volume Types ⭐⭐ (가장 중요!)

**부트 볼륨으로 쓸 수 있는 건 gp2/gp3, io1/io2만 가능!**

|타입|종류|특징|용도|
|---|---|---|---|
|**gp3**|SSD|3,000 IOPS / 125 MiB/s 기본. IOPS·처리량 **독립** 조정 가능. 최대 16,000 IOPS|범용|
|**gp2**|SSD|크기와 IOPS **연동**. 최대 16,000 IOPS (5,334 GB에서 도달)|범용|
|**io1/io2**|SSD|Nitro 기준 최대 64,000 IOPS. 크기와 독립적으로 IOPS 설정|DB, 고성능|
|**io2 Block Express**|SSD|최대 256,000 IOPS, 밀리초 미만 지연|최고 성능|
|**st1**|HDD|최대 500 MiB/s, 500 IOPS|빅데이터, 로그|
|**sc1**|HDD|최대 250 MiB/s, 250 IOPS|콜드 아카이브|

> **gp2 vs gp3 차이 자주 나옴**: gp2는 크기↑ = IOPS↑ (연동), gp3는 독립적으로 설정 가능

### 6. EBS Multi-Attach (io1/io2)

- **동일 AZ** 내 최대 **16개** 인스턴스에 동시 연결
- 각 인스턴스가 읽기/쓰기 권한 모두 가짐
- **클러스터 인식 파일 시스템** 필요 (XFS, EXT4 불가)
- 용도: 고가용성 클러스터링 애플리케이션

---

### 7. EBS Encryption

- KMS로 AES-256 암호화
- 암호화 시 **저장 데이터 + 전송 중 데이터 + 스냅샷 + 스냅샷에서 만든 볼륨** 모두 암호화
- 성능 영향 거의 없음

**미암호화 볼륨 → 암호화로 전환하는 방법:** 스냅샷 생성 → 스냅샷 복사 시 암호화 옵션 선택 → 새 볼륨 생성 → 인스턴스에 연결

---

### 8. EFS (Elastic File System) ⭐

- **여러 AZ의 여러 EC2 인스턴스**에 동시 마운트 가능 (EBS와 가장 큰 차이!)
- **Linux 전용** (Windows 불가, POSIX)
- 사용량만큼 비용 청구 — 용량 사전 프로비저닝 불필요
- 가격은 gp2의 약 3배

**성능 모드:**

- 범용 (기본): 짧은 지연, 웹서버/CMS
- Max I/O: 높은 처리량, 빅데이터/미디어

**처리량 모드:**

- Bursting: 스토리지 크기에 비례
- Provisioned: 크기와 무관하게 처리량 고정
- Elastic: 워크로드에 따라 자동 조정 (예측 불가 워크로드에 적합)

**스토리지 클래스:**

- Standard: 자주 접근하는 파일
- EFS-IA (Infrequent Access): 저장 비용 낮지만 조회 비용 발생
- One Zone: 단일 AZ, 개발용, 비용 90% 절감

### 최종 비교 — 시험 단골 ⭐⭐⭐


|     | EBS         | EFS       | Instance Store |
| --- | ----------- | --------- | -------------- |
| 연결  | 1개 인스턴스     | 수백 개 인스턴스 | 해당 인스턴스만       |
| AZ  | 단일 AZ 한정    | 멀티 AZ     | 해당 서버에 물리 연결   |
| 영속성 | 영구          | 영구        | 임시(종료 시 삭제)    |
| 성능  | 좋음          | 좋음        | 최고             |
| OS  | 무관          | Linux만    | 무관             |
| 비용  | 프로비저닝 크기 기준 | 사용량 기준    | 무료             |


---

> **시험 TIP**: "여러 AZ에 걸쳐 여러 EC2가 공유해야 한다" → EFS, "고성능 임시 스토리지" → Instance Store, "단일 인스턴스, 영구 저장" → EBS 로 구분하면 됩니다.
