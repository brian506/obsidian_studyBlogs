### 1. Lifecycle Rules (수명 주기 규칙) ⭐⭐

스토리지 클래스 간 객체 이동을 자동화하는 규칙.

**두 가지 액션 유형:**

**Transition Actions (전환)**: 다른 스토리지 클래스로 자동 이동

- 예: 생성 후 60일 → Standard-IA, 6개월 후 → Glacier

**Expiration Actions (만료/삭제)**: 일정 시간 후 자동 삭제

- 예: 365일 후 액세스 로그 삭제
- 버전 관리 활성화 시 이전 버전(noncurrent) 삭제
- **완료되지 않은 멀티파트 업로드 자동 정리**

규칙은 특정 **접두사(prefix)** 또는 **객체 태그** 단위로 적용 가능.

---

**시나리오 문제 — 시험에 그대로 나오는 패턴** ⭐

**시나리오 1** — 원본/썸네일 이미지 관리

- 소스 이미지: **Standard** → 60일 후 → **Glacier** (6시간 대기 허용)
- 썸네일: **One-Zone IA** → 60일 후 **삭제** (재생성 가능하므로)

**시나리오 2** — 삭제 후 30일 즉시 복구 + 365일간 48시간 내 복구

- **버저닝 활성화** → 삭제 = 삭제 마커 추가, 이전 버전 보존
- 이전 버전(noncurrent) → **Standard-IA** 이동
- 그 후 → **Glacier Deep Archive** 이동 (48시간 복원 가능)

---

**S3 Analytics (스토리지 클래스 분석)**

- **Standard ↔ Standard-IA 전환** 권장 사항 제공
- One-Zone IA, Glacier에는 적용 안 됨
- 분석 시작까지 **24~48시간** 소요, 보고서는 매일 업데이트
- 수명 주기 규칙 설계 전 첫 번째 참고 도구로 활용

---

### 2. S3 Requester Pays ⭐

기본적으로 버킷 소유자가 스토리지 + 데이터 전송 비용 전부 부담. **Requester Pays** 버킷으로 설정하면 데이터를 다운로드하는 **요청자가 전송 비용 부담**. 대규모 데이터셋을 다른 계정과 공유할 때 유용. 요청자는 **반드시 AWS 인증된 상태**여야 함 (익명 불가).

---

### 3. S3 Event Notifications ⭐⭐

S3 이벤트 발생 시 자동으로 다른 서비스에 알림. 이벤트 유형: `ObjectCreated`, `ObjectRemoved`, `ObjectRestore`, `Replication` 등. 객체 이름 필터링도 가능 (예: `.jpg`만 필터링).

전달 대상: **SNS, SQS, Lambda**. 이벤트는 보통 수 초 내 전달되지만 최대 1분 이상 걸릴 수도 있음.

**Amazon EventBridge와 연동 시 추가 기능:**

- 메타데이터, 객체 크기, 이름 등 **고급 JSON 필터링**
- **여러 목적지 동시 전송**: Step Functions, Kinesis Streams/Firehose 등
- 이벤트 아카이브, 이벤트 재생, 안정적 전송 기능

> **시험 TIP**: 다양한 서비스로 동시에 이벤트를 보내야 하거나 복잡한 필터가 필요하면 → **EventBridge** 사용

---

### 4. S3 성능 최적화 ⭐⭐

**Baseline Performance (기본 성능):**

- S3는 자동으로 확장, 지연 시간 **100~200ms**
- **접두사(prefix)당** 초당 3,500 PUT/COPY/POST/DELETE 또는 5,500 GET/HEAD
- 접두사 수에 제한 없음 → 접두사를 여러 개로 분산하면 성능 비례해서 향상
    - 4개 접두사 균등 분산 시 GET 요청 초당 22,000개 처리 가능

**업로드 성능 향상:**

|방법|설명|
|---|---|
|**Multi-Part Upload**|100MB 이상 권장, **5GB 이상 필수**. 병렬 업로드로 대역폭 최대화. 완료 시 자동 조합|
|**Transfer Acceleration**|가까운 **엣지 로케이션**으로 먼저 전송 → AWS 내부망으로 S3 버킷 전달. 멀티파트와 호환|

**다운로드 성능 향상:**

**Byte-Range Fetches**: GET 요청을 바이트 단위로 나눠서 **병렬** 다운로드. 실패 시 작은 범위로 재시도 → 복원력 높음. 파일 헤더 부분만 가져오는 용도로도 활용.

---

### 5. S3 Select & Glacier Select ⭐

S3 객체 전체를 다운받지 않고 **서버 사이드에서 SQL로 필터링**한 결과만 가져오는 기능. 행/열 기준 간단한 필터링 가능. 데이터 전송량 감소 + 클라이언트 CPU 비용 절감.

> 복잡한 쿼리가 필요하면 Athena 사용, **간단한 필터링**은 S3 Select / Glacier Select 추천

---

### 6. S3 Batch Operations ⭐

기존 S3 객체에 **단일 요청으로 대량 작업** 수행. 직접 스크립트 작성 없이 재시도 관리, 진행 상황 추적, 완료 알림, 보고서 생성을 자동으로 처리해줌.

할 수 있는 작업들: 객체 메타데이터/ACL/태그 수정, 버킷 간 복사, **암호화되지 않은 객체 일괄 암호화**, Glacier에서 객체 복원, Lambda 함수 호출.

**실전 활용 패턴 (시험에 나옴):** `S3 Inventory`로 암호화 안 된 객체 목록 생성 → `S3 Select`로 필터링 → `S3 Batch Operations`으로 일괄 암호화

---

### 시험 TIP 최종 정리

- **수명 주기 + 시나리오 설계** 문제는 반드시 나옴 — 접두사로 분리, 클래스 전환 시점, 삭제 타이밍 조합 기억
- 멀티파트 업로드: **100MB 이상 권장, 5GB 이상 필수**
- 접두사 늘리면 성능 올라감 (4개 prefix × 5,500 = 22,000 GET/s)
- S3 이벤트 → 여러 서비스 동시 라우팅 필요 시 → **EventBridge**
- 대용량 객체 일부만 필요할 때 → **Byte-Range Fetches**
- 객체 안에서 일부 데이터만 쿼리할 때 → **S3 Select**
- 기존 객체 일괄 작업 → **S3 Batch Operations**
