# 총정리 보충 (고빈출 누락 영역)

> 기존 「총 정리.md」에 거의 안 다뤄졌지만 SAA 시험에 자주 나오는 영역만 모았음. 상황 → 해결 형식.

## Route 53 라우팅 정책 (★시험 단골)
- 단순히 도메인 하나를 IP/리소스로 연결 : **Simple 라우팅**
- 트래픽을 비율(%)로 나눠 보내기, A/B 테스트·점진적 배포 : **Weighted(가중치) 라우팅**
- 사용자에게 가장 **지연 시간이 낮은 리전**으로 연결 : **Latency 기반 라우팅**
- 주 리소스 장애 시 자동으로 예비(DR) 리소스로 전환 : **Failover 라우팅** + 헬스 체크
- 사용자의 **물리적 위치(국가/대륙)** 기준으로 다른 응답 : **Geolocation 라우팅** (저작권·언어 분리)
- 리소스 위치 기준 + **바이어스(편향)로 트래픽 비율 조정** : **Geoproximity 라우팅**
- 여러 IP를 반환하고 클라이언트가 고르게 함(헬스체크 기반 간이 LB) : **Multivalue Answer 라우팅**
- **Alias vs CNAME** : Alias는 **Zone Apex(루트 도메인, example.com)** 에서 사용 가능 + 무료, AWS 리소스(ELB/CloudFront/S3) 가리킬 때. CNAME은 루트 불가, 서브도메인만.
- TTL 짧게 = 변경 빠르게 반영(but 쿼리 비용↑) / TTL 길게 = 비용↓ 반영 느림

## ELB 종류 비교 (★)
- **HTTP/HTTPS, 경로·호스트 기반 라우팅**, 마이크로서비스/컨테이너 : **ALB (L7)**
- **초고성능, 초저지연, 고정 IP(Elastic IP), 수백만 RPS, TCP/UDP** : **NLB (L4)**
- 타사 가상 방화벽 어플라이언스로 트래픽 검사 : **GWLB** (GENEVE, 3계층)
- 동일 사용자를 같은 인스턴스로 계속 보내기(세션 유지) : **스티키 세션(Sticky Session)**
- 인스턴스 제거 시 진행 중 요청 끊기지 않게 : **연결 드레이닝 / Deregistration Delay**
- AZ 간 트래픽 균등 분배 : **Cross-Zone Load Balancing** (ALB 기본 ON, NLB 기본 OFF)
- 하나의 LB에서 여러 도메인 인증서 처리 : **SNI**
- ALB는 고정 IP 없음 → **고정 IP 필요하면 NLB**

## Auto Scaling (ASG) 정책 (★)
- 특정 지표(CPU 50% 등)를 목표치로 유지 : **Target Tracking(목표 추적)** ← 가장 무난한 정답
- 지표 구간별로 단계적으로 증감 : **Step Scaling**
- 정해진 시간/요일에 미리 확장(예: 매일 9시 트래픽 증가) : **Scheduled Scaling**
- 과거 패턴을 ML로 학습해 미리 확장 : **Predictive Scaling**
- 인스턴스 시작/종료 직전에 작업(로그 수집 등) 끼워넣기 : **Lifecycle Hook**
- 최소·희망·최대 용량 설정 / 헬스체크로 비정상 인스턴스 자동 교체
- **Launch Template** 사용 (구버전 Launch Configuration 아님)

## VPC 심화 (★)
- **NACL vs 보안그룹** : NACL은 **서브넷** 레벨, **Stateless**(인/아웃 둘 다 규칙 필요), 허용+거부 가능. 보안그룹은 **인스턴스** 레벨, **Stateful**(응답 자동 허용), 허용만 가능.
- 프라이빗 서브넷의 인스턴스가 **아웃바운드 인터넷**(패치 등) 필요 : **NAT Gateway** (관리형, 고가용성, AZ별 배치) / NAT Instance는 구식·직접관리
- VPC를 인터넷에 연결 : **Internet Gateway(IGW)**
- IPv6 프라이빗 아웃바운드 전용 : **Egress-Only Internet Gateway**
- **수십~수백 개 VPC + 온프레미스를 허브-스포크로 한 번에 연결** : **Transit Gateway(TGW)** ← VPC 피어링이 너무 많아질 때 정답
- 네트워크 트래픽 로그 캡처/분석(거부된 트래픽 디버깅) : **VPC Flow Logs**
- S3·DynamoDB 내부 접속(무료) : **Gateway 엔드포인트** / 그 외 서비스 사설 접속 : **Interface 엔드포인트(PrivateLink)**

## Direct Connect & VPN 복원력 (★)
- 일관된 저지연·고대역폭 전용선 (초기 구축 시간 김, 1개월+) : **Direct Connect(DX)**
- **DX 구축 전 즉시** 연결 필요 or DX 백업 : **Site-to-Site VPN**
- 최고 복원력 : **DX 2개(서로 다른 로케이션) + VPN 백업**
- 여러 리전/여러 VPC를 DX 하나로 : **Direct Connect Gateway**
- DX는 기본 **암호화 안 됨** → 암호화 필요시 위에 VPN 얹기

## 재해 복구(DR) 전략 — RTO/RPO (★시험 단골)
RTO=복구까지 걸리는 시간, RPO=허용 데이터 손실 시간. 비용 낮은 순:
- **Backup & Restore** : 가장 저렴, RTO/RPO 길다(수 시간). 백업만 보관.
- **Pilot Light** : 핵심(DB)만 항상 켜두고 나머지는 꺼둠. 중간 비용/속도.
- **Warm Standby** : 축소된 전체 시스템 상시 가동 → 장애 시 스케일업. 빠름.
- **Multi-Site / Hot Standby (Active-Active)** : 양쪽 풀가동, RTO/RPO ≈ 0, 가장 비쌈.
- "비용 최소 + 손실 어느정도 허용" → Backup & Restore / "거의 무중단" → Multi-Site

## ElastiCache: Redis vs Memcached (★)
- **고가용성·Multi-AZ·자동 장애조치·복제·영속성·정렬셋 등 자료구조** 필요 : **Redis**
- **단순 캐시, 멀티스레드로 수평 확장, 데이터 보존 불필요** : **Memcached**
- DB 읽기 부하 줄이기(쿼리 캐싱) : **Lazy Loading / Write-Through**
- 세션 상태를 여러 서버가 공유(상태 비저장 앱) : **ElastiCache에 세션 저장**

## S3 추가 포인트 (★)
- 다른 리전으로 복제(지연·DR·규정) : **CRR(교차 리전 복제)** / 같은 리전 : **SRR**
  - 복제 전제조건 : **양쪽 버킷 버전관리 ON**
- 암호화 종류 : **SSE-S3**(기본, AWS 관리키) / **SSE-KMS**(키 감사·접근제어 필요) / **SSE-C**(고객이 키 제공) / **클라이언트 측 암호화**
- 임시로 특정 사용자에게만 업/다운로드 허용 : **Pre-signed URL**
- 대용량 객체 일부만 조회 : **S3 Select**
- 업로드/삭제 시 Lambda·SNS·SQS 트리거 : **S3 이벤트 알림**
- 정적 웹사이트 호스팅 가능 / 교차 출처 요청 : **CORS 설정**

## EBS 핵심 (★)
- **gp3/gp2** 범용, **io1/io2** 고성능·고IOPS(프로비저닝), **st1** 처리량(빅데이터·로그), **sc1** 콜드·최저가
- 여러 인스턴스에 한 볼륨 동시 연결 : **io1/io2 Multi-Attach** (같은 AZ만)
- 백업 : **스냅샷**(S3에 증분 저장, 리전 간 복사 가능)
- 스냅샷 자동화 스케줄 : **Data Lifecycle Manager(DLM)**
- 스냅샷에서 복원 시 첫 접근 지연 제거 : **Fast Snapshot Restore(FSR)**
- EBS는 **같은 AZ 인스턴스에만** 연결 / 다른 AZ는 스냅샷으로 이동
- **EBS vs 인스턴스 스토어** : 인스턴스 스토어는 휘발성(중지 시 소멸)·초고속, EBS는 영속적

## IAM / STS 핵심 (★)
- EC2·Lambda가 다른 서비스 접근 : **IAM Role**(키 하드코딩 금지)
- 다른 계정 리소스 임시 접근 / 역할 전환 : **STS AssumeRole**
- 모바일·웹 앱 사용자 임시 자격증명 : **Cognito(Identity Pool)** 또는 **Web Identity Federation**
- 기업 AD 사용자가 AWS 로그인(SSO) : **IAM Identity Center(구 AWS SSO) + SAML 2.0**
- 권한 상한선 설정 : **Permission Boundary** / 조직 전체 강제 : **SCP**
- 정책 충돌 시 : **명시적 Deny가 항상 우선**

## CloudFront 추가 (★)
- 특정 사용자에게만 콘텐츠 접근(유료 영상 등) : **Signed URL / Signed Cookie**
- S3 오리진을 CloudFront로만 접근 허용 : **OAC**(권장) / OAI(구식)
- 특정 국가 차단/허용 : **지리적 제한(Geo Restriction)**
- 엣지에서 코드 실행(요청 변형) : **Lambda@Edge / CloudFront Functions**

## 컨테이너 선택 (★)
- 서버(EC2) 관리 없이 컨테이너 실행 : **Fargate**
- AWS 자체 오케스트레이션, 간단 : **ECS**
- **쿠버네티스(Kubernetes)** 표준·멀티클라우드 호환 필요 : **EKS**
- 컨테이너 이미지 저장소 : **ECR**

## 자주 헷갈리는 한 줄 정리
- **SQS(당겨가기/버퍼) vs SNS(밀어주기/펍섭) vs EventBridge(이벤트 라우팅·스케줄·SaaS 연동)**
- **CloudWatch(지표·로그·모니터링) vs CloudTrail(API 호출 감사) vs Config(리소스 설정 규정준수)**
- **Inspector(취약점 스캔) vs GuardDuty(위협 탐지) vs Macie(S3 PII) vs Detective(원인 분석)**
- **Cognito(앱 사용자 인증) vs IAM(AWS 리소스 접근) vs IAM Identity Center(직원 SSO)**
- **Kinesis(실시간 스트리밍) vs SQS(작업 큐, 재처리·순서)**
