## 문제 3

한 회사가 MySQL 데이터베이스를 기반으로 하는 온프레미스 애플리케이션을 실행하고 있습니다. 회사는 애플리케이션의 탄력성과 가용성을 높이기 위해 애플리케이션을 AWS로 마이그레이션하고 있습니다. 현재 아키텍처는 정상 운영 기간 동안 데이터베이스에 과도한 읽기 작업이 발생함을 보여줍니다. 4시간마다 회사의 개발 팀은 프로덕션 데이터베이스의 전체 익스포트(export)를 가져와 스테이징 환경의 데이터베이스를 채웁니다. 이 기간 동안 사용자는 허용할 수 없는 애플리케이션 지연 시간을 겪습니다. 개발 팀은 해당 절차가 완료될 때까지 스테이징 환경을 사용할 수 없습니다. 솔루션 아키텍트는 애플리케이션 지연 시간 문제를 완화할 수 있는 대체 아키텍처를 권장해야 합니다. 또한 이 대체 아키텍처는 개발 팀이 지연 없이 스테이징 환경을 계속 사용할 수 있는 기능을 제공해야 합니다. 이러한 요구 사항을 충족하는 솔루션은 무엇입니까?

- **A.** 프로덕션용으로 다중 AZ(Multi-AZ) Aurora 복제본이 있는 Amazon Aurora MySQL을 사용합니다 . mysqldump 유틸리티를 사용하는 백업 및 복원 프로세스를 구현하여 스테이징 데이터베이스를 채웁니다.
    
- **B.** 프로덕션용으로 다중 AZ(Multi-AZ) Aurora 복제본이 있는 Amazon Aurora MySQL을 사용합니다. 데이터베이스 복제(database cloning)를 사용하여 필요할 때 스테이징 데이터베이스를 생성합니다.
    
- **C.** 프로덕션용으로 다중 AZ(Multi-AZ) 배포 및 읽기 전용 복제본이 있는 Amazon RDS for MySQL을 사용합니다. 스테이징 데이터베이스로 스탠바이 인스턴스를 사용합니다.
    
- **D.** 프로덕션용으로 다중 AZ(Multi-AZ) 배포 및 읽기 전용 복제본이 있는 Amazon RDS for MySQL을 사용합니다 . mysqldump 유틸리티를 사용하는 백업 및 복원 프로세스를 구현하여 스테이징 데이터베이스를 채웁니다.

**스테이징** : 프로덕션과 최대한 똑같은 환경에서 마지막으로 테스트해보는 곳

>[!.questions]- 정답
>정답 : B
> 기존 방식은 데이터를 다 뽑아서 새 DB에 넣게 되어 시간이 오래걸리고, 운영 DB에 부하 발생
> Amazon Aurora 의 데이터 복제
> 즉각성 : TB이상도 몇분만의 복제 끝. 
> 	- 실제 데이터를 복사하는 게 아니라, 데이터가 저장된 스토리지 층에서 포인터만 새로 만들기 때문
> 부하 제로 : 실제 DB의 데이터를 추출하는 방식이 아니므로, 운영 DB에 지연 없음
> 비용 효율성 : 복제본 생성 이후에는 추가 비용 거의 없음.

> C : 스탠바이 인스턴스는 장애 시 교체되기 위해 대기하는 백업용임. 스테이징 환경에서 사용 불가

## 문제 98

한 이미지 처리 회사가 사용자들이 이미지를 업로드하는 데 사용하는 웹 애플리케이션을 가지고 있습니다. 애플리케이션은 이미지를 Amazon S3 버킷에 업로드합니다. 회사는 Amazon Simple Queue Service(Amazon SQS) 표준 대기열에 객체 생성 이벤트를 게시하도록 S3 이벤트 알림을 설정했습니다. SQS 대기열은 이미지를 처리하고 이메일을 통해 사용자에게 결과를 보내는 AWS Lambda 함수의 이벤트 소스 역할을 합니다. 사용자들은 업로드된 모든 이미지에 대해 여러 통의 이메일 메시지를 받고 있다고 보고합니다. 솔루션 아키텍트는 SQS 메시지가 Lambda 함수를 두 번 이상 호출하여 여러 이메일 메시지가 전송된다고 판단했습니다. 가장 적은 운영 오버헤드로 이 문제를 해결하려면 솔루션 아키텍트는 어떻게 해야 합니까?

- **A.** ReceiveMessage 대기 시간을 30초로 늘려 SQS 대기열에 긴 폴링(long polling)을 설정합니다.
    
- **B.** SQS 표준 대기열을 SQS FIFO 대기열로 변경합니다. 중복 메시지를 폐기하려면 메시지 중복 제거 ID를 사용합니다.
    
- **C.** SQS 대기열의 가시성 제한 시간(visibility timeout)을 함수 제한 시간과 배치 창 제한 시간의 합보다 큰 값으로 늘립니다.
    
- **D.** 메시지를 읽은 후 처리하기 전에 SQS 대기열에서 메시지를 즉시 삭제하도록 Lambda 함수를 수정합니다.

>[!.questions]- 정답
>정답 : C
>
>동작 순서
>1. Lambda 함수가 SQS에서 메시지를 읽어오면, 그 메시지는 다른 소비자가 볼 수 없는 '가시성 제한 시간'상태로 들어감
>2. Lambda 함수가 완료되면, SQS에서 해당 메시지 자동 삭제
>
>SQS 가시성 제한 시간
>문제 발생 지점 : 만약 Lambda 함수의 처리 시간이 SQS의 가시성 제한 시간보다 길면, Lambda가 아직 일을 하고 있는데도, SQS는 해당 메시지들 다시 대기열에 노출 시킴
> -> 이렇게 되면 다른 Lambda 가 일을 가져가서 처리하면 중복 처리가 발생
> 해결 방법 : 가시성 제한 시간을 함수의 제한 시간 + 배치 창보다 넉넉하게 설정해야 함. 

>[!.questions]- 문항별 풀이
>B : FIFO 대기열은 정확히 한 번 처리, 순서 보장을 제공하므로 해결은 되지만, 다 새로 만들어야 해서 운영 오버헤드가 큼

## 문제 100

회사의 컨테이너화된 애플리케이션이 Amazon EC2 인스턴스에서 실행되고 있습니다. 애플리케이션은 다른 비즈니스 애플리케이션과 통신하기 전에 보안 인증서를 다운로드해야 합니다. 회사는 거의 실시간으로 인증서를 암호화 및 복호화할 수 있는 매우 안전한 솔루션을 원합니다. 또한 이 솔루션은 데이터가 암호화된 후 고가용성 스토리지에 데이터를 저장해야 합니다. 운영 오버헤드가 가장 적은 상태로 이러한 요구 사항을 충족하는 솔루션은 무엇입니까?

- **A.** 암호화된 인증서를 위해 AWS Secrets Manager 보안 암호를 생성합니다. 필요에 따라 인증서를 수동으로 업데이트합니다. 세분화된 IAM 액세스를 사용하여 데이터에 대한 액세스를 제어합니다.
    
- **B.** Python 암호화 라이브러리를 사용하여 암호화 작업을 수신하고 수행하는 AWS Lambda 함수를 생성합니다. 해당 함수를 Amazon S3 버킷에 저장합니다.
    
- **C.** AWS Key Management Service(AWS KMS) 고객 관리형 키를 생성합니다. EC2 역할이 암호화 작업에 KMS 키를 사용하도록 허용합니다. 암호화된 데이터를 Amazon S3에 저장합니다.
    
- **D.** AWS Key Management Service(AWS KMS) 고객 관리형 키를 생성합니다. EC2 역할이 암호화 작업에 KMS 키를 사용하도록 허용합니다. 암호화된 데이터를 Amazon Elastic Block Store(Amazon EBS) 볼륨에 저장합니다.

>[!.questions]- 문항별 풀이
> C : KMS + S3 는 가능하고 안전하지만, 운영 오버헤드 큼. S3 버킷 정책과 KMS 키 정책을 각각 관리해야 하며, 코드도 복잡해짐
> D : EBS는 특정 가용 영역에 묶여 있는 블록 스토리지. 리전 전체에 걸쳐 복제되는 S3나 SM보다 불리. 고가용성 아키텍처에 부적합
