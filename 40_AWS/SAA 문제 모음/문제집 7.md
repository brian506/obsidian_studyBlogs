## 문제 1

한 회사가 새로운 퍼블릭 웹 애플리케이션을 AWS에 배포하고 있습니다. 애플리케이션은 Application Load Balancer(ALB) 뒤에서 실행됩니다. 애플리케이션은 외부 인증 기관(CA)에서 발급한 SSL/TLS 인증서를 사용하여 엣지(edge)에서 암호화되어야 합니다. 인증서는 만료되기 전에 매년 교체되어야 합니다. 이러한 요구 사항을 충족하기 위해 솔루션 아키텍트는 무엇을 해야 합니까?

- **A.** AWS Certificate Manager(ACM)를 사용하여 SSL/TLS 인증서를 발급합니다. 인증서를 ALB에 적용합니다. 관리형 갱신 기능을 사용하여 인증서를 자동으로 교체합니다.
    
- **B.** AWS Certificate Manager(ACM)를 사용하여 SSL/TLS 인증서를 발급합니다. 인증서에서 키 구성 요소를 가져옵니다. 인증서를 ALB에 적용합니다. 관리형 갱신 기능을 사용하여 인증서를 자동으로 교체합니다.
    
- **C.** AWS Certificate Manager(ACM) Private Certificate Authority를 사용하여 루트 CA에서 SSL/TLS 인증서를 발급합니다. 인증서를 ALB에 적용합니다. 관리형 갱신 기능을 사용하여 인증서를 자동으로 교체합니다.
    
- **D.** AWS Certificate Manager(ACM)를 사용하여 SSL/TLS 인증서를 가져옵니다(import). 인증서를 ALB에 적용합니다. 인증서 만료가 다가오면 Amazon EventBridge(Amazon CloudWatch Events)를 사용하여 알림을 보냅니다. 인증서를 수동으로 교체합니다.

>[!.questions]- 정답
>정답 : D
>ACM(AWS Certificate Manager) 
>- 직접 인증서를 발급할 수도 있지만, 외부에서 발급받은 인증서를 가져와서 적용할수도 있음
>- 인증서 자동 갱신은 ACM이 직접 발급한 인증서에 대해서만 동작
>- 외부 인증서는 자동 갱신이 안되기 때문에, 만료 전에 EventBridge로 알림을 받아야함
>
>정답 이유
>- 문제에서 외부에서 받은 인증서라고 했으므로, ACM에서 직접 발급받는  A,B,C 는 답이 아님

## 문제 3

한 회사가 AWS에서 인프라를 실행하고 있으며 문서 관리 애플리케이션에 등록된 사용자가 700,000명입니다. 이 회사는 대용량 pdf 파일을 .jpg 이미지 파일로 변환하는 제품을 만들 계획입니다. .pdf 파일의 평균 크기는 5MB입니다. 회사는 원본 파일과 변환된 파일을 모두 저장해야 합니다. 솔루션 아키텍트는 시간이 지남에 따라 빠르게 증가할 수요를 수용할 수 있는 확장 가능한 솔루션을 설계해야 합니다. 가장 비용 효율적으로 이러한 요구 사항을 충족하는 솔루션은 무엇입니까?

- **A.** .pdf 파일을 Amazon S3에 저장합니다. 파일을 .jpg 형식으로 변환하고 Amazon S3에 다시 저장하는 AWS Lambda 함수를 호출하도록 S3 PUT 이벤트를 구성합니다.
    
- **B.** .pdf 파일을 Amazon DynamoDB에 저장합니다. DynamoDB Streams 기능을 사용하여 파일을 .jpg 형식으로 변환하고 DynamoDB에 다시 저장하는 AWS Lambda 함수를 호출합니다.
    
- **C.** Amazon EC2 인스턴스, Amazon Elastic Block Store(Amazon EBS) 스토리지 및 Auto Scaling 그룹이 포함된 AWS Elastic Beanstalk 애플리케이션에 .pdf 파일을 업로드합니다. EC2 인스턴스의 프로그램을 사용하여 파일을 .jpg 형식으로 변환합니다. EBS 스토어에 pdf 파일과 .jpg 파일을 저장합니다.
    
- **D.** Amazon EC2 인스턴스, Amazon Elastic File System(Amazon EFS) 스토리지 및 Auto Scaling 그룹이 포함된 AWS Elastic Beanstalk 애플리케이션에 .pdf 파일을 업로드합니다. EC2 인스턴스의 프로그램을 사용하여 파일을 .jpg 형식으로 변환합니다. EBS 스토어에 .pdf 파일과 .jpg 파일을 저장합니다.

>[!.questions]- 정답
>정답 : A
>S3 + Lambda
>- 스토리지 최적화 : 객체 데이터를 저장하는데 가장 저렴하고 확장성
>- 서버리스 : 사용자가 언제 파일을 올릴 지 모르는 상황에서, 파일이 업로드될 때만 실행되는 Lambda
>- S3 PUT : 이건 이벤트 트리거로, 별도의 큐나 폴링 로직 없이도 파일 업로드 즉시 변환 프로세스 시작 가능

>[!.questions]- 문항별 풀이
>B : DynamoDB는 개별 아이템 크기 제한이 400KB. 
>C & D : 너무 많은 운영 비용

## 문제 5

- **B.** Amazon Textract를 사용하여 보고서에서 텍스트를 추출합니다. Amazon SageMaker를 사용하여 추출된 텍스트에서 PHI를 식별합니다.

- **D.** Amazon Rekognition을 사용하여 보고서에서 텍스트를 추출합니다. Amazon Comprehend Medical을 사용하여 추출된 텍스트에서 PHI를 식별합니다.

>정답 : C

>[!.questions]- 문항별 풀이

## 문제 6

한 회사에 각각 약 5MB 크기의 파일을 대량으로 생성하는 애플리케이션이 있습니다. 이 파일들은 Amazon S3에 저장됩니다. 회사 정책에 따라 파일은 삭제되기 전까지 4년 동안 저장되어야 합니다. 파일에는 재현하기 쉽지 않은 중요한 비즈니스 데이터가 포함되어 있으므로 항상 즉각적인 접근성이 필요합니다. 이 파일들은 객체 생성 후 처음 30일 동안 자주 액세스되지만 처음 30일 이후에는 거의 액세스되지 않습니다. 가장 비용 효율적인 스토리지 솔루션은 무엇입니까?

- **A.** 객체 생성 30일 후 파일을 S3 Standard에서 S3 Glacier로 이동하는 S3 버킷 수명 주기 정책을 생성합니다. 객체 생성 4년 후 파일을 삭제합니다.
    
- **B.** 객체 생성 30일 후 파일을 S3 Standard에서 S3 One Zone-Infrequent Access(S3 One Zone-IA)로 이동하는 S3 버킷 수명 주기 정책을 생성합니다. 객체 생성 4년 후 파일을 삭제합니다.
    
- **C.** 객체 생성 30일 후 파일을 S3 Standard에서 S3 Standard-Infrequent Access(S3 Standard-IA)로 이동하는 S3 버킷 수명 주기 정책을 생성합니다. 객체 생성 4년 후 파일을 삭제합니다.
    
- **D.** 객체 생성 30일 후 파일을 S3 Standard에서 S3 Standard-Infrequent Access(S3 Standard-IA)로 이동하는 S3 버킷 수명 주기 정책을 생성합니다. 객체 생성 4년 후 파일을 S3 Glacier로 이동합니다.

>[!.questions]- 정답
>정답 : C
> S3 Standard - IA(Infrequent Access)
> - 즉각적인 엑세스 
> - 최소 3개의 가용 영역에 데이터 분산 저장
> 0~30일 : 자주 엑세스하므로 S3 Standard
> 30일 이후 : 거의 엑세스하지 않지만, 즉시 필요하므로 S3 Standard IA
> 4년 후 : 삭제

## 문제 7

한 회사가 여러 Amazon EC2 인스턴스에서 애플리케이션을 호스팅합니다. 애플리케이션은 Amazon SQS 대기열의 메시지를 처리하고, Amazon RDS 테이블에 쓰고, 대기열에서 메시지를 삭제합니다. RDS 테이블에서 가끔 중복 레코드가 발견됩니다. SQS 대기열에는 중복 메시지가 없습니다. 메시지가 한 번만 처리되도록 보장하기 위해 솔루션 아키텍트는 무엇을 해야 합니까?

- **A.** CreateQueue API 호출을 사용하여 새 대기열을 생성합니다.
    
- **B.** AddPermission API 호출을 사용하여 적절한 권한을 추가합니다.
    
- **C.** ReceiveMessage API 호출을 사용하여 적절한 대기 시간(wait time)을 설정합니다.
    
- **D.** ChangeMessageVisibility API 호출을 사용하여 가시성 제한 시간(visibility timeout)을 늘립니다.

**SQS에서 가시성이란?**
- SQS에서 인스턴스가 메시지를 가지고 가게 되면, 그 해당 메시지를 투명하게 취급한다(다른 인스턴스가 가지고 가지 못하도록)
- 이때 투명하게 취급되는 시간을 가시성 제한 시간이라고 한다.
**중복 레코드가 발생하는 원인**
- 애플리케이션의 처리 시간이 SQS에 설정된 가시성 제한 시간보다 길어질때
	- 인스턴스 처리 시간이 가시성 제한 시간보다 길어지면, SQS는 해당 메시지가 실패했다고 간주하여 대기열에 다시 노출시킨다.
- 이때, 다른 인스턴스가 그 메시지를 가지고 가서 중복 데이터가 쌓이게 되는것

>[!.questions]- 정답
>정답 : D
>가시성 제한 시간 연장
>- 해당 메시지에 대해 설정된 숨김 제한 시간을 늘림

## 문제 8

솔루션 아키텍트가 회사의 온프레미스 인프라를 AWS로 확장하기 위해 새로운 하이브리드 아키텍처를 설계하고 있습니다. 회사는 AWS 리전에 대해 일관되게 짧은 지연 시간(low latency)을 제공하는 고가용성 연결이 필요합니다. 회사는 비용을 최소화해야 하며, 주 연결(primary connection)이 실패할 경우 더 느린 트래픽을 수용할 의향이 있습니다. 이러한 요구 사항을 충족하기 위해 솔루션 아키텍트는 무엇을 해야 합니까?

A. 리전에 대한 AWS Direct Connect 연결을 프로비저닝합니다. 기본 Direct Connect 연결이 실패할 경우에 대비하여 백업으로 VPN 연결을 프로비저닝합니다.

B. 프라이빗 연결을 위해 리전에 VPN 터널 연결을 프로비저닝합니다. 기본 VPN 연결이 실패할 경우에 대비하여 프라이빗 연결 및 백업용으로 두 번째 VPN 터널을 프로비저닝합니다.

C. 리전에 대한 AWS Direct Connect 연결을 프로비저닝합니다. 기본 Direct Connect 연결이 실패할 경우에 대비하여 동일한 리전에 백업용으로 두 번째 Direct Connect 연결을 프로비저닝합니다.

D. 리전에 대한 AWS Direct Connect 연결을 프로비저닝합니다. AWS CLI의 Direct Connect 장애 조치(failover) 속성을 사용하여 기본 Direct Connect 연결이 실패할 경우 자동으로 백업 연결을 생성합니다.

>[!.questions]- 정답
>정답 : A
>문제 조건 : 저지연 + 비용 최적화 + 장애 조치
>Direct Connect 
>- AWS와 온프레미스를 잇는 전용선으로 매우 일관되고 짧은 지연 시간 보장
>Site to Site VPN 
>- 공용 인터넷을 사용하기 때문에 네트워크 상황에 따라 지연시간 들쑥날쑥

## 문제 9

한 회사가 Application Load Balancer 뒤에 있는 Amazon EC2 인스턴스에서 비즈니스에 필수적인 웹 애플리케이션을 실행하고 있습니다. EC2 인스턴스는 Auto Scaling 그룹에 속해 있습니다. 애플리케이션은 단일 가용 영역에 배포된 Amazon Aurora PostgreSQL 데이터베이스를 사용합니다. 회사는 가동 중지 시간과 데이터 손실을 최소화하면서 애플리케이션의 고가용성을 보장하고자 합니다. 최소한의 운영 노력으로 이러한 요구 사항을 충족하는 솔루션은 무엇입니까?

- **A.** EC2 인스턴스를 다른 AWS 리전에 배치합니다. Amazon Route 53 상태 확인을 사용하여 트래픽을 리디렉션합니다. Aurora PostgreSQL 리전 간 복제(Cross-Region Replication)를 사용합니다.
    
- **B.** 다중 가용 영역을 사용하도록 Auto Scaling 그룹을 구성합니다. 데이터베이스를 다중 AZ(Multi-AZ)로 구성합니다. 데이터베이스를 위한 Amazon RDS 프록시 인스턴스를 구성합니다.
    
- **C.** 단일 가용 영역을 사용하도록 Auto Scaling 그룹을 구성합니다. 데이터베이스의 스냅샷을 매시간 생성합니다. 장애 발생 시 스냅샷에서 데이터베이스를 복구합니다.
    
- **D.** 다중 AWS 리전을 사용하도록 Auto Scaling 그룹을 구성합니다. 애플리케이션의 데이터를 Amazon S3에 씁니다. S3 이벤트 알림을 사용하여 데이터를 데이터베이스에 쓰는 AWS Lambda 함수를 실행합니다.

>[!.questions]- 정답
>정답 : B
>정답 조건 : 가동 중지 시간 최소화 + 데이터 손실 최소화
>Amazon Aurora
>- 기본적으로 3개의 AZ에 데이터를 복제하여 저장
>- 여기에 다중 AZ(보조 인스턴스)를 구성하면, 주 인스턴스에 장애가 발생해도 다른 AZ의 복제본으로 자동으로 failover 되어 데이터 손실 없이 서비스 유지
>
>RDS 프록시
>DB가 Failover 발생할 때 DB 엔드포인트 찾는 과정이 시간이 걸리는데, 이 시간을 최대 66퍼까지 단축시켜줌

>[!.questions]- 문항별 풀이
>A : 리전 간 복제는 리전 전체에 마비가 생겼을 때 복구 솔루션이며, 설정이 복잡하고 비용이 많이 듬
>B : 스냅샷 복구는 시간이 많이 들고, 유실 데이터 존재 가능성


## 문제 10

한 회사의 HTTP 애플리케이션이 Network Load Balancer(NLB) 뒤에 있습니다. NLB의 대상 그룹(target group)은 웹 서비스를 실행하는 여러 EC2 인스턴스가 포함된 Amazon EC2 Auto Scaling 그룹을 사용하도록 구성되어 있습니다. 회사는 NLB가 애플리케이션의 HTTP 오류를 감지하지 못하고 있다는 것을 인지했습니다. 이러한 오류가 발생하면 웹 서비스를 실행하는 EC2 인스턴스를 수동으로 다시 시작해야 합니다. 회사는 사용자 지정 스크립트나 코드를 작성하지 않고도 애플리케이션의 가용성을 개선해야 합니다. 이러한 요구 사항을 충족하기 위해 솔루션 아키텍트는 무엇을 해야 합니까?

- **A.** 회사 애플리케이션의 URL을 제공하여 NLB에서 HTTP 상태 확인(health check)을 활성화합니다.
    
- **B.** EC2 인스턴스에 cron 작업을 추가하여 매분 로컬 애플리케이션의 로그를 확인합니다. HTTP 오류가 감지되면 애플리케이션이 다시 시작됩니다.
    
- **C.** NLB를 Application Load Balancer로 교체합니다. 회사 애플리케이션의 URL을 제공하여 HTTP 상태 확인을 활성화합니다. 비정상 인스턴스를 교체하도록 Auto Scaling 작업을 구성합니다.
    
- **D.** NLB의 UnhealthyHostCount 지표를 모니터링하는 Amazon CloudWatch 경보를 생성합니다. 경보가 ALARM 상태일 때 비정상 인스턴스를 교체하도록 Auto Scaling 작업을 구성합니다.

>[!.questions]- 정답
>정답 : C
>NLB : 4계층(TCP/UDP)
>- 애플리케이션 내부에서 발생하는 구체적인 오류는 감지하지 못함. 포트의 생존 여부만 판단 가능
>ALB : 7계층(HTTP/HTTPS)
>Auto Scaling Group
>- ASG의 상태 확인 유형을 ELB로 설정하면, 로드밸런서가 Unhealthy라고 판단한 인스턴스를 ASG가 자동으로 종료하고 새 인스턴스로 교체
