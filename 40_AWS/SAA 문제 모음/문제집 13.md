## 문제 131

한 회사가 Amazon S3 버킷을 스토리지로 사용할 파일 공유 애플리케이션을 개발하고 있습니다. 이 회사는 Amazon CloudFront 배포(distribution)를 통해 모든 파일을 제공하고자 합니다. 회사는 S3 URL로 직접 이동하여 파일에 액세스하는 것을 원하지 않습니다. 솔루션 아키텍트는 이러한 요구 사항을 충족하기 위해 무엇을 해야 합니까?

- **A.** CloudFront 액세스에 대해서만 읽기 권한을 부여하도록 각 S3 버킷에 대한 개별 정책을 작성합니다.
    
- **B.** IAM 사용자를 생성합니다. 해당 사용자에게 S3 버킷의 객체에 대한 읽기 권한을 부여합니다. 이 사용자를 CloudFront에 할당합니다.
    
- **C.** CloudFront 배포 ID를 보안 주체(Principal)로 지정하고 대상 S3 버킷을 Amazon 리소스 이름(ARN)으로 지정하는 S3 버킷 정책을 작성합니다.
    
- **D.** 원본 액세스 ID(OAI, Origin Access Identity)를 생성합니다. OAI를 CloudFront 배포에 할당합니다. OAI만 읽기 권한을 갖도록 S3 버킷 권한을 구성합니다.

>[!.questions]- 정답
>정답 : D
>정답 조건 : S3로의 직접적인 접근을 차단하고 오직 CloudFront로만 접근할 수 있게 해야됨
>원본 엑세스 ID (OAI, Origin Access Identity)
>1. CloudFront 전용 신분증(OAI) 생성
>2. S3 버킷 정책에 OAI을 가진 사람만 파일을 읽을 수 있도록 설정
>3. 그 외의 접근 모두 거부
>-> 이로써 사용자가 S3 URL로 직접 접속하면 '권한 없음'이 뜨고, CloudFront를 통해서 접속할 때만 CloudFront가 자신의 신분증(OAI)을 S3에 보여주며 파일을 가져다줌

>[!.questions]- 문항별 풀이
>A : 방향은 맞지만, 구체적인 식별자가 필요
>B : IAM 사용자는 사람이나 특정 애플리케이션에 쓰는 것. 서비스 사이의 통신을 위해 사용하지는 않음

## 문제 133 - RDS Custom

한 회사가 온프레미스에서 Oracle 데이터베이스를 실행하고 있습니다. AWS 마이그레이션의 일환으로 회사는 데이터베이스를 사용 가능한 최신 버전으로 업그레이드하고자 합니다. 회사는 또한 데이터베이스에 대한 재해 복구(DR)를 설정하려고 합니다. 회사는 정상 운영 및 DR 설정에 대한 운영 오버헤드를 최소화해야 합니다. 또한 회사는 데이터베이스의 기본 운영 체제에 대한 액세스를 유지해야 합니다. 어떤 솔루션이 이러한 요구 사항을 충족합니까?

- **A.** Oracle 데이터베이스를 Amazon EC2 인스턴스로 마이그레이션합니다. 다른 AWS 리전으로의 데이터베이스 복제를 설정합니다.
    
- **B.** Oracle 데이터베이스를 Amazon RDS for Oracle로 마이그레이션합니다. 교차 리전 자동 백업을 활성화하여 스냅샷을 다른 AWS 리전으로 복제합니다.
    
- **C.** Oracle 데이터베이스를 Amazon RDS Custom for Oracle로 마이그레이션합니다. 다른 AWS 리전에 데이터베이스에 대한 읽기 전용 복제본(read replica)을 생성합니다.
    
- **D.** Oracle 데이터베이스를 Amazon RDS for Oracle로 마이그레이션합니다. 다른 가용 영역에 대기(standby) 데이터베이스를 생성합니다.

>[!.questions]- 정답
>정답 : C 
> 정답 조건 : 재해 복구 + 운영 오버헤드 최소화 + OS에 대한 엑세스 유지
> 운영 오버헤드 최소화
> - 표준 RDS : AWS가 OS와 DB엔진을 모두 관리하므로, 사용자는 SSH를 통해 서버에 접속하거나 OS 설정 파일을 직접 건드릴 수 없음.
> - RDS Custom : 관리형 서비스이면서, SSH나 AWS System Manager 를 통해 OS에 직접 접속 가능
> OS에 직접 접속 가능한 솔루션을 구해야함!\

## 문제 134 - CRR + KMS (리전 각 복제)

한 회사가 애플리케이션을 서버리스 솔루션으로 이동하려고 합니다. 서버리스 솔루션은 SQL을 사용하여 기존 데이터와 새로운 데이터를 분석해야 합니다. 회사는 Amazon S3 버킷에 데이터를 저장합니다. 데이터는 암호화가 필요하며 다른 AWS 리전으로 복제되어야 합니다. 가장 적은 운영 오버헤드(LEAST operational overhead)로 이러한 요구 사항을 충족하는 솔루션은 무엇입니까?

- **A.** 새 S3 버킷을 생성합니다. 새 S3 버킷에 데이터를 로드합니다. S3 교차 리전 복제(CRR)를 사용하여 암호화된 객체를 다른 리전의 S3 버킷으로 복제합니다. AWS KMS 다중 리전 키(SSE-KMS)를 사용한 서버 측 암호화를 사용합니다. Amazon Athena를 사용하여 데이터를 쿼리합니다.
    
- **B.** 새 S3 버킷을 생성합니다. 새 S3 버킷에 데이터를 로드합니다. S3 교차 리전 복제(CRR)를 사용하여 암호화된 객체를 다른 리전의 S3 버킷으로 복제합니다. AWS KMS 다중 리전 키(SSE-KMS)를 사용한 서버 측 암호화를 사용합니다. Amazon RDS를 사용하여 데이터를 쿼리합니다.
    
- **C.** 기존 S3 버킷에 데이터를 로드합니다. S3 교차 리전 복제(CRR)를 사용하여 암호화된 객체를 다른 리전의 S3 버킷으로 복제합니다. Amazon S3 관리형 암호화 키(SSE-S3)를 사용한 서버 측 암호화를 사용합니다. Amazon Athena를 사용하여 데이터를 쿼리합니다.
    
- **D.** 기존 S3 버킷에 데이터를 로드합니다. S3 교차 리전 복제(CRR)를 사용하여 암호화된 객체를 다른 리전의 S3 버킷으로 복제합니다. Amazon S3 관리형 암호화 키(SSE-S3)를 사용한 서버 측 암호화를 사용합니다. Amazon RDS를 사용하여 데이터를 쿼리합니다.

>[!.questions]- 정답
>정답 : A
>복제 및 암호화 : S3 CRR + KMS
>CRR(Cross-Region Replication) : S3 버킷 간에 데이터를 다른 리전으로 자동 복제하는 표준 기능
>KMS 다중 리전 키 : 복제된 데이터를 다른 리전에서도 쉽게 복호화하고 관리할 수 있게 해줌. 동일한 키 ID를 여러 리전에서 공유할 수 있어, 리전마다 다른 키를 생성하고 권한을 관리해야 하는 번거로움을 줄임
>
>기존 데이터 복제 문제 : S3의 일반적인 복제 기능은 설정 이후에 업로드된 객체만 복제

## 문제 135

한 회사가 AWS에서 워크로드를 실행하고 있습니다. 회사는 외부 공급자의 서비스에 연결해야 합니다. 해당 서비스는 공급자의 VPC에 호스팅되어 있습니다. 회사의 보안 팀에 따르면 연결은 비공개(private)여야 하며 대상 서비스로만 제한되어야 합니다. 연결은 회사의 VPC에서만 시작되어야 합니다. 어떤 솔루션이 이러한 요구 사항을 충족합니까?

- **A.** 회사의 VPC와 공급자의 VPC 간에 VPC 피어링(peering) 연결을 생성합니다. 라우팅 테이블을 업데이트하여 대상 서비스에 연결합니다.
    
- **B.** 공급자에게 해당 VPC에 가상 프라이빗 게이트웨이(virtual private gateway)를 생성하도록 요청합니다. AWS PrivateLink를 사용하여 대상 서비스에 연결합니다.
    
- **C.** 회사의 VPC 내 퍼블릭 서브넷에 NAT 게이트웨이를 생성합니다. 라우팅 테이블을 업데이트하여 대상 서비스에 연결합니다.
    
- **D.** 공급자에게 대상 서비스에 대한 VPC 엔드포인트를 생성하도록 요청합니다. AWS PrivateLink를 사용하여 대상 서비스에 연결합니다.

>[!.questions]- 정답
>정답 : D
>정답 조건 : 비공개 연결 + 대상 서비스로만 제한 +단방향 연결
>AWS PrivateLink(가장 강력한 보안 연결)
>- 기능 : PrivateLink는 서로 다른 VPC 간에 통신할 때, 네트워크 전체를 연결하는 것이 아니라 '특정 서비스'만 집어서 연결
>- 트래픽이 공용 인터넷을 거치지 않고 AWS 내부 네트워크망을 통해서만 이동하므로 비공개 연결임
>대상 서비스로만 제한
>- 공급자가 노출한 특정 서비스에만 접근 가능. 즉, 공급자 VPC의 다른 자원에는 접근 불가. 

>[!.questions]- 문항별 풀이
> A : VPC 피어링은 두 VPC를 전체적으로 연결되기 때문에, 특정 서비스에 대한 엑세스 제한이 어렵다. 또한, 네트워크 대역이 겹치면 연결 불가
> B : 가상 프라이빗 게이트웨이는 온프레미스와 AWS 사이를 VPN이나 Direct Connect 로 연결할 때 사용
