## 문제 1 - 고트래픽 상황

한 전자상거래 회사가 AWS에 '매일 한 가지 상품만 할인 판매하는' 웹사이트를 구축하려고 합니다. 매일 24시간 동안 정확히 한 가지 상품만 할인 판매할 예정입니다. 이 회사는 피크 시간대에 시간당 수백만 건의 요청을 처리하면서도 밀리초 단위의 지연 시간만 유지해야 합니다.  
이러한 요구 사항을 충족하면서 운영 오버헤드를 최소화하는 솔루션은 무엇일까요?

- A. Amazon S3를 사용하여 전체 웹사이트를 여러 S3 버킷에 호스팅합니다. Amazon CloudFront 배포를 추가하고, 해당 S3 버킷을 배포의 원본으로 설정합니다. 주문 데이터를 Amazon S3에 저장합니다.
- B. 전체 웹사이트를 여러 가용 영역에 걸쳐 Auto Scaling 그룹으로 실행되는 Amazon EC2 인스턴스에 배포합니다. 웹사이트 트래픽 분산을 위해 애플리케이션 로드 밸런서(ALB)를 추가합니다. 백엔드 API용으로 또 다른 ALB를 추가합니다. 데이터는 Amazon RDS for MySQL에 저장합니다.
- C. 전체 애플리케이션을 컨테이너에서 실행되도록 마이그레이션합니다. 컨테이너는 Amazon Elastic Kubernetes Service(Amazon EKS)에 호스팅합니다. Kubernetes Cluster Autoscaler를 사용하여 트래픽 급증에 맞춰 Pod 수를 늘리거나 줄입니다. 데이터는 Amazon RDS for MySQL에 저장합니다.
- D. 웹사이트의 정적 콘텐츠를 호스팅하기 위해 Amazon S3 버킷을 사용합니다. Amazon CloudFront 배포를 구축하고, S3 버킷을 오리진으로 설정합니다. 백엔드 API에는 Amazon API Gateway와 AWS Lambda 함수를 사용하고, 데이터는 Amazon DynamoDB에 저장합니다.

>[!.questions]- 정답
>정답 : D
> 폭발적인 트래픽 + 빠른 응답 속도 + 운영 최소화 -> '서버리스' 서비스
> 1. 정적 컨텐츠 최적화 
> 	- CloudFront 에 정적 페이지를 넣고, 엣지 로케이션으로 응답하면 개빠름
> 2. 무한 확장 가능한 API
> 	 - API Gateway, Lambda는 서버를 띄울 필요 없이, 요청이 들어오는 순간에만 실행되고 알아서 인스턴스를 늘림
> 3. 고성능 데이터베이스
> 	- DynamoDB는 초당 수백만건을 처리해도 한 자릿수 밀리초 응답 속도 보장
> 	-> 빠른 응답속도에 최적화 되어 있음

> [!.questions]- 문항별 풀이
> B : 수백만 건의 요청을 처리하려면 엄청난 규모의 Auto Scaling과 RDS 인스턴스 필요(운영 오버헤드)
> C : 쿠버네티스는 운영 오버헤드

## 문제 2 - S3

솔루션 아키텍트가 새로운 디지털 미디어 애플리케이션의 스토리지 아키텍처를 설계하기 위해 Amazon S3를 사용하고 있습니다. 미디어 파일은 가용성 영역 손실에도 복원력이 있어야 합니다. 일부 파일은 자주 액세스되는 반면, 다른 파일은 예측할 수 없는 패턴으로 드물게 액세스됩니다. 솔루션 아키텍트는 미디어 파일의 저장 및 검색 비용을 최소화해야 합니다.  
이러한 요구 사항을 충족하는 스토리지 옵션은 무엇일까요?

- A. S3 표준
- B. S3 지능형 계층화
- C. S3 표준-간헐적 접근(S3 Standard-IA)
- D. S3 원존-간헐적 접근(S3 원존-IA)

>[!.questions]- 정답
>정답 : B
> 문제 조건 - S3  지능형 계층화
> 1. 가용성 영역 손실에도 복원력
> 	- 대부분의 S3는 최소 3개의 가용영역에 분산 저장
> 2. 예측할 수 없는 패턴으로 엑세스
> 	- S3 지능형 계층화는 알아서 파일의 라이프사이클을 토대로 계층 이동
> 3. 비용 최소화
> 	- 모니터링 비용만 조금 내고, 검색 비용은 발생하지 않음

>[!.questions]- 문항별 풀이
>A : S3 표준은 자주 접근할 때는 좋지만, 드물게 접근하는 파일에는 요금 발생
>C: S3 Standard-IA는 예측 불가능한 상황에서는 검색 요금 많이 발생 가능성
>D : S3 원존 I-A는 하나의 영역에서만 복원 (가용성 X)

## 문제 3 - S3

한 회사가 Amazon S3 Standard 스토리지를 사용하여 백업 파일을 저장하고 있습니다. 해당 파일은 1개월 동안 자주 액세스되지만, 1개월 후에는 액세스되지 않습니다. 회사는 파일을 무기한으로 보관해야 합니다.  
이러한 요구 사항을 가장 비용 효율적으로 충족하는 스토리지 솔루션은 무엇일까요?

- A. S3 인텔리전트 티어링을 구성하여 객체가 자동으로 마이그레이션되도록 합니다.
- B. 1개월 후 S3 Standard에서 S3 Glacier Deep Archive로 객체를 전환하는 S3 라이프사이클 구성을 생성합니다.
- C. 1개월 후 객체를 S3 Standard에서 S3 Standard-Infrequent Access(S3 Standard-IA)로 전환하는 S3 라이프사이클 구성을 생성합니다.
- D. 1개월 후 S3 Standard에서 S3 One Zone-IA(S3 One Zone-IA)로 객체를 전환하는 S3 라이프사이클 구성을 생성합니다.

>[!.questions]- 정답
>정답 : B
> 정답 조건 : 가장 저렴한 비용으로 오랫동안 보관할 수 있는 방법
> S3 Glacier Deep Archive 는 가장 저렴하고 오랫동안 보관할 때 쓰임

>[!.questions]- 문항별 풀이
>A : S3 인텔리전트 티어링(Inteligent Tiering)은 예측할 수 없을 때 사용
>C : Standard I-A도 저렴하긴 한데, 자주 접근 안하기도 하고 Glacirer 가 더 쌈

참고 -[S3](../SAA/S3.md)
## 문제 4 - 비용 청구

한 회사가 최근 청구서에서 Amazon EC2 비용이 증가한 것을 확인했습니다. 청구팀은 몇몇 EC2 인스턴스에서 원치 않는 인스턴스 유형의 수직 확장이 발생한 것을 발견했습니다. 솔루션 아키텍트는 지난 2개월간의 EC2 비용을 비교하는 그래프를 작성하고 심층 분석을 통해 수직 확장의 근본 원인을 파악해야 합니다.  
솔루션 아키텍트는 운영 오버헤드를 최소화하면서 어떻게 이 정보를 생성해야 할까요?

- A. AWS Budgets를 사용하여 예산 보고서를 생성하고 인스턴스 유형별 EC2 비용을 비교합니다.
- B. Cost Explorer의 세부 필터링 기능을 사용하여 인스턴스 유형별로 EC2 비용을 심층 분석합니다.
- C. AWS 청구 및 비용 관리 대시보드의 그래프를 사용하여 지난 2개월 동안 인스턴스 유형별 EC2 비용을 비교합니다.
- D. AWS 비용 및 사용량 보고서를 사용하여 보고서를 생성하고 Amazon S3 버킷으로 전송합니다. Amazon QuickSight를 Amazon S3와 함께 사용하여 인스턴스 유형을 기반으로 하는 대화형 그래프를 생성합니다.

>[!.questions]- 풀이
>정답 : B
> A : AWS Budgets 는 설정한 임계치를 지출이 넘었을 때 알림을 보내는 용도
> B : Cost Explorer는 비용 및 사용량 데이터를 그래프로 시각화하고 분석하는데 최적화된 도구
> C : Billing Dashboard는 전반적인 청구 요약과 이달의 예상 비용을 보여주는 요약판. 상세 분석 기능은 제공하지 않음
> D : 너무 많은 작업이 필요하므로 운영 복잡도 높음


## 문제 5 - Lambda

한 회사가 애플리케이션을 개발 중입니다. 이 애플리케이션은 AWS Lambda 함수를 사용하여 Amazon API Gateway를 통해 정보를 수신하고 Amazon Aurora PostgreSQL 데이터베이스에 저장합니다.  
개념 증명(PoC) 단계에서 회사는 데이터베이스에 로드해야 하는 대량의 데이터를 처리하기 위해 Lambda 할당량을 크게 늘려야 합니다. 솔루션 아키텍트는 확장성을 향상시키고 구성 작업을 최소화하는 새로운 설계를 제안해야 합니다.  
이러한 요구 사항을 충족하는 솔루션은 무엇일까요?

- A. Lambda 함수 코드를 Amazon EC2 인스턴스에서 실행되는 Apache Tomcat 코드로 리팩토링합니다. 네이티브 Java Database Connectivity(JDBC) 드라이버를 사용하여 데이터베이스에 연결합니다.
- B. 플랫폼을 Aurora에서 Amazon DynamoDB로 변경합니다. DynamoDB Accelerator(DAX) 클러스터를 프로비저닝합니다. DAX 클라이언트 SDK를 사용하여 기존 DynamoDB API 호출이 DAX 클러스터를 가리키도록 설정합니다.
- C. 두 개의 Lambda 함수를 설정합니다. 하나는 정보를 수신하도록 구성하고, 다른 하나는 정보를 데이터베이스에 로드하도록 구성합니다. Amazon Simple Notification Service(Amazon SNS)를 사용하여 Lambda 함수를 통합합니다.
- D. Lambda 함수 두 개를 설정합니다. 하나는 정보를 수신하도록 구성하고, 다른 하나는 정보를 데이터베이스에 로드하도록 구성합니다. Amazon Simple Queue Service(Amazon SQS) 큐를 사용하여 Lambda 함수를 통합합니다.

>[!.questions]- 정답
>정답 : D
>정답 조건 : 병목 현상을 비동기식 결합(Lambda + SQS)으로 해결한다.
>SQS
>	- DB가 즉시 처리할 필요 없이, 일단 SQS 안에 넣어둔다.
>	- Lambda가 SQS에서 메시지를 가지고 올때 DB 상황에 맞춰서 조절해 가며 가져올 수 있따.

>[!.questions]- 문항별 풀이
>A : 구성 작업 너무 많음
>B : DB 자체를 바꾸는 것은 리팩토링 비용 너무 큼. 그리고 DAX는 읽기 성능 향상 용도
>C : SNS는 메시지를 뿌리는 용도(Fan out)

## 문제 6 - 감사

한 회사가 AWS 클라우드 배포 환경을 검토하여 Amazon S3 버킷에 무단 구성 변경이 없는지 확인해야 합니다.  
솔루션 아키텍트는 이 목표를 달성하기 위해 무엇을 해야 할까요?

- A. 적절한 규칙을 사용하여 AWS Config를 활성화하십시오.
- B. 적절한 확인 절차를 거쳐 AWS Trusted Advisor를 활성화합니다.
- C. 적절한 평가 템플릿을 사용하여 Amazon Inspector를 활성화합니다.
- D. Amazon S3 서버 액세스 로깅을 활성화합니다. Amazon EventBridge(Amazon Cloud Watch Events)를 구성합니다.

>[!.questions]- 정답
>정답 : A
>AWS Config
>- 리소스 구성 추적 : AWS 리소스의 설정 변경 사항을 실시간으로 기록.(누가,언제,무엇을 바꿨는지 추적)\
>- 규정 준수 : 리소스의 규정을 정하여 Config 규칙을 설정할 수 있음
>- 무단 변경 감지 : 누군가 허가 없이 설정을 바꾸면, Config가 이를 감지

>[!.questions]- 문항별 풀이
>B : AWS Trusted Advisor 는 보안, 성능, 비용 면에서 개선사항에 대한 권장 사항을 제안하는 도구
>C : Amazon Inspector는 보안 취약점 진단 서비스. 주로 EC2 인스턴스 내부에 깔린 소프트웨어의 취약점이나 노출된 포트 검사할 때 사용
>D : S3 서버 엑세스 로깅은 누가 파일에 접근했는지 기록하는 용도

## 문제 7 - CloudWatch 보안

한 회사가 새로운 애플리케이션을 출시하고 Amazon CloudWatch 대시보드에 애플리케이션 지표를 표시할 예정입니다. 회사 제품 관리자는 이 대시보드에 주기적으로 접근해야 합니다. 제품 관리자는 AWS 계정이 없습니다. 솔루션 아키텍트는 최소 권한 원칙에 따라 제품 관리자에게 접근 권한을 부여해야 합니다.  
이러한 요구 사항을 충족하는 솔루션은 무엇일까요?

- A. CloudWatch 콘솔에서 대시보드를 공유합니다. 제품 관리자의 이메일 주소를 입력하고 공유 단계를 완료합니다. 제품 관리자에게 대시보드를 공유할 수 있는 링크를 제공합니다.
- B. 제품 관리자 전용 IAM 사용자를 생성합니다. 해당 사용자에게 CloudWatchReadOnlyAccess AWS 관리형 정책을 연결합니다. 새 로그인 자격 증명을 제품 관리자와 공유합니다. 올바른 대시보드의 브라우저 URL을 제품 관리자와 공유합니다.
- C. 회사 직원용 IAM 사용자를 생성합니다. 해당 IAM 사용자에게 ViewOnlyAccess AWS 관리형 정책을 연결합니다. 새 로그인 자격 증명을 제품 관리자와 공유합니다. 제품 관리자에게 CloudWatch 콘솔로 이동하여 대시보드 섹션에서 해당 대시보드 이름을 찾도록 요청합니다.
- D. 공용 서브넷에 배스천 서버를 배포합니다. 제품 관리자가 대시보드에 액세스해야 할 경우 서버를 시작하고 RDP 자격 증명을 공유합니다. 배스천 서버에서 브라우저가 대시보드 URL을 열 때 대시보드를 볼 수 있는 적절한 권한이 있는 캐시된 AWS 자격 증명을 사용하도록 구성되어 있는지 확인합니다.

>[!.questions]- 정답
>정답 : A
>CloudWatch는 AWS 계정이 없는 사람과 대시보드 공유 기능 제공.
>- 보안성 : 이메일 주소를 지정하고 비밀번호를 설정하여 특정 사람만 볼 수 있게 제한
>- 최소 권한 : 불필요한 IAM 계정을 만들 필요 없고, 별도의 AWS 로그인 없이 확인 가능

>[!.questions]- 문항별 풀이
>B,C : 현재 제품 관리자는 AWS 계정이 없으므로 IAM 계정 할당은 운영 오버헤드
>	- 문항에 나와있는 정책은 다른 리소스 정보도 볼 수 있으므로 최소 권한 원칙 위배
>: D : 너무 복잡

## 문제 9 - Global Accelerator

한 회사가 UDP 연결을 사용하는 VoIP(Voice over Internet Protocol) 서비스를 제공합니다. 이 서비스는 Auto Scaling 그룹에서 실행되는 Amazon EC2 인스턴스로 구성됩니다. 회사는 여러 AWS 리전에 서비스를 배포했습니다.  
이 회사는 사용자를 지연 시간이 가장 짧은 리전으로 연결해야 하며, 리전 간 자동 장애 조치 기능도 필요합니다.  
이러한 요구 사항을 충족하는 솔루션은 무엇일까요?

- A. 네트워크 로드 밸런서(NLB)와 연결된 대상 그룹을 배포합니다. 대상 그룹을 Auto Scaling 그룹과 연결합니다. 각 리전에서 NLB를 AWS Global Accelerator 엔드포인트로 사용합니다.
- B. 애플리케이션 로드 밸런서(ALB)와 연결된 대상 그룹을 배포합니다. 대상 그룹을 Auto Scaling 그룹과 연결합니다. 각 리전에서 ALB를 AWS Global Accelerator 엔드포인트로 사용합니다.
- C. 네트워크 로드 밸런서(NLB)와 연결된 대상 그룹을 배포합니다. 대상 그룹을 Auto Scaling 그룹과 연결합니다. 각 NLB에 대한 별칭을 가리키는 Amazon Route 53 지연 시간 레코드를 생성합니다. 해당 지연 시간 레코드를 원본으로 사용하는 Amazon CloudFront 배포를 생성합니다.
- D. 애플리케이션 로드 밸런서(ALB)와 연결된 대상 그룹을 배포합니다. 대상 그룹을 Auto Scaling 그룹과 연결합니다. 각 ALB에 대한 별칭을 가리키는 Amazon Route 53 가중치 레코드를 생성합니다. 가중치 레코드를 오리진으로 사용하는 Amazon CloudFront 배포를 배포합니다.

>[!.questions]- 정답 
>정답 : A
>정답 조건 : UDP(L4) + 초저지연
>Global Accelerator
>- 초저지연 : AWS 내부 글로벌 네트워크를 통해 사용자 트래픽을 가장 가까운 리전으로 안내
>- 장애 조치 : 각 리전의 엔드포인트 상태를 지속적으로 확인하며, 특정 리전에 문제가 생기면 10초 이내에 다른 건강한 리전으로 트래픽을 자동으로 돌려줌
>- 애니캐스트 : 고정된 IPwnth wprhd

## 문제 10

개발팀은 Performance Insights가 활성화된 범용 Amazon RDS for MySQL DB 인스턴스에서 매달 리소스 집약적인 테스트를 실행합니다. 이 테스트는 한 달에 한 번 48시간 동안 진행되며, 해당 데이터베이스를 사용하는 유일한 프로세스입니다. 개발팀은 DB 인스턴스의 컴퓨팅 및 메모리 용량을 줄이지 않고 테스트 실행 비용을 절감하고자 합니다.  
이러한 요구 사항을 가장 비용 효율적으로 충족하는 솔루션은 무엇일까요?

- A. 테스트가 완료되면 데이터베이스 인스턴스를 중지하십시오. 필요한 경우 데이터베이스 인스턴스를 다시 시작하십시오.
- B. 테스트가 완료되면 자동으로 확장되도록 DB 인스턴스에 자동 확장 정책을 적용합니다.
- C. 테스트가 완료되면 스냅샷을 생성합니다. 필요한 경우 DB 인스턴스를 종료하고 스냅샷을 복원합니다.
- D. 테스트가 완료되면 DB 인스턴스를 용량이 작은 인스턴스로 변경합니다. 필요한 경우 DB 인스턴스를 다시 변경합니다.

>[!.questions]- 정답
>정답 : C
> 정답 조건 : 비용 효율 
> 만들어둔 스냅샷으로 DB 인스턴스를 종료시키고 다시 실행시킬때 스냅샷으로 복구하여 비용을 절감한다.

>[!.questions]- 문항별 풀이
>A:DB 인스턴스를 종료해도 EBS 스토리지 요금은 계속 부과되며, 7일이 지나면 자동으로 restart 되므로 수동으로 다시 끄지 않으면 인스턴스 자체 요금도 플러스로 부과된다.
