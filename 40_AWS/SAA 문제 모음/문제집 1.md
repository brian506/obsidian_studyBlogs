## 문제 1 - S3 전송 효율

한 회사가 여러 대륙에 걸쳐 있는 도시들의 온도, 습도, 기압 데이터를 수집합니다. 각 사이트에서 매일 수집하는 데이터 양은 평균 500GB입니다. 모든 사이트는 고속 인터넷에 연결되어 있습니다.  
이 회사는 전 세계 모든 사이트에서 수집한 데이터를 가능한 한 빨리 하나의 Amazon S3 버킷에 통합하고자 합니다. 솔루션은 운영 복잡성을 최소화해야 합니다.  
이러한 요구 사항을 충족하는 솔루션은 무엇일까요?

- A. 대상 S3 버킷에서 S3 전송 가속을 활성화합니다. 멀티파트 업로드를 사용하여 사이트 데이터를 대상 S3 버킷에 직접 업로드합니다.
- B. 각 사이트의 데이터를 가장 가까운 리전의 S3 버킷에 업로드합니다. S3 리전 간 복제를 사용하여 객체를 대상 S3 버킷으로 복사합니다. 그런 다음 원본 S3 버킷에서 데이터를 삭제합니다.
- C. AWS Snowball Edge Storage Optimized 디바이스 작업을 매일 예약하여 각 사이트의 데이터를 가장 가까운 리전으로 전송합니다. S3 교차 리전 복제를 사용하여 객체를 대상 S3 버킷으로 복사합니다.
- D. 각 사이트의 데이터를 가장 가까운 리전의 Amazon EC2 인스턴스에 업로드합니다. 데이터를 Amazon Elastic Block Store(Amazon EBS) 볼륨에 저장합니다. 정기적으로 EBS 스냅샷을 생성하여 대상 S3 버킷이 있는 리전으로 복사합니다. 해당 리전에서 EBS 볼륨을 복원합니다.

> [!.question]- 정답
> 답 : A
> 정답 조건 
> - 빠른 전송 : S3 전송 가속(S3TA)은 엣지 로케이션을 활용
> - 대용량 데이터 : 멀티파트 업로드로 파일을 조각내어 병렬로 올리고, 중간에 끊겨도 이어서 가능
> - 최소한의 운영 복잡성  : S3TA는 체크 한번이면 적용

> [!.question]- 문항별 풀이
> B : 운영 복잡성 불만족
>      리전마다 S3 버킷 생성 -> 복제 -> 삭제 과정 복잡함
> C : Snowball Edge 는 인터넷 속도가 느려 많은 양(TB~)을 보낼 때 사용하는 물리적 전송 수단
> D : 그냥 개복잡


## 문제 2 - 로그 분석

어떤 회사는 자사 애플리케이션의 로그 파일을 분석할 수 있는 기능이 필요합니다. 로그는 JSON 형식으로 Amazon S3 버킷에 저장되어 있습니다. 쿼리는 간단하며 필요에 따라 실행될 수 있습니다. 솔루션 아키텍트는 기존 아키텍처를 최소한으로 변경하면서 이러한 분석을 수행해야 합니다.  
운영 오버헤드를 최소화하면서 이러한 요구 사항을 충족하려면 솔루션 아키텍트는 어떻게 해야 할까요?

- A. 아마존 레드시프트를 사용하여 모든 콘텐츠를 한 곳에 로드하고 필요에 따라 SQL 쿼리를 실행합니다.
- B. Amazon CloudWatch Logs를 사용하여 로그를 저장합니다. 필요에 따라 Amazon CloudWatch 콘솔에서 SQL 쿼리를 실행합니다.
- C. 필요에 따라 쿼리를 실행하려면 Amazon Athena를 Amazon S3와 직접 연결하여 사용하십시오.
- D. AWS Glue를 사용하여 로그를 카탈로그화합니다. 필요에 따라 Amazon EMR에서 임시 Apache Spark 클러스터를 사용하여 SQL 쿼리를 실행합니다.

>[!.question]- 정답
> 답 : C
> 정답 조건 
> - 기존 아키텍처 최소 변경 : Athena 는 S3에 저장된 데이터를 바로 쿼리할 수 있음
> - 서버리스(Serverless) : Athena 는 서버를 띄우거나 관리할 필요 없음
> - JSON 형식 지원

> [!.question]- 문항별 풀이
> A : Redshift 는 클러스터를 구축해야 하고, S3 데이터를 로드(Copy) 하는 과정 필요 (오버헤드, 아키텍처 증가)
> B : 로그가 이미 S3에 저장되어 있기 때문에, CloudWatch로 옮기는 것은 비효율적임
> D : EMR은 대규모 데이터 처리를 위한 클러스터임. 간단한 쿼리에는 부적합

## 문제 3 - S3 권한 정책

한 회사가 AWS Organizations를 사용하여 여러 부서의 AWS 계정을 관리하고 있습니다. 관리 계정에는 프로젝트 보고서가 저장된 Amazon S3 버킷이 있습니다. 이 회사는 AWS Organizations 내 조직 계정 사용자만 해당 S3 버킷에 접근할 수 있도록 접근 권한을 제한하고자 합니다.  
이러한 요구 사항을 충족하면서 운영 오버헤드를 최소화하는 솔루션은 무엇일까요?

- A. 조직 ID를 참조하는 AWS PrincipalOrgID 전역 조건 키를 S3 버킷 정책에 추가합니다.
- B. 각 부서별로 조직 단위(OU)를 생성합니다. S3 버킷 정책에 aws:PrincipalOrgPaths 전역 조건 키를 추가합니다.
- C. AWS CloudTrail을 사용하여 계정 생성, 계정 초대, 조직 탈퇴 및 계정 삭제 이벤트를 모니터링하고, S3 버킷 정책을 accordingly 업데이트하십시오.
- D. S3 버킷에 접근해야 하는 각 사용자에게 태그를 지정합니다. S3 버킷 정책에 aws:PrincipalTag 전역 조건 키를 추가합니다.

> [!.question]- 정답
> 정답 : A
> aws:PrincipalOrgId? : 특정 사용자(Principal)가 속한 AWS조직의 고유 ID를 확인하는 조건 키 (전체 조직 식별)
> 정답 조건
> - 빠른 권한 체크 : 전체 조직의 ID를 식별하여 접근 허용
> -> 버킷 정책 수정 필요없음

>[!.questions]- 문항별 풀이
> B : aws:PrincipalOrgPaths 는 조직 내의 특정 부서를 확인할 때 쓰므로, 조직 전체 계정에 대한 여부에서는 굳이 쓰지 않음 (특정 조직 식별)
> C : 계정을 추가할때마다 버킷 정책을 계정 ID에 추가해야 하므로 운영 비용 증가
> D : aws:PrincipalTag는 모든 계정에 태그를 달아야 하므로, 관리 힘듬

## 문제 4 - VPC

VPC 내의 Amazon EC2 인스턴스에서 애플리케이션이 실행됩니다. 이 애플리케이션은 Amazon S3 버킷에 저장된 로그를 처리합니다. EC2 인스턴스는 인터넷 연결 없이 S3 버킷에 액세스해야 합니다.  
Amazon S3에 대한 프라이빗 네트워크 연결을 제공하는 솔루션은 무엇일까요?

- A. S3 버킷에 대한 게이트웨이 VPC 엔드포인트를 생성합니다.
- B. 로그를 Amazon CloudWatch Logs로 스트리밍합니다. 로그를 S3 버킷으로 내보냅니다.
- C. S3 접근을 허용하도록 Amazon EC2에 인스턴스 프로필을 생성합니다.
- D. S3 엔드포인트에 액세스할 수 있도록 비공개 링크를 사용하는 Amazon API Gateway API를 생성합니다.

>[!.questions]- 정답
>정답 : A
>정답 조건
> S3는 기본적으로 VPC 밖에 있는 AWS 공용망에 있는 서비스이다.
> 
> 1. VPC 내의 프라이빗 서브넷에 있는 EC2는 외부와 차단되어 있다. 하지만 S3는 외부 서비스라 원래는 IGW로 연결되어야 한다.
> 2. VPC 엔드포인트 : VPC와 S3 사이에 AWS 전용 통로를 뚫어주는 서비스. AWS 내부 네트워크 안에서 통신하게 한다.

>[!.questions]- 문항별 풀이
>B : S3에 있는 로그를 처리하는 연결 통로를 묻는 것과 관련이 없음
>C : 인스턴스 프로필 생성은 S3에 들어갈 자격을 임명하는 것
>D : 가능은 하지만 복잡함

## 문제 5 - EFS

한 회사가 AWS에서 웹 애플리케이션을 호스팅하고 있으며, 사용자가 업로드한 문서를 Amazon EBS 볼륨에 저장하는 단일 Amazon EC2 인스턴스를 사용하고 있습니다. 확장성과 가용성을 향상시키기 위해 회사는 아키텍처를 복제하여 다른 가용 영역에 두 번째 EC2 인스턴스와 EBS 볼륨을 생성하고, 둘 다 애플리케이션 로드 밸런서 뒤에 배치했습니다. 이 변경 후 사용자들이 웹사이트를 새로 고칠 때마다 문서의 일부만 볼 수 있고 모든 문서를 동시에 볼 수 없다고 보고했습니다. 솔루션 아키텍트는  
사용자가 모든 문서를 한 번에 볼 수 있도록 어떤 해결책을 제시해야 할까요?

- A. 두 EBS 볼륨 모두에 모든 문서가 포함되도록 데이터를 복사합니다.
- B. 애플리케이션 로드 밸런서를 구성하여 사용자가 문서가 있는 서버로 연결되도록 합니다.
- C. 두 EBS 볼륨의 데이터를 Amazon EFS로 복사합니다. 새 문서를 Amazon EFS에 저장하도록 애플리케이션을 수정합니다.
- D. 애플리케이션 로드 밸런서를 구성하여 요청을 두 서버 모두로 전송합니다. 각 문서는 올바른 서버에서 반환합니다

> [!.questions]- 정답
> 정답 : C
> 정답 조건
> EBS : 특정 인스턴스에 종속된 전용 디스크
> EFS 는 여러 인스턴스가 동시에 연결해서 사용할 수 있는 네트워크 공유 드라이브
> 
> 문제 상황 : 사용자가 A 서버에 파일을 업로드 하면 B서버에서는 해당 파일을 읽어오지 못함
> 해결 : 서버 A,B 모두 하나의 EFS를 바라보게 하면 모두 같은 파일을 읽을 수 있음

## 문제 6 - 데이터 전송(크기, 시간)

한 회사가 사내 네트워크 연결 스토리지(NAS)에 대용량 비디오 파일을 저장하기 위해 NFS를 사용하고 있습니다. 각 비디오 파일의 크기는 1MB에서 500GB까지 다양합니다. 총 저장 용량은 70TB이며 더 이상 증가하지 않습니다. 이 회사는 비디오 파일을 Amazon S3로 마이그레이션하기로 결정했습니다. 마이그레이션은 최대한 신속하게 진행하면서 네트워크 대역폭 사용량을 최소화해야 합니다.  
이러한 요구 사항을 충족하는 솔루션은 무엇일까요?

- A. S3 버킷을 생성합니다. S3 버킷에 쓰기 권한이 있는 IAM 역할을 생성합니다. AWS CLI를 사용하여 로컬의 모든 파일을 S3 버킷으로 복사합니다.
- B. AWS Snowball Edge 작업을 생성합니다. 온프레미스에 Snowball Edge 디바이스를 수령합니다. Snowball Edge 클라이언트를 사용하여 디바이스로 데이터를 전송합니다. AWS에서 데이터를 Amazon S3로 가져올 수 있도록 디바이스를 반환합니다.
- C. 온프레미스에 S3 파일 게이트웨이를 배포합니다. S3 파일 게이트웨이에 연결하기 위한 공용 서비스 엔드포인트를 생성합니다. S3 버킷을 생성합니다. S3 파일 게이트웨이에 새 NFS 파일 공유를 생성합니다. 새 파일 공유가 S3 버킷을 가리키도록 설정합니다. 기존 NFS 파일 공유의 데이터를 S3 파일 게이트웨이로 전송합니다.
- D. 온프레미스 네트워크와 AWS 간에 AWS Direct Connect 연결을 설정합니다. 온프레미스에 S3 파일 게이트웨이를 배포합니다. S3 파일 게이트웨이에 연결하기 위한 공용 가상 인터페이스(VIF)를 생성합니다. S3 버킷을 생성합니다. S3 파일 게이트웨이에 새 NFS 파일 공유를 생성합니다. 새 파일 공유가 S3 버킷을 가리키도록 설정합니다. 기존 NFS 파일 공유의 데이터를 S3 파일 게이트웨이로 전송합니다

> [!.questions]- 정답
> 정답 : B
> 문제 상황 
> - 70TB를 인터넷 회선으로 보내려면 몇 주 걸리고, 네트워크 대역폭을 최소화해야하는 상황임
> 해결
> Snowball Edge는 데이터를 물리적인 전송 장치에 담아 옮기는 방식이므로 네트워크 대역폭을 전혀 사용하지 않음.
> 또한, 최대 가용 크기 80TB

**데이터 양,추천 솔루션**
수십 GB ~ 수 TB : 인터넷/사이트 간 VPN (충분히 보낼 만함)
10TB ~ 수 PB : AWS Snowball Edge (물리 장비 배송)
수십 PB ~ EB(엑사바이트) : AWS Snowmobile (진짜 거대한 트럭이 옵니다)
실시간/지속적 전송 : AWS DataSync (네트워크 최적화 전송 서비스)

## 문제 7 - Fan Out 패턴

한 회사의 애플리케이션이 수신 메시지를 처리합니다. 수십 개의 다른 애플리케이션과 마이크로서비스가 이 메시지를 빠르게 소비합니다. 메시지 수는 변동이 심하고 때로는 초당 10만 건까지 급증하기도 합니다. 회사는 솔루션의 결합도를 낮추고 확장성을 향상시키고자 합니다.  
이러한 요구 사항을 충족하는 솔루션은 무엇일까요?

- A. 메시지를 Amazon Kinesis Data Analytics에 저장합니다. 소비자 애플리케이션이 메시지를 읽고 처리하도록 구성합니다.
- B. CPU 사용량 지표에 따라 EC2 인스턴스 수를 확장할 수 있도록 Auto Scaling 그룹의 Amazon EC2 인스턴스에 데이터 수집 애플리케이션을 배포합니다.
- C. 메시지를 단일 샤드를 사용하여 Amazon Kinesis Data Streams에 기록합니다. AWS Lambda 함수를 사용하여 메시지를 전처리하고 Amazon DynamoDB에 저장합니다. 소비자 애플리케이션이 DynamoDB에서 메시지를 읽어 처리하도록 구성합니다.
- D. 여러 개의 Amazon Simple Queue Service(Amazon SOS) 구독이 있는 Amazon Simple Notification Service(Amazon SNS) 토픽에 메시지를 게시합니다. 소비자 애플리케이션이 큐에서 메시지를 처리하도록 구성합니다.

> [!.questions]- 정답
> 정답 : D 
> 정답 조건
> 
> Fan out : SNS는 메시지를 한 번 발행하면 한 번에 구독하고 있는 대상들에게 뿌려줌. 
> 결합도 낮추기 : 메시지 발행자는 누가 받는 지 몰라도 됨. 또한, SQS를 두면 소비자가 죽어도 메시지가 큐에 안전하게 보관되어 의존성 분리됨
> 확장성 및 트래픽 처리 : SNS,SQS는 모두 severless이므로 별도의 설정없이 자동으로 확장

>[!.questions]- 문항별 풀이
>A : Kinesis Analytics는 실시간 SQL 쿼리나 분석할 때 사용. 메시징 목적이 아님
>B : Auto Scaling으로 서버를 늘리는 것으로 메시지의 분산 전송 해결 안됨. 결합도가 낮아지지도 않음
>C : Kinesis Data Streams는 초당 1000건이 최대이고, DynamoDB에 또 저장하는 건 메시징 처리와 관련없음.

참고 - [SQS, SNS, Kinesis, Amazon MQ](../SAA/SQS,%20SNS,%20Kinesis,%20Amazon%20MQ.md)
## 문제 8 - 디커플링, 동적 스케일링

한 회사가 분산 애플리케이션을 AWS로 마이그레이션하고 있습니다. 이 애플리케이션은 가변적인 워크로드를 처리합니다. 기존 플랫폼은 여러 컴퓨팅 노드에 걸쳐 작업을 조정하는 기본 서버로 구성되어 있습니다. 회사는 복원력과 확장성을 극대화하는 솔루션으로 애플리케이션을 현대화하고자 합니다.  
솔루션 아키텍트는 이러한 요구 사항을 충족하는 아키텍처를 어떻게 설계해야 할까요?

- A. 작업 대상으로 Amazon Simple Queue Service(Amazon SQS) 큐를 구성합니다. Auto Scaling 그룹에서 관리되는 Amazon EC2 인스턴스를 사용하여 컴퓨팅 노드를 구현합니다. 예약된 스케일링을 사용하도록 EC2 Auto Scaling을 구성합니다.
- B. 작업 대상으로 Amazon Simple Queue Service(Amazon SQS) 큐를 구성합니다. Auto Scaling 그룹에서 관리되는 Amazon EC2 인스턴스를 사용하여 컴퓨팅 노드를 구현합니다. 큐 크기에 따라 EC2 Auto Scaling을 구성합니다.
- C. AWS Auto Scaling 그룹으로 관리되는 Amazon EC2 인스턴스를 사용하여 기본 서버와 컴퓨팅 노드를 구현합니다. 작업 대상으로 AWS CloudTrail을 구성합니다. 기본 서버의 부하에 따라 EC2 Auto Scaling을 구성합니다.
- D. Amazon Auto Scaling 그룹에서 관리되는 Amazon EC2 인스턴스를 사용하여 기본 서버와 컴퓨팅 노드를 구현합니다. Amazon EventBridge(Amazon CloudWatch Events)를 작업 대상으로 구성합니다. 컴퓨팅 노드의 부하에 따라 EC2 Auto Scaling을 구성합니다.

>[!.questions]- 정답
> 정답 : B
> 문제 상황 :
> 현재 구조는 기본 서버는 워크 노드들에게 일을 임명하는 방식임. 기본 서버, 즉 대장 서버가 죽으면 아예 시스템 마비됨.(SPOF) - 복원력X
> 해결 :
> - 대장 서버를 없애고, SQS를 도입하여 노드들이 작업을 큐에서 가져가는 형식으로 '복원력'을 갖추게 됨
> - 동적 스케일링 : 큐에 쌓인 일의 양으로, 일이 많을 때는 서버를 늘리고, 없을 때는 줄이는 방식으로 '확장성' 갖춤

>[!.questions]- 문항별 풀이
> A : Scheduled Scaling 은 예측 가능한 상황에 쓰는 것. 문제는 가변적인 워크로드라고 명시되어 있으므로 부적합
> C & D : '기본 서버'를 두는 것이 복원력을 떨어트리는 것임.
> CloudTrail 은 API 호출 로그를 기록하는 것
> EventBridge 는 이벤트 버스 역할을 하는 것(작업 처리가 아닌 이동 수단)

## 문제 9 - 스토리지(S3 File gateway)

한 회사가 데이터 센터에서 SMB 파일 서버를 운영하고 있습니다. 이 파일 서버에는 파일 생성 후 처음 며칠 동안 자주 액세스되는 대용량 파일이 저장됩니다. 7일 후에는 해당 파일에 대한 액세스가 거의 발생하지 않습니다.  
전체 데이터 크기는 계속 증가하여 회사의 총 저장 용량에 근접하고 있습니다. 솔루션 설계자는 가장 최근에 액세스된 파일에 대한 저지연 액세스를 유지하면서 회사의 가용 저장 공간을 늘려야 합니다. 또한 향후 저장 문제 발생을 방지하기 위해 파일 수명 주기 관리 기능도 제공해야 합니다.  
이러한 요구 사항을 충족하는 솔루션은 무엇일까요?

- A. AWS DataSync를 사용하여 7일 이상 된 데이터를 SMB 파일 서버에서 AWS로 복사합니다.
- B. 회사 스토리지 공간을 확장하기 위해 Amazon S3 파일 게이트웨이를 생성합니다. 7일 후 데이터를 S3 Glacier Deep Archive로 전환하는 S3 라이프사이클 정책을 생성합니다.
- C. 회사 저장 공간을 확장하기 위해 Amazon FSx for Windows 파일 서버 파일 시스템을 생성합니다.
- D. 각 사용자의 컴퓨터에 Amazon S3에 접근할 수 있는 유틸리티를 설치합니다. 7일 후 데이터를 S3 Glacier Flexible Retrieval로 전환하는 S3 라이프사이클 정책을 생성합니다.

>[!.questions]- 정답
> 정답 : B
> S3 File Gateway
> - SMB 프로토콜 호환성 
> - 저지연 엑세스 (로컬 캐싱) : 파일 게이트웨이는 자주 사용하는 파일을 로컬에 캐싱해둠.-> 빠른 처리
> - 확장성 : 파일은 용량 제한이 없는 S3에 저장
> - 수명 주기 관리 : S3의 Lifecycle 정책으로, 7일 지난 파일을 자동으로 Glacier Deep Archive에 저장 -> 저장 문제 방지 및 비용 절감

>[!.questions]- 문항별 분석
>C : Amazon FSx 는 윈도우 파일 서버로, 서버를 하나 더 만들어서 관리해야함. 단순히 서버 하나를 더 만들어서 관리하는 것은 지연 문제를 해결하기 어렵고, 수명 주기 관리가 포함되어 있지 않음

## 문제 10 - SQS

한 회사가 AWS에서 전자상거래 웹 애플리케이션을 구축하고 있습니다. 이 애플리케이션은 신규 주문 정보를 Amazon API Gateway REST API로 전송하여 처리합니다. 회사는 주문이 접수된 순서대로 처리되도록 보장하고자 합니다.  
이러한 요구 사항을 충족하는 솔루션은 무엇일까요?

- A. API 게이트웨이 통합을 사용하여 애플리케이션이 주문을 수신하면 Amazon Simple Notification Service(Amazon SNS) 토픽에 메시지를 게시합니다. AWS Lambda 함수를 해당 토픽에 구독하여 처리를 수행합니다.
- B. 애플리케이션이 주문을 수신하면 API 게이트웨이 통합을 사용하여 Amazon Simple Queue Service(Amazon SQS) FIFO 큐로 메시지를 전송합니다. SQS FIFO 큐가 AWS Lambda 함수를 호출하여 처리하도록 구성합니다.
- C. API 게이트웨이 인증자를 사용하여 애플리케이션이 주문을 처리하는 동안 모든 요청을 차단합니다.
- D. 애플리케이션이 주문을 수신하면 API Gateway 통합을 사용하여 Amazon Simple Queue Service(Amazon SQS) 표준 큐로 메시지를 전송합니다. SQS 표준 큐가 처리를 위해 AWS Lambda 함수를 호출하도록 구성합니다.

> [!.questions]- 정답
> 정답 : B
> 문제 조건 -> 순서대로 처리 보장
> SQS FIFO : First in First Out 으로 순서대로 한번씩 소비자에게 전달된다.
> API Gateway : 받은 API를 바로 SQS에 집어 넣어서 사용할 수 있음

>[!.questions]- 문항별 분석
> A : SNS는 보통 순서를 보장하지 않음. 
> D : SQS 표준 큐는 순서를 100% 보장하지 않음
