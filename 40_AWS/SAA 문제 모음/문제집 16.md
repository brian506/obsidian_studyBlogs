- **A.** AWS CloudFormation 템플릿을 사용하여 전체 솔루션을 배포합니다. 30일마다 CloudFormation 스택을 다시 배포하고 원본 스택을 삭제합니다.
    
- **B.** AWS Marketplace의 모니터링 애플리케이션을 실행하는 EC2 인스턴스를 사용합니다. 테이블에 새 항목이 생성될 때의 타임스탬프를 저장하기 위해 Amazon DynamoDB Streams를 사용하도록 모니터링 애플리케이션을 구성합니다. EC2 인스턴스에서 실행되는 스크립트를 사용하여 30일 이상 지난 타임스탬프를 가진 항목을 삭제합니다.
    
- **C.** 테이블에 새 항목이 생성될 때 AWS Lambda 함수를 호출하도록 Amazon DynamoDB Streams를 구성합니다. 30일 이상 지난 테이블의 항목을 삭제하도록 Lambda 함수를 구성합니다.

>정답 : D
>DynamoDB TTL 기능 : 항목의 만료 시간을 지정하면 해당 시간이 지났을 때 자동으로 데이터를 삭제해준다.
>	- TTL에 의한 삭제는 쓰기 용량 유닛을 소모하지 않으므로 비용 없음

수집 프로세스는 신속하게 발생하지만 주문 이행 프로세스는 시간이 더 오래 걸릴 수 있습니다. 확장(scaling) 이벤트로 인해 데이터가 손실되어서는 안 됩니다. 솔루션 아키텍트는 주문 수집 프로세스와 주문 이행 프로세스 모두 피크 트래픽 시간 동안 적절하게 확장될 수 있도록 보장해야 합니다. 솔루션은 회사의 AWS 리소스 활용을 최적화해야 합니다. 어떤 솔루션이 이러한 요구 사항을 충족합니까?

- **D.** AWS Management Console에서 Amazon S3로 이동합니다. S3 버킷의 객체를 찾아봅니다. 암호화 필드를 기준으로 정렬합니다. 암호화되지 않은 각 객체를 선택합니다. 수정(Modify) 버튼을 사용하여 S3 버킷에 있는 암호화되지 않은 모든 객체에 기본 암호화 설정을 적용합니다.

>[!.questions]- 정답

## 문제 217

한 회사가 Application Load Balancer 뒤에 있는 Amazon EC2 인스턴스에서 글로벌 웹 애플리케이션을 실행하고 있습니다. 애플리케이션은 Amazon Aurora에 데이터를 저장합니다. 회사는 재해 복구 솔루션을 생성해야 하며 최대 30분의 다운타임 및 잠재적인 데이터 손실을 허용할 수 있습니다. 이 솔루션은 기본(primary) 인프라가 정상일 때는 로드를 처리할 필요가 없습니다. 솔루션 아키텍트는 이러한 요구 사항을 충족하기 위해 무엇을 해야 합니까?

- **A.** 필수 인프라 요소가 준비된 상태로 애플리케이션을 배포합니다. Amazon Route 53을 사용하여 활성-수동(active-passive) 장애 조치를 구성합니다. 두 번째 AWS 리전에 Aurora 읽기 전용 복제본(Aurora Replica)을 생성합니다.
    
- **B.** 두 번째 AWS 리전에 축소된 규모(scaled-down)로 애플리케이션을 호스팅합니다. Amazon Route 53을 사용하여 활성-활성(active-active) 장애 조치를 구성합니다. 두 번째 리전에 Aurora 읽기 전용 복제본을 생성합니다.
    
- **C.** 두 번째 AWS 리전에 기본 인프라를 복제합니다. Amazon Route 53을 사용하여 활성-활성 장애 조치를 구성합니다. 최신 스냅샷에서 복원된 Aurora 데이터베이스를 생성합니다.
    
- **D.** AWS Backup으로 데이터를 백업합니다. 해당 백업을 사용하여 두 번째 AWS 리전에 필요한 인프라를 생성합니다. Amazon Route 53을 사용하여 활성-수동 장애 조치를 구성합니다. 두 번째 리전에 Aurora 보조 기본 인스턴스(second primary instance)를 생성합니다.

1. **Backup & Restore:** 가장 저렴, 가장 느림 (RPO/RTO 수 시간~수 일)
    
2. **Pilot Light:** 핵심(DB 등)만 켜둠. 장애 시 나머지를 신속히 가동. (RPO/RTO 수십 분)
    
3. **Warm Standby:** 작은 규모로 전체 시스템을 가동 중. 장애 시 바로 확장. (RPO/RTO 수 분)
    
4. **Multi-Site (Hot Standby):** 두 곳 모두 풀가동. 즉시 전환. (RPO/RTO 실시간, 가장 비쌈)

>[!.questions]- 정답
>정답 : A
>RPO(데이터 손실) 해결 : Aurora 교차 리전 읽기 복제본은 비동기식으로 실시간 복제하므로 데이터 손실을 최소화
>다운타임 해결 : Route53이 장애가 발생할 때 자동으로 트래픽을 넘기고, Active-Passive 구조로 최소한의 자원만 유지하다가 필요할 때만 확장한다.

>[!.questions]- 문항별 풀이
> C : 스냅샷으로 복원하는 것은 데이터 양에 따라 시간 많이 걸림. 스냅샷 생성 시점과 장애 시점 사이의 데이터가 모두 유실되므로 30분 데이터 유실 방지 보장 어려움
> D : AWS Backup도 스냅샷 기반

## 문제 218

한 회사에 탄력적 IP 주소가 있는 퍼블릭 서브넷의 Amazon EC2 인스턴스에서 실행되는 웹 서버가 있습니다. 기본 보안 그룹이 EC2 인스턴에 할당되어 있습니다. 기본 네트워크 ACL이 수정되어 모든 트래픽을 차단하도록 되어 있습니다. 솔루션 아키텍트는 포트 443을 통해 어디에서나 웹 서버에 액세스할 수 있도록 해야 합니다. 이 작업을 달성할 수 있는 단계의 조합은 무엇입니까? (2개 선택)

- **A.** 소스 0.0.0.0/0에서 TCP 포트 443을 허용하는 규칙을 가진 보안 그룹을 생성합니다.
    
- **B.** 대상 0.0.0.0/0으로 TCP 포트 443을 허용하는 규칙을 가진 보안 그룹을 생성합니다.
    
- **C.** 소스 0.0.0.0/0에서 TCP 포트 443을 허용하도록 네트워크 ACL을 업데이트합니다.
    
- **D.** 소스 0.0.0.0/0에서 들어오는 인바운드 TCP 포트 443과 대상 0.0.0.0/0으로 나가는 아웃바운드 TCP 포트 443을 허용하도록 네트워크 ACL을 업데이트합니다.
    
- **E.** 소스 0.0.0.0/0에서 들어오는 인바운드 TCP 포트 443과 대상 0.0.0.0/0으로 나가는 아웃바운드 TCP 포트 32768-65535를 허용하도록 네트워크 ACL을 업데이트합니다.

>[!.questions]- 정답 
>정답 : A,E
> A : 보안그룹은 Stateful함. 인바운드가 허용되면 그에 대한 응답은 아웃바운드 규칙과 상관없이 자동으로 허용
> B : NACL은 Stateless함. 인바운드와 아웃바운드를 따로 열어줘야함. 아웃바운드로 443으로 응답을 보내는 것이 아니라, 클라이언트가 임시로 열어둔 랜덤 포트로 응답을 보냄

## 문제 219

한 회사의 애플리케이션에 성능 문제가 발생하고 있습니다. 이 애플리케이션은 상태를 저장하며(stateful) Amazon EC2 인스턴스에서 인메모리(in-memory) 작업을 완료해야 합니다. 회사는 AWS CloudFormation을 사용하여 인프라를 배포했고 M5 EC2 인스턴스 패밀리를 사용했습니다. 트래픽이 증가함에 따라 애플리케이션 성능이 저하되었습니다. 사용자들이 애플리케이션에 액세스하려고 할 때 지연이 발생한다고 보고하고 있습니다. 운영상 가장 효율적인 방법(MOST operationally efficient way)으로 이러한 문제를 해결하는 솔루션은 무엇입니까?

- **A.** EC2 인스턴스를 Auto Scaling 그룹에서 실행되는 T3 EC2 인스턴스로 교체합니다. AWS Management Console을 사용하여 변경 사항을 적용합니다.
    
- **B.** CloudFormation 템플릿을 수정하여 EC2 인스턴스가 Auto Scaling 그룹에서 실행되도록 합니다. 증가가 필요할 때 수동으로 Auto Scaling 그룹의 원하는 용량과 최대 용량을 늘립니다.
    
- **C.** CloudFormation 템플릿을 수정합니다. EC2 인스턴스를 R5 EC2 인스턴스로 교체합니다. 향후 용량 계획을 위해 애플리케이션 성능을 추적하려면 Amazon CloudWatch 내장 EC2 메모리 지표를 사용합니다.
    
- **D.** CloudFormation 템플릿을 수정합니다. EC2 인스턴스를 R5 EC2 인스턴스로 교체합니다. 향후 용량 계획을 위해 사용자 지정 애플리케이션 지연 시간 지표를 생성하도록 EC2 인스턴스에 Amazon CloudWatch 에이전트를 배포합니다.

>[!.questions]- 정답
>정답 : D
>- 인스턴스 사양 : 인메모리 작업을 해야하므로 R5 인스턴스를 사용하여 RAM 자원을 더 효율적으로 사용
>- CloudWatch 에이전트 를 사용하여 OS 내부지표나 지연시간 등을 수집한다.
>- IaC 유지 : 이미 CloudFormation으로 배포되었으므로, 콘솔에서 수동으로 바꾸지 않고 템플릿 코드를 수정하여 배포한다.

>[!.questions]- 문항별 풀이
>A : T3는 버스트 가능 인스턴스로, 헤비한 인메모리 작업에는 적합하지 않음
>B : '상태 저장' 애플리케이션은 스케일 아웃을 하게 되면, 세션 동기화나 데이터 일관성 문제가 발생할 수 있음. 스케일 아웃이 우선임
>C : CloudWatch에는 내장된 메모리 지표가 없음
