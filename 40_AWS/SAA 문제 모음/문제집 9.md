## 문제 2

한 회사가 AWS 클라우드에서 웹 애플리케이션을 호스팅하고 있습니다. 이 회사는 AWS Certificate Manager(ACM)로 가져온 인증서를 사용하도록 Elastic Load Balancer를 구성했습니다. 각 인증서가 만료되기 30일 전에 회사 보안 팀에 알림이 전송되어야 합니다. 솔루션 아키텍트는 이 요구 사항을 충족하기 위해 무엇을 권장해야 합니까?

- **A.** 인증서가 만료되기 30일 전부터 매일 Amazon Simple Notification Service(Amazon SNS) 주제에 사용자 지정 메시지를 게시하도록 ACM에 규칙을 추가합니다.
    
- **B.** 30일 이내에 만료되는 인증서를 확인하는 AWS Config 규칙을 생성합니다. AWS Config가 규정을 준수하지 않는 리소스를 보고할 때 Amazon Simple Notification Service(Amazon SNS)를 통해 사용자 지정 알림을 호출하도록 Amazon EventBridge(Amazon CloudWatch Events)를 구성합니다.
    
- **C.** AWS Trusted Advisor를 사용하여 30일 이내에 만료되는 인증서를 확인합니다. 검사 상태 변경에 대한 Trusted Advisor 지표를 기반으로 Amazon CloudWatch 경보를 생성합니다. Amazon Simple Notification Service(Amazon SNS)를 통해 사용자 지정 알림을 보내도록 경보를 구성합니다.
    
- **D.** 30일 이내에 만료되는 모든 인증서를 감지하는 Amazon EventBridge(Amazon CloudWatch Events) 규칙을 생성합니다. AWS Lambda 함수를 호출하도록 규칙을 구성합니다. Amazon Simple Notification Service(Amazon SNS)를 통해 사용자 지정 알림을 보내도록 Lambda 함수를 구성합니다.

>[!.questions]- 정답
>정답 : B
>가져온 인증서는 AWS가 자동으로 갱신해주지 않기 때문에, 별도의 만료 모니터링 시스템이 필수
>AWS Config
> : 리소스의 설정 상태가 정책에 부합하는지 실시간으로 확인하는 서비스
> `acm-certificate-expiration-check` 라는 관리형 규칙을 사용하여, 인증서 만료 며칠 전부터 '비준수'로 판단해서 이벤트를 생성한다. 이 이벤트를 EventBridge가 감지하고 SNS를 통해 알림을 보낸다.

>[!.questions]- 문항별 풀이
> A : ACM에는 해당 메시지를 게시하는 규칙이 존재하지 않음
> C : Trusted Advisor 에 인증서 만료 확인 항목이 ㅆ지만, 알림 생성 로직 복잡함
> D : EventBridge는 이벤트를 감지하는 것이 아니라 전달하는 역할임. ACM은 기본적으로 45일 전부터 이벤트를 감지시키지만, 30일전부터 감지하려면 Lambda가 매번 확인해야하므로 관리 부담큼


## 문제 3

한 회사의 동적 웹사이트가 미국의 온프레미스 서버를 사용하여 호스팅되고 있습니다. 이 회사는 유럽에서 제품을 출시할 예정이며, 새로운 유럽 사용자를 위해 사이트 로딩 시간을 최적화하려고 합니다. 사이트의 백엔드는 미국에 유지되어야 합니다. 제품 출시가 며칠 남지 않았으며 즉각적인 솔루션이 필요합니다. 솔루션 아키텍트는 무엇을 권장해야 합니까?

- **A.** us-east-1에 Amazon EC2 인스턴스를 시작하고 사이트를 마이그레이션합니다.
    
- **B.** 웹사이트를 Amazon S3로 이동합니다. 리전 간 교차 리전 복제(Cross-Region Replication)를 사용합니다.
    
- **C.** 온프레미스 서버를 가리키는 사용자 지정 오리진과 함께 Amazon CloudFront를 사용합니다.
    
- **D.** 온프레미스 서버를 가리키는 Amazon Route 53 지리 근접(geoproximity) 라우팅 정책을 사용합니다

>[!.questions]- 정답
>정답 : C
>정답 조건 : 지연시간 최적화 + 기존 인프라 유지 + 빠른 배포
>CloudFront
>- 엣지 로케이션을 통해 콘텐츠를 캐싱하여 사용자와 가장 가까운 곳에서 데이터를 전달
>	- 미국 본 서버가 아닌, 유럽 내 엣지 로케이션에서 콘텐츠를 받을 수 있음
>- 여기서 동적 컨텐츠는 미국 서버까지 가야하긴 하지만, 엣지 로케이션의 AWS 전용 케이블로 통신하기 때문에 더 빠름
>
>Custom Origin
>- AWS 서비스뿐만아니라, 온프레미스에 있는 외부 서버도 'Custom Origin'으로 설정하여 사용 가능

>[!.questions]- 문항별 풀이
> B : S3 는 정적 웹 호스팅에 적합
> D : 라우팅 정책은 사용자를 특정 서버로 안내할 뿐, 물리적인 거리로 인한 네트워크 지연 시간 자체를 줄여주지 않는다.


## 문제 4

한 회사가 기존의 3계층 웹 아키텍처 비용을 절감하고자 합니다. 웹, 애플리케이션 및 데이터베이스 서버는 개발, 테스트 및 프로덕션 환경을 위해 Amazon EC2 인스턴스에서 실행되고 있습니다. EC2 인스턴스는 피크 시간에 평균 30%, 피크가 아닌 시간에는 평균 10%의 CPU 사용률을 보입니다. 프로덕션 EC2 인스턴스는 하루 24시간 실행됩니다. 개발 및 테스트 EC2 인스턴스는 매일 최소 8시간 동안 실행됩니다. 회사는 개발 및 테스트 EC2 인스턴스를 사용하지 않을 때 중지하도록 자동화를 구현할 계획입니다. 다음 중 회사의 요구 사항을 가장 비용 효율적으로 충족하는 EC2 인스턴스 구매 솔루션은 무엇입니까?

- **A.** 프로덕션 EC2 인스턴스에는 스팟 인스턴스를 사용합니다. 개발 및 테스트 EC2 인스턴스에는 예약 인스턴스를 사용합니다.
    
- **B.** 프로덕션 EC2 인스턴스에는 예약 인스턴스를 사용합니다. 개발 및 테스트 EC2 인스턴스에는 온디맨드 인스턴스를 사용합니다.
    
- **C.** 프로덕션 EC2 인스턴스에는 스팟 블록(Spot blocks)을 사용합니다. 개발 및 테스트 EC2 인스턴스에는 예약 인스턴스를 사용합니다.
    
- **D.** 프로덕션 EC2 인스턴스에는 온디맨드 인스턴스를 사용합니다. 개발 및 테스트 EC2 인스턴스에는 스팟 블록을 사용합니다.

>[!.questions]- 정답
>정답 : B
> 프로덕션(운영) EC2 인스턴스
> 상태: 하루 24시간, 일주일 내내 중단없이 작동
> -> 가동시간이 일정한 워크로드는 '예약 인스턴스'나 'Saving Plans'를 사용하여 긴 약정 기간을 가진 타입이 적합
> 
> 개발 및 테스트 EC2 인스턴스
> 상태 : 매일 최소 8시간 작동, 사용안할 땐 중지
> -> 사용한 시간에만 과금나오고, 사용안할 땐 중지 가능

## 문제 8

한 설문조사 회사가 수년 동안 미국 내 여러 지역에서 데이터를 수집해 왔습니다. 이 회사는 크기가 3TB이고 계속 증가하고 있는 Amazon S3 버킷에 데이터를 호스팅합니다. 회사는 S3 버킷을 보유한 유럽의 마케팅 회사와 데이터를 공유하기 시작했습니다. 회사는 데이터 전송 비용을 가능한 한 낮게 유지하고자 합니다. 어떤 솔루션이 이러한 요구 사항을 충족합니까?

- **A.** 회사의 S3 버킷에 요청자 지불(Requester Pays) 기능을 구성합니다.
    
- **B.** 회사의 S3 버킷에서 마케팅 회사의 S3 버킷 중 하나로 S3 교차 리전 복제(Cross-Region Replication)를 구성합니다.
    
- **C.** 마케팅 회사가 회사의 S3 버킷에 액세스할 수 있도록 마케팅 회사에 교차 계정 액세스(cross-account access)를 구성합니다.
    
- **D.** 회사의 S3 버킷에서 S3 Intelligent-Tiering을 사용하도록 구성합니다. S3 버킷을 마케팅 회사의 S3 버킷 중 하나와 동기화(Sync)합니다.

>[!.questions]- 정답
>정답 : A
>기본 요금 부과 방식 : S3에서 데이터를 다운하면, 데이터를 보관하고 있는 '버킷 소유자(회사)'가 데이터 전송 비용을 부담한다.
>해결 : 요청자 지불 기능으로 데이터를 요청하고 다운하는 상대방(마케팅 회사)이 요금을 부담하도록 한다.

>[!.questions]- 문항별 풀이
>B : S3 교차 리전 복제를 하더라도 복제하는 과정에서 발생하는 데이터 전송 비용을 버킷 소유자가 지불해야 한다.
>C : 교차 계정 엑세스로 마케팅이 접근은 가능하지만, 요금은 회사가 부담
>D : Intelligent-Tiering은 저장 비용을 최적화하는 것. 전송 비용과는 무관


## 문제 9

한 회사가 Amazon S3를 사용하여 기밀 감사 문서를 저장합니다. S3 버킷은 최소 권한 원칙에 따라 감사 팀 IAM 사용자 자격 증명에 대한 액세스를 제한하기 위해 버킷 정책을 사용합니다. 회사 관리자들은 S3 버킷 내 문서의 우발적인 삭제를 우려하고 있으며 더 안전한 솔루션을 원합니다. 솔루션 아키텍트는 감사 문서를 보호하기 위해 무엇을 해야 합니까?

- **A.** S3 버킷에서 버전 관리(versioning) 및 MFA 삭제(MFA Delete) 기능을 활성화합니다.
    
- **B.** 각 감사 팀 IAM 사용자 계정의 IAM 사용자 자격 증명에 다중 인증(MFA)을 활성화합니다.
    
- **C.** 감사 날짜 동안 s3:DeleteObject 작업을 거부하도록 감사 팀의 IAM 사용자 계정에 S3 수명 주기(Lifecycle) 정책을 추가합니다.
    
- **D.** AWS Key Management Service(AWS KMS)를 사용하여 S3 버킷을 암호화하고 감사 팀 IAM 사용자 계정이 KMS 키에 액세스하지 못하도록 제한합니다.

>[!.questions]- 정답
>정답 : A
>S3 버전 관리 
>기능 : 파일을 덮어쓰거나 삭제하더라도 이전 버전이 그대로 보존
>효과 : 누군가 실수로 파일을 지워도 '삭제 마커'만 생성되고 실제 데이터는 살아있어서 복구 가능
>
>MFA 삭제
>기능 : 객체의 버전을 영구적으로 삭제하거나 버전 관리 설정을 변경하려고 할 때, MFA 인증
>효과 : 물리적인 MFA 장치가 없으면 삭제 불가능(가장 높은 수준의 삭제 방지 솔류션)
