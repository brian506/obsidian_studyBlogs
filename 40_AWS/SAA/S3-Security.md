## S3 Security — SAA 핵심 정리

### 1. 암호화 4가지 방식 ⭐⭐⭐ (초빈출)

|방식|키 관리|HTTP 헤더|특징|
|---|---|---|---|
|**SSE-S3**|AWS 완전 관리|`x-amz-server-side-encryption: AES256`|**신규 버킷/객체 기본값**, AES-256|
|**SSE-KMS**|AWS KMS (사용자가 키 제어 가능)|`x-amz-server-side-encryption: aws:kms`|CloudTrail로 키 사용 감사 가능|
|**SSE-C**|**고객이 직접 제공**|매 요청마다 헤더에 키 포함|**HTTPS 필수**, AWS는 키 저장 안 함|
|**Client-Side**|**고객이 완전 관리**|-|업로드 전 직접 암호화, S3에 도달할 때 이미 암호화된 상태|

**SSE-KMS 주의사항 — 시험 포인트:**

- 업로드: `GenerateDataKey` API 호출
- 다운로드: `Decrypt` API 호출
- KMS 초당 할당량에 포함됨 (리전별 5,500 / 10,000 / 30,000 req/s)
- 처리량 매우 높은 버킷에서 KMS 스로틀링 발생 가능 → 할당량 증가 요청 필요

**버킷 정책 vs 기본 암호화 우선순위:** 버킷 정책이 항상 먼저 평가됨. 암호화 헤더 없는 업로드를 버킷 정책이 거부하면 기본 암호화 단계까지 도달하지 못하고 차단됨.

**전송 중 암호화**: HTTP 엔드포인트(암호화 없음) vs HTTPS 엔드포인트(암호화). **SSE-C는 HTTPS 필수.**

### 2. CORS ⭐⭐ (시험에 자주 나옴!)

**Origin = 프로토콜 + 도메인 + 포트**. 웹 브라우저 보안 메커니즘으로 다른 오리진 간의 요청을 제어함.

- Same origin: `http://example.com/app1` ↔ `http://example.com/app2` (프로토콜·호스트·포트 모두 동일)
- Different origin: `http://www.example.com` ↔ `http://other.example.com`

**S3 CORS 시나리오:** 클라이언트가 정적 웹사이트 버킷(버킷 A)에 접속하고, 거기서 다른 S3 버킷(버킷 B)의 이미지를 요청하는 경우 → **버킷 B에 CORS 헤더를 설정**해야 함. 특정 오리진 또는 `*`(전체)로 허용 가능.

> **시험에 그대로 나오는 패턴**: "S3에서 다른 S3로 자원 요청 시 오류" → CORS 설정 문제

---

### 3. MFA Delete ⭐

**버전 관리 활성화 필수.** MFA가 **필요한** 작업과 **필요 없는** 작업을 구분하는 게 핵심.

|MFA 필요|MFA 불필요|
|---|---|
|객체 버전 **영구 삭제**|버전 관리 활성화|
|버전 관리 **중단**|삭제된 버전 목록 조회|

- MFA Delete 활성화/비활성화 권한: **버킷 소유자(루트 계정)에게만** 있음
- 특정 객체 버전의 영구 삭제 방지 용도

---

### 4. S3 Access Logs ⭐

감사 목적으로 S3 버킷의 **모든 요청(승인/거부 무관)** 을 다른 버킷에 로깅. 로깅 대상 버킷은 **같은 리전**에 있어야 함.

> ⚠️ **절대 주의**: 로깅 버킷을 모니터링하는 버킷과 **동일하게 설정하면 안 됨** → 로깅 루프 발생 → 버킷 크기 기하급수적 증가 + 요금 폭탄

---

### 5. Pre-Signed URLs ⭐⭐

S3 객체에 일시적으로 접근할 수 있는 **서명된 URL**. URL을 받은 사람은 URL 생성자의 권한을 상속받아 GET/PUT 가능.

**만료 시간:**

- S3 콘솔: 최대 **720분(12시간)**
- AWS CLI: 기본 3600초, 최대 **604800초(168시간/7일)**

**사용 사례:** 프리미엄 영상 다운로드 허용, 특정 사용자에게 일시적 업로드 허용, 동적으로 변하는 사용자 목록에 파일 공유

---

### 6. S3 Glacier Vault Lock vs S3 Object Lock ⭐⭐

둘 다 **WORM (Write Once Read Many)** 모델. 한 번 쓴 데이터를 수정/삭제 못 하게 잠그는 개념.

**Glacier Vault Lock**: Glacier 볼트 전체에 잠금 정책 적용. 한 번 잠그면 정책 자체를 수정/삭제 불가. 규정 준수, 데이터 보존용.

**S3 Object Lock** (버전 관리 필수): 객체 단위로 잠금. 두 가지 보관 모드가 있음.

|모드|설명|
|---|---|
|**Compliance (규정 준수)**|**루트 포함 누구도** 삭제/덮어쓰기 불가. 보존 모드 자체도 변경 불가. 보존 기간 단축 불가|
|**Governance (거버넌스)**|대부분 사용자는 불가. **특별 권한을 가진 일부 사용자**는 보존 기간 변경/삭제 가능|

**Legal Hold (법적 보관):** 보관 기간과 무관하게 **무기한** 보호. `s3:PutObjectLegalHold` IAM 권한으로 설정/해제 가능.

> **시험 구분 포인트**: "루트도 못 삭제" → **Compliance** / "일부 권한자는 가능" → **Governance**

---

### 7. S3 Access Points ⭐

버킷에 대한 접근을 사용자/그룹별로 분리 관리. 각 액세스 포인트마다 고유한 DNS 이름과 정책을 가짐. 하나의 복잡한 버킷 정책 대신 여러 개의 단순한 액세스 포인트 정책으로 관리가 훨씬 쉬워짐.

예: Finance 팀용 액세스 포인트, Sales 팀용 액세스 포인트를 각각 별도 정책으로 운영.

---

### 8. S3 Object Lambda ⭐

S3에서 객체를 꺼내 애플리케이션에 전달하기 **직전에 Lambda 함수로 객체를 변환**. 버킷은 하나만 있으면 되고 S3 Access Point + Lambda를 연결하는 구조.

사용 사례: PII(개인식별정보) 삭제, XML → JSON 변환, 호출자별 이미지 동적 크기 조정 및 워터마킹.

---

### 시험 TIP 최종 정리

- **SSE-C는 HTTPS 필수** — HTTP로 보내면 거부됨
- **SSE-KMS = KMS 할당량 소모** → 고처리량 버킷에서 스로틀링 주의
- **기본 암호화 < 버킷 정책** 순서로 평가
- **S3 CORS** = 다른 S3 버킷 자원 요청 시 대상 버킷에 설정 필요
- **MFA Delete** 활성화/비활성화 = **루트 계정만** 가능
- **로깅 루프** = 모니터링 버킷과 로깅 버킷을 같게 설정하면 안 됨
- **Pre-Signed URL 최대 만료**: 콘솔 12시간, CLI **7일**
- **Object Lock Compliance** = 루트도 못 건드림 / **Governance** = 특권 사용자는 가능
- **Object Lambda** = S3에서 꺼낼 때 실시간 변환
