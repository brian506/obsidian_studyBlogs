## Route 53

### Route 53으로만 로드밸런싱할 때
Route 53은 DNS 캐싱으로 인해 죽은 IP에게 계속 요청을 보낸다.
- 헬스체크 불가

**단순 라우팅(Simple Routing)**
- 하나의 도메인에 하나의 리소르 매핑
- *상태 검사 지원 안함(헬스체크 X)*

**가중치 기반 라우팅(Weighted Routing)**
- 각 인스턴스에 지정한 비율대로 트래픽 분산해서 보냄
- 카나리 배포, 블루-그린 배포

**장애 조치 라우팅(Failover Routing)**
- Active-Passive 구조
- 평소에는 메인(active)로 보내다가 *상태 검사 실패 시* 재해 복구용(Passive)로 보냄
- 다운타임 해결

**다중 가치 응답 라우팅(Multi-value Answer Routing)**
- 정상적인 리소스 IP를 최대 8개까지 무작위로 반환
- *DNS 수준에서 간단한 로드밸런싱 구현*

**지연 시간 기반 라우팅(Latency-based Routing)**
- 네트워크 지연 시간이 가장 짧은 리전으로 트래픽 보냄

**지리 위치 라우팅**
- 대륙이나 국가 기반 리전으로 보냄
- 물리적 위치가 기준

## 대규모 요청 처리 아키텍처(티켓팅 플젝 클라우드 버전)

### 웹 서버
웹 서버는 요청이 많아지면, Auto Scaling으로 CPU사용량에 따라 증설한다.

### 백그라운드 서버
DB 처리 시에 발생하는 락 때문에 대기하게 될 때는 CPU는 사용하지 않는다. 
이렇기 때문에 **CPU 사용량에 따라 ASG를 구성하면 안된다.**

**디커플링 구성** : SQS에 작업 큐를 적재하고, 큐에 메시지가 얼마나 쌓여있는지(Backlog)를 인스턴스 개수로 나눈 값을 기준으로 ASG를 구성한다.
- Backlog / 인스턴스 수 기준
백로그 : 밀려 있는 일의 총량(총 메시지)

## 태그 붙이는 이유

- 비용 배분
    - 한 계정에서 여러 플젝할 때, 플젝에 대한 태그별 비용 확인 가능
- 리소스 그룹화 및 관리
    - 운영 or 개발 서버를 그룹화하여 관리
- 세밀한 보안 제어
    - 태그에 따라서 리소스 수정 및 보안 관리
- 자동화
    - 태그에 따라서 해당 볼륨만 찾아서 자동 수행 가능

**Resource Groups Tag Editor** : 태그를 기반으로 리소스를 관리하는 도구

## 재해 복구 솔루션

1. **Backup & Restore:** 가장 저렴, 가장 느림 (RPO/RTO 수 시간~수 일)
2. **Pilot Light:** 핵심(DB 등)만 켜둠. 장애 시 나머지를 신속히 가동. (RPO/RTO 수십 분)
3. **Warm Standby:** 작은 규모로 전체 시스템을 가동 중. 장애 시 바로 확장. (RPO/RTO 수 분)
4. **Multi-Site (Hot Standby):** 두 곳 모두 풀가동. 즉시 전환. (RPO/RTO 실시간, 가장 비쌈)

AWS Backup 같은 **스냅샷 기반** 복원 특징
- 데이터양에 따라 시간 많이 걸림
- 스냅샷 생성 시점과 장애 시점 사이의 데이터가 모두 유실되므로 30분 데이터 유실 방지 보장 어려움

 Aurora 교차 리전 읽기 복제본
 - 비동기식으로 실시간 복제

## 보안그룹 / NACL

### 보안그룹
- 허용 규칙만 만들 수 있음
- **Statefull(상태 저장) : Inbound 허용하면, Outbound 자동 허용**

### 네트워크 ACL (NACL)
- 서브넷 단위
- 허용 규칙, 거부 규칙 가능
- Stateless(상태 비저장) : Inbound,Oubound 따로 설정
	- 클라이언트에게 나가는 아웃바운드 포트도 따로 열어줘야함


## HTTPS 외의 암호화

### CloudFront 필드 수준 암호화
HTTPS는 전송 중인 데이터를 암호화하지만, 데이터가 웹서버나 로드밸런서에 도착하여 SSL/TLS가 종료되면 데이터는 일시적으로 복호화된 평문 상태가 된다. 
-> **필드 수준 암호화**는 데이터가 이미 암호화되어 원본까지 전달

## Amazon CloudWatch 복합 경보  
>기능 : 여러 개의 개별 경보를 논리 연산자로 결합하여 하나의 상위 경보를 만드는 기능  

**경보 방식**  
경보 1 : CPU 사용률 > 50 일 때 발생  
경보 2 : 디스크 읽기 IOPS 높으면 발생  
복합 경보 : 경보1 AND 경보2  
-> 이렇게 하면 복합 경보일 때의 조건만 알림을 발생시킴

## 악성 봇 요청 차단 방식

### AWS WAF : API Gateway 바로 앞단에서 L7 공격 방어

- 해결 방식 : 특정 봇넷이 사용하는 IP 주소 대역을 차단
    - 비정상 패턴 탐지 : 짧은 시간 동안 많은 요청을 보내는 봇을 자동으로 차단
    - Bot Control : AWS WAF에는 이미 알려진 봇들의 패턴을 식별하여 차단하는 전용 규칙 세트가 있어서 미리 걸러냄

### API 키 및 사용량 계획 : 누가 API를 호출할 수 있는 지 식별하고, 호출 횟수 제어

- 해결 방식 :
    - API 키 : 허가된 클라이언트에게만 API 키를 발급하고, 요청 헤더에 키가 포함될때만 API Gateway 통과
    - 사용량 계획 : 특정 키를 가진 사용자가 분당/일단 보낼 수 있는 요청 수를 제한하여, 특정 사용자의 요청이 급증하더라도 전체 시스템에 영향을 주지 않음
