#면접 
## Q. 예매 요청을 Kafka로 비동기 처리할 때, 사용자에게는 어떤 응답을 내려줬나요?

- PG 결제 이후 단계까지 갔을 경우에 예매 성공 응답을 줍니다. 

### 꼬리: 사용자가 "예매 성공" 응답을 받았는데 DB에 데이터가 없는 시간이 존재하는 거잖아요. 이 구간에서 장애가 나면 어떻게 처리했나요?
- 컨슈머가 메시지를 소비할 때 재시도 3번 정책을 수행합니다. 재시도마저 실패했을 경우에 DLT 로 가게 되면, 해당 메시지를 바라보고 있는 컨슈머가 다시 발행을 시도합니다.
### 꼬리: 이 방식이 결국 최종 일관성(Eventual Consistency)을 허용하는 건데, 이게 괜찮은 설계라고 생각하시나요?
- 결제는 PG사에서 이미 완료된 상태이고, DB 저장은 최종적으로 반드시 이루어지도록 재시도와 DLT로 보장했습니다. 사용자 입장에서는 결제가 완료됐다는 사실이 중요하고, DB 동기화는 수 초 내에 완료되므로 이 서비스 특성상 허용 가능한 트레이드오프라고 판단했습니다

## Q. Kafka Batch 처리와 Bulk INSERT를 도입했는데, Lag이 여전히 발생했던 근본 원인이 뭐였나요?

- Bulk Insert 로 처리량은 늘었지만, 여전히 DB I/O 처리 속도로 인해 발행 속도와 소비 속도의 차이 때문에 Lag 가 발생했습니다.

### 꼬리: Consumer를 Scale-out하는 방법은 왜 선택하지 않으셨나요?
- Consumer 개수 또한 파티션 수에 맞춰서 늘렸습니다.
### 꼬리: Kafka 파티션 수와 Consumer 수의 관계를 설명해주세요.
- 컨슈머 하나는 파티션 하나에 대해서 작업을 수행합니다. 따라서 컨슈머가 놀고 있지 않게 하기 위해서는 컨슈머 수를 파티션 수보다 같거나 적게 할당해야 합니다. 파티션 수와 같게 설정하는 것이 이상적입니다.
### 꼬리: Bulk INSERT 시 배치 크기(batch size)는 어떻게 결정했나요?
- 부하 테스트를 통해 여러 배치 사이즈를 실험했고, 너무 크면 단일 트랜잭션 처리 시간이 길어져 타임아웃 위험이 있고, 너무 작으면 I/O 횟수가 늘어나는 트레이드오프가 있어서 실측 기반으로 결정했다

## Q. Thread Sleep으로 Polling 대기를 시도했을 때 스레드 풀이 고갈된 이유를 자세히 설명해주세요.

- 결제 요청이 들어올 때마다 새 스레드가 할당되고, 그 스레드가 sleep 상태로 DB 동기화를 기다리는 동안 반납되지 않습니다. 요청이 계속 들어오면 sleep 중인 스레드가 누적되어 새 요청을 처리할 스레드가 없어지고 서버 전체가 멈추게 됩니다.
### 꼬리: 스레드 풀 크기를 늘리는 방안은 생각해보시지 않았나요?
- 현재 상황은 비동기적으로 수행되는 시간 차이 발생 때문에 발생한 문제라서 스레드 풀 크기를 늘린다고 해결되지 않을 거라고 생각합니다.

### 꼬리: 이 상황에서 CompletableFuture나 WebFlux를 고려하지 않은 이유가 있나요?
- CompletableFuture 를 통해 응답을 미리 줘도 되지만, 외부 PG 사 호출로 인해서 받은 응답에 따라서 섣불리 성공 응답을 수행하지 못하는 상황이었고, WebFlux 를 사용하게 되면 전체적인 구조를 바꾸는 비용 때문에 고려하지 않았습니다.

## Q. Kafka를 도입했는데 Producer가 메시지를 발행하고 실패하면 어떻게 처리했나요?

- 재시도 3번으로 처리하도록 설정했습니다.

### 꼬리: Kafka acks 설정을 어떻게 했나요? acks=all로 하면 어떤 트레이드오프가 생기나요?
- acks = 1 로 설정했습니다. acks = all 로 설정하게 되면, 메시지 유실 가능성은 줄어들지만 지연속도가 늘어납니다.

### 꼬리: Consumer가 메시지를 처리하다가 실패하면 offset은 어떻게 관리했나요?
- acks=1로 설정했고, 메시지 유실보다 처리 속도를 우선시했습니다. 대신 Consumer에서 수동 커밋 방식으로 처리 완료 후 offset을 커밋하도록 했고, 실패 시 재시도 3회 후 DLT로 보내는 방식으로 유실을 방지했습니다.
