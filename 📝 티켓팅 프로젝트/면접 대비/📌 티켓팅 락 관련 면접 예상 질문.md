## Q. 초기에 synchronized, 비관적 락, 낙관적 락, DB 원자적 연산을 비교했는데, 낙관적 락이 DB 원자적 연산보다 TPS가 낮았던 이유가 뭐라고 생각하세요?

- 낙관적락은 동시 요청이 몰리는 상황에서 version 불일치로 인한 충돌이 매우 빈번하게 발생하고, 충돌한 트랜잭션이 처음부터 재시도 하면서 DB I/O 가 급격히 증가합니다. DB 원자적 연산은 충돌 자체가 발생하지 않기 때문에 상대적으로 높은 TPS 가 나왔습니다.
### 꼬리: 낙관적 락에서 충돌이 자주 발생하면 어떤 문제가 생기나요?
- 충돌이 많아지면 재시도가 많이 발생하여 급격한 DB 부하가 발생하여 전체 시스템 응답 시간이 늘어납니다.
### 꼬리: 낙관적 락이 유리한 상황은 언제인가요?
- 낙관적 락은 저장 시점에 version 만 비교하므로 충돌 가능성이 낮은 상황에서 읽기가 쓰기보다 많은 경우, 또는 사용자가 데이터를 오래 가지고 수정하는 것 처럼 락 유지 시간이 길어질 수 있는 상황이 유리합니다. 

## Q. synchronized, 비관적 락, 낙관적 락, DB 원자적 연산을 비교했는데, 왜 synchronized는 정합성이 보장되지 않았나요?

- Spring AOP 기반의 @Transactional과 synchronized 혼용했기 때문입니다. synchronized로 락을 획득한 스레드가 로직을 마치고 락을 반납하더라도, 아직 트랜잭션 커밋이 완료되지 않은 틈이 발생합니다. 이때 다른 스레드가 락을 획득하고 DB를 읽게 되면, 앞선 스레드의 변경 사항이 반영되지 않은 과거 데이터를 읽게 되어 정합성이 깨지게 됩니다.

### 꼬리: synchronized가 단일 서버에서도 정합성이 깨진 건가요, 멀티 서버 환경이라서 깨진 건가요?
- 단일 서버에서 트랜잭션 위의 메서드 내에서 수행되어 정합성이 깨졌습니다. 스레드 기반으로 락을 획득/해제하기 때문에 멀티 서버 환경에서도 정합성이 깨질 것 입니다.

## Q. DB 원자적 연산이 88.2 TPS로 가장 높았는데 왜 이걸 채택하지 않고 Redis로 전환했나요?

- 단일 서버 기준 88TPS, 평균응답 시간이 약 4초로 측정되었고, 스레드/커넥션 점유 시간이 길어져 동시 요청이 몰릴  경우 커넥션 풀 고갈로 이어질 수 있다고 판단했습니다. 메모리 기반인 Redis로 재고 차감을 이전하면 DB부하를 줄이면서 목표 TPS를 달성할 수 있다고 판단했습니다.

### 꼬리: 4초 지연시간이 왜 리소스 고갈 위험으로 이어지나요? 구체적으로 설명해주세요.
- 동기식으로 많은 요청을 4초의 응답시간을 가지게 된다면, 스레드와 커넥션 점유로 인한 리소스 고갈이 이어질 것 입니다.
### 꼬리: TPS 목표치를 300으로 잡은 기준이 뭔가요? 어떻게 산정했나요?
- 정확한 트래픽 데이터가 없는 프로젝트였기 때문에 가정 기반으로 산정했습니다. 티켓팅 오픈 시 동시 접속자를 3,000명으로 가정했고, 그 중 10%가 동시에 예매 요청을 보낸다고 가정하면 초당 300건이 됩니다. 이를 단일 서버에서 감당할 수 있는 최소 목표치로 설정했습니다.

## Q. Gap Lock이 왜 발생했는지 메커니즘을 설명해주세요.왜 expiredAt 단일 인덱스로 바꾸면 범위가 분리되나요?

- 현재 쿼리는 인덱스 내에 있는 expiredAt 컬럼에 대해서 현재날짜 < expiredAt 인 모든 데이터를 조회합니다. 여기서 조회할 때 innoDB는 현재 날짜 < expiredAt을 만족하는 모든 데이터에 gap lock을 겁니다. 
- 기존에는 status 가 선두컬럼에 있어서 status 기준으로 먼저 정렬되었지만, expiredAt 단일 인덱스를 걸게 되면 expiredAt 기준으로 정렬하게 됩니다. 스케줄러 트랜잭션은 유효기간이 지난 데이터를 탐색하고, 결제 트랜잭션은 유효기간이 지나지 않은 컬럼들을 대상으로 탐색하기 때문에 탐색 범위가 서로 분리됩니다.

### 꼬리: Gap Lock은 어떤 격리 수준(Isolation Level)에서 발생하나요?
- Gap lock 은 REPEATABLE READ 수준에서 발생합니다. REPEATABLE READ 는 같은 트랜잭션 안에서 조회했을 때 항상 같은 값을 반환해야 합니다. 그렇기 때문에 다른 트랜잭션이 해당 범위에 새로운 값을 넣지 못하도록 쓰기 쿼리를 제한하기 위해 gap lock 을 걸어야 합니다.

### 꼬리: Gap Lock 대신 Next-Key Lock이 걸리는 조건은 뭔가요?
- Next-Key Lock은 Gap Lock + Record Lock의 조합입니다. 인덱스 레코드 자체와 그 앞의 gap을 함께 잠급니다. InnoDB는 기본적으로 범위 스캔 시 Next-Key Lock을 걸고, 실제 레코드가 없는 구간만 있을 때 Gap Lock으로 축소됩니다.

### 꼬리: Gap Lock은 INSERT만 막나요, UPDATE도 막나요?
- UPDATE도 막습니다.

### 꼬리: READ COMMITTED로 격리 수준을 낮추지 않은 이유가 뭔가요?
- 특정 트랜잭션만 격리 수준을 낮추게 되면 다른 팀원이 봤을 때 혼돈이 있을 수 있고, 근본적인 문제로 탐색 범위를 분리시켜 인덱스에서 status를 제외시키는 것이 낫다고 판단했습니다.

## Q. 트래픽이 적은 시간대에 복구 로직을 수행하는 정책으로 변경했는데, 그 사이 취소된 티켓 재고는 어떻게 관리됐나요?

- 1차적으로 티켓 예매 진행 후에, 취소된 티켓과 결제까지 완료하지 못한 티켓에 대해서 2차적으로 티켓 예매를 진행할 예정입니다.

## Q. 트랜잭션 범위를 3단계로 분리했다고 하셨는데, 그 사이에 장애가 나면 데이터 일관성은 어떻게 보장했나요?

- 레디스 재고 감소와 티켓 상태 UPDATE 정합성을 맞추기 위해서, UPDATE 커밋 이후에 레디스 재고 감소를 수행하도록 했습니다. 만약 레디스 재고 감소에서 네트워크 오류로 인해 반영이 되지 않는다면, DB에 반영된 취소 티켓에 대해 개수를 세고 다시 레디스 재고를 수정하는 방향으로 보장했습니다.

### 꼬리: 1단계(조회)와 3단계(쓰기) 사이에 다른 트랜잭션이 끼어들 수 있는 race condition은 없나?
- 현재 구조에서 조회와 쓰기 사이 race condition 가능성은 있습니다. 하지만 현재 정책상 재고 복구 스케줄러는 시간대를 분리해서 수행되기 때문에 race condition 위험 가능성이 낮다고 판단했습니다. 

## Q. [status, expiredAt] 복합 인덱스에서 쓰기 시 데드락이 발생했다고 하셨는데, 인덱스 구조와 락의 관계를 설명해주세요.

- 해당 복합 인덱스에서 범위 탐색 시에 next-key lock 으로 인해 데드락이 발생했습니다. 인덱스 테이블은 두 컬럼에 대해서 정렬되어 있고 REPEATABLE READ 격리수준에서 유령 데이터가 발생하지 않게 하기 위해 next-key lock, gap lock, row lock 으로 데이터 정합성을 지킵니다.

### 꼬리: 인덱스가 없었다면 데드락이 발생하지 않았을까요? (Full Scan과 Table Lock의 관계)
- 인덱스가 없었으면 Full Scan 으로 인한 next-key lock으로 마찬가지로 데드락이 더 심하게 발생했을 것입니다. Full Scan 을 하게 되면 모든 테이블의 레코드를 탐색하는 동안 해당 테이블에 락이 걸려버려서 데드락이 발생했을 것입니다.

## Q. Full Table Scan이 발생했을 때 왜 커넥션 풀 고갈로 이어지나요?

- Full Table Scan을 하게 되면 해당 테이블에 락이 걸립니다. 테이블의 모든 레코드들에 대해서 조건에 맞는 탐색을 해야 하므로, 그만큼 수행시간이 길어져 하나의 트랜잭션이 커넥션을 점유하는 시간이 길어지게 되므로 커넥션 풀 고갈이 발생합니다.

### 꼬리: Full Table Scan 시 걸리는 락은 Gap Lock인가요 Table Lock인가요?
- InnoDB는 Full Table Scan 시에도 기술적으로는 테이블 락이 아니라 모든 레코드에 Next-Key Lock을 겁니다. 결과적으로 테이블 전체가 잠기는 효과가 나타나서 사실상 Table Lock과 동일하게 동작합니다. MyISAM은 명시적 Table Lock을 사용하지만 InnoDB는 레코드/갭 락의 집합으로 동작합니다.

## Q. [status, expiredAt] 복합 인덱스에서 status가 선두 컬럼인데, 왜 이 순서로 인덱스를 설계했나요?

- 처음 설계했을 땐, 티켓 상태에 대해서 먼저 정렬한 상태로 수직 탐색하는 것이 인덱스 범위 탐색에 있어서 유리할 것이라고 생각했습니다. 또한 where 조건 절에서 status 를 기준으로 필터링을 선제적으로 하고 있으므로 인덱스 범위 탐색에 있어서 유리할 것이라고 생각했습니다.

### 꼬리: 처음부터 [expiredAt] 단일 인덱스로 설계했다면 데드락이 발생하지 않았을까요?
- 근본적인 원인은 status 로 먼저 정렬하게 되어 expiredAt의 정렬 순서가 꼬이게 되어, 두 트랜잭션 사이의 스캔 범위가 겹치게 되어 데드락이 발생했습니다. 따라서 단일 인덱스로 설계했다면 데드락이 발생하지 않았을 것입니다. 
### 꼬리: 복합 인덱스에서 선두 컬럼 순서가 쿼리 성능에 미치는 영향을 설명해주세요.
- 선두 컬럼에 대한 수직 탐색 이후에, 선두 컬럼의 카디널리티가 클수록 수평 탐색하는 범위가 좁아지게 됩니다. 따라서 스캔을 많이 하지 않아도 되기 때문에 쿼리 성능이 개선될 것입니다. 

## Q. status를 인덱스에서 제거했는데, status 조건으로 조회하는 다른 쿼리 성능에는 영향이 없나요?

- 다른 쿼리들도 후행 컬럼에 대해서 카디널리티가 더 작은 컬럼을 가지고 있습니다. 또한 쓰기 작업이기 때문에 카디널리티가 작은 status 컬럼에 대해서 조회 성능을 향상 하는 것보다는 쓰기 작업에 대한 효율을 높여 인덱스에 포함시키지 않는 것이 맞습니다.

## Q. 격리 수준을 특정 트랜잭션에만 낮추는 것도 가능한데 왜 선택하지 않으셨나요?

- 특정 트랜잭션에만 격리 수준을 낮추게 되면 왜 이 트랜잭션만 다른지 맥락을 파악하기 어려워 유지보수가 힘들 것이라고 판단했습니다. 또한, 단순히 status 컬럼을 인덱스에서 제거하는 것이 근본적인 원인에 대해서 가장 정확한 해결 방안입니다.

### 꼬리: Spring에서 트랜잭션별로 격리 수준을 다르게 설정하는 방법을 알고 계세요?
- 해당 메서드 위에 `@Transactional(isolation = Isolation.READ_COMMITTED)` 처럼 설정해주면 됩니다.
### 꼬리: READ COMMITTED로 낮추면 이 서비스에서 실제로 Phantom Read가 발생할 수 있는 시나리오가 있나요?
- 현재 결제 트랜잭션과 티켓 재고 복구 트랜잭션은 서로 수행하는 데이터 범위가 다르기 때문에 Phantom Read는 발생하지 않을 것이라고 생각합니다. 
- 시스템 전체적으로 봤을 때도 티켓 재고 복구 트랜잭션을 티켓 발행과 동시에 수행하게 되었을 땐 phantom read 가 발생할 수 있지만, 현재 정책 상 별도의 시간대에 수행하게 되므로 Phantom read가 발생하지 않습니다.

## Q. 결국 정책 변경으로 해결했는데, 취소된 티켓 재고가 즉시 반영되지 않는 구간이 생기잖아요. 비즈니스적으로 문제가 없나요?

- 티켓 발행하는 비즈니스 로직이 더 중요하다고 생각합니다. 티켓을 취소하는 건보다 새롭게 발행하는 건이 더 많을 것이라고 판단되어 별도의 시간을 두고 수행하는 것이 안정적인 서비스를 운영하는 데에 있어서 더 효과적일 것입니다.

### 꼬리: 취소된 자리를 즉시 재판매하지 못하는 게 손실 아닌가요?
- 트래픽이 몰리는 서비스라면 그만큼 취소표에 대해 재시도 할 사용자가 많을 것이기 때문에 손실이라고 생각하지 않습니다.

## Q. MySQL이 데드락을 감지하고 해소하는 방식을 알고 계세요?

- InnoDB는 대기 그래프를 실시간으로 유지합니다. 트랜잭션들이 서로의 락을 대기하는 순환 구조가 감지되면 즉시 데드락으로 판단합니다. innodb_deadlock_detect 옵션으로 활성화되며 기본값은 ON입니다.
### 꼬리: 데드락이 감지됐을 때 어떤 트랜잭션이 롤백되나요? 그 기준은 뭔가요?
- InnoDB는 롤백 비용이 가장 적은 트랜잭션을 희생자로 선택합니다. 기준은 undo lo의 양으로, 변경한 행 수가 적은 트랜잭션이 롤백됩니다. innodb_deadlock_detect로 감지 후 해당 트랜잭션에 deadlock 에러를 반환하고 애플리케이션에서 재시도 처리를 해야 합니다.
- 희생자가 롤백되면서 순환대기가 해소되기 때문에, 나머지 트랜잭션은 기다리던 락을 획득하고 정상 수행됩니다.
