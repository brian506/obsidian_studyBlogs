## 1. 유통기한 쿼리 최적화

### 핵심 요약
- `DATEDIFF(expiry_date, CURDATE()) = n` → 컬럼 가공으로 인덱스 풀스캔
- `expiry_date = DATE_ADD(CURDATE(), INTERVAL n DAY)` → 우변만 연산, 인덱스 정상 탐색
- 최종 인덱스: `[memberKey, expiry_date]` 복합 인덱스

### 자주 나오는 질문

**Q. ref vs range 차이는?**
- `ref`: 비고유 인덱스를 **동등 조건(=)** 으로 탐색할 때 발생. 해당 값을 가진 여러 row가 있을 수 있어 리프 노드에서 수평 탐색
- `range`: 인덱스를 **범위 조건** (`>`, `<`, `BETWEEN`)으로 탐색할 때 발생. 시작점부터 끝점까지 수평 탐색

> ⚠️ "묶음 단위 스캔"이라는 표현은 부정확. ref의 핵심은 "동등 조건 + 비고유 인덱스"

**Q. status 컬럼을 인덱스에서 뺀 이유?**
- status 카디널리티가 낮아 (`ACTIVE`/`INACTIVE`) 선두 컬럼으로 두면 수평 탐색 범위가 넓어짐
- `[memberKey, expiry_date]` 25ms vs `[memberKey, status, expiry_date]` 27ms → 거의 차이 없는데 인덱스 쓰기 비용 고려

**Q. 옵티마이저가 인덱스를 무시하는 경우는?**
- Random I/O 비용 > Sequential Full Scan 비용으로 판단될 때
- 조회 비율이 높을 때 (전체 데이터의 20~30% 이상)
- 테이블 자체가 매우 작을 때
- 통계 정보가 오래되어 부정확할 때

**Q. 운영 환경 인덱스 추가 시 주의사항?**
- `ALTER TABLE ... ADD INDEX ... ALGORITHM=INPLACE, LOCK=NONE` (Online DDL)
- 일반 DDL은 테이블 락이 발생해 서비스 중단 → Online DDL로 락 없이 추가

**Q. OR 조건 인덱스 동작 방식?**
- MySQL은 OR 조건에 대해 **Index Merge** 최적화 지원
  - `union`: 각 OR 조건 결과를 합집합
  - `intersection`: AND 조건과 혼용 시 교집합
- 각 OR 조건에 대해 개별 인덱스 수직 탐색 후 병합

## 2. Cursor 기반 페이지네이션

### Offset vs Cursor 비교

| 항목 | Offset | Cursor |
|------|--------|--------|
| 특정 페이지 이동 | 가능 | ❌ 불가 |
| 대용량 뒤 페이지 성능 | 느림 (앞 전체 스캔) | 빠름 (커서 이후만 스캔) |
| 실시간 데이터 삽입 시 | 데이터 중복/누락 위험 | 안정적 |
| 구현 복잡도 | 단순 | 상대적으로 복잡 |

**Cursor Pagination 주요 단점 (면접에서 자주 물어봄)**
1. **특정 페이지로 직접 이동 불가** (랜덤 접근 불가)
2. 정렬 기준 컬럼이 고유하지 않으면 중복 발생 가능 → **클러스터링 인덱스(PK) 사용 필수**
3. 검색 조건과 결합이 offset보다 복잡

## 3. 알림 발송 비동기 최적화

### 핵심 흐름
```
Scheduler(@Scheduled) → Cursor 기반 450개 단위 조회
  → Virtual Thread 비동기 발송
  → @Retryable 3회 재시도
  → 실패 시 로그 기록
```

### 자주 나오는 질문

**Q. 왜 가상 스레드를 선택했나?**
- FCM 발송은 **I/O 대기 위주** 작업 → OS 스레드 블로킹 비효율
- 가상 스레드는 I/O 대기 시 캐리어 스레드를 반납 → 수천 개 생성해도 OS 스레드는 소수만 사용
- OS 스레드 1개 ≈ 1MB 메모리, 컨텍스트 스위칭 비용 그대로 → 단순 스레드 풀 확장으로는 근본 해결 불가

**Q. 가상 스레드 주의사항?**
- CPU 연산 위주 작업에는 오히려 오버헤드 증가
- `synchronized` 사용 시 **pinning 현상** (캐리어 스레드에 고정) → `ReentrantLock`으로 대체
- DB 커넥션 풀과 함께 사용 시 커넥션 풀 고갈 주의 (커넥션 수 이상의 가상 스레드 생성 무의미)

**Q. 450개 기준?**
- Firebase 배치 최대 500개 제한 → 여유 두고 450개 설정

**Q. 타임아웃 시 중복 발송 위험은?**
- 타임아웃은 서버 응답을 못 받은 것일 뿐, FCM 서버에서 실제 발송됐을 수 있음
- 알림 특성상 중복 발송이 치명적이지 않아 재시도 정책 유지
- 단, 실제 금전 관련 서비스라면 **멱등성 키**로 중복 방지 필요

## 4. Spring AI & 4계층 아키텍처

### @Async 주의사항
- 같은 클래스 내부 호출 시 프록시를 거치지 않아 동작 안 함 → **별도 Bean으로 분리 필수**
- 비동기 예외는 `AsyncUncaughtExceptionHandler`로 처리 가능

### 4계층 아키텍처 의존성 강제 방법
- 디렉토리 구조만으로는 코드 레벨 강제 **불가**
- 실제 강제 방법:
  - **인터페이스 기반 추상화** (상위 계층은 인터페이스에만 의존)
  - **ArchUnit** 라이브러리로 테스트 코드에서 의존성 방향 검증
  - **모듈 분리** (Gradle multi-module)

### Prometheus pull 방식 (수정된 설명)
- **Pull 방식**: Prometheus가 주기적으로 애플리케이션의 `/actuator/prometheus` 엔드포인트를 **scrape**하여 수집
  - 즉, 애플리케이션이 "보내는" 것이 아니라 Prometheus가 "가져오는" 것
- **Push 방식**: 애플리케이션이 Pushgateway 등 중간 컴포넌트에 능동적으로 데이터를 전송

## 5. 스케줄러 & Scale-out

**Q. 서버 2대 이상 시 스케줄러 중복 실행 해결?**
1. **ShedLock**: DB/Redis에 락 레코드 생성, 별도 Redis 없이도 사용 가능
2. **Quartz 클러스터 모드**: DB 기반 분산 스케줄링
3. **Redis 분산락**: Redisson

**Q. 서버 다운으로 알림 누락 시?**
- `@Scheduled`는 서버 재시작 후 다음 주기까지 대기 → 당일 알림 누락
- 해결: Quartz의 `misfire` 정책으로 누락된 실행 재처리 가능

## 6. 인프라 구성

### Public / Private Subnet 분리 이유
- DB, 모니터링(Prometheus/Grafana) → Private Subnet (외부 직접 접근 차단)
- NAT Gateway: Private → 외부 아웃바운드 허용, 외부 → Private 인바운드 차단

### NAT Gateway vs Internet Gateway
| 항목 | Internet Gateway | NAT Gateway |
|------|-----------------|-------------|
| 대상 | Public Subnet | Private Subnet |
| 양방향 통신 | 가능 | 아웃바운드만 |
| 용도 | 외부 공개 서버 | 내부 서버의 외부 통신 |
