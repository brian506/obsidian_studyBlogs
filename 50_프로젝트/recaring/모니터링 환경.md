#### 두 가지 패러다임: PULL과 PUSH

**PULL (Prometheus 방식)**

- 모니터링 도구가 주기적으로 대상에게 "지표 줘"라고 요청
- 대상은 가만히 있다가 요청 받으면 응답
- 대상이 추가/제거되면 모니터링 도구가 그걸 알아야 함

**PUSH (Loki + Alloy 방식)**

- 대상이 자기 데이터를 모니터링 도구로 적극적으로 보냄
- 모니터링 도구는 받는 입장
- 대상이 어디 있는지 몰라도 됨

#### Prometheus가 App에서 메트릭 가져오는 과정

```
[15초마다 반복]
1. Prometheus가 자기 config 읽음:
   "spring-app.local:8080/actuator/prometheus를 scrape하라"

2. Prometheus가 spring-app.local의 IP 찾음
   → Cloud Map 통해 현재 살아있는 App Task IP 받음
   → 예: 10.0.2.78

3. Prometheus → 10.0.2.78:8080/actuator/prometheus 호출
   ↓ HTTP GET

4. Spring App의 actuator/prometheus 엔드포인트가 응답
   ↓ Prometheus exposition format으로 메트릭 텍스트 반환
   예:
   http_requests_total{method="GET",status="200"} 1234
   jvm_memory_used_bytes 567890

5. Prometheus가 받아서 자기 TSDB에 저장
   → /prometheus-data 볼륨에 저장 (EFS)
```

핵심: **Prometheus가 능동적으로 가서 가져온다.** App은 그냥 자기 메트릭을 `/actuator/prometheus`에 노출해두고 가만히 있음.

#### Redis Exporter

Redis 자체는 Prometheus 메트릭 형식을 모른다. Redis는 자기 형식(INFO 명령어 결과)으로만 정보를 준다. 그래서 **Exporter**가 중간에 들어간다.

```
[Redis Exporter의 역할]
1. Redis Exporter → Redis: "INFO 명령어 줘" (Redis 프로토콜로)
2. Redis → Exporter: 자기 상태 정보 응답
3. Exporter가 그걸 Prometheus 형식으로 변환
4. Exporter가 자기 9121 포트에 /metrics 엔드포인트로 노출

[Prometheus의 동작]
5. Prometheus → Redis Exporter:9121/metrics 호출 (PULL)
6. 변환된 메트릭 받음
```

즉 **Exporter는 "프로토콜 번역기"**다. Redis 외에도 MySQL Exporter, Node Exporter(호스트 메트릭), JMX Exporter(JVM 메트릭) 등 다양함.

#### Node Exporter

호스트 OS 자체의 메트릭(CPU, 메모리, 디스크, 네트워크)을 노출.

```
Node Exporter → 호스트 OS의 /proc, /sys 읽음
            → CPU 사용률, 메모리, 디스크 IO 등 추출
            → :9100/metrics로 노출

Prometheus → Node Exporter:9100/metrics 호출 (PULL)
```

너의 그림에서 Node Exporter가 따로 있는 게 호스트 레벨 메트릭을 보기 위함.

#### Alertmanager

Prometheus와 짝꿍이다. **알람을 사람에게 전달하는 책임**을 분리.

```
1. Prometheus가 PromQL로 조건 체크
   예: "CPU 사용률 > 80% for 5분"

2. 조건 충족 시 Prometheus → Alertmanager: 알람 발송
   ↓ HTTP POST

3. Alertmanager가 알람을 분류/중복제거/그룹화
   ↓
4. Discord, Slack, Email 등으로 전송
```

Prometheus는 메트릭 수집과 조건 체크까지, Alertmanager는 알람 전달의 라우팅과 노이즈 제거 담당. 분리해두는 이유는 알람 전달 로직이 의외로 복잡하기 때문 (같은 알람 30분간 묶기, 야간엔 무시, 특정 라벨은 다른 채널로 등).

#### Grafana

Grafana는 **시각화**만 한다. 데이터를 직접 저장하지 않음.

```
[사용자가 Grafana 대시보드 접속]
1. Grafana → Prometheus: "CPU 사용률 데이터 줘" (PromQL 쿼리)
2. Prometheus가 자기 TSDB에서 데이터 꺼내서 응답
3. Grafana가 받아서 그래프로 그림

[Grafana 자체가 저장하는 것]
- 대시보드 정의 (JSON)
- 사용자 계정, 권한
- 데이터 소스 설정
→ /grafana-data 볼륨에 저장 (EFS)
```

Grafana는 Prometheus, Loki, CloudWatch 등 여러 데이터 소스를 동시에 볼 수 있다. 데이터 소스가 어디 있든 쿼리만 던질 수 있으면 OK.

#### Loki + Alloy (로그 수집, PUSH 방식)

Prometheus가 메트릭의 PULL이라면, Loki는 로그의 PUSH다.

```
[Spring App이 로그를 파일로 씀]
1. Spring App → /var/log/re-caring/app.log 에 로그 작성

[Alloy가 파일 감시]
2. Alloy(Grafana Agent의 후속작)가 그 파일을 tail
   - 새 줄이 추가될 때마다 감지
3. Alloy → Loki: 새 로그 라인 전송 (PUSH)

[Loki 저장]
4. Loki가 받아서 자기 저장소에 인덱싱하여 저장
```

여기서 **Spring App과 Alloy가 같은 EFS 경로를 공유**하는 게 핵심이다. App 컨테이너의 `/var/log/re-caring`을 EFS에 마운트하고, Alloy 컨테이너도 같은 EFS 경로를 read-only로 마운트. 그러면 두 컨테이너가 같은 파일을 본다.

#### CloudWatch Logs (그림에 있는 거)

이건 약간 다른 레이어. **컨테이너의 stdout/stderr를 AWS가 자동 수집**해주는 서비스.

```
ECS Task 정의에 logConfiguration 설정:
  logDriver: awslogs
  options:
    awslogs-group: /ecs/recaring
    awslogs-region: ap-northeast-2

→ 컨테이너의 stdout/stderr가 자동으로 CloudWatch Logs로 흘러감
```

Loki와 역할이 겹치는 면이 있다. 차이:

- **CloudWatch Logs**: AWS 관리형, 비용은 GB당, 검색/필터링은 좀 약함
- **Loki**: 자체 운영, EFS만 쓰면 비용 적음, Grafana와 통합 좋음

보통 둘을 같이 쓴다. CloudWatch는 ECS 자체 동작(컨테이너 시작/실패 등) 로그용, Loki는 애플리케이션 로그용.
