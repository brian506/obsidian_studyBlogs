## 🔥 Track 1. AI Ops 자동 장애 복구

### Q. "왜 AlertManager 받자마자 LLM으로 보내지 않고 3개 에이전트로 나눴어요?"
**A.** 단일 LLM이 모든 걸 판단하면 환각(hallucination) 위험이 큽니다. 그래서 원인을 **Device / SSE / Infra 세 관점으로 분리**해서 병렬로 조사하고, 각자의 판정을 **우선순위 기반 Verdict Engine**으로 합의시켜 결론을 냅니다. 자동 복구는 가장 안전한 `CONTAINER_ISSUE` 한 가지에만 적용했고, 나머지는 알림만 보냅니다. LLM은 '판단'에만 쓰고 '실행'은 안전 등급별로 분리한 거죠.

### Q. "비용은요? LLM 호출 비싸지 않아요?"
**A.** 같은 알림이 반복돼서 호출이 폭주하는 걸 막기 위해 **Runbook 캐시 + fingerprint 중복 방지**를 넣었습니다.
- Alertmanager가 firing 상태에서 반복 webhook을 쏘는데, Redis에 `investigation:{fingerprint}` 형태로 RUNNING 상태를 캐싱해서 중복 조사를 막습니다.
- 이미 해결책이 알려진 알림은 RunbookManager 캐시 hit으로 LLM 호출 없이 SSM 명령만 실행합니다.
- cache miss일 때만 LLM이 개입합니다.

### Q. "Virtual Threads는 왜 썼어요?"
**A.** Webhook 응답을 빠르게 돌려줘야 하는데(Alertmanager가 timeout 두는 게 있어서), `orchestrate()` 전체가 동기로 끝나길 기다리면 안 됩니다. 그래서 응답은 미리 반환하고 조사·복구는 가상 스레드로 비동기 실행했습니다. I/O 위주의 작업(SSM 호출, LLM 호출)이라 OS 스레드를 점유하지 않는 Virtual Thread가 적합했습니다.

### Q. "한계는요?"
**A.** 현재 룰은 `POST /api/v1/location/gps`만 감시해서 **피보호자 → 서버 GPS 업로드 경로만** 잡습니다. **보호자 앱의 SSE 끊김은 모니터링 사각지대**입니다. 다음 개선으로는 `sse_active_connections` 커스텀 Micrometer 메트릭을 export해서 보호자 단위 룰을 추가할 계획입니다. 또 LLM 비용 폭주 방지를 위한 Rate Limit / Circuit Breaker도 보강해야 합니다.

## 🔥 Track 2. 실시간 위치 SSE

### Q. "왜 WebSocket 안 쓰고 SSE 썼어요?"
**A.** GPS 좌표는 **서버 → 클라이언트 단방향**이라 양방향 채널인 WebSocket의 오버헤드가 불필요합니다. SSE는 단순 HTTP 위에서 동작해서 ALB·CDN·프록시 호환도 좋고 재연결도 표준이라 운영 부담이 적습니다.

### Q. "보호자 여러 명이 같은 피보호자를 보는 구조는 어떻게 모델링했어요?"
**A.** `Map<wardKey, CopyOnWriteArrayList<SseEmitter>>` 구조입니다. wardKey가 피보호자, 리스트가 그 피보호자를 구독 중인 보호자들의 emitter 모음입니다. CopyOnWriteArrayList를 쓴 이유는 broadcast가 리스트를 순회하는 동안 다른 스레드가 emitter를 추가·제거해도 ConcurrentModificationException이 안 나도록 하기 위해서입니다.

### Q. "죽은 SseEmitter는 어떻게 정리해요?"
**A.** 세 가지 콜백을 다 걸어둡니다 — `onCompletion`(정상 종료), `onTimeout`(30분 timeout), `onError`(네트워크 오류). 모든 경로에서 `remove()`가 호출되어 메모리에서 빠집니다. 또 send() 중 IOException이 나면 그 자리에서 즉시 remove합니다. emitter는 매번 새 객체라 같은 클라이언트라도 주소가 달라서 비교는 객체 identity로 합니다.

### Q. "보호자가 처음 켰을 때 빈 화면 나오는 문제는요?"
**A.** 첫 연결 시 **Redis에서 최신 GPS 1건을 조회**해서 `sendInitialEvent`로 즉시 쏴줍니다. Redis가 장애면 **DB로 폴백**합니다. 이후로는 피보호자가 GPS 보낼 때마다 broadcast로 받게 됩니다.

### Q. "동시 연결 몇 개까지 받을 수 있어요? 측정해봤어요?"
**A.** (※ 측정 후 채우기) 현재 Gatling으로 부하 테스트 진행 중입니다. 주요 지표는 ① 동시 연결 한계 ② broadcast P95 지연 ③ 24시간 heap 누수 여부 입니다.

### Q. "한계는요?"
**A.** broadcast가 동기 순회라 emitter 한 명의 send가 늦으면 뒤 사람들도 지연됩니다. Virtual Thread로 emitter별 비동기 send로 분리할 계획입니다. 또 단일 인스턴스 메모리에 emitter를 들고 있어서 스케일아웃 시 한 ward의 보호자가 여러 인스턴스에 분산되는 문제가 있습니다. Redis Pub/Sub으로 인스턴스 간 broadcast를 보강해야 합니다.

## 🔥 Track 3. AWS Production 인프라

### Q. "왜 ALB 안 쓰고 nginx EC2를 직접 띄웠어요?"
**A.** 세 가지 이유입니다.
1. **SSE 호환** — ALB는 idle timeout이 기본 60초라 SSE 장기 연결과 안 맞습니다. NLB로 가야 하는데 그러면 ACM 인증서 + Let's Encrypt 자동 갱신을 다시 설계해야 합니다.
2. **Let's Encrypt HTTP-01 챌린지** — 80번 포트로 직접 들어와야 해서 nginx가 받아주는 게 단순합니다.
3. **비용** — ALB 약 $60/월 vs nginx t3.nano 약 $4 + NAT GW $43. NAT가 있어야 외부 API(CoolSMS, FCM) 호출이 가능하므로 NAT는 어차피 필수였고, ALB 비용을 절감했습니다.

### Q. "왜 awsvpc 모드?"
**A.** Rolling Update 때문입니다. bridge 모드는 EC2 호스트 포트를 공유해서 V1, V2가 동시에 8080을 못 씁니다. awsvpc는 컨테이너마다 ENI를 따로 받아서 각자 IP를 갖기 때문에 포트 충돌 없이 V1·V2가 공존할 수 있습니다. 단점은 인스턴스 타입별 ENI 한도와 awsvpc + EC2 launch type 조합에선 Task ENI에 Public IP 자동 할당이 안 된다는 점이고, 이건 Private Subnet + NAT GW로 풀었습니다.

### Q. "Cloud Map은 왜 도입했어요?"
**A.** awsvpc 모드 + Rolling Update면 Task 교체마다 Private IP가 바뀝니다. nginx가 매번 새 IP를 알아야 하는데, Cloud Map이 Task의 생사를 실시간으로 추적해서 `app.local` 도메인을 자동 갱신해줍니다. nginx는 `resolver 169.254.169.253 valid=5s`로 5초마다 DNS를 다시 해석해서 최신 IP를 받습니다.

### Q. "Terraform 왜 도입했어요?"
**A.** 1) 콘솔 클릭으로는 내가 뭘 바꿨는지 추적이 안 됩니다. 2) 팀원이 같은 인프라를 만들 때 손으로 따라 만들면 빠뜨리는 게 생깁니다. 3) AWS 비용 폭탄을 한 번 맞은 적이 있어서 인프라 변경을 코드 리뷰로 통제할 필요가 있었습니다. S3 backend + DynamoDB lock으로 팀 동시 작업 충돌도 막았습니다.

### Q. "EFS 왜 썼어요? 비싸지 않아요?"
**A.** Redis AOF / Prometheus TSDB / Grafana 설정 / Loki 로그 / Let's Encrypt 인증서를 EC2 로컬에 두면 EC2가 내려갈 때 다 날아갑니다. **컴퓨팅과 스토리지를 분리하는 원칙** (RDS로 PostgreSQL 분리한 것과 같음)을 EC2 데이터에도 적용한 거죠. ElastiCache는 MVP 규모에 과합니다 (월 단위 $30+). EFS는 저장량 기반 과금이라 현재 규모에서 월 $5 이하입니다. nginx + certbot이 분리된 컨테이너인데 인증서를 공유해야 해서 **NFS 기반 공유 스토리지가 자연스러운 선택**이었습니다.

### Q. "한계는요?"
**A.** ① nginx가 단일 EC2 SPOF입니다. ② App Service가 단일 AZ입니다. ③ EC2 launch type ENI 한도로 동시 Task 수가 제약됩니다. 트래픽이 늘면 nginx ASG + 2대 / multi-AZ ECS Service / Fargate 마이그레이션 검토가 다음 단계입니다.
