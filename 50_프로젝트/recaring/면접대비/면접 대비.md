## ECS 네트워크 설계

### awsvpc 모드와 bridge 모드의 차이가 정확히 뭐냐, 왜 awsvpc를 쓰면 ENI 한도를 초과하냐?
- 일단 bridge 모드는 도커 가상 네트워크 안에서 가상 IP주소를 할당 받지만, 외부와의 통신은 호스트의 ENI를 공유해서 처리합니다. 반면에 awsvpc모드는 인스턴스의 프라이빗IP와 별개로 개인의 private IP를 사용하게 됨으로써, 현재 medium 사양의 ENI 개수는 최대 3개로 알고 있습니다. 따라서 awsvpc모드의 컨테이너가 3개를 넘어가게 되면 할당받는 ENI개수가 초과됩니다.

### bridge 모드의 포트 충돌은 ECS의 동적 포트 매핑으로 해결 가능한데 왜 그 방법을 안씀?
- ECS 동적 포트 매핑은 ALB를 사용해야 하는 것으로 알고 있습니다. 하지만 저는 비용상의 문제 때문에 ALB를 쓰지 않고 Nginx 컨테이너를 띄워서 트래픽을 받고 있기 때문에 쓰지 않았습니다.

### host 모드는 왜 고려하지 않았나?
- host 모드는 인스턴스의 Private IP를 그대로 쓰기 때문에, 한 컨테이너가 해킹될 경우 다른 컨테이너에 접근을 할 수 있게 되어 보안상 우려가 있어서 사용하지 않았습니다. host 모드는 대체로 네트워크 지연을 최소화할 경우에만 사용해야합니다.

### 왜 하필 Nginx는 bridge, Spring은 awsvpc로 나눔? 반대로 하면 어케됨?
- Spring 컨테이너는 현재 롤링 업데이트 방식의 무중단 배포를 사용하기 때문에, 동시에 두 개의 같은 포트를 가진 컨테이너가 띄워지게 됩니다. 근데 만약 bridge 모드를 하게 되면 포트번호가 겹치기 때문에 안됩니다.따라서 spring은 awsvpc모드로 별도의 ENI를 할당받아, 같은 포트번호가 동시에 띄워질 수 있게 하고, ENI 개수 충돌로 인해 bridge모드로 할당했습니다. 

## 동적 라우팅 / CloudMap

### CloudMap이 정확히 뭐고, Service Discovery가 어떤 원리로 동작함?
- CloudMap은 VPC내에서만 동작하는 사설 DNS 서비스입니다. Service Discovery는 ECS 태스크가 생성되거나 종료될 때 CloudMap에 해당 태스크의 private IP를 자동으로 등록/해제합니다. CloudMap은 해당 private IP를 갱신하고, DNS 쿼리에서 최신 IP를 가져가는 방식입니다.

### 5초 주기로 IP 갱신을 했는데, 그 5초 사이에 들어온 트래픽은 어떻게 됨?
- 이 부분은 설정하지 못했지만, 구버전의 컨테이너가 모든 요청을 다 처리할때까지 컨테이너를 다운되지 않도록 설정할 것입니다.

### Nginx는 기본적으로 시작 시점에 DNS를 한 번 resolve하고 캐싱하는데, 어떻게 5초마다 갱신되게 만들었나? 
- Nginx resolver에 AWS VPC 내부 DNS 주소를 넣고, 변수에 도메인 주소를 5초마다 맞는 도메인 주소인지 확인할 수 있도록 했습니다. 이렇게 구현함으로써 변수에 담긴 도메인 주소가 맞는지 매 요청마다 확인하고, 5초마다 DNS 쿼리로 spring의 프라이빗 IP가 변경되는 것을 방지하기 위해 옳바른 주소를 읽도록 했습니다.

## 무중단 배포 / 안정성

### 롤링 업데이트와 블루-그린의 차이, 그리고 왜 ECS 롤링?(전환 이유)
- 블루 - 그린 : 구버전과 신버전 환경을 완전히 별도로 구성해두고, 준비가 끝나면 트래픽을 한 번에 전환합니다. 롤백이 빠르지만 리소스가 두 배 필요합니다.
- 롤링 : 구버전 인스턴스를 신버전으로 점진적인 교체를 합니다. 리소스 효율적이지만 전환 중 구/신 버전이 동시에 서비스됩니다.
전환 이유 : 블루-그린을 그대로 사용하려면 ALB와 별도의 스크립트 작성이 필요합니다. ALB를 사용하지 않는 상태이기 때문에, 롤링 + CloudMap DNS 갱신 조합으로 배포 전체를 자동화할 수 있어서 전환했습니다.

### ECS 서킷 브레이커의 실패 임계치를 몇으로 잡았고 왜 그 값이냐
- ECS 배포 서킷 브레이커와 자동 롤백 옵션을 활성화해서, 배포 중 태스크가 반복적으로 실패하면 이전 태스크로 자동 복구되도록 했습니다.


## 보안 / 네트워크

### 퍼블릭 서브넷 + 보안그룹에만 의존하는 구성의 구체적 위험?
- 보안그룹은 L3 ~ L4인 IP와 포트 레벨만 제어하기 때문에 허용된 포트를 통한 L7(응용 계층) 공격은 방어하지 못합니다. 또 퍼블릭 IP가 노출되면 포트 스캔, DDos 공격도 받을 위험이 있습니다. 

### 보안그룹과 NACL의 차이
**보안그룹과의 핵심 차이 세 가지**

첫째, **적용 단위**가 달라. 보안그룹은 ENI(인스턴스) 단위, NACL은 **서브넷** 단위야. 그 서브넷에 들어오고 나가는 모든 트래픽에 적용돼.

둘째, **Stateful vs Stateless**. 이게 제일 중요해. 보안그룹은 Stateful이라 인바운드를 허용하면 응답(아웃바운드)이 자동으로 허용돼. NACL은 Stateless라 **인바운드랑 아웃바운드 규칙을 각각 따로 설정**해야 해. 예를 들어 외부에서 80포트로 들어오는 HTTP를 허용하려면, 인바운드 80 허용뿐 아니라 **아웃바운드 응답 포트(임시 포트, 1024~65535)도 별도로 허용**해야 해. 이걸 모르면 NACL 설정했는데 통신이 안 된다고 당황하는 상황이 생겨.

셋째, **Allow + Deny 규칙**. 보안그룹은 Allow 규칙만 있고 나머지는 암묵적으로 Deny야. NACL은 Allow와 **Deny를 명시적으로** 설정할 수 있어. 특정 IP를 명시적으로 차단하고 싶을 때 유용해.

**규칙 평가 방식**

NACL은 규칙 번호 순서대로 평가하고 매칭되는 첫 번째 규칙에서 결정이 나. 예를 들어 이렇게 설정되어 있으면:

```
규칙 100: 192.168.1.1/32 → DENY
규칙 200: 0.0.0.0/0     → ALLOW
```

192.168.1.1은 100번에서 DENY되고 끝나. 200번까지 가지 않아.

**보안그룹이랑 같이 동작하는 방식**

트래픽이 서브넷으로 들어올 때 NACL을 먼저 통과하고, 그 다음 인스턴스의 보안그룹을 통과해야 해. 즉 **NACL이 1차 방어선, 보안그룹이 2차 방어선**이야. NACL에서 막히면 보안그룹까지 도달조차 못 해.

### SSM 포트포워딩이 어떻게 SSH없이 접근을 가능하게 하느냐
- SSM Agent가 AWS SSM 서비스 엔드포인트에 아웃바운드로 연결을 유지하고 있어서, 인바운드 포트가 열려있지 않아도 됩니다. 로컬에서 명령을 실행하면 SSM이 아웃바운드 채널을 이용해서 세션을 만들어줍니다. 

### NAT Gateway가 어떻게 동작하느냐, private subnet에서 인터넷으로 어떻게 나가느냐
- 프라이빗 서브넷에 있는 컨테이너들은 퍼블릭 ip 주소가 없어서 인터넷 게이트웨이로의 인바운드 및 아웃바운드가 제한됩니다. 만약 외부와의 통신이 필요한 경우에 NAT Gateway를 IGW 사이에 배치시키고, NAT Gateway가 별도의 퍼블릭 IP를 가지고 private subnet의 요청을 받아서 외부로 통신합니다. VPC 내부에서는 NAT Gateway와 컨테이너는 private IP로 통신합니다.


### 네트워크 CS (네 에세이와 가장 직접 연결됨)

- TCP와 UDP의 차이, 그리고 너의 위치 추적 서비스는 둘 중 뭐가 적합하냐 (실시간 좌표 전송이니까 흥미로운 질문)
- TCP 3-way handshake가 어떻게 동작하느냐
- HTTP와 HTTPS의 차이, TLS handshake가 어떻게 일어나느냐 (ACM/SSL 얘기 꺼냈으니 연결됨)
- DNS가 도메인을 IP로 바꾸는 전체 과정 (CloudMap도 결국 DNS라 자연스럽게 이어짐)
- 로드밸런싱 알고리즘에는 뭐가 있느냐 (라운드로빈, least connection 등 — Nginx 직접 썼으니)
- 포트가 정확히 뭐냐, 왜 필요하냐 (포트 충돌을 그렇게 고생해서 풀었으니 개념 자체를 물을 수 있음)
- REST API가 뭐고 멱등성(idempotency)이 뭐냐
