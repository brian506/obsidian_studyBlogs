# 단일 t3.medium에서 무중단 배포하기 — ASG를 들였다 걷어낸 기록

EC2 한 대에 앱·Redis·모니터링을 욱여넣은 채로 무중단 배포를 하려다, ASG(Auto Scaling Group)까지 도입했다가 결국 다시 단일 인스턴스로 돌아온 이야기다. 결론부터 말하면 **무중단의 발목을 잡은 건 인프라가 아니라 앱 메모리 한 줄**이었다.

## 먼저, 결론

- **무중단 배포는 본질적으로 "앱 1벌분 메모리"를 항상 비워둘 것을 요구한다.** 신·구 두 벌이 잠깐 공존해야 하기 때문이다. 전체 메모리를 앱 하나에 다 줄 수 없다.
- ASG로 "배포 때만 임시 인스턴스를 빌렸다 반납"하는 그림은 매력적이지만, **ECS는 돌고 있는 태스크를 다시 모아주지(consolidate) 않는다.** "자동 반납"은 환상이고, 수동 drain·파이프라인 자동화가 따라붙는다.
- 진짜 원인은 **앱 메모리를 1024 → 2048로 올린 것**이었다. 그 전엔 단일 박스에서 롤링이 멀쩡히 돌았다. 메모리를 다시 right-size하니 ASG 없이도 무중단이 복원됐다.

최종적으로 도착한 구조는 **t3.medium 1대 + 단일 인스턴스 롤링(min=100/max=200) + 서킷브레이커**, ASG 없음이다. ASG는 우회로였을 뿐, 답이 아니었다.


## 0. 배경

보호 대상자의 GPS를 백그라운드로 수신해, 보호자에게 SSE로 실시간 위치를 보내는 케어 서비스다.

**인프라**

- ECS (EC2 launch type)
- t3.medium 1대에 `app` + `Redis` + `모니터링(Grafana/Prometheus/exporters)` 동거
- nginx는 별도 t3.nano

**제약 조건**

- SSE 끊김은 허용
- Redis 캐시는 반드시 보존 (앱 재시작과 무관해야 함)
- 몇 초 다운은 허용

목표는 무중단(롤링) 배포였다.


## 1. 무중단의 근본 법칙과 ECS의 결

### 무중단 = make-before-break = 컨테이너 2벌 공존

무중단 배포의 정의 자체가 "신버전이 healthy해진 뒤 구버전을 내린다"이다. 그러려면 신·구 두 벌이 잠깐 동시에 떠 있어야 한다. blue-green이든 rolling이든 같다. 이 2벌이 어딘가엔 있어야 하고, 둘 곳은 두 군데뿐이다.

|2벌을 어디에|비용|
|---|---|
|같은 인스턴스|그 박스에 앱 2벌분 메모리가 상시 필요 → 전체 메모리를 한 앱에 다 못 줌|
|다른 인스턴스|배포 때만 임시 인스턴스(ASG scale-out)|

t3.medium(4GB, ECS 등록 메모리 ≈ 3.7GB)에 `app 2GB + Redis + 모니터링`이면 앱 2벌이 한 박스에 안 들어간다. 그래서 처음엔 **ASG로 임시 인스턴스를 띄우는 방향**으로 갔다.

### ASG의 함정 — "자동 반납"은 없다

ECS managed scaling은 scale-out과 **빈** 인스턴스 종료는 자동이지만, 돌고 있는 태스크를 더 적은 인스턴스로 다시 모으는(consolidate / rebalance) 동작은 하지 않는다. 즉 배포가 끝나도 임시 인스턴스로 올라간 앱을 메인으로 되돌려 임시 인스턴스를 비우는 일은 자동으로 일어나지 않는다.

> "상시 1대 + 배포 때만 임시 1대 → 끝나면 자동 반납" 구조는 ECS의 결을 거스른다. 반납하려면 명시적 drain(수동 또는 파이프라인 자동화)이 필요하고, 메인이 ASG 라이프사이클에 들어가면 Redis 영속성까지 위협받는다.

## 2. 방식별 트러블슈팅

실제로 두 방향을 다 밟아봤다. 막힌 지점과 우회법을 순서대로 정리한다.

### 방식 A — 다운타임 허용 (break-before-make, 단일 인스턴스)

`minimumHealthyPercent=0`, `maximumPercent=100`으로 구버전을 먼저 내리고 신버전을 올린다.

**막힌 지점들**

**1) "몇 초 다운"이라는 가정이 틀렸다.** 실제 다운타임은 앱의 콜드 스타트 전체 시간이다. CloudWatch 로그 실측:

```
Started RecaringApplication in 99.5s / 94.5s / 100.5s ...
```

break-before-make의 다운타임은 **약 2분**이었다. 서비스 영향은, 디바이스가 GPS를 버퍼링하지 않고 30초 간격으로 보내므로 배포당 좌표 4개가량 유실, SSE는 재연결, 패턴 분석은 SQS라 유실 없음.

**2) `maximumPercent=100`이 거부된다.**

```
InvalidParameterException: Availability Zone Rebalancing does not support
maximumPercent <= 100% as deployment configuration.
```

AZ 리밸런싱을 같이 꺼야 한다: `--availability-zone-rebalancing DISABLED`

**3) `min=0`만 바꾸면 안 된다.** `max=200`을 그대로 두면 ECS가 신버전 태스크를 먼저 띄우려다(200%) 한 박스에 메모리가 없어 capacity provider scale-out을 유발한다. **없애려던 ASG가 다시 깨어난다.** `min=0` + `max=100`이 한 세트다.

### 방식 B — ASG 걷어내고 단일 인스턴스 롤링으로 복원

ASG 삭제 → 임시 인스턴스 종료 → 서비스를 capacity provider에서 `launchType=EC2`로 환원.

**막힌 지점들**

**4) 태스크 정의를 EC2 타입으로 바꿔야 하나? → 아니다.** 태스크 정의는 처음부터 `requiresCompatibilities: [EC2]`였다. ASG 도입 때 바뀐 건 **서비스의 실행 방식**(`launchType=EC2` → `capacityProviderStrategy`)이지 태스크 정의가 아니다. 되돌리는 건 서비스 레벨이고, launch type 전환은 보통 **서비스 재생성**이 필요하다.

**5) 좀비 서비스.** 서비스를 재생성해 새 `recaring-service(launchType=EC2)`를 만들었더니, 옛 `recaring-task-service`가 `desired=0` + 이미 삭제된 ASG의 capacity provider에 묶인 채 남았다.

**6) 배포 파이프라인이 좀비를 가리킴.** `deploy.yml`이 여전히 옛 서비스명을 타겟으로 해, 다음 배포가 빈 서비스에 적용됐다. 서비스명을 새 서비스로 수정해야 한다.

**7) `deploy.yml`에 push 트리거가 없음.** `develop`에 직접 푸시해도 워크플로우가 안 돈다(트리거가 `workflow_dispatch`/PR-merge뿐). `gh workflow run`으로 수동 디스패치.

**8) (핵심) 메모리 벽 — 롤링 배포 실패.**

```
unable to place a task because no container instance met all of its
requirements. The closest matching container-instance ... has
insufficient memory available.
```

원인은 **ECS 배치가 OS의 free 메모리가 아니라 "예약(reservation)" 기준**이라는 것. `free -m`상 available이 2GB여도, ECS는 태스크의 memory **예약 합**으로 배치를 판단한다.

|태스크|예약|실사용|
|---|---|---|
|app(구)|2048|626|
|Redis|512|4|
|모니터링|1024|~173|
|**합**|**3584 / 등록 ~3700**|—|

남는 예약이 약 150MiB뿐. 신버전(1024 예약)을 못 올린다 → 서킷브레이커 롤백, **구버전은 계속 가동(무중단 유지)**.

**9) 모니터링 태스크의 6배 과할당.** 1024를 예약하지만 실사용 ~173MiB(Grafana 122 / Prometheus 32 / exporters ~18). 실측+여유로 줄이면 약 512MiB를 회수해 롤링 여유를 만들 수 있다.

**10) Redis는 트림 불가.** 512를 예약하지만 실사용 4MiB. 그런데 재시작하면 캐시가 날아가므로(하드 제약) 예약을 줄일 수 없다. 과할당이 "동결"된다.

**11) 첫 전환(2048 → 작은 사이즈)은 오버랩 불가.** 모니터링을 줄여도, 돌고 있는 구버전이 2048이라 신버전과 동시 배치가 안 된다(`2048 + 신 + Redis + 모니터링 > 등록`). 이 **1회만** `desired=0` 짧은 다운이 필요하고, 그 뒤 작은 사이즈끼리는 롤링이 무중단으로 된다.

**12) "예전엔 잘 됐다"의 진실.** 단일 인스턴스 롤링이 원래 잘 돌던 건 앱이 ~1024이던 시절이라 `2×1024`가 박스에 들어갔기 때문이다. 로드테스트하면서 앱을 2048로 올린 것(JFR·virtual thread 옵션 흔적)이 단일 박스 fit을 깨뜨렸고, 그게 ASG 우회의 발단이었다.

### 숨은 두 번째 천장 — ENI

t3.medium은 ENI 3개 한도다. `awsvpc` 앱 태스크는 1개씩 ENI를 먹고, 호스트 primary가 1개. 롤링 중 앱 2벌이면 `2 + primary = 3 = 정확히 한계`. (Redis·모니터링은 bridge 모드라 ENI를 안 먹어서 가능.) **메모리뿐 아니라 ENI도 오버랩 구간엔 여유가 0이다.**

## 3. 교훈

1. **무중단은 본질적으로 N+1 용량을 요구한다.** 바뀌는 서비스의 2벌이 잠깐 공존하므로, 단일 박스라면 앱 1벌분 메모리를 항상 비워둬야 한다. 전체 메모리를 한 앱에 다 줄 수 없다.
2. **ECS 배치는 예약 기준이다.** `free -m`의 available ≠ 배치 가능량. memory를 실측+여유로 right-size하지 않으면 용량을 낭비하고 배포가 막힌다.
3. **ECS managed scaling은 모으지 않는다.** scale-out·빈 인스턴스 종료는 자동이지만 running 태스크 consolidate는 없다. "임시 인스턴스 자동 반납"은 환상이다.
4. **launch type은 서비스 속성이지 태스크 정의가 아니다.** capacity provider ↔ launchType 전환은 서비스 재생성을 부르고, 좀비 서비스 + 파이프라인 서비스명 불일치를 유발한다.
5. **`maximumPercent ≤ 100`은 AZ 리밸런싱 비활성화가 선행되어야 한다.**
6. **콜드 스타트를 실측하라.** "몇 초"라 가정했지만 실제는 ~100초였다.
7. **상태 있는 사이드카(Redis 캐시)가 전략 전체를 제약한다.** 재시작이 곧 데이터 손실이면 그 태스크의 과할당은 줄일 수 없다.
8. **서킷브레이커 + `min=100%`가 안전망이다.** 이 모든 실험 내내 실패한 배포가 구버전을 죽이지 않아 다운타임 0이 유지됐다.

---

## 4. 최종 구조

- **t3.medium 1대**에 app + Redis + 모니터링 동거, **ASG 없음**, `launchType=EC2`, 롤링(`min=100`/`max=200`) + 서킷브레이커
- 앱 메모리 `2048 → ~1152–1280`, 모니터링 `1024 → ~512–640`(Prometheus는 히스토그램 카디널리티 고려해 여유), **Redis는 캐시 보존 위해 그대로**
- `2048 → 축소` 첫 전환만 1회 `desired=0` 짧은 다운, 이후 단일 박스 무중단 롤링 복원

결국 ASG는 문제의 우회로였고, 진짜 해법은 **앱 메모리를 다시 박스에 맞춘 것**이었다. 인프라를 늘리기 전에, 늘어난 게 정말 필요한 메모리였는지부터 의심하는 게 먼저였다.
