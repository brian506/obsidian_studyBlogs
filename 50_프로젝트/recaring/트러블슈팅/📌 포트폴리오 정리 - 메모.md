## 2. 어필 포인트 (강한 순서)

### ★★★ AI Ops 기반 자동 장애 복구 (최강 차별점)
- Prometheus 룰 → AlertManager → Webhook → **3-Agent 병렬 조사 → Verdict Engine 우선순위 판정 → 자동 복구**
- 키워드:
  - **Spring AI `@Tool` 패턴** — LLM이 메서드 스펙 보고 스스로 어떤 tool 호출할지 결정
  - **Virtual Threads** — alert orchestration 비동기화
  - **다중 에이전트 합의** — LLM 환각 방어 (단일 LLM 판단 X)
  - **Runbook 캐시 (cache hit/miss)** — 동일 알림 반복 시 LLM 호출 안 함 → 비용 절감
  - **fingerprint 중복 방지** — Alertmanager 반복 webhook 방어
  - **자동 복구 범위 제한** — `CONTAINER_ISSUE` 만 docker restart, 나머지는 알림만 → 안전 우선
- 면접 한 줄: *"LLM을 운영 자동화에 쓰되, 단일 판단을 신뢰하지 않고 다중 에이전트 + Verdict Engine + 복구 범위 제한으로 안전망을 만들었다."*

### ★★★ 실시간 위치 추적 SSE 아키텍처
- 자료 구조: `Map<wardKey, CopyOnWriteArrayList<SseEmitter>>` — 1:N broadcast
- **선택 근거 정리됨**: CopyOnWriteArrayList 이유, onCompletion/Timeout/Error 메모리 누수 방어, Redis 폴백
- 면접 한 줄: *"SSE의 가장 큰 함정인 죽은 emitter 누수를 onError/onCompletion + Redis 폴백 패턴으로 막았다."*

### ★★ AWS Production 인프라 의사결정
- **awsvpc + NAT GW + VPC Endpoint** 조합을 단순 적용이 아니라 [bridge → assignPublicIp → ALB → NAT GW] 대안 비교 후 선택
- **ALB 대신 Nginx EC2** — SSE idle timeout 회피 + Let's Encrypt HTTP-01 직접 처리 + 비용
- **ECS Rolling Update** — Blue-Green 자체 스크립트(GA) → ECS 네이티브로 진화 (마이그레이션 스토리)
- **IaC** — Terraform + S3 backend + DynamoDB lock (팀 협업 동기)

### ★★ 비용 vs 안정성 트레이드오프
- ElastiCache 대신 EFS + Redis AOF (`appendfsync everysec` — NFS 레이턴시 고려)
- ALB(~$60) 대신 nginx EC2 t3.nano(~$4) + NAT GW(~$43)
- 비용 의식이 의사결정 곳곳에 있음 — 시니어가 좋아하는 신호

## 3. SSE 성능 테스트 방향

### 측정 목표 (면접에서 답할 수 있어야 함)
1. 한 EC2(t3.medium)당 동시 SSE 연결 최대 N개
2. broadcast 1회 → 전체 emitter 도달 P95 지연
3. JVM Heap 사용량 증가 곡선 (연결 수에 따라)
4. 24시간 누수 검증 (heap 그래프 평탄한가)

### 시나리오
| # | 시나리오 | 무엇을 보는가 |
|---|---|---|
| A | 동시 연결 1k → 10k → 30k 점진 증가 | 한계 동시연결 수, OOM/FD 한도 |
| B | 1회 GPS 업로드 → broadcast 지연 측정 | 처리 지연 (P50/P95/P99) |
| C | 랜덤 연결 끊기 (10% 비율 유지) | onError/remove 비용, 누수 여부 |
| D | 24시간 + 주기적 broadcast | 메모리 누수 검증, GC 압박 |

### 도구
- **K6** — HTTP 부하용이지만 SSE 시뮬 가능 (`http.get` 대신 stream 클라이언트 작성)
- **Gatling** — SSE protocol native 지원 (Scala/Java 친화)
- **자체 스크립트** — Node/Java로 N개 클라이언트 동시 띄우기 (가장 통제 쉬움)
- 추천: **Gatling** — SSE 네이티브 + 리포트 깔끔 → 포폴 첨부용으로 적합

### 측정 메트릭 노출
- 커스텀 Micrometer 메트릭 필요:
  - `sse_active_connections` (gauge, wardKey별)
  - `sse_broadcast_duration` (timer)
  - `sse_emitter_send_failures_total` (counter)
- Prometheus → Grafana 대시보드로 가시화 → 포폴 스크린샷용

### 추가 분석 포인트
- `broadcast()` 가 동기 순회면 emitter 1개 늦으면 다 늦어짐 → 비동기 전환 검토
- `CopyOnWriteArrayList` 는 write O(N) → 보호자 수 늘 때 성능 측정 필요
- send() I/O 가 broadcast 쓰레드 점유 → Virtual Threads로 분리 검토

## 4. GPS 끊김 AI 에이전트 파이프라인 — 평가

### 잘된 점 (면접 어필)
- **알림 정의가 명확** — 단순 "GPS 끊김" 이 아니라 `POST /api/v1/location/gps status=200 == 0 for 15min` 이라는 메트릭 기반 정의
- **fingerprint 중복 방지** — Alertmanager 반복 webhook 방어
- **3-Agent 분리 + Verdict Engine** — LLM 환각 방어 설계
- **자동 복구 범위 제한** — `CONTAINER_ISSUE` 만 자동, 나머지는 알림 → "AI 신뢰 안 함" 표명
- **모니터링 사각지대 자각** — "보호자 SSE 끊김은 룰 밖" 이라고 명시 → 메타인지 점수 ★★★ ("다음 개선" 답 직결)

### 보완 필요
- **보호자 SSE 끊김 알림** — 커스텀 메트릭(`sse_active_connections`) 필요. ward는 GPS 올리는데 보호자만 못 보는 케이스 → 따로 룰 추가
- **클라이언트 reconnect 정책** — exponential backoff + jitter 명시
- **LLM 비용 폭주 방지** — cache miss가 짧은 시간에 폭주하면 비용 위험. Rate limit / Circuit breaker 필요
- **DEVICE_TOKEN_ISSUE** 자동 처리 — 단순 알림보다 푸시 전송으로 재로그인 유도까지 자동화 가능

### 면접 답안 한 줄
*"AI를 자동화의 끝단이 아니라 '판단'에만 쓰고, '실행'은 안전 등급별로 인간/자동 분리했다. 다중 에이전트로 LLM 환각을 방어하고, 동일 알림에 대해선 runbook 캐시로 LLM 비용을 줄였다."*
