## 플랫폼 스레드를 이용한 SSE 통신(실시간 이동 경로)

### 동작 순서
1. 클라이언트 -> 서버 : HTTP 통신 요청(핸드셰이크 후 수립) - 플랫폼 스레드A
2. 서버 : HTTP 통신 수락 후 SseEmitter 객체 반환 - 플랫폼 스레드A 반납
3. **여기까지 SSE 연결 수립**
	- 이후 SSE 자체는 비동기로 수행되어 스레드 A를 반납하지만, 데이터를 쏴주기 위해 임의로 스레드 B 생성
4. 서버 : 초기 GPS 값(Look Aside 캐싱 전략) 조회 - 플랫폼 스레드 B
	- 캐시 먼저 조회
	- 없으면 DB 조회
5. 서버 -> 클라이언트 : 이후에 5분 타임아웃동안(while) 캐시에서 최근 위치 조회 - 플랫폼 스레드 B
6. 서버 : 5분이 지나면 연결 해제 - 플랫폼 스레드 B 반납
7. 클라이언트 : 네트워크 오류나 화면에서 벗어나거나 여러가지 이유로 연결 해제 - 플랫폼 스레드 B 반납

#### 플랫폼 스레드 메모리 할당 위치

플랫폼 스레드를 생성하면 메모리 상, 두 군데에 나누어서 데이터가 생긴다.
1. **힙 영역** : 스레드를 제어하기 위한 `java.lang.Thread` 객체
2. **Native Memory(OS 메모리)** : 플랫폼 스레드는 OS 스레드와 1:1 매핑되기 때문에, 생성될때마다 JVM 외부의 메모리 공간을 할당 받음

### 결과
SSE 연결은 5분동안 지속되어야 하기 때문에 **스레드를 반납하지 않고 점유**한다.
JVM이 GC를 할 것도 없이, 순수 스레드 사용량에 대한 메모리가 급증하게 된다.

플랫폼 스레드의 크기는 ~ 약 2MB이다.
이렇게 지속적으로 쌓인 스레드는 총 메모리 공간을 초과해 **Out Of Memory(OOM)** 으로 앱이 중단되는 결과를 낳는다. 
또한, 수많은 스레드가 CPU를 차지하기 위해 컨텍스트 스위칭을 경쟁하여 서버의 CPU 사용률이 치솟는다.

## 가상 스레드를 이용한 SSE 통신(실시간 이동 경로)

위의 문제를 해결하기 위해 스레드를 오랫동안 물리지 않게 두 가지정도의 방식이 있다.
데이터가 들어올 때만 잠깐 일하고 빠질 수 있도록 **이벤트 기반(Redis Pub/Sub, Kafka 등)** 이 있지만 별도의 인프라 없이 가상 스레드를 이용해서 해결해볼 것이다.

### 동작 순서
1. 클라이언트 -> 서버 : HTTP 통신 요청 - 가상 스레드 A 할당
2. 서버 : HTTP 통신 수락 후 SseEmitter 객체 반환 - 가상 스레드 A 반납
3. **여기까지 SSE 연결 수립**
4. 서버 : 초기 GPS 값(Look Aside 캐싱 전략) 조회 - 가상 스레드 B 할당 후 캐시 먼저 조회
5. 서버 -> 클라이언트 : 이후에 5분 타임아웃 동안 캐시에서 최근 위치 조회 후 전송 - 가상 스레드 B가 담당
	- 5분 동안 루프를 돌며 `Thread.sleep()`이나 I/O 작업을 만나 **대기 상태가 되는 순간,** 가상 스레드는 **캐리어 스레드를 놓아주고** 자신의 상태를 **힙(Heap)에 저장**한 채 **대기(Unmount)** 함. 즉, OS 스레드는 다른 일을 하러 감
6. 서버 : 5분이 지나면 연결 해제 - 가상 스레드 B 종료(**일반 객체처럼 GC의 대상**)
7. 클라이언트 : 네트워크 오류 등 여러 이유로 연결 해제 - 가상 스레드 B 종료

#### 메모리 할당 위치

가상 스레드의 크기는 ~10KB 수준이다.
가상 스레드는 플랫폼 스레드와 달리 OS 메모리가 필요하지 않고 **힙 영역에 할당된다.**
- 가상 스레드의 스택 데이터 전체가 힙 영역에 일반 자바 객체처럼 저장
- JVM이 미리 만들어둔 소수의 캐리어 스레드가 OS 메모리를 점유하고, 가상 스레드는 이 캐리어 스레드 위에 번갈아 올라타서 실행

### 결과

- SSE 연결이 5분 동안 지속되며 `while`문으로 스레드를 점유하더라도, **대기하는 시간 동안**에는 **OS 스레드를 물고 늘어지지 않음**

- 가상 스레드의 크기는 플랫폼 스레드(약 1MB)의 1,000분의 1 수준인 수백 바이트 ~ 수 킬로바이트(KB)에 불과하며 힙 영역에 저장

- 따라서 연결이 수만 개로 늘어나 가상 스레드가 수만 개 생성되더라도 메모리를 거의 차지하지 않아 **Native Memory 고갈로 인한 Out Of Memory(OOM)가 발생하지 않음.** (힙 메모리만 넉넉하면 수백만 개도 생성 가능)

- 또한, 스레드가 대기 상태에 들어갈 때 무거운 OS 수준의 컨텍스트 스위칭이 일어나지 않고, JVM 힙 영역에서 객체 포인터만 바꾸는 수준으로 아주 가볍게 처리되므로 **서버의 CPU 사용률도 평온하게 유지**

## Serial GC의 한계

Serial GC는 컨테이너의 CPU/메모리가 작으면 JVM이 자동으로 Serial을 선택한다. 
현재 app 컨테이너에 할당된 메모리는 1GB이므로 자동으로 Serial이 선택되었다.

가상 스레드가 힙 영역에 저장되는 것 자체는 메모리를 크게 차지하지 않는다. (몇 MB ~ 수십 MB 수준)
whlie 문의 sse polling 과정에서 캐시를 조회하고, DTO를 만들고, JSON으로 변환하는 작업등 수많은 단명 객체들이 힙 영역(Young Generation)에 쏟아진다.

가상 스레드가 힙 영역에 Unmount 되어 있는 동안에는 해당 스레드가 참조하고 있는 지역 변수들을 **살아 있는 객체**로 취급되어 지워지지 않는다. 따라서 힙 메모리가 계속 차오르고, GC는 자주 동작한다.

### 수 많은 broadcast() 에서 힙 사용량 증가

사용자에게 위치 값을 보낼 때 , 캐시에서 조회하고 새로운 객체를 만들고 등 힙 영역에 데이터가 쌓인다. 여기서 SseEmitter 객체는 계속해서 연결을 유지하고 있기 때문에 힙 영역의 객체를 장시간 참조하게 되어 GC를 할 수 있는 객체들이 적어서 공간을 계속 차지하고 있게 된다. 

Serial GC에서의 Full GC는 힙 공간이 다 차야 동작한다. 빠르게 차는 객체들 떄문에 STW는 여러번하게 되고 길게 하기 때문에 다른 애플리케이션 동작까지 영향을 준다.

```
가상 스레드 2,000 continuation 객체 (수 KB × 2K)
  + GPS 직렬화 byte[] copyOfRange 매 송신 신규 생성
  + emitter 내부 상태
  ↓
Young Gen 빠르게 채워짐 → Allocation Failure
  ↓
Serial GC 단일 스레드 → Full STW 910ms
  ↓
STW 중 모든 스레드 정지 → GPS 트랜잭션 멈춤
  ↓
DB 커넥션 쥔 채 pause → HikariCP 반납 지연
  ↓
새 GPS 요청 커넥션 대기 → pending 150 스파이크
  ↓
HikariPool timeout → k6 connection error
```

## G1GC

### 적용한 처방

1. **G1GC 전환** — `-XX:+UseG1GC -XX:MaxGCPauseMillis=200`. STW를 단일 스레드 포화형에서 region 선별 임계값형으로 바꿈
2. **Pre-serialization** — Jackson 직렬화를 1,500회 → 1회로. byte[] 할당량 절반 감소 → Eden 포화 빈도 ↓ → GC 호출 빈도 ↓
3. **Virtual Threads broadcast** — 동기 순차 send → 병렬 send
4. **computeIfPresent** — wardKey 맵 누수 차단 (장기 운영 Heap baseline 우상향 완화)

### 배운 점

- **컨테이너의 작은 힙 + 적은 코어 환경에서는 JVM 기본 GC가 Serial로 떨어진다** → 명시적으로 G1GC 지정 필요
- **`MaxRAMPercentage` 같은 컨테이너 인지 옵션과 GC 옵션은 분리된 결정** — 둘 다 챙겨야 한다
- **JFR + JMC 자동 분석**은 실시간 메트릭(p95, STW)이 잡지 못하는 메모리·할당 패턴까지 보여준다. Threads Allocating·Allocated Classes·GC Pressure 룰이 가장 신호 강함
- **Long-lived/Short-lived 객체 구분이 처방의 갈래를 만든다** — 둘을 하나로 묶어 보면 잘못된 처방으로 간다

#### Parallel GC를 안 쓰는 이유

```
Serial GC:    [스레드 1개로 힙 스캔]   → 500ms STW
Parallel:     [스레드 2개로 힙 스캔]   → ~250ms STW (절반)
G1GC:         [concurrent marking 병행] → 200ms 이하 목표 제어
```

Parallel은 STW를 **짧게 만들 뿐** 모든 앱 스레드가 멈추는 근본 구조는 동일하다. 운영 분포에서 thundering herd가 발생하는 메커니즘은 그대로다 — STW가 250ms로 줄어도 그 동안 들어온 25건이 큐에 적체된다. 게다가 Parallel은 처리량(throughput) 최적화 GC라 응답 지연이 중요한 서버에는 맞지 않는다.

#### ZGC를 안 쓰는 이유 (3가지)

1. **768MB 힙은 ZGC 최적 구간이 아님** — ZGC는 수 GB 이상 대용량 힙에서 concurrent 작업 이점이 커진다. 768MB에서는 G1GC 대비 이점이 거의 없다.
2. **vCPU 2 환경에서 CPU 경합** — ZGC는 앱 스레드와 동시에 도는 GC 백그라운드 스레드를 쓴다. 코어 2개뿐인 t3.medium에서 ZGC 스레드가 Tomcat 워커의 CPU를 잠식한다.
3. **Over-engineering** — 우리 목표는 "500ms STW를 200ms로 줄이자"다. ZGC의 <10ms는 실시간 트레이딩·게임 서버용이지, SSE GPS push에 필요한 정밀도가 아니다.

#### G1GC를 선택한 이유 (4가지)

1. **환경에 정확히 맞는 설계** — vCPU 2개 중 하나로 concurrent marking을 하면서 다른 하나로 앱을 계속 돌린다. Serial이 코어를 독점하던 구조를 해소.
2. **STW 목표 제어 가능** — `-XX:+UseG1GC -XX:MaxGCPauseMillis=200`으로 목표 시간 안에 수집할 region을 선별한다. "힙 전체 한 번에" 스캔하지 않는다. 운영 분포에서 thundering herd 적체량이 STW × RPS에 비례하므로, STW를 200ms 이하로 제어하면 적체 폭이 79건 → 20건으로 줄어들어 폭주가 자연 완화된다.
3. **SSE 서버 특성에 최적** — SSE 연결 객체는 장기간 살아남아 Old Gen으로 승격된다. G1의 **Mixed GC**(Young + 일부 Old region 선별 수집)는 SerialOld의 전체 Old Gen 스캔과 달리 1,500 연결 + 누적 wardKey 맵 상황에 효율적이다.
4. **한 줄 변경, 검증된 표준** — Java 9부터 서버 기본 GC. Spring Boot 4 + Java 21 조합에서 사실상 표준. 리스크 없이 즉시 적용 가능.

```bash
JAVA_OPTS=-XX:+UseG1GC -XX:MaxGCPauseMillis=200
```

> **단, 메모리 여유 확보가 선행돼야 한다.** 97% 메모리 상황에서는 G1의 region 메타데이터 오버헤드 때문에 오히려 GC 빈도가 늘 수 있다. 1차에서 끊은 메모리 천장 라인이 있기에 G1GC의 효과가 온전히 나온다.
