## 연산 함수를 제거하고 동등조건으로 바꾼 이유는?

인덱스를 타려면 컬럼이 where 절에서 가공된 상태이면 안됩니다. 정렬된 상태로 인덱스 탐색을 해야 하는데, 해당 컬럼이 가공되면 값이 바뀌기 때문에 인덱스를 탈 수 없는 환경이게 되어 풀테이블 스캔을 하게 됩니다. 
## OR 조건이 여러 개일 때 인덱스가 어떻게 동작하는가?

각각의 or 조건을 만족하는 첫 번째 데이터를 찾아서 각각 수직 탐색을 합니다.

## Cursor 기준 컬럼을 어떤 것으로 잡았나? 정렬 기준이 중요한 이유는?

현재 cursorId는 ticketId 기준입니다.
 row 값들이 순차적으로 정렬되어야 다음 커서값 기준으로 데이터 묶음을 불러올때 순차적으로 책갈피 역할을 할 수 있습니다. 따라서 클러스터링 인덱스 값인 ticketId를 기준으로 반드시 고유한 값이어야 하고 중간에 스킵하는 데이터 없이 가져와야 합니다.
## Offset 대신 Cursor Pagination을 선택한 이유는?

현재 요구사항 중 하나가 메인 페이지에서 식재료들을 날짜 내림차순으로 스크롤 형식으로 구현하도록 되어있습니다. 또한 식재료 데이터 값이 증대함에 따라 offset 으로 구현하게 되면 이후의 페이지에서 앞에 있는 식재료들까지 조회하게 되어 조회 성능 저하의 우려가 있습니다. 커서 기반으로 조회하게 되면 앞의 데이터는 조회할 필요없이 다음 커서값 기준으로 조회하게 되어 조회 성능 우려는 미미합니다. 따라서 커서 기반으로 조회하는 것이 합리적이라고 생각합니다
## Cursor Pagination의 단점은 없나?

cursorId 값을 순차적으로 알아야 하기 때문에 특정 페이지로의 데이터를 한번에 불러오지 못하고 앞의 데이터를 한번씩 다 불러와야 합니다.
## 450개 단위 기준이 뭔가?

firebase에서 배치로 보낼 수 있는 최대 크기가 500개입니다. 초과되지 않고 넉넉하게 여유를 두고 보내기 위해 페이지네이션으로 조회받는 개수를 450개로 정했습니다.

## 기존 Blocking I/O 방식의 문제가 뭐였나?

기존의 Blocking I/O 방식은 메인 스레드에서 외부 네트워크 통신을 진행했습니다. Firebase 문자 알림에서 다량의 식재료 알림을 보내야 했습니다. 처음에 firebase 서버에 요청할때 FcmToken의 유효성을 검증하는 단계에서 유효하지 않은 토큰들에 한해서 실패 응답을 받고 해당 Fcmtoken을 DB에서 지웁니다. 다량의 식재료와 fcmtoken을 요청하고 응답하는 과정에서 스레드와 커넥션 풀을 점유 하게 되어 이후에 병목 지점이 될 수 있을  것을 확인했습니다. 서비스 규모에 따라서 하루에 한번 수행되는 로직에 대해서 불필요한 최적화일 수도 있지만, 비즈니스 로직 중 중요한 알림 기능을 안정적으로 처리하기 위해서는 다른 API 요청과 분리되어 전송할 수 있는 환경에서 처리해야 한다고 생각했습니다.

## 스레드 풀 고갈이 발생하면 어떤 증상이 나타나나?

수행해야할 상태의 작업들이 큐에 쌓이게 되고, 만약 해당 로직이 커넥션 풀을 사용하게 된다면 커넥션 풀 점유 기간도 늘어나게 될 것입니다.

## OS 스레드와 가상 스레드의 차이는?

OS 스레드는 커널이 관리하며 컨텍스트 스위칭 비용이 큽니다. 반면, 가상 스레드는 JVM이 스케줄링합니다. 가상 스레드는 캐리어 스레드에 마운트되어 실행되다가, I/O 작업 등으로 블로킹이 발생하면 언마운트되어 캐리어 스레드를 양보합니다. 따라서 OS 스레드의 블로킹 없이 자원을 효율적으로 쓸 수 있습니다.

## 가상 스레드의 단점이나 주의할 점은?

가상 스레드는 연산 위주의 작업에는 사용하면 안된다. CPU를 지속적으로 점유하는 연산 위주의 작업에서는 컨텍스트 스위칭 오버헤드만 늘어나 오히려 성능이 저하될 수 있다.
또한 가상 스레드가 캐리어 스레드에 고정되어서 OS와 격리되는 pinning 현상이 발생할 수 있다.
pinning 현상을 방지하기 위해 synchronized 대신 reentrantlock 을 사용해야 합니다.

## 기존 스레드 풀을 늘리지 않고 가상 스레드를 바로 사용한 이유는?

스레드 풀을 늘리는 건 OS스레드를 더 만드는건데, OS 스레드는 하나당 약 1MB 스택 메모리를 차지하고 컨텍스트 스위칭 비용도 그대로입니다. 알림 발송처럼 I/O 대기가 대부분인 작업에서는 스레드 수를 늘려봤자 대기 중인 스레드만 늘어나는 거라 근본적인 해결이 안됩니다. 반면 가상 스레드는 I/O 대기 중에 캐리어 스레드를 반납하기 때문에, 수천 개를 만들어도 실제 OS 스레드는 소수만 사용합니다. 같은 비용으로 훨씬 많은 동시 요청을 처리할 수 있어서 스레드 풀 확장 대신 가상 스레드를 선택했습니다.

## 450개 배치 중 일부 FCM 전송이 실패하면 어떻게 처리했나?

`@Retryable`어노테이션으로 프록시 객체가 해당 메서드를 가로채서 가상 스레드 내에서 3번 재시도 처리하도록 했습니다. 만약 3번의 재시도에도 불구하고 전송에 실패했다면, 해당 메시지에 대해서 로그로 남겼습니다.
FcmToken 이 유효하지 않은 메시지는 재시도 로직을 처리 하지 않고 해당 토큰들을 리스트에 담아서 한번에 제거했습니다.

## 만약 네트워크 타임아웃으로 인해 FCM 서버로부터 응답을 받지 못하면 발송한 450건은 어떻게 처리되나?

타임아웃을 일반 예외와 구분해서 재시도 대상에 포함시키고, 3회 재시도에도 실패하게 되면, 해당 배치를 실패 로그로 남겨서 수동으로 해당 메시지에 대한 재시도를 추후에 재처리하도록 합니다.
