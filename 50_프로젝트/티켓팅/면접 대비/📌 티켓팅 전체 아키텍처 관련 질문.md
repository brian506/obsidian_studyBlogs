# Redis & 동시성 제어

## Q. Redis Lua Script로 재고를 관리한 이유가 뭔가요?

### 꼬리: Redis 자체가 Single Thread인데 굳이 Lua Script를 쓴 이유가 있나요? INCR/DECR 명령어와 뭐가 다른가요?
- 레디스가 싱글 스레드에서 단일 명령어에 대해 원자성을 보장하지만, 실제 비즈니스 로직은 조회/검증/수정 등 여러 단계의 명렁어로 이루어집니다. 명령어 사이에 다른 명령어가 들어올 수 있으므로 데이터 정합성이 깨지게 됩니다. Lua Script 는 이 여러 단계의 명령어를 하나의 스크립트로 바꾸어 하나의 명령어로 취급하게 되므로 원자성이 보장됩니다.
- INCR/DECR 는 단순히 숫자를 올리고 내리는 것은 가능하지만, 복잡한 조건부 비즈니스 로직을 처리할 수 없습니다.
### 꼬리: Redis가 장애났을 때 재고 데이터는 어떻게 복구하나요?
- 서비스 특성상 단기간 내에 티켓팅이 이루어질 것 이므로 장애 발생 시 즉각적인 대응이 필요합니다.  
- 따라서 먼저 클러스터가 아닌 sentinel을 두어 master 노드가 장애가 발생하더라도 바로 replica 노드로 대체할 수 있게 합니다.
- 또한 RDB 방식의 스냅샷을 이용하는 것보단 AOF 로그 파일로 everysec로 설정을 하여 1초 분량의 데이터까지 유실되는 데이터를 복구할 수 있도록 할 것 입니다. 

### 꼬리: Redisson의 분산 락 대신 Lua Script를 선택한 트레이드오프를 설명해주세요.
- Lua Script 를 선택한 가장 큰 이유는 네트워크 I/O 감소와 성능 극대화 때문이었습니다. Redisson을 사용하면 락 획득, 비즈니스 로직 수행, 락 해제 등 3번 이상의 네트워크 왕복이 발생합니다. 또한 락을 대기하면서 스레드가 블로킹 되어 전체적인 처리량 감소 우려가 있습니다. 
- 반면에 Lua Script 를 사용하게 되면 한 번의 네트워크 요청으로 락 획득/해제 과정 없이 원자적 처리가 가능해져서 성능이 향상됩니다.
- 하지만 스크립트 실행 시간이 길어지만 다른 명령어가 블로킹 되므로, 스크립트 자체를 가볍게 작성해야 했습니다.
## Q. Redis에 우선 마킹 후 검증하는 방식에서 Redis 데이터와 DB 데이터 사이의 정합성은 어떻게 보장했나요?

- Redis에는 티켓 발행 여부를 마킹해두고, 실제 영구 저장은 Kafka Consumer가 DB에 비동기로 동기화합니다. 결제 검증 시점에는 DB 대신 Redis를 먼저 조회하기 때문에 Consumer Lag이 발생해도 결제 로직이 정상 동작합니다. 최종적으로 Consumer가 DB에 저장을 완료하면 두 데이터가 일치하게 되므로 Eventual Consistency 방식으로 정합성을 보장했습니다.

### 꼬리: Kafka Consumer가 계속 Lag이 쌓이면 Redis 마킹 데이터는 언제까지 유지되나요? TTL은 어떻게 설정했나요?
- TTL 은 10분으로 설정했습니다. 사용자가 티켓 요청, PG 결제까지의 적정 완료 시간은 10분이면 될 것이라고 생각했습니다.

### 꼬리 : TTL이 만료됐는데 Consumer Lag으로 DB에 아직 저장이 안 됐으면 어떻게 되나요?
- TTL 만료 전에 Consumer가 DB에 저장을 완료하는 것이 정상 케이스이고, 만약 10분 내에 처리가 안 될 만큼 Lag이 심각하다면 이미 시스템 장애 상황이므로 DLT와 재처리 로직으로 대응합니다


## Q. Outbox 패턴과 Debezium CDC를 결합한 이유가 뭔가요?

- 결제 상태 변경과 메시지 발행은 독립적으로 수행되어야 합니다. DB는 성공하고 Kafka는 실패했을 경우에 데이터 불일치 문제가 발생합니다. 이를 해결하기 위해 DB에 저장되어 있는 데이터를 CDC가 읽는 방식으로 데이터 일관성을 지켰습니다.

### 꼬리: Outbox 패턴 없이 Kafka로 직접 이벤트 발행하면 어떤 문제가 생기나요?
- 하나의 트랜잭션 안에서 이벤트 발행까지 했을 경우에, 마지막에 DB 커밋은 못했는데, 카프카 이벤트 발행이 될 수도 있고, 커밋은 했지만 이벤트 발행이 안될 경우가 생길 수 있습니다. 따라서 아웃박스 테이블에 이벤트를 저장하는 것과 하나의 트랜잭션 안에서 실행되기 때문에, DB 커밋이 성공한 경우에만 Debezium이 이벤트를 읽어서 발행해야합니다.

### 꼬리: Outbox 패턴의 단점은 뭔가요? (At-least-once delivery → 중복 처리 문제)
- Outbox 패턴은 At-least-once delivery를 보장합니다. Debezium이 binlog를 읽고 Kafka에 발행한 뒤 오프셋을 커밋하기 전에 장애가 나면, 재시작 시 같은 이벤트를 다시 읽어서 중복 발행이 발생할 수 있습니다. 즉 최소 한 번은 전달되지만 두 번 이상 전달될 수도 있다는 것이 핵심 단점입니다.

### 꼬리: CDC 방식 말고 Transactional Outbox를 Polling으로 구현하는 방식과 비교했을 때 어떤 트레이드오프가 있나요?
- polling 방식은 별도의 인프라 설정이 없어도 되지만, 주기적으로 아웃박스 테이블에 SELECT 를 하기 때문에 실시간성이 떨어지고, polling 주기가 짧을수록 DB에 부하가 발생합니다. 
- CDC 방식은 추가 인프라 설정과 binlog 용량 관리와 커넥터 장애 시 별도 대응이 필요합니다. 반면에 실시간성과 DB에 부하가 일어나지 않습니다.

## Q. DLT(Dead Letter Topic)는 어떤 상황에서 메시지가 들어가도록 설계했나요?

- 티켓 발행, 결제 발행 이벤트를 소비할 때 3번의 시도 끝에 소비하지 못하는 상황에 DLT 로 들어갑니다.

### 꼬리: DLT에 쌓인 메시지는 어떻게 재처리했나요? 수동으로 했나요, 자동으로 했나요?
- DLT 에 쌓인 메시지는 DLT 토픽을 바라보는 별도의 컨슈머가 자동으로 처리도록 했습니다.

### 꼬리: DLT 메시지를 재처리할 때 멱등성은 어떻게 보장했나요?
- 해당 이벤트의 orderId 가 DB 에 먼저 존재하는지 확인 후 재처리합니다.

### 꼬리: Debezium 커넥터가 다운됐을 때 재시작 후 데이터 누락이 생기지 않나요?
- Debezium 커넥터가 중단되더라도 MySQL 의 Binlog 는 보존된다.
커넥터가 복구되면 마지막으로 기록된 오프셋 지점부터 다시 읽기 시작하므로 데이터 유실이 발생하지 않는다.
