## 1. 동시성 제어 - 락 비교

### 방식별 결과 요약

| 방식                   | TPS            | 문제점                           |
| -------------------- | -------------- | ----------------------------- |
| synchronized         | 정합성 깨짐         | @Transactional 커밋 전 락 해제 틈 발생 |
| 비관적 락                | 낮음             | 락 대기 → 커넥션 점유 시간 증가           |
| 낙관적 락                | DB 원자적 연산보다 낮음 | 충돌 시 재시도 폭주 → DB 부하 급증        |
| DB 원자적 연산            | 88 TPS, 평균 4초  | 커넥션 풀 고갈 위험                   |
| **Redis Lua Script** | **940 TPS**    | 채택                            |

### 자주 나오는 질문

**Q. synchronized가 단일 서버에서도 정합성이 깨진 이유?**
- `@Transactional` + `synchronized` 혼용 문제
- synchronized 락 해제 → 커밋 완료 전 다른 스레드 진입 → 커밋 안 된 데이터 읽음
- 멀티 서버에서는 JVM이 달라 synchronized 자체가 무의미

**Q. 낙관적 락이 유리한 상황?**
- 읽기 > 쓰기, 충돌 빈도 낮은 경우
- 사용자가 오래 들고 수정하는 시나리오 (락 유지 시간 길어지는 상황)

**Q. 88 TPS인데 Redis를 도입한 이유?**
- 단일 서버 4초 응답 → 동시 요청 시 스레드/커넥션 점유 → 풀 고갈 위험
- 목표 300 TPS 달성 불가
- Redis(메모리 기반)로 재고 이전 → DB 부하 분리

## 2. Redis Lua Script

### 왜 INCR/DECR 대신 Lua Script?
- Redis는 **단일 명령어**에 대해 원자성 보장
- 실제 로직은 `조회 → 검증 → 차감` 다단계 → 명령어 사이에 다른 명령어 삽입 가능
- Lua Script는 여러 명령어를 **하나의 원자적 단위**로 실행

### Redisson 분산락 vs Lua Script

| 항목 | Redisson 분산락 | Lua Script |
|------|----------------|------------|
| 네트워크 왕복 | 락 획득 + 로직 + 해제 (3회+) | 1회 |
| 락 대기 | 스레드 블로킹 발생 | 없음 |
| 코드 복잡도 | 상대적으로 간단 | 스크립트 직접 작성 |
| 적합한 경우 | 복잡한 비즈니스 로직 | 단순 원자적 연산 |

> 주의: Lua Script 실행 중 다른 명령어가 블로킹됨 → 스크립트는 가볍게 유지

### Redis 장애 대응
- **Sentinel**: Master 장애 시 Replica로 자동 Failover
- **AOF (everysec)**: 최대 1초 분량 데이터 유실로 제한 (RDB 스냅샷 대비 유실 최소화)
- Cluster vs Sentinel: 단기 이벤트 서비스 → 빠른 복구가 목적 → Sentinel 선택

## 3. Gap Lock & 데드락

### 데드락 발생 원인
- `[status, expiredAt]` 복합 인덱스에서 두 트랜잭션의 범위 스캔이 겹침
  - 스케줄러 트랜잭션: `expiredAt < NOW()` 범위 탐색
  - 결제 트랜잭션: `expiredAt > NOW()` 범위 탐색
  - status가 선두 컬럼이라 expiredAt 정렬이 뒤섞임 → 스캔 범위 겹침 → 데드락

### 해결: `expiredAt` 단일 인덱스
- expiredAt 기준으로 정렬 → 스케줄러와 결제 트랜잭션의 **스캔 범위가 물리적으로 분리**

### Gap Lock vs Next-Key Lock
- **Gap Lock**: 레코드 사이의 빈 공간만 잠금 (INSERT 방지)
- **Next-Key Lock**: Gap Lock + Record Lock 조합 (InnoDB 기본 동작)
- **발생 조건**: REPEATABLE READ 격리 수준에서 범위 스캔 시
- Gap Lock은 INSERT뿐 아니라 **UPDATE도 차단**

### InnoDB 데드락 감지 & 해소
- 대기 그래프(Wait-for Graph)로 순환 탐지
- 롤백 대상 기준: **undo log 양이 적은 트랜잭션** (변경 행 수 기준)
- 롤백된 트랜잭션은 deadlock 에러 반환 → 애플리케이션에서 재시도 처리 필요
- 확인 명령어: `SHOW ENGINE INNODB STATUS`

### Full Table Scan 시 락
- InnoDB: Table Lock이 아닌 **모든 레코드에 Next-Key Lock** → 사실상 테이블 전체 잠금 효과
- MyISAM: 명시적 Table Lock 사용
- Full Scan 시 트랜잭션 점유 시간 증가 → 커넥션 풀 고갈로 이어짐

## 4. Outbox 패턴 + Debezium CDC

### 왜 Outbox 패턴?
- DB 저장 성공 + Kafka 발행 실패 → 데이터 불일치 문제
- 해결: Outbox 테이블 저장 + DB 커밋을 **하나의 트랜잭션**으로 묶음
- Debezium이 binlog 읽어서 Kafka에 발행 → DB 커밋 성공한 이벤트만 발행 보장

### CDC vs Polling Outbox

| 항목 | CDC (Debezium) | Polling |
|------|----------------|---------|
| 실시간성 | 높음 | 낮음 (주기에 의존) |
| DB 추가 부하 | 없음 (binlog 읽기) | 있음 (SELECT 반복) |
| 인프라 복잡도 | 높음 (커넥터 관리) | 낮음 |
| 커넥터 장애 시 | 오프셋 기록으로 재시작 후 이어서 처리 | - |

### At-least-once & 멱등성
- Debezium: 오프셋 커밋 전 장애 → 재시작 시 **중복 발행 가능**
- 대응: `orderId` 기존 존재 여부 확인 후 처리 (멱등성 보장)

## 5. Kafka & 비동기 처리

### Consumer Lag 원인 & 해결

| 원인 | 해결 |
|------|------|
| 발행 > 소비 속도 | Consumer Scale-out (파티션 수와 동일하게) |
| DB I/O 병목 | Bulk INSERT 도입 |
| 단일 파티션 | 파티션 수 증가 + Consumer 수 동일하게 증가 |

### Kafka 파티션 : Consumer 관계
- Consumer 1개 = 파티션 1개 담당
- Consumer 수 > 파티션 수 → 놀고 있는 Consumer 발생
- 이상적: **파티션 수 = Consumer 수**

### acks 설정 트레이드오프

| 설정 | 내구성 | 지연 |
|------|--------|------|
| acks=0 | 없음 | 최소 |
| acks=1 | Leader 기록 확인 | 중간 |
| acks=all | 모든 ISR 기록 확인 | 최대 |

### Thread Sleep Polling 문제
- 요청마다 스레드 할당 → sleep 중 스레드 미반납 → 누적 → 스레드 풀 고갈
- 해결: SSE(Server-Sent Events), WebSocket, 또는 결제 검증 시 Redis 먼저 조회

### DLT (Dead Letter Topic)
- 3회 재시도 후 실패 시 DLT로 이동
- 별도 Consumer가 DLT를 구독하여 자동 재처리
- 멱등성: `orderId` 존재 여부 체크

## 6. 부하 테스트 지표

### TPS 목표 산정 근거
- 티켓팅 오픈 시 동시 접속자 3,000명 가정
- 그 중 10%가 동시 예매 요청 → **300 TPS** 목표

### P95 vs 평균(avg)
- 평균: 이상치(outlier)에 크게 영향받아 왜곡 가능
- **P95**: 95%의 사용자가 경험하는 응답 속도 → 실사용자 경험 반영
- 결제 서비스라면 **P99, P99.9까지 측정** 권장 (금융/결제는 단 한 번의 지연도 신뢰 손실)

### k6 설정 방식
- `ramping-arrival-rate`: 응답 속도와 무관하게 초당 요청 수를 강제 → **한계 TPS 측정에 적합**
- VU 기반: 응답 느려지면 자연스럽게 TPS 감소 → 트래픽 급증 상황 재현 어려움

## 7. Redis ↔ DB 정합성 (Eventual Consistency)

### 흐름
```
티켓 예매 요청
  → Redis에 마킹 (TTL 10분)
  → PG 결제
  → Kafka Consumer가 DB에 비동기 저장
  → 결제 검증 시 Redis 기준으로 판단
```

### TTL 만료 전 Consumer Lag 심각 시
- 정상 케이스: TTL 내 DB 저장 완료
- 장애 케이스: DLT + 재처리 로직으로 최종 일관성 보장

### 왜 이 설계를 선택했나?
- PG 결제 완료가 비즈니스 핵심 → DB 동기화 수 초 지연은 허용 가능 트레이드오프
- 결제 롤백 없이 DB 저장 실패 → 재시도/DLT로 반드시 저장 보장
