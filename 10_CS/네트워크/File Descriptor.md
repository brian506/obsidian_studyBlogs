## File Descriptor(FD)란?

  운영체제가 **열린 자원**마다 부여하는 *정수 번호*다.
  소켓(네트워크 연결), 파일, 파이프 등 모든 I/O 자원을 OS는 FD로 관리한다.

  fd 0 → stdin
  fd 1 → stdout
  fd 2 → stderr
  fd 3 → 클라이언트 A의 SSE 소켓
  fd 4 → 클라이언트 B의 SSE 소켓
  fd 5 → DB 커넥션
  ...

  **소켓**은 연결 통로임.
  - HTTP API 요청
	  - 연결 1개 위에서 요청을 여러 번 주고 받음
	  - 소켓 != 요청
- WebSocket
	- 연결 1개가 끊기지 않고 계속 유지

  SSE 연결 1개 = TCP 소켓 1개 = FD 1개 소비.
	- SSE 연결은 계속 붙들고 있으므로
  FD가 고갈되면 새 연결을 수락하는 `accept()` 시스템 콜이 실패한다.

## ulimit이란?

>  프로세스 하나가 **최대 몇 개의 FD를 열 수 있는지** OS 커널이 강제하는 한도다.

  기본값은 보통 1024다.
  서버 애플리케이션에선 이 기본값이 병목이 되는 경우가 많다.

  1024 (총 FD 한도)
  - 약 935개 (시스템 예약: stdin/stdout/stderr, 로그 파일, 라이브러리 소켓 등)
  = 실제 SSE 연결에 쓸 수 있는 FD ≈ 89개

### EC2 인스턴스 사양과 ulimit의 관계

  ulimit은 EC2 인스턴스 타입(t3.micro, t3.medium 등)과 직접적인 연관이 없다.
  t3.micro든 c5.4xlarge든 OS 기본값은 동일하게 1024다.

  다만 간접적인 한계는 존재한다.

  FD 1개 = 소켓 버퍼 등 메모리 소비
  ulimit을 65536으로 올렸어도 RAM이 부족하면 실제로는 그만큼 못 씀

  ulimit은 "최대 허용치"이고, 실제 가능한 연결 수는 RAM·CPU에 의해 더 낮아질 수 있다.

### 메모리 기반 한계 계산 예시 (t3.medium, 4GB RAM)

  JVM Heap:          2GB
  OS/기타:           500MB
  남은 여유:         1.5GB = 1,536,000KB

  SSE 연결당 8KB 가정:
  1,536,000 / 8 ≈ 192,000개 이론적 한계

  실제 안전 운용: 이론치의 50~60% = ~100,000개

## 부하 테스트 도구(k6)와 메모리

  VU당 메모리 소비

  k6는 VU 하나당 **독립적인 런타임(JS 엔진 + HTTP 클라이언트 + 상태)**을
  메모리에 올린다.

  VU 1: [JS 런타임] + [소켓] + [헤더/쿠키 상태] + [메트릭 버퍼] ≈ 443KB
  VU 2: ≈ 443KB
  ...
  VU 1000: ≈ 443KB
  ────────────────────────────────────
  총합: 1000 × 443KB ≈ 433MB

### SSE 테스트가 특히 메모리를 많이 먹는 이유

  일반 HTTP: 연결 → 요청 → 응답 → 연결 종료 → VU 대기 (메모리 일부 해제)
  SSE:       연결 → 요청 → 응답 계속 유지 → 연결 절대 안 닫힘

  일반 HTTP 테스트라면 VU들이 돌아가며 쉬기 때문에 실제 메모리 점유는 낮다.
  SSE 테스트는 **모든 VU가 연결을 동시에 붙들고 있으므로** VU 1개당
  메모리가 전부 동시에 점유된다.
