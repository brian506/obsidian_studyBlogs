# HTTP 1.1 vs 2 vs 3 (QUIC)

## 한 줄 비교

| 버전 | 전송 | 핵심 특징 | 출시 |
| --- | --- | --- | --- |
| HTTP/1.1 | TCP | Keep-Alive, 파이프라이닝, 텍스트 | 1997 |
| HTTP/2 | TCP + TLS | 멀티플렉싱, 바이너리, 헤더 압축, 서버 푸시 | 2015 |
| HTTP/3 | **UDP (QUIC)** + TLS 1.3 내장 | 진정한 멀티플렉싱, 빠른 핸드셰이크, 연결 마이그레이션 | 2022 |

## HTTP/1.1 (가장 오래 쓰임)

### 특징
- **Keep-Alive**: 한 TCP 연결로 여러 요청 (기본). 연결 비용 절감
- **Pipelining**: 응답 기다리지 않고 연속 요청 — 이론상 가능하지만 **HOL Blocking** 으로 사실상 X
- **텍스트 기반**: 사람이 읽기 쉽지만 파싱 비용

### 문제 — Head-of-Line (HOL) Blocking
한 TCP 연결에서 앞 요청이 느리면 뒤 요청들이 모두 대기.
→ 브라우저는 도메인당 **6개 정도의 병렬 TCP 연결** 로 우회 (도메인 샤딩 트릭)

## HTTP/2

### 핵심 개선
1. **바이너리 프레이밍** — 텍스트 → 바이너리 (파싱 빠르고 효율적)
2. **멀티플렉싱** — 하나의 TCP 연결에서 **여러 요청·응답 동시** 처리 (스트림 단위)
3. **헤더 압축 (HPACK)** — 반복 헤더(쿠키, 호스트) 압축
4. **서버 푸시** — 클라이언트가 요청하기 전에 서버가 리소스 미리 푸시 (실제론 잘 안 쓰여 deprecated 중)
5. **스트림 우선순위** — 중요 리소스 먼저

### HTTP/2의 한계 — TCP 레벨 HOL Blocking
HTTP/2가 멀티플렉싱이라 해도, **TCP 패킷 손실 시 모든 스트림이 멈춤** (TCP가 순서 보장하기 때문).
→ HTTP/3가 UDP로 가는 이유.

## HTTP/3 (QUIC)

### QUIC = UDP + 신뢰성·암호화 재구현
구글이 만들고 IETF 표준화. 사실상 **TCP를 대체하는 새 전송 프로토콜**.
- TCP의 패킷 도달 확인(ACK), 순서 보장, 유실 시 재전송 로직 모두 구현
	- 패킷이 유실되면 QUIC이 재전송

### 개선점
1. **진정한 멀티플렉싱** — UDP 기반이라 한 스트림 손실이 다른 스트림에 영향 X
2. **0-RTT 핸드셰이크** — 이전에 연결한 적 있으면 첫 패킷부터 데이터 전송 가능
   (TCP+TLS는 보통 2~3 RTT)
3. **TLS 1.3 내장** — 분리된 핸드셰이크 없음
4. **Connection Migration** — IP/포트 바뀌어도 (Wi-Fi → LTE) 연결 유지 (Connection ID 사용)
5. **개선된 혼잡 제어** — 더 정확한 RTT 측정

### 단점
- CPU 오버헤드 
- TCP에 최적화된 기존 네트워크 장비 및 방화벽과의 충돌
- 
### 사용 현황
- Google, YouTube, Facebook, Cloudflare, Akamai 광범위 사용
- HTTPS 사이트의 25%+ 가 HTTP/3 지원 (2024 기준)
- 단점: 일부 방화벽이 UDP 차단 → fallback 필요

## 정리 — 한눈에 보기

| 항목 | 1.1 | 2 | 3 |
| --- | --- | --- | --- |
| 전송 | TCP | TCP | **UDP (QUIC)** |
| 포맷 | 텍스트 | 바이너리 | 바이너리 |
| 다중 요청 | 6개 TCP 병렬 | 1개 TCP 멀티플렉싱 | 멀티플렉싱 + **스트림 독립** |
| TLS | 별도 (HTTPS) | 별도 + ALPN | **내장** |
| 헤더 압축 | 없음 | HPACK | QPACK |
| HOL Blocking | 애플리케이션 + TCP | TCP | **없음** |
| 핸드셰이크 RTT | TCP 1 + TLS 2 | TCP 1 + TLS 2 | **1 (또는 0)** |

## 면접 빈출 질문
1. **HTTP/1.1과 2의 가장 큰 차이?** → 바이너리 + 멀티플렉싱. 텍스트 파싱 부담과 HOL 일부 해소
2. **HTTP/2의 HOL Blocking은?** → 애플리케이션 레벨은 해결했지만 **TCP 레벨은 여전**. TCP 패킷 손실 시 모든 스트림 정지
3. **HTTP/3가 UDP를 쓰는 이유?** → TCP의 HOL 해결 + 핸드셰이크 단축 + 연결 마이그레이션
4. **QUIC가 신뢰성을 어떻게 보장?** → UDP 위에서 자체 순서·재전송·암호화 구현 (TCP를 다시 만든 셈)
5. **HTTP/2의 서버 푸시는 왜 잘 안 쓰이나?** → 클라이언트가 이미 있는 리소스도 푸시되어 비효율, 캐시와 충돌 → 표준에서 사실상 deprecated
6. **HTTP/3의 0-RTT란?** → 재방문 시 첫 패킷에 데이터를 실어 보내 핸드셰이크 없이 통신 시작

## 프로젝트 연결
- AWS CloudFront / ALB는 HTTP/2 기본 지원, HTTP/3는 설정 가능
- Nginx HTTP/2 활성화: `listen 443 ssl http2;`
- WebSocket은 HTTP/1.1 Upgrade 기반 → HTTP/2·3 환경에서도 사용 가능하지만 별도 처리
