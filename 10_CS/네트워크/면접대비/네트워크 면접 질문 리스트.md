
## 문제

실시간 알림 기능과 양방향 채팅 기능을 각각 구현한다고 가정했을 때, SSE(Server-Sent Events)와 WebSocket 중 어떤 기술을 선택할 것이며 그 기준은 무엇입니까? 또한, 다중 서버 환경에서 WebSocket을 사용할 경우 사용자의 연결 상태가 서버별로 파편화되는 문제가 발생하는데, 이를 메시지 브로커(Pub/Sub 등)를 활용해 어떻게 해결할 수 있는지 아키텍처 관점에서 설명해 주세요.

>[!.questions]- Keyword
> - 연결된 서버가 다를 때 메시지 전달 불가

## 문제

HTTP/2는 멀티플렉싱을 통해 HTTP/1.1의 애플리케이션 레벨 HOL(Head-of-Line) Blocking을 해결했지만, 여전히 TCP 레벨의 HOL Blocking 한계를 가지고 있습니다. 이 TCP 레벨의 한계가 구체적으로 어떤 상황에서 발생하는지 설명하고, HTTP/3(QUIC)가 UDP를 도입하여 이 문제를 어떻게 근본적으로 극복했는지 설명해 주세요.

>[!.questions]- Keyword
> HTTP/1.1 : 1개의 TCP에서 한 번에 1개의 요청/응답 처리
> 	- 한계 : 앞 요청이 느리면 뒤 요청 대기
> HTTP/2 : 단일 TCP에서 여러 개의 스트림(여러 개의 요청/응답) 처리 (멀티플렉싱)
> 	- 한계 : TCP 패킷 손실 시 모든 스트림 대기
> HTTP/3 : UDP기반 QUIC프로토콜 사용하여 여러 개의 스트림
