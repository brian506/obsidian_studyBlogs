# TCP vs UDP

> handshake 자세히는 [[TCP - IP 4계층]] 참고. 이 노트는 **비교·선택 기준** 중심.

## 한 줄 정의
- **TCP**: **연결 지향 + 신뢰성**. 정확하지만 느림 (HTTP, SMTP, FTP, SSH)
- **UDP**: **비연결 + 빠름**. 손실 가능 (DNS, 영상 스트리밍, 게임, VoIP)

## 비교표

| | TCP | UDP |
| --- | --- | --- |
| 연결 | 연결 지향 (3-way) | 비연결 |
| 신뢰성 | **보장** (재전송·순서) | 보장 X |
| 순서 | 보장 | 보장 X |
| 흐름 제어 | O | X |
| 혼잡 제어 | O | X |
| 헤더 크기 | 20~60 byte | **8 byte** (가벼움) |
| 전송 방식 | 스트림 (바이트) | 데이터그램 (패킷 단위) |
| 속도 | 느림 (오버헤드) | **빠름** |
| 용도 | 정확성 필요 | 실시간성 필요 |

## TCP가 신뢰성을 보장하는 4가지 메커니즘

### 1. 순서 번호 (Sequence Number)
- 모든 바이트에 번호 부여 → 받는 쪽이 순서 맞춰 재조립
- 도착 순서가 뒤바뀌어도 재정렬 가능

### 2. 확인 응답 (ACK)
- 받았다는 신호. 받지 못한 데이터는 **재전송**
- **누적 ACK**: "여기까지 다 받음" 한 번에 표시
- **SACK (Selective ACK)**: 부분적으로 받은 범위 알림

### 3. 흐름 제어 (Flow Control)
- **수신자가 처리 가능한 만큼만** 보내도록 조절
- **Sliding Window**: 수신자가 ACK와 함께 윈도우 크기(rwnd)를 알림 → 송신자는 그 만큼만 미리 보냄
- 빠른 송신자가 느린 수신자를 압도하는 것 방지

### 4. 혼잡 제어 (Congestion Control)
- **네트워크 상태**(혼잡)를 보고 송신 속도 조절
- 알고리즘:
  - **Slow Start**: 작은 윈도우(cwnd=1)에서 시작해 ACK 받을 때마다 2배 (지수 증가)
  - **Congestion Avoidance**: 임계점(ssthresh) 넘으면 선형 증가
  - **Fast Retransmit**: 중복 ACK 3번 받으면 즉시 재전송 (타임아웃 기다리지 않음)
  - **Fast Recovery**: 손실 후 천천히 회복
- 변형: **TCP Reno, Cubic(Linux 기본), BBR(Google)**

## UDP는 왜 빠른가
- 핸드셰이크 없음 (즉시 전송)
- 헤더 8 byte (TCP의 1/3 이하)
- 흐름·혼잡 제어 없음 → 보낼 수 있는 만큼 그냥 보냄
- 트레이드오프 = 손실·순서 책임을 **애플리케이션이** 직접 처리

## UDP 사용 사례
- **DNS**: 작고 빠른 질의·응답 (TCP fallback 있음)
- **DHCP**: IP 할당
- **NTP**: 시간 동기화
- **실시간 미디어**: VoIP, 화상회의, 게임 — 약간 손실되어도 멈추는 것보다 나음
- **QUIC (HTTP/3)**: UDP 위에 TCP 같은 신뢰성 + TLS를 새로 구현 → [[HTTP 1.1 vs 2 vs 3 (QUIC)]]

## TCP 사용 사례
- HTTP/1.1, HTTP/2, HTTPS
- SMTP / IMAP (메일)
- FTP / SFTP
- SSH
- DB 통신 (MySQL, PostgreSQL)

## 선택 기준 정리
1. **정확성·순서가 중요** → TCP
2. **약간의 손실 OK + 지연 민감** → UDP
3. **둘 다 필요** → UDP 위에 직접 구현 (QUIC, RTP)

## 면접 빈출 질문
1. **TCP와 UDP 차이?** → 연결성·신뢰성·헤더 크기·속도. 한 줄 정의로 시작
2. **TCP가 신뢰성을 보장하는 방법 4가지?** → 순서번호, ACK, 흐름 제어(슬라이딩 윈도우), 혼잡 제어
3. **흐름 제어와 혼잡 제어 차이?** → 흐름 제어=수신자 처리 한도 / 혼잡 제어=네트워크 상태
4. **Slow Start가 뭔가?** → 작은 윈도우로 시작해 ACK마다 2배씩 늘려가는 혼잡 제어 시작 단계
5. **UDP가 빠른 이유?** → 핸드셰이크·확인 응답·제어 없음, 헤더 작음
6. **DNS는 왜 UDP인가?** → 작은 질의·응답이라 신뢰성보다 속도. 512 byte 초과 시 TCP 사용
7. **HTTP/3가 UDP를 쓰는 이유?** → TCP의 HOL(Head-of-Line) blocking 회피, 핸드셰이크 단축
