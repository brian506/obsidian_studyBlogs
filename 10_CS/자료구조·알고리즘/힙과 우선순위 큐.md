# 힙과 우선순위 큐

## 힙 (Heap)
**완전 이진 트리** 형태의 자료구조. 부모-자식 간 대소 관계만 지킴 (형제끼리는 무관).

- **Max Heap**: 부모 ≥ 자식 → 루트가 최댓값
- **Min Heap**: 부모 ≤ 자식 → 루트가 최솟값

> ⚠️ "정렬된 트리"가 아님. 루트만 최대/최소가 보장됨.

## 배열로 구현 (0-indexed)
- 부모: `(i - 1) / 2`
- 왼쪽 자식: `2i + 1`
- 오른쪽 자식: `2i + 2`

연속 메모리 → **캐시 친화적**, 포인터 없이 단순.

## 연산
| | 시간 |
| --- | --- |
| 최댓값/최솟값 조회 (peek) | **O(1)** |
| 삽입 (push) | O(log N) — 위로 올라가며 swap |
| 삭제 (pop) | O(log N) — 마지막 원소를 루트로 옮긴 뒤 아래로 swap |
| 임의 원소 검색 | O(N) |
| 배열로부터 힙 만들기 | **O(N)** (heapify, bottom-up) |

## 우선순위 큐 (Priority Queue)
- "우선순위가 가장 높은 원소를 먼저 꺼냄"이라는 **추상 자료형(ADT)**
- 내부 구현은 보통 **힙**
- 이름은 큐지만 **FIFO 아님**

## 활용 사례
- **Dijkstra 최단 경로**, Prim MST
- **Top-K** 문제: "가장 큰/작은 K개" — Min Heap을 K 크기로 유지
- **작업 스케줄러** (우선순위 기반)
- **Median finder**: Max Heap + Min Heap 두 개로 중앙값 O(log N)
- **이벤트 시뮬레이션**, A* 탐색

## Java PriorityQueue
```java
// Min Heap (기본)
PriorityQueue<Integer> minHeap = new PriorityQueue<>();

// Max Heap
PriorityQueue<Integer> maxHeap = new PriorityQueue<>(Comparator.reverseOrder());

// 객체 + 커스텀 정렬
PriorityQueue<Task> pq = new PriorityQueue<>(Comparator.comparingInt(t -> t.priority));

pq.offer(t);  // O(log N)
pq.poll();    // O(log N)
pq.peek();    // O(1)
```

⚠️ Java PriorityQueue는 **순회 시 정렬 순서 보장 X** (peek만 보장). 정렬된 순회는 poll 반복.

## 힙 정렬 (Heap Sort)
1. 배열을 힙으로 만들기 — O(N)
2. 루트(최대) pop을 N번 — O(N log N)
- **전체 O(N log N)**, **In-place**, **불안정 정렬**

## 면접 빈출 질문
1. **힙과 BST의 차이?** → BST는 좌·우 대소 관계 완전 유지 / 힙은 부모-자식 관계만 (형제 무관). 힙은 최댓값/최솟값 조회에 특화
2. **PriorityQueue의 내부 구조?** → 보통 이진 힙 (배열 기반)
3. **힙에서 최댓값 조회와 삭제 복잡도?** → 조회 O(1), 삭제 O(log N)
4. **Top-K 문제 푸는 법?** → 크기 K짜리 힙 유지 → 시간 O(N log K), 공간 O(K)
5. **힙 정렬과 퀵 정렬 비교?** → 둘 다 평균 O(N log N), 힙은 최악도 O(N log N)·in-place지만 보통 퀵이 캐시 친화로 더 빠름
6. **배열을 힙으로 만드는 데 O(N)이 가능한 이유?** → bottom-up heapify의 비용 합이 등비 수렴 → O(N) (각 노드 sift-down O(log N) ×N이 아님)
7. **Java PriorityQueue를 그냥 순회하면 정렬되어 나오나?** → 아니오. poll을 반복해야 정렬 순서

## 응용
- Kafka 파티션 내 메시지 처리도 일종의 우선순위 큐로 모델링 가능 (priority topic)
- 작업 스케줄러, 리소스 할당, 게임 이벤트 큐
