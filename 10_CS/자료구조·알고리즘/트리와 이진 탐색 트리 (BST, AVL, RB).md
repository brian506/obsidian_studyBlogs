# 트리와 이진 탐색 트리 (BST, AVL, RB)

## 용어
- **노드, 간선, 루트, 리프**
- **깊이(Depth)**: 루트에서 해당 노드까지 거리
- **높이(Height)**: 해당 노드에서 가장 먼 리프까지 거리
- **이진 트리(Binary Tree)**: 자식이 최대 2개

## 이진 탐색 트리 (BST)
- 각 노드 기준: **왼쪽 < 자기 < 오른쪽**
- 탐색·삽입·삭제 평균 **O(log N)**
- ⚠️ **최악 O(N)**: 정렬된 데이터를 그대로 삽입 → 한쪽으로 치우친 트리 (사실상 연결 리스트)
  → 해결책 = **균형 트리 (Self-Balancing)**

## 균형 이진 탐색 트리

### AVL Tree
- 모든 노드의 좌·우 서브트리 **높이 차이가 1 이하**
- 삽입/삭제 후 어긋나면 **회전(Rotation)** 으로 복구
- 매우 엄격한 균형 → 탐색 빠름, 삽입/삭제 비용은 다소 ↑

### Red-Black Tree
- 노드에 빨강/검정 색칠 + 5가지 속성으로 균형 유지
  1. 루트는 검정
  2. 빨강 노드의 자식은 검정 (빨강 연속 X)
  3. 모든 리프(NIL)는 검정
  4. 루트→리프 모든 경로의 검정 노드 수가 같음
- AVL보다 균형은 느슨하지만 **삽입/삭제가 더 효율적**
- **Java TreeMap, TreeSet, HashMap의 트리화 버킷**, Linux 커널 스케줄러 등에서 사용

### AVL vs Red-Black 정리
| | AVL | Red-Black |
| --- | --- | --- |
| 균형 | 엄격 | 느슨 |
| 탐색 속도 | 약간 빠름 | 약간 느림 |
| 삽입/삭제 | 느림 (회전 자주) | 빠름 |
| 용도 | 조회 위주 | 균형 잡힌 워크로드 |

## B-Tree / B+ Tree
- 한 노드에 여러 키 저장 + 자식이 여러 개 (다진 트리)
- **디스크/페이지 I/O 최소화**가 목적 → DB 인덱스 표준
- B+ Tree: 데이터는 **리프에만**, 리프끼리 연결 리스트 → 범위 검색 빠름
- 자세히는 [[10_CS/데이터베이스/인덱스/인덱스 구조 (B-Tree 자료구조)]] 참고

## 트리 순회 (Traversal)
이진 트리 기준:
- **전위(Pre-order)**: 자기 → 왼쪽 → 오른쪽 — 트리 복사
- **중위(In-order)**: 왼쪽 → 자기 → 오른쪽 — **BST에서 정렬된 순서** 출력
- **후위(Post-order)**: 왼쪽 → 오른쪽 → 자기 — 트리 삭제
- **레벨 순회(BFS)**: 같은 깊이부터

## 기타 트리
- **Trie**: 문자열 prefix 검색에 특화 (자동완성, 사전)
- **Segment Tree**: 구간 합·최소 등 range query O(log N)
- **Fenwick Tree (BIT)**: 구간 합 전용, 더 가볍게

## 면접 빈출 질문
1. **BST의 시간 복잡도와 최악 케이스?** → 평균 O(log N), 최악 O(N) — 한쪽 치우침
2. **균형 BST가 필요한 이유?** → 최악 O(N) 방지 → 항상 O(log N) 보장
3. **AVL과 Red-Black 차이?** → 균형 엄격도. AVL은 조회 빠름, RB는 삽입/삭제 빠름
4. **DB 인덱스가 BST 대신 B+ Tree를 쓰는 이유?** → 디스크 I/O 최소화 (한 노드에 많은 키 저장 → 트리 높이 ↓), 리프 연결로 range 빠름
5. **In-order traversal로 BST를 순회하면?** → 정렬된 순서로 출력됨
6. **Java TreeMap의 내부 구조?** → Red-Black Tree
7. **Trie를 쓰는 경우?** → prefix 기반 검색 (자동완성, IP 라우팅, 사전)

## Java Collections 매핑
- `TreeMap` / `TreeSet` → Red-Black Tree
- `HashMap`의 충돌 시 트리화 → Red-Black Tree

## 프로젝트 연결
- DB 인덱스 = B+ Tree → 인덱스 노트와 함께 학습
- MongoDB도 기본 인덱스는 B-Tree 계열
