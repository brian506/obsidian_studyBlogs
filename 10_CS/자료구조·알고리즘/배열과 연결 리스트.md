# 배열과 연결 리스트

## 한 줄 요약
- **배열(Array)**: 메모리에 **연속적**으로 배치, 인덱스 접근 O(1)
- **연결 리스트(Linked List)**: 노드들이 **포인터로 연결**, 삽입/삭제 O(1) (위치 알면)

## 비교표

| 연산 | 배열 (정적/동적) | 단일 연결 리스트 |
| --- | --- | --- |
| 인덱스 접근 | **O(1)** | O(N) |
| 검색(값) | O(N) | O(N) |
| 끝에 삽입 | O(1) amortized | O(1) (tail 포인터 있으면) |
| 중간 삽입/삭제 | **O(N)** (밀기) | **O(1)** (위치 알면) — 위치 찾기는 O(N) |
| 메모리 | 연속, 캐시 친화 | 분산, 노드마다 포인터 오버헤드 |

## 동적 배열 (ArrayList, Vector)
- 내부적으로 배열을 갖고, 꽉 차면 **새 배열(보통 1.5~2배)을 만들어 복사**
- `add()`는 **amortized O(1)** — 가끔 발생하는 O(N) 확장 비용을 평균 내면 상수
- 중간 삽입/삭제는 여전히 O(N) (뒤 원소 밀어야 함)

## 연결 리스트의 변형
- **단일(Singly)**: next만 — 역방향 순회 불가
- **이중(Doubly)**: prev, next — Java `LinkedList`
- **원형(Circular)**: 끝이 처음을 가리킴 — 라운드 로빈 스케줄러 등

## 캐시 친화성 (중요)
- 배열: 연속 메모리 → **공간 지역성** 활용, CPU 캐시 히트율 ↑
- 연결 리스트: 노드가 흩어져 있음 → 캐시 미스 빈발
- **현실에서는 N이 같아도 배열이 훨씬 빠른 경우가 많음** (이론적 복잡도가 같아도)
- → 면접에서 "LinkedList가 중간 삽입 빠르니까 좋다"는 단순 답변은 주의

## Java Collections 매핑
| | 자료구조 | 비고 |
| --- | --- | --- |
| `ArrayList` | 동적 배열 | 기본 선택, 인덱스 접근 빠름 |
| `LinkedList` | 이중 연결 리스트 + Deque | 실무에선 거의 안 씀 — 캐시 친화 X, 오버헤드 ↑ |
| `Vector` | 동적 배열 + synchronized | 레거시. 동시성 필요하면 `Collections.synchronizedList` 또는 `CopyOnWriteArrayList` |

## 언제 무엇을 쓸까
- **인덱스 접근 / 순회 위주 → ArrayList** (대부분의 경우)
- **양끝 삽입·삭제 자주 → ArrayDeque** (LinkedList보다 빠른 큐·스택)
- **중간 삽입·삭제가 진짜 빈번하고 위치 포인터를 들고 있다 → LinkedList**

## 면접 빈출 질문
1. **배열과 연결 리스트의 차이?** → 메모리 배치(연속 vs 분산), 인덱스 접근 vs 삽입/삭제 트레이드오프
2. **ArrayList의 add가 O(1)인가 O(N)인가?** → amortized O(1) — 평소엔 O(1), 확장 시 O(N)이지만 평균 내면 O(1)
3. **LinkedList가 ArrayList보다 빠른 상황?** → 위치 포인터를 들고 있는 상태에서 중간 삽입/삭제. 그 외엔 대부분 ArrayList가 빠름
4. **왜 캐시 측면에서 배열이 유리한가?** → 연속 메모리는 공간 지역성을 활용 → 캐시 히트율 ↑
5. **Java의 LinkedList를 Stack/Queue로 써도 되나?** → 가능하지만 `ArrayDeque`가 더 빠르고 권장
