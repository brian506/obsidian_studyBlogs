# 해시 테이블 (Hash Table)

## 핵심
- **해시 함수**가 키 → **버킷 인덱스**로 변환 → 배열의 그 위치에 저장
- 평균 **O(1)** 의 삽입·조회·삭제

```
key → hash(key) % bucketSize → bucket index → 저장
```

## 해시 함수의 좋은 조건
1. **고른 분포** (충돌 적음)
2. **빠른 계산**
3. **결정성** (같은 입력 → 같은 출력)
4. Java에서는 `Object.hashCode()` 가 이 역할

## 해시 충돌 (Collision)
서로 다른 키가 같은 버킷에 매핑되는 현상.

### 해결 1: Chaining (분리 연결법)
- 같은 버킷에 **연결 리스트**(또는 트리)로 매달기
- Java HashMap이 사용. Java 8+는 한 버킷 노드가 8개 넘으면 **레드-블랙 트리로 변환** → 최악 O(log N)

### 해결 2: Open Addressing (개방 주소법)
- 충돌 시 **다른 빈 버킷 탐색**
- 방식: **선형 탐사**, 이차 탐사, 더블 해싱
- 캐시 친화적이지만 **클러스터링** 문제 가능

## Load Factor와 Resize
- **Load Factor** = `저장된 원소 수 / 버킷 수`
- Java HashMap 기본: **0.75 초과 시 버킷 2배 확장 + 재해싱**
- 확장은 비용이 크지만 (전체 재배치) 평균적으로 amortized O(1) 유지

## 시간 복잡도
| 연산 | 평균 | 최악 |
| --- | --- | --- |
| get / put / remove | **O(1)** | O(N) → Java 8+ 트리화 시 O(log N) |
| 순회 | O(N + bucketSize) | — |

## Java HashMap의 핵심 메커니즘
1. `key.hashCode()` → 32bit 해시
2. 추가 해시 함수로 분포 개선 (`hash ^ (hash >>> 16)`)
3. `hash & (n - 1)` 로 버킷 인덱스 (n은 2의 거듭제곱)
4. 충돌 시 연결 리스트 → 8 이상이면 트리화
5. Load factor 0.75 초과 시 resize

## equals와 hashCode 계약
- `a.equals(b) == true` → 반드시 `a.hashCode() == b.hashCode()`
- 역은 성립 안 함 (충돌 가능)
- **둘 중 하나만 오버라이드하면 버그** — HashMap에서 값 못 찾음
- Java에서는 `@EqualsAndHashCode` (Lombok), `record` 활용

## HashMap vs HashSet vs LinkedHashMap vs TreeMap
| | 자료구조 | 순서 | 정렬 | 비고 |
| --- | --- | --- | --- | --- |
| HashMap | 해시 테이블 | 없음 | X | 가장 일반적 |
| LinkedHashMap | 해시 + 이중 연결 리스트 | **삽입 순서** | X | 캐시 구현 시 유용 (LRU) |
| TreeMap | **레드-블랙 트리** | 키 정렬 순서 | O(log N) | range query 가능 |
| HashSet | HashMap의 키만 사용 | 없음 | X | 중복 제거 |

## 멀티스레드 안전성
- `HashMap` — **스레드 안전 X**, 동시 쓰기 시 무한 루프 가능 (Java 7 이슈)
- `Hashtable` — 모든 메서드 synchronized → 느림, 레거시
- **`ConcurrentHashMap`** — **세그먼트/노드 단위 락**, 동시성 안전 + 빠름

## 면접 빈출 질문
1. **HashMap이 O(1)인 이유와 최악은 왜 O(N)?** → 해시로 직접 인덱싱 → 평균 O(1). 충돌 다 한 버킷에 몰리면 리스트 순회 O(N), Java 8+는 트리화로 O(log N)
2. **충돌 해결 방법 두 가지?** → Chaining(연결 리스트/트리), Open Addressing(선형 탐사 등)
3. **equals만 오버라이드하면 어떻게 되나?** → 같은 객체로 인식해도 hashCode가 달라 다른 버킷 → 못 찾음
4. **HashMap의 resize 시점과 비용?** → Load factor(기본 0.75) 초과 시 2배 확장 + 전체 재해싱. amortized O(1)
5. **HashMap vs ConcurrentHashMap?** → 후자는 멀티스레드 안전. 락 분할 + CAS로 동시성 성능 ↑
6. **LinkedHashMap의 용도?** → 삽입/접근 순서 유지 → LRU 캐시 구현 가능
7. **TreeMap을 쓰는 경우?** → 키 정렬 / range query (subMap, headMap)

## 응용
- **Redis** = 분산 해시 테이블이 본질 (단일 노드는 dict 자료구조)
- **HashMap을 캐시로** — LinkedHashMap의 accessOrder 옵션으로 LRU 구현 가능
- **DB 인덱스** 중 Hash Index — 정확 매칭은 빠르지만 range 못 함 (B-Tree와 비교)
