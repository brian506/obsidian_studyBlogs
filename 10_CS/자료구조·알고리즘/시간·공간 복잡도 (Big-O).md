# 시간·공간 복잡도 (Big-O)

## 왜 필요한가
입력 크기 N이 커질수록 알고리즘이 얼마나 느려지는지를 **추세**로 나타내는 표기법.
→ 머신·언어·상수 무시하고 **알고리즘의 본질적 성능**을 비교.

## 표기법 3종 (셋 다 알아야 함)
- **Big-O (O)**: **상한** — 최악의 경우 N이 무한히 커질 때 이 이상으로는 안 느려진다
- **Big-Ω (Ω)**: **하한** — 최선의 경우
- **Big-Θ (Θ)**: **상·하한 일치** — 정확한 차수

면접에서는 보통 **Big-O = 최악 시간복잡도** 의미로 씀.

## 복잡도 순위 (빠름 → 느림)
```
O(1)  <  O(log N)  <  O(N)  <  O(N log N)  <  O(N²)  <  O(2^N)  <  O(N!)
상수      로그        선형       선형로그       이차        지수        팩토리얼
```

## 예시와 직관
| 복잡도 | 예시 | N=10⁶일 때 대략 |
| --- | --- | --- |
| O(1) | 배열 인덱스 접근, HashMap get | 즉시 |
| O(log N) | 이진 탐색, BST 탐색, 힙 push/pop | ~20번 |
| O(N) | 단순 순회, 한 번의 for | 10⁶번 |
| O(N log N) | 병합·퀵·힙 정렬 | ~2×10⁷ |
| O(N²) | 이중 for, 버블·삽입 정렬 | 10¹² → 불가 |
| O(2^N) | 부분집합 모두 탐색 | N=30도 한계 |
| O(N!) | 순열 모두 탐색 | N=10도 한계 |

> 일반적으로 코딩 테스트: **N ≤ 10⁶ → O(N log N) 이하** 목표

## 분할 상환 분석 (Amortized)
- 평균적으로는 빠른데 가끔 한 번 오래 걸리는 경우, **여러 연산 평균**으로 계산
- 예: ArrayList의 add는 보통 O(1), 가끔 크기 2배 확장으로 O(N) → 평균 **amortized O(1)**

## 평균 vs 최악
같은 자료구조도 케이스마다 다름. 면접에서 자주 묻는 차이:

| 자료구조/연산 | 평균 | 최악 |
| --- | --- | --- |
| HashMap get/put | O(1) | O(N) (충돌 폭주) → Java 8+ 트리화 시 O(log N) |
| QuickSort | O(N log N) | O(N²) (피벗 편향) |
| 동적 배열 add | O(1) amortized | O(N) (확장 시) |

## 공간 복잡도
- 입력 외 **추가로 쓰는 메모리**의 함수
- 재귀의 콜 스택도 포함 (DFS 깊이 = O(N))

## 실수하기 쉬운 포인트
- `O(N + M)` ≠ `O(N) + O(M)` 가 아니라 **둘 다 같이 표기**해야 함 (다른 입력 두 개일 때)
- 상수항·계수는 무시: O(2N) = O(N), O(N + 100) = O(N)
- **로그의 밑은 무시**: log₂N과 log₁₀N은 상수배 차이라 모두 O(log N)
- "**N**이 무엇인가" 명확히: 노드 수? 간선 수? 입력 문자열 길이?

## 면접 빈출 질문
1. **Big-O가 뭔가?** → 입력 크기 증가에 따른 최악 시간 복잡도의 상한
2. **O(log N)이 나오는 대표 알고리즘?** → 이진 탐색, BST·힙·B-Tree 연산
3. **HashMap이 O(1)인데 왜 최악은 O(N)?** → 해시 충돌이 다 한 버킷에 몰리면 리스트 순회. Java 8+는 트리화로 O(log N)
4. **QuickSort가 평균 N log N인데 왜 N² 최악?** → 피벗이 항상 최소/최대로 선택되면 분할이 안 됨. 랜덤 피벗·중앙값 피벗으로 완화
5. **재귀의 공간 복잡도는?** → 호출 깊이만큼 콜 스택 차지
