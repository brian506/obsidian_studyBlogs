# 메모리 관리와 가상 메모리

## 왜 필요한가
- 물리 메모리(RAM)는 한정 → 여러 프로세스를 안전·효율적으로 올려야 함
- 각 프로세스가 **독립된 주소 공간**을 가져야 함 (서로 침범 방지)
- 메모리보다 큰 프로그램도 실행 가능해야 함

## 연속 할당의 문제 → 단편화
- **외부 단편화(External)**: 사용 가능한 공간이 잘게 흩어져 큰 요청을 못 받음
- **내부 단편화(Internal)**: 할당 단위가 요청보다 커서 안에 공간이 남음
→ 해결: **페이징 / 세그멘테이션**

## 페이징 (Paging)
- 메모리를 **고정 크기 페이지**로 분할 (보통 4KB)
- 프로세스의 가상 주소 = `페이지 번호 + 페이지 내 오프셋`
- **페이지 테이블**: 가상 페이지 → 물리 프레임 매핑
- 외부 단편화 X, 내부 단편화 ↑ (마지막 페이지)

## 세그멘테이션 (Segmentation)
- **논리 단위(Code/Data/Stack)** 별로 가변 크기 분할
- 외부 단편화 발생, 내부 단편화 X
- 현대 OS는 **페이징 + 세그먼트** 혼용

## 가상 메모리 (Virtual Memory)
- **요구 페이징(Demand Paging)**: 필요할 때만 디스크 → RAM으로 로드
- 효과: 실제 메모리보다 큰 프로그램 실행 가능, 다중 프로그래밍 ↑

## 페이지 폴트 (Page Fault)
가상 주소 접근 시 페이지가 RAM에 없으면 발생.

처리 흐름:
1. MMU가 페이지 테이블 확인 → invalid
2. OS가 디스크에서 페이지 로드
3. 메모리에 빈 프레임 없으면 **페이지 교체 알고리즘**으로 victim 선택
4. 페이지 테이블 갱신 후 명령어 재실행

## 페이지 교체 알고리즘
- **FIFO**: 가장 먼저 들어온 페이지 교체. **Belady's Anomaly** (프레임 늘려도 폴트 증가) 가능
- **Optimal(OPT)**: 앞으로 가장 늦게 쓸 페이지 → 미래 알아야 하므로 이론적 기준
- **LRU (Least Recently Used)**: 가장 오래 안 쓴 페이지 → 시간 지역성 활용. 실제로 가장 많이 쓰임
- **LFU**: 사용 빈도 가장 낮은 것

## TLB (Translation Lookaside Buffer)
- 페이지 테이블의 **캐시** (CPU MMU 내부)
- TLB 히트: 메모리 접근 1회 / TLB 미스: 메모리 접근 2회 (테이블 + 데이터)

## Thrashing (스래싱)
- 페이지 폴트가 너무 자주 발생해 CPU가 페이지 교체에만 시간을 씀 → 처리량 급락
- 원인: 메모리에 비해 너무 많은 프로세스 동시 실행
- 해결: **Working Set 모델** (자주 쓰는 페이지 집합을 메모리에 유지), **Page Fault Frequency** 조절

## 면접 빈출 질문
1. **가상 메모리가 왜 필요한가?** → 메모리보다 큰 프로그램 실행, 프로세스 간 격리, 효율적인 메모리 사용
2. **페이지와 세그먼트의 차이?** → 고정 크기 vs 가변 크기, 단편화 종류 다름
3. **페이지 폴트 처리 과정?** → MMU 확인 → OS 트랩 → 디스크 로드 → 교체 알고리즘 → 테이블 갱신 → 재실행
4. **LRU와 FIFO 차이? LRU가 왜 더 좋은가?** → LRU는 시간 지역성을 반영, FIFO는 Belady's Anomaly 가능
5. **TLB가 무엇이고 왜 필요한가?** → 페이지 테이블 캐시. 매 접근마다 페이지 테이블 읽기 비용을 줄임
6. **스래싱(Thrashing)이란?** → 페이지 폴트가 폭증해 실제 작업보다 페이지 교체에 시간이 더 드는 현상

## 추가 개념
- **MMU(Memory Management Unit)**: 가상 → 물리 주소 변환 담당 하드웨어
- **Copy-on-Write**: fork() 시 부모/자식이 메모리를 공유하다가 변경 시점에 복사 → 효율적
- **mmap**: 파일을 메모리에 매핑 (DB·검색엔진에서 활용)
