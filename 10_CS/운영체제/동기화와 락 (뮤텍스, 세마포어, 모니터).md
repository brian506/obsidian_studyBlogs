

## 왜 필요한가
멀티스레드가 **공유 자원**(전역 변수, DB 레코드 등)에 동시에 접근하면 **Race Condition** 발생 → 결과가 실행 순서에 따라 달라짐.
→ **임계 구역(Critical Section)** 을 한 번에 하나의 스레드만 진입하도록 제어해야 함.

## 임계 구역 해결 조건 (3가지)
1. **상호 배제(Mutual Exclusion)**: 동시에 하나만 진입
2. **진행(Progress)**: 진입 가능한 스레드가 있으면 즉시 진입 결정
3. **한정 대기(Bounded Waiting)**: 대기하는 스레드는 유한한 시간 안에 진입

## Spinlock
- 락을 얻을 때까지 무한 루프를 돌며 계속 화인 (= Busy Waiting)
- 스레드가 CPU를 쥔 상태로 계속 대기하며 확인하므로, **컨텍스트 스위칭 발생 안함**
- 단점 : 안에서 동작하는 스레드를 계속 기다려야 하므로, CPU 계속 낭비

## Mutex (뮤텍스)
- 락을 얻지 못하면 스레드는 대기상태로 들어가고, 커널은 다른 스레드에게 CPU 할당
	- 컨텍스트 스위칭 발생
- **0과 1만 가지는 락** (binary lock). lock 보유자만 unlock 가능.
	- 0일때 락 소유
- "**잠금/소유 개념**" — 한 명에게 화장실 열쇠 하나.
- 예: `synchronized`, `ReentrantLock`

## Semaphore (세마포어)
- Busy Waiting이 아닌 대기 큐에 들어가서 대기
- **카운터** 기반. 0 이상의 정수 (Counting Semaphore).
- "**자원 개수 제한**" — N개 자원을 N명까지 동시 사용.
- 소유 개념 없음 → 누구나 signal(V) 가능. **잘못 쓰면 동기화 깨짐**.
- 종류:
  - **Binary Semaphore (0/1)** ≈ Mutex이지만 소유 개념 없음
  - **Counting Semaphore(S>=0)** — 커넥션 풀, 제한된 워커 수 제어

## Mutex vs Semaphore 핵심 차이

| | Mutex | Semaphore |
| --- | --- | --- |
| 값 | 0/1 | 0 이상 정수 |
| 소유권 | O (lock한 스레드만 unlock) | X |
| 용도 | 임계 구역 보호 | 자원 카운팅 + 신호 전달 |

## Monitor (모니터)
- **고수준 동기화 기법**. 공유 자원 + 그 자원에 접근하는 메서드를 묶고, 한 번에 하나의 스레드만 진입 가능.
- 자바의 `synchronized` 키워드 + `wait()` / `notify()` / `notifyAll()` 가 모니터의 구현.
- `wait()`: 조건 안 맞으면 락 풀고 대기, `notify()`: 대기 중인 스레드 깨움.

#### `synchronized` 동작 시나리오

1. **진입 시도:** 스레드 A가 `synchronized` 메서드를 호출합니다. 모니터의 락(열쇠)이 비어있으므로 스레드 A가 **Owner**가 됩니다.
    
2. **대기 발생:** 이때 스레드 B와 C가 해당 메서드를 호출하면, 이미 A가 Owner이므로 B와 C는 밖의 **Entry Set**에서 대기(Block)합니다.
    
3. **`wait()` 호출:** 스레드 A가 작업을 하다가, 외부 API 응답이 필요해서 잠시 기다려야 합니다. 이때 A가 `wait()` 메서드를 호출하면, A는 쥐고 있던 락을 반납하고 Wait Set(수면실)으로 가서 잠듭니다 (Waiting 상태).
    
4. **다음 스레드 진입:** 락이 비었으므로, 밖의 Entry Set에서 기다리던 스레드 B가 락을 얻고 **Owner**가 됩니다.
    
5. **`notify()` 호출:** 스레드 B가 자신의 볼일을 다 보고, 아까 수면실에 있던 A가 깰 조건이 되었다고 판단하여 `notify()`를 호출합니다.
    
6. **깨어남:** `notify()`를 들은 스레드 A는 Wait Set(수면실)에서 빠져나옵니다. 단, 바로 코드를 실행하는 게 아니라 **다시 밖의 Entry Set으로 가서** 락을 얻기 위해 줄을 섭니다.
    
7. **종료:** 스레드가 `synchronized` 블록을 완전히 빠져나가면 락은 자동으로 반납됩니다

## Java의 동기화 도구
- **synchronized**: JVM 레벨 모니터, 메서드/블록 단위
- **volatile**: 메모리 가시성만 보장 (원자성 X)
- **java.util.concurrent.locks.ReentrantLock**: 명시적 락, `tryLock`, 공정성(fair) 설정 가능
- **Atomic 클래스 (AtomicInteger 등)**: CAS(Compare-And-Swap) 기반 lock-free
- **ConcurrentHashMap**: 세그먼트/버킷 단위 락

## 면접 빈출 질문
1. **Race Condition이란?** → 둘 이상의 스레드가 공유 자원에 동시 접근해 결과가 실행 순서에 따라 달라지는 현상
2. **Mutex와 Semaphore 차이?** → 값의 범위, 소유 개념 유무
3. **synchronized와 ReentrantLock 차이?** → ReentrantLock은 tryLock·timeout·공정성 지원, 명시적 unlock 필요
4. **volatile이 동기화를 보장하는가?** → 아니오. 가시성만 보장, 원자성은 보장 X (i++는 read-modify-write라서 안전 X)
5. **CAS란?** → Compare And Swap. 메모리 값과 예상 값을 비교해 같으면 새 값으로 교체. lock 없이 원자성 보장 → AtomicInteger 등에서 사용.
