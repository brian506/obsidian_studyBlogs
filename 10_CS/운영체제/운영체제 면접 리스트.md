## 문제 

멀티스레드 환경에서 임계 구역(Critical Section)을 보호하기 위한 Mutex와 Semaphore의 결정적인 차이는 무엇인가요? 추가로 대규모 트래픽이 몰리는 티켓팅 서비스처럼 분산된 서버 환경에서는 단일 JVM의 `synchronized`나 `ReentrantLock`만으로 Race Condition을 막을 수 없는데, 이때 락을 어떻게 관리하고 동기화를 보장할 수 있는지 설명해 주세요.

>[!.questions]- Keyword
> Mutex : 1개 자원에 스레드가 하나씩 점유
> Semaphore : 1개 자원에 스레드 개수 제한

## 문제

수많은 클라이언트와 장기 연결(예: SSE)을 유지하며 부하 테스트를 진행할 때, File Descriptor(FD) 고갈이나 메모리 부족 문제가 발생할 수 있습니다. 운영체제 관점에서 File Descriptor란 정확히 무엇인가요? 일반적인 짧은 호흡의 HTTP REST API 통신과 비교했을 때, SSE처럼 연결이 닫히지 않는 통신 방식이 시스템 자원(FD, 메모리)에 어떤 영향을 미치는지 논블로킹 I/O 모델과 연관 지어 설명해 주세요.

## 문제

시스템에서 데드락(Deadlock)이 발생하기 위한 4가지 필요조건은 무엇입니까? MySQL InnoDB와 같은 RDBMS에서 두 트랜잭션 간에 갭락(Gap Lock)이나 행락(Row Lock) 경합으로 데드락이 감지되었을 때 DBMS는 내부적으로 이를 어떻게 처리하나요? 또한, 애플리케이션 개발자로서 비관적 락 등을 사용할 때 데드락을 사전에 회피하기 위해 어떤 전략을 취해야 하는지 설명해 주세요.
