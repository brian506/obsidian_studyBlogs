# 시스템콜과 인터럽트

## 유저 모드와 커널 모드
- **유저 모드(User Mode)**: 일반 애플리케이션이 동작하는 권한 제한 모드. 하드웨어·OS 자원 직접 접근 불가
- **커널 모드(Kernel Mode)**: OS 커널이 동작하는 최고 권한 모드. 메모리·I/O 등 모든 자원 접근 가능

→ **CPU 보호 링(Protection Ring)** 으로 구현 (x86은 Ring 0 ~ Ring 3, 보통 0=커널 / 3=유저)

## 시스템콜 (System Call)
유저 프로그램이 커널 기능(파일 I/O, 네트워크, 프로세스 생성 등)을 요청하는 인터페이스.

### 동작 순서
1. 유저 프로그램이 `read()` 같은 라이브러리 함수 호출
2. 라이브러리가 시스템콜 번호를 레지스터에 넣고 **trap 명령** 실행
3. CPU가 유저 모드 → **커널 모드 전환**
4. 커널이 시스템콜 테이블에서 핸들러 찾아 실행
5. 결과 반환 후 유저 모드로 복귀

### 종류
- 프로세스: `fork`, `exec`, `wait`, `exit`
- 파일: `open`, `read`, `write`, `close`
- 네트워크: `socket`, `bind`, `listen`, `accept`
- 메모리: `mmap`, `brk`

### 왜 비싼가
- 모드 전환 비용 (레지스터·스택 저장/복원)
- 캐시·TLB 일부 무효화 가능
→ 따라서 I/O는 가능한 한 **buffered**, **batched** 로 처리

## 인터럽트 (Interrupt)
CPU가 현재 작업을 중단하고 특정 이벤트를 처리하도록 알리는 신호.

### 분류
- **하드웨어 인터럽트 (Asynchronous)**: 외부 장치 발생. 예: 키보드 입력, 네트워크 패킷, 타이머
- **소프트웨어 인터럽트 (Synchronous, Trap/Exception)**: CPU가 명령 실행 중 발생. 예: 시스템콜, 0으로 나누기, 페이지 폴트

### 처리 흐름
1. 인터럽트 발생 → 현재 명령 종료까지 대기
2. **PC·레지스터 저장** (스택 또는 PCB)
3. 인터럽트 벡터 테이블에서 핸들러(ISR) 주소 찾기
4. **ISR(Interrupt Service Routine)** 실행
5. 저장된 컨텍스트 복원 후 원래 작업 재개

### Polling vs Interrupt
- **Polling**: CPU가 주기적으로 장치 상태 확인 → 단순하지만 자원 낭비
- **Interrupt**: 장치가 알려줄 때만 처리 → 효율적 (현대 시스템 기본)

## DMA (Direct Memory Access)
- 대량 데이터 전송 시 CPU 개입 없이 디바이스 ↔ 메모리 직접 전송
- CPU는 다른 작업 가능, 전송 완료 시 인터럽트로 알림

## 동기 / 비동기 / 블로킹 / 논블로킹 I/O
| | 블로킹 | 논블로킹 |
| --- | --- | --- |
| **동기** | read() → 데이터 올 때까지 대기 | read() 즉시 반환, 결과 폴링 |
| **비동기** | 거의 안 씀 | aio_read() — 완료 시 콜백/시그널로 통지 |

- **I/O Multiplexing**: select, poll, **epoll(Linux), kqueue(BSD)** — 한 스레드로 여러 소켓 감시
- Spring WebFlux, Node.js, Nginx의 비결 = epoll 기반 이벤트 루프

## 면접 빈출 질문
1. **시스템콜이 무엇이고 왜 필요한가?** → 유저 프로그램이 커널 기능을 안전하게 사용하기 위한 진입점
2. **유저 모드와 커널 모드를 나누는 이유?** → 보안·안정성. 응용 프로그램이 OS·하드웨어를 직접 건드리지 못하게 함
3. **시스템콜이 비싼 이유?** → 모드 전환 비용, 레지스터 저장/복원, TLB·캐시 일부 무효화
4. **인터럽트와 시스템콜의 차이?** → 인터럽트는 외부/내부 이벤트로 발생, 시스템콜은 유저가 의도적으로 발생 (소프트웨어 인터럽트의 한 종류)
5. **Polling vs Interrupt 비교?** → 폴링은 주기적 확인(자원 낭비), 인터럽트는 이벤트 기반(효율적)
6. **DMA가 무엇인가?** → CPU 없이 장치-메모리 간 직접 데이터 전송. 대용량 I/O 성능 ↑

## 백엔드 연결
- Spring MVC(Tomcat)의 **블로킹 I/O + 스레드풀** 구조 → 요청당 스레드
- Spring WebFlux의 **논블로킹 I/O (Netty + epoll)** → 적은 스레드로 많은 연결 처리
- 둘 다 결국 OS의 I/O 모델과 시스템콜에 의존
