

## 정의
둘 이상의 스레드/프로세스가 서로 상대가 가진 자원을 기다리며 **무한 대기** 상태에 빠진 것.

> A는 자원1을 잡고 자원2를 기다림 / B는 자원2를 잡고 자원1을 기다림 → 둘 다 영원히 대기

## 발생 조건 (4가지, 모두 동시에 성립해야 함)
1. **상호 배제(Mutual Exclusion)**: 자원을 한 번에 한 프로세스만 사용
2. **점유와 대기(Hold and Wait)**: 자원을 가진 채로 다른 자원을 기다림
3. **비선점(No Preemption)**: 다른 프로세스의 자원을 강제로 뺏을 수 없음
4. **순환 대기(Circular Wait)**: 대기 그래프가 사이클을 이룸

→ 4개 중 **하나라도 깨면 데드락 발생 X**

## 해결 전략

### 1. 예방 (Prevention) — 4조건 중 하나를 사전에 차단
- 상호 배제 제거: 자원을 공유 가능하게 (현실적으로 어려움)
- 점유 대기 제거: 자원을 **한 번에 모두** 요청
- 비선점 제거: 기다리게 되면 가진 자원을 모두 반납
- 순환 대기 제거: **자원에 순서를 부여**하고 그 순서대로만 요청 ← 가장 현실적

### 2. 회피 (Avoidance) — 런타임에 안전 상태 유지
- **은행원 알고리즘(Banker's Algorithm)**: 자원 요청 시 안전 상태로 갈 수 있는지 시뮬레이션 후 할당
- 단점: 자원 종류·최대 요청량 미리 알아야 함 → 실제 시스템에는 적용 어려움

### 3. 탐지 + 복구 (Detection & Recovery)
- 주기적으로 **자원 할당 그래프**에서 사이클 탐지
- 복구: 프로세스 강제 종료, 자원 선점 회수, 롤백

### 4. 무시 (Ignorance) — Ostrich Algorithm
- 발생 빈도가 낮으면 그냥 둠. 대부분의 OS(리눅스 포함) 기본 전략

## DB에서의 데드락
- 트랜잭션 A: row1 락 → row2 대기 / 트랜잭션 B: row2 락 → row1 대기
- MySQL InnoDB: **데드락 탐지기**가 자동 감지 후 **롤백할 victim** 선정 → 한쪽 트랜잭션을 실패시킴
- 대응: 일관된 락 획득 순서 + 짧은 트랜잭션 + 인덱스 활용 (불필요한 갭락 방지)

## Java/Spring에서의 데드락 예
```
Thread A: synchronized(lock1) { synchronized(lock2) { ... } }
Thread B: synchronized(lock2) { synchronized(lock1) { ... } }
```
→ 락 획득 순서를 통일하면 해결.

## Livelock vs Starvation
- **Livelock**: 서로 양보만 하다가 진행 안 됨 (대기는 아님)
- **Starvation(기아)**: 우선순위 낮은 스레드가 영원히 자원 못 얻음 → 해결: **에이징(Aging)** — 대기 시간에 따라 우선순위 ↑

## 면접 빈출 질문
1. **데드락 발생 조건 4가지?** → 상호배제 / 점유대기 / 비선점 / 순환대기
2. **가장 현실적인 예방 방법?** → 자원에 순서를 부여 (순환 대기 차단)
3. **DB에서 데드락이 발생하면 어떻게 처리?** → DBMS가 탐지 후 victim 트랜잭션 롤백
4. **데드락과 기아 상태(Starvation)의 차이?** → 데드락: 서로 자원을 기다리며 멈춤 / 기아: 우선순위 낮아 자원 못 얻음
5. **은행원 알고리즘이 실제로 잘 안 쓰이는 이유?** → 모든 프로세스의 최대 자원 요청량을 사전에 알아야 하기 때문
