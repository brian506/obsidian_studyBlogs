
## 메모리 계층 구조 (위로 갈수록 빠르고 비쌈·작음)
```
CPU 레지스터       (1 cycle)
L1 캐시           (~4 cycles, 수십 KB)
L2 캐시           (~10 cycles, 수백 KB)
L3 캐시           (~40 cycles, 수~수십 MB)
메인 메모리(RAM)   (~100 cycles, GB)
SSD/HDD           (수만~수백만 cycles, TB)
```

## 캐시가 동작하는 원리: 지역성 (Locality)
- **시간 지역성 (Temporal)**: 최근 접근한 데이터는 곧 다시 접근될 가능성이 높다 → 반복문 변수, 자주 호출되는 함수
- **공간 지역성 (Spatial)**: 접근한 데이터 근처를 곧 접근할 가능성이 높다 → 배열 순회

→ 두 지역성을 활용해 자주 쓰는 데이터를 빠른 메모리에 미리 가져다 둠

## 캐시 히트 / 미스
- **Hit**: 캐시에 있어 빠르게 응답
- **Miss**: 캐시에 없어 하위 계층에서 가져옴 (지연 발생)
  - **Compulsory miss**: 최초 접근
  - **Capacity miss**: 캐시 용량 부족
  - **Conflict miss**: 매핑 충돌

## 캐시 매핑 방식
- **Direct Mapped**: 메모리 블록 → 하나의 캐시 라인 (충돌 많음)
- **Fully Associative**: 어디든 저장 가능 (충돌 적지만 탐색 비용)
- **Set Associative (N-way)**: 절충안. 보통 N=4 ~ 16

## 캐시 쓰기 정책
- **Write-Through**: 캐시와 메모리 동시 기록 → 안전, 느림
- **Write-Back**: 캐시에만 기록 후 나중에 메모리 반영 → 빠름, 일관성 관리 필요

## 코드에서 캐시 친화적인 패턴
```java
// 캐시 친화 (행 우선 접근, 공간 지역성 활용)
for (int i = 0; i < N; i++)
    for (int j = 0; j < N; j++)
        sum += arr[i][j];

// 캐시 비친화 (열 우선)
for (int j = 0; j < N; j++)
    for (int i = 0; i < N; i++)
        sum += arr[i][j];   // 한 페이지에서 멀리 떨어진 메모리 반복 접근
```

## 면접 빈출 질문
1. **캐시 메모리가 빠른 이유?** → CPU 가까이 위치 + SRAM 기반 + 지역성을 활용
2. **시간 지역성 vs 공간 지역성?** → 시간: 같은 데이터 재접근 / 공간: 인접 데이터 접근
3. **캐시 미스의 종류?** → Compulsory / Capacity / Conflict
4. **Write-Through와 Write-Back 차이?** → 즉시 vs 지연 기록, 안전성 vs 성능
5. **2차원 배열 순회에서 행 우선이 더 빠른 이유?** → 메모리에 행 단위로 연속 저장 → 공간 지역성

## 응용 (CS와 백엔드의 연결)
- **DB 버퍼 풀, OS 페이지 캐시**: 같은 캐시 원리
- **Redis 캐싱**: 자주 조회되는 데이터를 빠른 메모리에 — 결국 같은 지역성 아이디어
- **인덱스 B-Tree의 leaf 노드 순회**: 공간 지역성 활용해 디스크 I/O 최소화
