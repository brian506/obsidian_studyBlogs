
## 한 줄 요약
- **프로세스**: 실행 중인 프로그램. OS로부터 자원(메모리·CPU 시간)을 할당받는 **독립적 단위**.
	- 스레드를 제어하는 역할
- **스레드**: 프로세스 내부의 **실행 흐름**. 같은 프로세스의 스레드들은 메모리(Heap, Data, Code)를 공유.

프로세스 : 집 / 스레드 : 방
-> 하나의 프로세스 안에 종속된 스레드가 생성되고 소멸되는 느낌
## 핵심 비교표

| 구분                   | *프로세스*                               | *스레드*                                    |
| -------------------- | ------------------------------------ | ---------------------------------------- |
| 메모리                  | 독립 (Code/Data/Heap/Stack 각자)         | Code/Data/Heap **공유**, Stack·PC·레지스터만 독립 |
| 통신                   | IPC 필요 (Pipe, Socket, 공유메모리, Signal) | 변수 공유로 직접 통신                             |
| Context Switching 비용 | 큼 (페이지 테이블·캐시 flush)                 | 작음 (메모리 공간 그대로)                          |
| 안정성                  | 한 프로세스 죽어도 다른 프로세스 영향 X              | 한 스레드 죽으면 프로세스 전체 영향                     |
|                      |                                      |                                          |

## 프로세스의 메모리 구조

- **Code(Text)**: 실행 명령어 (read-only)
- **Data**: 전역/static 변수 (초기화/BSS)
- **Heap**: 동적 할당, 런타임에 증가 (위로 자람)
- **Stack**: 지역 변수·함수 호출 정보, 함수 호출마다 프레임 쌓임 (아래로 자람)

*JVM 자체가 하나의 프로세스라고 볼 수 있음*

## 프로세스 상태 (5 state)
`new → ready → running → waiting → terminated`
- ready ↔ running: 스케줄러가 결정
- running → waiting: I/O 요청 등
- waiting → ready: I/O 완료
### 1. Created State
> 작업을 커널에 등록한 상태
- PCB 할당 및 프로세스 생성
- **커널** : 가용 메모리 공간 체크 후 **Ready** 또는 **Suspended ready**로 이동
	- 메모리 있으면 Ready로 이동
	- 메모리 없으면 Suspended Ready 이동

### 2. Ready State
> CPU(프로세서에 할당 되기 전 대기 상태
- 즉시 실행 가능 상태
- 여기서 할당되면(dispatch) Running으로 이동

### 3. Running State
> CPU와 필요한 자원을 모두 할당 받은 상태

Preemption
- Running -> Ready (state)
Block
- Running -> asleep(state)
	- I/O 등 응답을 받기 전 sleep상태

### 4. Blocked / Asleep State
> 다른 자원을 기다리는 상태

Asleep -> Ready 
- I/O가 끝나면 바로 Running이 아니라, Ready 상태로 다시 이동돼서 작업 할당을 기다린다.

### 2-1. Suspended State
> 메모리를 할당 받지 못한 상태

- 메모리가 뻇겼을 때 swap device라는 하드웨어 공간에 넣어뒀다가 swap in으로 메모리를 할당 받아 다시 run에 참여

### 5. Terminated
> 프로세스 수행이 끝난 상태

- 모든 자원 반납 후, 커널 내에 일부 PCB 정보만 남아 있는 상태


## PCB (Process Control Block)
OS가 프로세스 관리를 위해 유지하는 자료구조.
**저장 항목**: PID, 프로세스 상태, Program Counter, 레지스터 값, 메모리 정보, 열린 파일, 스케줄링 정보

PCB 참조 및 갱신 속도는 OS의 성능을 결정 짓는 중요한 요소 중 하나

## Context Switching
> 실행 중인 프로세스의 context를 저장하고, 앞으로 실행할 프로세스의 context를 복구 하는 일
> -> 커널이 개입하여 수행한다.

1. 현재 프로세스의 상태(레지스터, PC)를 PCB에 저장
2. 다음 프로세스의 PCB에서 상태 복원
3. **비용**: 캐시·TLB flush, 메모리 접근 → 잦으면 오버헤드 ↑

## 스레드

![](../../images/스크린샷%202026-06-19%2018.20.42.png)
- 하나의 프로세스 안에 여러 개의 resource(스레드)가 존재한다.

### 스레드 독립 자원

- **스택 영역** 
- **PC 레지스터**
	- 스레드가 현재 코드의 어느 부분을 실행하고 있는지, 다음에 실행할 명령어의 주소를 가리킴
- **레지스터 셋**
	- CPU가 연산을 수행하는 동안 임시로 데이터를 저장하는 공간
		- 컨텍스트 스위칭이 일어날 때, 자기의 작업하던 상태를 기록해두는 것
- 스레드 ID
	- 고유 ID, 현재 상태(Running, Ready 등)

### 프로세스 안의 공유 자원

- **코드 영역** : 소스 코드(바이트 코드)
- **데이터 영역** : 전역 변수와 정적 변수 등
- **힙 영역**
- **운영체제 자원** : 열려 있는 파일, 네트워크 소켓 연결, 입출력 장치 등 프로세스가 OS로부터 부여받은 자원

### 장점

- 사용자 응답성
	- 일부 스레드의 처리가 지연되어도, 다른 스레드는 작업 계속 처리 가능
- 자원 공유
	- 자원을 공유해서 효율성 증가(**커널의 개입을 피할 수 있음**)
- 경제성
	- 프로세스의 생성, 컨텍스트 스위칭에 비해 효율적
- 멀티 프로세서 활용
	- 병렬처리를 통해 성능 향상

### 서로 다른 프로세스끼리 협력하는 방법

*하나의 프로세스 안에서 생성된 스레드는 다른 프로세스에서 접근하지 못한다.*
대신에 다른 방식으로 프로세스를 공유할 수 있다.
1. 네트워크 소켓
2. 파이프 & 메시지 큐
3. 공유 메모리

## 멀티프로세스 vs 멀티스레드
- **멀티프로세스**: 안정성 ↑, 자원 사용량 ↑ (Chrome 탭 구조)
- **멀티스레드**: 자원 효율 ↑, 동기화 이슈 (공유 자원에 락 필요)

## 면접 빈출 질문
1. **프로세스와 스레드 차이?** → 자원 할당 단위 vs 실행 단위, 메모리 공유 여부
2. **Context Switching이 비싼 이유?** → 레지스터·PCB 저장 + 캐시/TLB 무효화
3. **멀티스레드의 단점?** → 공유 자원 접근 시 동기화 필요, 디버깅 어려움, 한 스레드 오류가 전체 프로세스 영향
4. **PCB가 왜 필요한가?** → Context Switching 시 이전 상태 복원하려면 저장소 필요
5. **스레드가 공유하는 것 / 공유하지 않는 것?** → 공유: Code/Data/Heap, 파일 디스크립터 / 독립: Stack, PC, 레지스터
