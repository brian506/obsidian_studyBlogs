# 프로세스와 스레드

## 한 줄 요약
- **프로세스**: 실행 중인 프로그램. OS로부터 자원(메모리·CPU 시간)을 할당받는 **독립적 단위**.
- **스레드**: 프로세스 내부의 **실행 흐름**. 같은 프로세스의 스레드들은 메모리(Heap, Data, Code)를 공유.

## 핵심 비교표

| 구분 | 프로세스 | 스레드 |
| --- | --- | --- |
| 메모리 | 독립 (Code/Data/Heap/Stack 각자) | Code/Data/Heap **공유**, Stack·PC·레지스터만 독립 |
| 통신 | IPC 필요 (Pipe, Socket, 공유메모리, Signal) | 변수 공유로 직접 통신 |
| Context Switching 비용 | 큼 (페이지 테이블·캐시 flush) | 작음 (메모리 공간 그대로) |
| 안정성 | 한 프로세스 죽어도 다른 프로세스 영향 X | 한 스레드 죽으면 프로세스 전체 영향 |

## 프로세스의 메모리 구조
- **Code(Text)**: 실행 명령어 (read-only)
- **Data**: 전역/static 변수 (초기화/BSS)
- **Heap**: 동적 할당, 런타임에 증가 (위로 자람)
- **Stack**: 지역 변수·함수 호출 정보, 함수 호출마다 프레임 쌓임 (아래로 자람)

## 프로세스 상태 (5 state)
`new → ready → running → waiting → terminated`
- ready ↔ running: 스케줄러가 결정
- running → waiting: I/O 요청 등
- waiting → ready: I/O 완료

## PCB (Process Control Block)
OS가 프로세스 관리를 위해 유지하는 자료구조.
**저장 항목**: PID, 프로세스 상태, Program Counter, 레지스터 값, 메모리 정보, 열린 파일, 스케줄링 정보

## Context Switching
1. 현재 프로세스의 상태(레지스터, PC)를 PCB에 저장
2. 다음 프로세스의 PCB에서 상태 복원
3. **비용**: 캐시·TLB flush, 메모리 접근 → 잦으면 오버헤드 ↑

## 멀티프로세스 vs 멀티스레드
- **멀티프로세스**: 안정성 ↑, 자원 사용량 ↑ (Chrome 탭 구조)
- **멀티스레드**: 자원 효율 ↑, 동기화 이슈 (공유 자원에 락 필요)

## 면접 빈출 질문
1. **프로세스와 스레드 차이?** → 자원 할당 단위 vs 실행 단위, 메모리 공유 여부
2. **Context Switching이 비싼 이유?** → 레지스터·PCB 저장 + 캐시/TLB 무효화
3. **멀티스레드의 단점?** → 공유 자원 접근 시 동기화 필요, 디버깅 어려움, 한 스레드 오류가 전체 프로세스 영향
4. **PCB가 왜 필요한가?** → Context Switching 시 이전 상태 복원하려면 저장소 필요
5. **스레드가 공유하는 것 / 공유하지 않는 것?** → 공유: Code/Data/Heap, 파일 디스크립터 / 독립: Stack, PC, 레지스터
