# 정규화와 반정규화

## 정규화 (Normalization)
중복을 줄이고 **이상 현상(Anomaly)** 을 막기 위해 테이블을 분해하는 과정.

### 이상 현상 3종
- **삽입 이상**: 새 데이터 넣으려면 불필요한 데이터까지 같이 넣어야 함
- **삭제 이상**: 한 행 지우면 의도하지 않은 정보까지 사라짐
- **갱신 이상**: 같은 정보가 여러 곳에 있어 일부만 수정되어 불일치 발생

## 단계별 정규화

### 1NF (제1정규형)
- **모든 컬럼이 원자값**을 가질 것 (배열·중복 컬럼 X)
- 위반 예: `취미 = "축구,야구,농구"` → 분리하거나 별 테이블

### 2NF (제2정규형) — 부분 함수 종속 제거
- 1NF 만족 + **모든 비주요 속성이 기본키 전체에 종속**
- 복합키일 때만 의미 있음
- 위반 예: 주문(상품ID, 고객ID, **상품명**, 주소)에서 상품명은 상품ID에만 종속 → 상품 테이블로 분리

### 3NF (제3정규형) — 이행 함수 종속 제거
- 2NF + **비주요 속성이 다른 비주요 속성에 종속되면 안 됨**
- 위반 예: 사원(사원ID, 부서ID, **부서명**) — 부서명은 부서ID에 종속 → 부서 테이블로 분리

### BCNF
- 3NF 보다 엄격: 모든 결정자가 후보키여야 함
- 실무에서는 보통 **3NF까지** 정규화하고, 필요 시 반정규화

## 반정규화 (Denormalization)
**성능을 위해 일부러 중복**을 허용하거나 테이블을 합치는 것.

### 언제 하나
- 자주 쓰는 조회에 **JOIN이 너무 많아서** 느림
- 집계 결과를 매번 계산하기 비쌈
- 통계·리포트 테이블 (자주 변하지 않는 결과)

### 방법
- **컬럼 추가** (자주 함께 조회하는 값 미리 저장: 상품명을 주문 테이블에)
- **요약 테이블** (월별 매출 집계)
- **테이블 병합** (1:1이 항상 함께 조회되면)

### 트레이드오프
- ↑ 조회 성능, ↓ 일관성 위험 (갱신 시 동기화 필요)
- 일관성은 트리거·배치·이벤트로 보완

## 정규화 vs 반정규화 선택 기준
| | 정규화 | 반정규화 |
| --- | --- | --- |
| 일관성 | 강함 | 약함 (관리 필요) |
| 저장 공간 | 작음 | 큼 |
| 쓰기 성능 | 좋음 | 중복 갱신 비용 |
| 읽기 성능 | JOIN 많아 느려질 수 있음 | 좋음 |
| 적용 | OLTP (트랜잭션 중심) | OLAP·통계, 읽기 위주 |

## 면접 빈출 질문
1. **정규화의 목적은?** → 중복 제거 + 이상 현상 방지 + 일관성 보장
2. **1·2·3 NF 차이?** → 원자값 / 부분 함수 종속 제거 / 이행 함수 종속 제거
3. **항상 정규화가 좋은가?** → 아니오. JOIN 비용·집계 비용이 크면 반정규화로 트레이드. **3NF + 필요 시 부분 반정규화**가 실무 패턴
4. **반정규화의 위험?** → 같은 데이터가 여러 곳에 → 갱신 누락 시 일관성 깨짐
5. **OLTP vs OLAP의 정규화 수준?** → OLTP는 정규화 중심, OLAP은 반정규화·집계 테이블 중심

## 프로젝트 연결
- 피드(게시물) 조회가 많은 위스키 프로젝트 → 게시물 수·좋아요 수 같은 카운터를 컬럼으로 저장(반정규화) 후 비동기 동기화
- 통계 대시보드 (recaring)도 별도 집계 테이블 운영이 유리
