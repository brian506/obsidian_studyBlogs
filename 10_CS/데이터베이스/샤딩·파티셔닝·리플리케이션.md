# 샤딩 · 파티셔닝 · 리플리케이션

대용량·고가용성을 위한 3대 분산 기법. 자주 헷갈리니 **개념을 정확히 구분**하기.

## 1. 파티셔닝 (Partitioning) — 한 DB 안에서 테이블 분할
하나의 DB 인스턴스 안에서, 한 **테이블을 여러 물리 조각**으로 나눔.

### 수평 파티셔닝 (Horizontal)
- 같은 스키마, **행을 분할** (예: 2024년 데이터 / 2023년 데이터)
- 키 기준: Range / List / Hash / Key

### 수직 파티셔닝 (Vertical)
- 같은 행, **컬럼을 분할** (자주 쓰는 컬럼 / TEXT 같이 큰 컬럼 분리)

### 효과
- 인덱스 크기 ↓, 특정 파티션만 스캔 → 조회 빠름
- 오래된 파티션 통째로 DROP 가능 (TTL 관리)
- 한 인스턴스 안이므로 **확장성 한계**는 여전히 존재

## 2. 샤딩 (Sharding) — 여러 DB로 데이터 분할
**여러 DB 인스턴스(샤드)** 에 데이터를 나눠 저장. 파티셔닝의 분산 버전.

### 샤드 키 전략
- **Range**: 키 범위로 분배 (user_id 1~10000 → shard A) — 핫스팟 가능
- **Hash**: hash(key) % N — 균등 분포, **샤드 추가/제거 시 리해싱 부담**
- **Consistent Hashing**: 일부만 재배치 → 확장에 유리 (Redis Cluster, Cassandra)
- **Directory**: 별도 매핑 테이블 (lookup 서비스)

### 어려운 점
- **조인 불가능** (서로 다른 샤드의 데이터)
- **트랜잭션 어려움** → 분산 트랜잭션 또는 Saga
- **샤드 추가** = 리밸런싱 작업 필요
- **글로벌 유니크 키** 생성 어려움 → Snowflake ID, UUID

→ 샤딩은 **최후의 수단**. 가능하면 read replica + 파티셔닝 + 캐싱으로 먼저 해결.

## 3. 리플리케이션 (Replication) — DB 복제
같은 데이터를 **여러 노드에 복제**해 가용성·읽기 분산 확보.

### Master-Slave (Primary-Replica)
- 쓰기 = Master, 읽기 = Replica로 분산
- Master 장애 시 Replica를 승격(failover)
- ⚠️ **복제 지연(Replication Lag)** → 방금 쓴 데이터를 Replica에서 못 읽을 수 있음

### 동기 / 비동기 / 반동기
- **동기(Sync)**: Replica가 받았다는 확인까지 기다림 → 일관성 ↑, 쓰기 성능 ↓
- **비동기(Async)**: Master는 일단 응답, 복제는 백그라운드 → 빠름, 데이터 손실 가능
- **반동기(Semi-sync)**: 최소 1개 Replica 확인 (MySQL의 기본 절충)

### Multi-Master
- 여러 노드가 쓰기 가능 → **충돌 해결** 필요 (Last-Write-Wins, CRDT)
- 어려워서 보통 회피

## 비교 정리

| | 파티셔닝 | 샤딩 | 리플리케이션 |
| --- | --- | --- | --- |
| 목적 | 큰 테이블 분할 | **수평 확장** | **가용성·읽기 확장** |
| 데이터 | 분할 | 분할 | **복제** |
| 위치 | 단일 DB | 다중 DB | 다중 DB |
| 트레이드오프 | 인스턴스 한계 | 조인·트랜잭션 어려움 | 복제 지연·일관성 |

## 자주 헷갈리는 포인트
- **파티셔닝 vs 샤딩**: 한 DB 안 vs 여러 DB. 본질은 같지만 운영 난이도가 다름
- **샤딩 vs 리플리케이션**: 데이터를 **나누느냐(샤딩) 복제하느냐(리플리케이션)**. 보통 같이 씀 (각 샤드를 또 복제)

## 캐싱과의 관계
대부분의 읽기 부하는 샤딩보다 **Redis 같은 캐시 한 층**으로 해결되는 경우가 많음 → 비용·복잡도 측면에서 캐시 먼저 검토.

## 면접 빈출 질문
1. **파티셔닝과 샤딩의 차이?** → 단일 인스턴스 내 분할 vs 여러 인스턴스 분할
2. **샤딩이 어려운 이유?** → 조인 불가, 분산 트랜잭션, 샤드 키 변경 시 리밸런싱
3. **리플리케이션의 복제 지연 문제?** → Master에 쓰고 Replica에서 못 읽음. 핵심 조회는 Master 강제(Read-Your-Writes) 또는 캐시
4. **샤드 키를 잘못 고르면?** → 핫스팟(특정 샤드 집중), 리밸런싱 폭발
5. **샤딩 외에 부하 분산 방법은?** → Replica 추가, 캐시(Redis), CQRS, 비동기 처리
6. **MySQL 복제 모드?** → 비동기(기본), 반동기, GTID 기반 복제

## 프로젝트 연결
- recaring: RDS Multi-AZ + Read Replica 구성. 쓰기 = Primary, 읽기 = Replica
- 티켓팅: 인기 콘서트의 쓰기 폭주 → 샤딩 대신 **Redis 큐 + 비동기 처리**로 해결
- MongoDB 위스키: 채팅 데이터는 사용자별·방별 샤드 키 후보 검토
