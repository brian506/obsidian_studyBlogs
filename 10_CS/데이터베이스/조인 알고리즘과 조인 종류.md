# 조인 알고리즘과 조인 종류

## 조인 종류 (논리적 관점)

| 종류 | 설명 | 결과 |
| --- | --- | --- |
| **INNER JOIN** | 양쪽 모두에 매칭되는 행 | 교집합 |
| **LEFT (OUTER) JOIN** | 왼쪽 전체 + 오른쪽 매칭 (없으면 NULL) | 왼쪽 기준 |
| **RIGHT JOIN** | 반대 | 거의 안 씀 (LEFT 뒤집기로 대체) |
| **FULL OUTER JOIN** | 양쪽 모두, 매칭 없는 쪽은 NULL | 합집합 (MySQL 미지원, UNION으로 흉내) |
| **CROSS JOIN** | 카티전 곱 (A × B 모든 조합) | 거의 안 씀, 실수로 발생 주의 |
| **SELF JOIN** | 같은 테이블끼리 별칭으로 JOIN | 계층 구조, 조직도 |

```sql
-- INNER (가장 일반적)
SELECT u.name, o.amount
FROM users u
INNER JOIN orders o ON o.user_id = u.id;

-- LEFT (주문 없는 사용자도 보고 싶을 때)
SELECT u.name, o.amount
FROM users u
LEFT JOIN orders o ON o.user_id = u.id;
```

## 조인 알고리즘 (물리적 관점)
옵티마이저가 상황에 따라 선택. EXPLAIN 으로 확인 가능.

### 1. Nested Loop Join (NLJ)
가장 단순. 바깥 테이블의 각 행마다 안쪽 테이블에서 매칭 행 찾기.
```
for r in outer:
    for s in inner:
        if r.key == s.key: emit(r, s)
```
- 복잡도: O(N × M) — 단, **inner에 인덱스가 있으면 O(N × log M)** 으로 매우 빠름
- MySQL InnoDB의 **기본 조인 알고리즘**
- ⚠️ **inner의 join key에 인덱스 필수**. 없으면 풀스캔 × N → 재앙

### 2. Hash Join
inner 테이블로 **해시 테이블 생성** → outer 행을 해시로 조회.
```
1. Build phase: 작은 테이블로 해시맵 만들기 (메모리)
2. Probe phase: 큰 테이블을 순회하며 해시맵 조회
```
- 복잡도: O(N + M)
- **인덱스가 없어도 효율적** → 대용량 일괄 처리에 적합
- 단점: 메모리 사용. PostgreSQL·Oracle은 오래전부터, **MySQL 8.0.18+** 부터 지원

### 3. Sort-Merge Join
두 테이블을 join key로 정렬 후 한 번에 머지.
- 복잡도: O(N log N + M log M) — 정렬이 비쌈
- **이미 정렬되어 있거나 인덱스로 정렬된 결과**가 있으면 매우 빠름
- 큰 데이터에 강함, 범위 조인(`>`, `<`)에도 사용

## 어떤 알고리즘이 언제?
| 상황 | 추천 |
| --- | --- |
| inner에 인덱스 있음 + outer 작음 | NLJ |
| 인덱스 없음 + 메모리 충분 + 작은 쪽이 메모리에 들어감 | Hash Join |
| 이미 정렬되어 있음 / 범위 조인 | Sort-Merge |
| 양쪽 다 큰 데이터 + 인덱스 없음 | Hash Join |

→ 실제로는 옵티마이저가 통계(테이블 크기, 인덱스, 메모리)를 보고 자동 선택. **개발자는 적절한 인덱스를 깔아 옵티마이저를 돕는 게 핵심**.

## ORM의 함정 — N+1 문제
JPA에서 List<Order> → 각 Order에서 lazy로 User 가져오면 **1번의 Order 쿼리 + N번의 User 쿼리**
- 해결:
  - **fetch join**: `select o from Order o join fetch o.user`
  - **EntityGraph**, BatchSize, `@BatchSize(size = 100)`
- 자세히: [[20_언어·프레임워크/Spring/data/연관관계 설정 - N+1 문제]]

## 면접 빈출 질문
1. **INNER와 LEFT JOIN 차이?** → 매칭만 vs 왼쪽 전체 + 매칭(없으면 NULL)
2. **세 가지 조인 알고리즘?** → Nested Loop / Hash / Sort-Merge
3. **NLJ에서 성능이 중요한 부분?** → 안쪽 테이블의 join key 인덱스. 없으면 풀스캔이 N번 발생
4. **Hash Join이 좋은 상황?** → 인덱스 없는 대용량 + 작은 쪽이 메모리에 적재 가능
5. **MySQL의 기본 조인 알고리즘과 8.0의 변화?** → 기본 NLJ, 8.0.18+ 부터 Hash Join 도입
6. **N+1 문제와 해결?** → ORM의 lazy loading으로 인한 다중 쿼리. fetch join, EntityGraph, BatchSize로 해결

## 프로젝트 연결
- 인덱스 없는 컬럼으로 join → NLJ 풀스캔 → 느린 쿼리. EXPLAIN으로 확인
- JPA `@OneToMany` lazy → fetch join 또는 BatchSize로 N+1 방어
