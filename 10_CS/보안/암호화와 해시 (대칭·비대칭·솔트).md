# 암호화와 해시 (대칭·비대칭·솔트)

## 큰 그림: 암호화 vs 해시

| | 암호화(Encryption) | 해시(Hash) |
| --- | --- | --- |
| 방향 | **양방향** (복호화 가능) | **단방향** (복호 불가) |
| 키 | 필요 | 불필요 |
| 용도 | 데이터 기밀성 보호 (전송, 저장) | 무결성 검증, 비밀번호 저장 |
| 예 | AES, RSA | SHA-256, bcrypt |

## 대칭키 암호화 (Symmetric)
- **같은 키**로 암호화·복호화
- 빠르고 가볍지만 **키 전달 문제** (둘 다 같은 키를 안전하게 공유해야 함)
- 알고리즘: AES (128/192/256), ChaCha20
- 용도: 대용량 데이터 암호화, TLS의 세션 데이터

## 비대칭키 암호화 (Asymmetric)
- **공개키 + 개인키** 쌍. 공개키로 암호화 → 개인키로 복호화 (또는 그 반대로 서명)
- 키 전달 문제 해결, 하지만 **느림** (수천 배)
- 알고리즘: RSA, ECC(타원곡선)
- 용도: TLS 키 교환, 디지털 서명, JWT 서명(RS256)

### 두 가지 사용 패턴
- **기밀성**: 받는 사람의 **공개키로 암호화** → 받는 사람의 **개인키로 복호화**
- **서명·인증**: 보낸 사람의 **개인키로 서명** → 보낸 사람의 **공개키로 검증**

## 하이브리드 (실제 사용)
대칭키의 속도 + 비대칭키의 키 전달 안전성 결합. **TLS가 대표적**:
1. 비대칭으로 **세션키(대칭키)** 안전하게 교환
2. 이후 통신은 대칭키로 빠르게

## 해시 함수 특성
1. **고정 길이 출력** (입력 길이 무관)
2. **결정성**: 같은 입력 → 같은 출력
3. **눈사태 효과**: 1 bit 바뀌면 출력 완전히 다름
4. **역상 저항성**: 출력 → 입력 추론 불가
5. **충돌 저항성**: 같은 출력을 내는 다른 입력 찾기 어려움

## 비밀번호 저장 — 단순 해시는 위험!
1. **레인보우 테이블 공격**: 자주 쓰이는 비번 ↔ 해시 매핑 미리 계산
2. **GPU 무차별 대입**: SHA-256은 너무 빨라서 초당 수십억 회 시도 가능

### 솔트(Salt)
- 비밀번호에 **랜덤 값** 추가 후 해시 → 같은 비번이어도 다른 해시
- 솔트는 **사용자별로 고유**, DB에 해시와 함께 저장
- 레인보우 테이블 무력화

### 키 스트레칭 (느리게 만들기)
- 해시를 **수천~수만 번 반복** → 무차별 대입 비용 ↑
- **bcrypt, scrypt, Argon2, PBKDF2** — 비밀번호 저장 전용 알고리즘
- Spring의 `BCryptPasswordEncoder`가 표준

### 잘못된 vs 올바른 저장
```
나쁨:  password
나쁨:  md5(password)        ← 깨진 알고리즘 + 빠름
나쁨:  sha256(password)     ← 빠름, 솔트 없음
좋음:  bcrypt(password)     ← 솔트 내장 + 느림
```

## 디지털 서명
- 데이터 해시 → 보낸 사람 **개인키로 암호화** = 서명
- 받는 사람: 보낸 사람 **공개키로 복호화** → 데이터 해시와 비교
- 보장: **무결성 + 부인 방지**

## 면접 빈출 질문
1. **대칭키와 비대칭키의 차이와 각 용도?** → 속도/키 전달 차이, 보통 키 교환은 비대칭, 데이터 암호화는 대칭
2. **해시와 암호화의 차이?** → 단방향 vs 양방향. 해시는 복호화 불가
3. **비밀번호를 SHA-256으로 저장해도 되나?** → 안 됨. 너무 빨라서 무차별 대입 취약. bcrypt 등 사용
4. **솔트(Salt)의 역할?** → 레인보우 테이블 방어 + 같은 비번도 다른 해시 만들기
5. **bcrypt가 좋은 이유?** → 솔트 자동 생성·저장, 작업량(work factor) 조정 가능, 의도적으로 느림
6. **HTTPS는 대칭·비대칭 중 무엇?** → 둘 다 (핸드셰이크=비대칭, 본 통신=대칭)
7. **디지털 서명의 역할?** → 무결성 + 부인 방지 (보낸 사람만 가진 개인키로 서명)

## 프로젝트 연결
- Spring Security `BCryptPasswordEncoder` → 솔트·반복 자동
- JWT 서명: HS256(대칭, 빠름) vs RS256(비대칭, 키 분리 가능) — 멀티 서비스 환경에선 RS256
